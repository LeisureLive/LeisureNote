# 小米 Java 研发工程师（数据服务）面试高频知识地图

## 1. 这份文档解决什么问题

这份文档不是替代 `01_1month_plan.md`，而是给那份计划补一层“知识导航”。

当前 1 个月计划的问题不在于日拆分不合理，而在于：

- 每天直接进入专题问答，缺少稳定的知识总览
- 进入问答前没有先明确“今天这个专题到底要答到什么深度”
- 输出动作偏早，导致你经常一边查漏、一边组织表达，耗时偏高

这份知识地图的作用是：

- 帮你先建立模块级框架，再进入当天计划
- 帮你区分哪些知识点只要会讲主线，哪些必须能承接深挖
- 让你每天复习时先有“总图”，再做“局部细节”

补充背景：

- 这份地图当前不只服务“Java 数据服务”这个单一标题
- 还会同时覆盖你已收集到的几类小米相关岗位：
  - Java 研发工程师（数据服务）
  - 数据平台开发工程师 Java 方向
  - 数据工场 / 数据平台后端研发方向
- 所以这份地图默认按“共性核心要求优先、数据平台广度第二、加分项最后”组织
- 这份地图同样只应视为当前版本，不是不能调整的固定大纲
- 后续如果补充新的岗位要求，模块优先级、深度目标和复习顺序都可以重排

## 2. 推荐使用方式

### 每天开始前先做 10-15 分钟模块预热

步骤固定如下：

1. 先定位今天对应的知识模块
2. 只看该模块的“核心框架”和“高频问题”
3. 明确今天要拿稳的 1-2 个主问题
4. 再进入当天专项复习和问答
5. 最后只输出提纲或口头回答，不要求每题都写成长文

### 答题深度统一分 3 层

- `L1`：1 分钟版本
  - 说清定义、作用、适用场景
- `L2`：3 分钟版本
  - 说清核心机制、关键流程、常见对比
- `L3`：深挖版本
  - 能承接边界、原理、线上案例、项目结合

建议默认目标：

- P0 模块至少达到 `L2`
- 你的强项模块尽量冲到 `L3`
- 当前短板模块先保 `L1 -> L2`，不要一上来追求细枝末节

### 掌握度标记

- `A`：主线稳定，追问也能展开
- `B`：主线能答，但深挖容易乱
- `C`：知道概念，但还讲不顺

每天复习后，给当天模块打一个 `A/B/C` 即可。

#### 常见追问

- `ThreadLocal` 的使用边界和泄漏风险

#### 你当前更该补的点

- 线程池参数配置要带容量视角，不要只背公式
- `volatile` / JMM / DCL 这组问题要能连起来讲
- AQS 不必抠源码，但要会说“队列 + 状态 + 唤醒”的主线

## 5. 和 1 个月计划如何配合

### 第 1 周

- 只盯 5 个模块：并发、JVM、MySQL、Redis、Kafka
- 每天先看知识地图对应模块，再进入当日复习文档
- 目标不是每题写完，而是先把主问题答稳

### 第 2 周

- 只盯框架链和分布式入口模块：
  - Spring
  - Spring Boot / MVC
  - MyBatis
  - Dubbo / Spring Cloud / RPC
  - 设计模式 / Java Web
- 输出以“流程图 + 回答提纲”为主
- 如果某天问答特别卡，先退回看本知识地图，不要硬扛

### 第 3 周

- 重点切到系统设计、分布式基础、数据服务场景和数据平台 / 大数据引擎
- 复用前两周的中间件和存储知识，不单独割裂复习

### 第 4 周

- 以混合复习和模拟为主
- 每天按模块抽查自己最弱的 2 个问题
- 只做查漏补缺，不再扩展低频知识
- 项目表达、排障案例和综合模拟放进主线
- BI / 报表平台 / 前端协作这类加分项放在主线完成后再补

## 6. 当前最推荐的执行顺序

如果你已经在执行 `01_1month_plan.md`，当前更合理的原则是：

1. 先看最新岗位要求，确认当前模块优先级有没有变化
2. 再决定是沿用原计划，还是重排知识地图和周次安排
3. 每天进入问答前，先看当前版本里对应模块的框架和高频题
4. 每周只挑 2-3 个最核心问题写成完整答案
5. 其余问题先做到“能口头讲 + 有提纲”

结论：

- 你确实需要单独补一块“面试高频知识点的系统梳理”
- 但这块梳理不需要被旧版计划绑死
- 最有效的方式是：以最新岗位要求为准，动态调整月计划和知识地图
