## 1. 这一块在解决什么问题

循环依赖这块不是在考你会不会背 `singletonFactories`、`earlySingletonObjects`、`singletonObjects`，而是在考你能不能把下面四件事讲清楚：

1. 什么叫循环依赖，为什么 Spring 容器创建 Bean 时会遇到这个问题。
2. 为什么 Spring 不是“所有循环依赖都能解决”，而是只能解决一部分。
3. 三级缓存到底在解决什么问题，为什么不是二级缓存就够了。
4. 构造器注入、原型 Bean、AOP 代理介入时，为什么循环依赖会有不同结果。

把这四个问题串起来，主线其实也很稳定：

- Spring 容器要创建 Bean，同时还要处理依赖注入。
- 如果 A 依赖 B，B 又依赖 A，容器在创建过程中就会出现“对象还没完全创建好，但另一个 Bean 已经要用它”的问题。
- 直接等对象完全创建完再暴露已经来不及，所以 Spring 需要“提前暴露一个可引用对象”。
- 但提前暴露的不一定是原始对象，还可能是后续需要增强的代理对象，这就引出了三级缓存。
- 即便如此，Spring 也不是万能的，因为有些依赖阶段太早，有些作用域不适合缓存复用。

所以这块真正要建立的是三种能力：

- 能解释循环依赖为什么会发生在 Bean 创建期。
- 能解释三级缓存是在解决“提前暴露”和“可能需要代理”这两个问题。
- 能说明 Spring 能解决什么场景，不能解决什么场景。

## 2. 先把主线串起来，再看知识树

### 2.1 先用一条因果链把全文串起来

如果你觉得循环依赖这块容易散，通常不是三级缓存名字没背住，而是没有先抓住下面这条因果链：

1. 先看循环依赖为什么会出现：Bean 创建和依赖注入是分阶段的。
2. 再看为什么必须“提前暴露”：否则另一个 Bean 在注入时拿不到依赖对象。
3. 再看为什么提前暴露不能只暴露原始对象：因为这个 Bean 后面可能还要被 AOP 等机制增强成代理对象。
4. 再看三级缓存为什么存在：它本质上是在管理“成品对象、提前暴露对象、可生成提前对象的工厂”三种状态。
5. 最后看哪些场景仍然解决不了：构造器注入、原型 Bean、某些代理和依赖链组合。

所以这篇文档不是在讲几个缓存名词，而是在回答一条连续问题链：

- 为什么会出现循环依赖。

- 为什么提前暴露还要考虑代理对象。

- 哪些场景 Spring 仍然救不了。

## 3. 核心重点展开

### 3.1 循环依赖为什么会出现在 Spring Bean 创建过程中

这一节在回答什么问题：

- 循环依赖为什么是容器创建期问题。
- 为什么它不是一个“普通引用关系”这么简单。

这个问题是怎么解决的：

循环依赖会出现在 Spring Bean 创建过程中，是因为 Bean 的“实例化”和“依赖注入完成”不是同一时刻。如果 A 在创建时需要 B，而 B 在创建时又反过来需要 A，就会出现一个典型问题：A 还没完全初始化完成，但 B 已经需要拿到 A 的引用。所以循环依赖的本质，不是“两个类互相认识”，而是“两个 Bean 都在创建过程中，同时依赖对方已经可用”。

放回最小场景里看，比如 `ServiceA` 注入 `ServiceB`，`ServiceB` 又注入 `ServiceA`。如果是 Setter 注入，Spring 可能先把 `ServiceA` 实例化出来，再在属性填充阶段注入 `ServiceB`。当它去创建 `ServiceB` 时，又需要 `ServiceA`，这时如果没有提前暴露机制，就会发现 `ServiceA` 还没走完整个生命周期。

从机制上可以拆成三层：

1. Spring 创建 Bean 不是一步到位，而是先实例化、再填充属性、再初始化。
2. 只要在属性填充阶段出现互相依赖，就可能触发循环依赖问题。
3. 问题的关键不在于“有没有对象”，而在于“对方要的那个对象能不能在当前阶段被安全引用”。

所以循环依赖题真正要回答的是：为什么对象创建分阶段这件事，会让“半成品对象引用”变成一个必须被处理的问题。

### 3.2 为什么 Spring 只能解决一部分循环依赖，而不是所有情况都能解决

这一节在回答什么问题：

- Spring 的能力边界到底在哪。
- 为什么面试里不能把这题答成“Spring 可以解决循环依赖”。

这个问题是怎么解决的：

Spring 只能解决一部分循环依赖，因为它依赖的前提是：对象能够先被实例化，再在后续阶段把“早期引用”暴露出去。这通常只适用于单例 Bean、且依赖发生在属性填充阶段的场景。对于构造器注入、原型 Bean 等情况，这个前提不成立，所以 Spring 解决不了。

把这个边界放回场景里会更直观。如果 A 和 B 都通过构造器注入依赖彼此，那么 A 连第一步实例化都过不去，因为创建 A 前就必须先拿到完整 B，而创建 B 前又必须先拿到完整 A。这个时候连“提前暴露一个半成品对象”的机会都没有，所以循环依赖不再是缓存策略问题，而是创建顺序本身无法展开的问题。

这也是为什么这题不能只背结论，而要讲清前提：

1. 必须先能拿到实例化后的对象。
2. 依赖最好发生在属性填充阶段，而不是构造阶段。
3. 作用域必须允许容器缓存和复用这个早期引用。

所以更准确的表述不是“Spring 能解决循环依赖”，而是“Spring 能解决一部分单例 Bean、属性注入场景下的循环依赖”。超出这个前提，容器就没有足够的时机和结构把问题化解掉。

### 3.3 三级缓存到底在解决什么问题，为什么不是二级缓存就够了

这一节在回答什么问题：

- 三级缓存的本质作用是什么。
- 为什么它不是单纯多放一层 Map。

这个问题是怎么解决的：

三级缓存的核心，不是为了把缓存层数搞复杂，而是为了同时处理两件事：第一，Bean 还没完成初始化时，要能提前暴露一个可引用对象；第二，这个提前暴露出去的对象，后面可能不是原始对象，而是需要经过代理包装的最终对象。所以三级缓存真正解决的是“提前暴露”和“代理一致性”这两个问题，而不是单纯的缓存分层。

把这件事放回场景里更容易理解。如果 A 需要 B，B 又需要 A，而 A 最终还要被 AOP 代理，那么 B 在注入 A 时，最好拿到的不是“将来会失效的原始对象”，而是和最终容器里一致的那个引用。三级缓存里对象工厂的作用，就是在真正需要早期引用时，再按需决定返回原始对象还是代理对象。

可以把三层缓存理解成三种状态管理：

1. 一级缓存放的是已经创建完成的单例成品对象。
2. 二级缓存放的是已经提前暴露出来的早期对象。
3. 三级缓存放的是还没真正暴露对象前、可以按需生成早期引用的对象工厂。

这里为什么不是二级缓存就够了，关键就在于“不能过早决定暴露什么”。如果没有第三级，只靠二级缓存，很多情况下你只能提前暴露原始对象，而无法优雅处理“这个 Bean 其实应该提前暴露代理对象”的问题。三级缓存的价值，不在于名字多一层，而在于把“早期引用怎么生成”的决策推迟到真正需要的时候。

### 3.4 哪些场景仍然解决不了，为什么代理介入后结果会不一样

这一节在回答什么问题：

- 为什么 Spring 不是万能修复器。
- 为什么构造器注入、原型 Bean、AOP 代理介入时，循环依赖会有不同结果。

这个问题是怎么解决的：

Spring 处理循环依赖的能力，建立在“单例 + 属性注入 + 可提前暴露引用”这套前提上。只要这套前提被破坏，结果就会变。构造器注入的问题在于对象连实例化都还没完成，就已经要求拿到完整依赖；原型 Bean 的问题在于它本来就不走单例缓存复用主线，容器也很难像单例一样稳定管理它的早期引用；AOP 代理介入时，问题则从“能不能暴露对象”进一步升级成“暴露出去的到底该是原始对象还是代理对象”。

所以这些场景的差异，不是零散边界，而是在检验循环依赖处理的三个核心前提有没有被满足：

1. 有没有机会先实例化出一个对象。
2. 容器能不能在后续阶段继续管理并复用这个对象引用。
3. 这个早期引用和最终容器里的对象语义能不能保持一致。

这也是为什么面试官会追问构造器注入、原型 Bean 和代理问题。它们不是额外刁难，而是在确认你有没有真正理解循环依赖处理依赖的前提条件。一旦把这三个前提讲清楚，很多边界就不会再显得像背诵题。

## 4. 边界、误区、常见追问

- 不要把循环依赖讲成“两个类互相引用”，重点是 Bean 创建期和依赖注入期的冲突。
- 不要把三级缓存答成单纯记忆题，关键是“提前暴露 + 代理一致性”。
- 构造器注入场景通常解决不了，因为连实例化都过不去。
- Prototype Bean 不适合依赖这套单例缓存思路。
- 面试官经常继续追问：如果有 AOP 代理，提前暴露出来的到底应该是谁。

## 5. 自测清单

- 你能不能先不背缓存名，直接讲清循环依赖为什么会发生。
- 你能不能解释为什么 Spring 只能解决一部分循环依赖。
- 你能不能说明三级缓存的关键不是层数，而是“提前暴露 + 代理一致性”。
- 你能不能讲清构造器注入为什么通常解决不了。
- 你能不能说明代理介入后，为什么循环依赖问题会进一步变复杂。
- 你能不能用 1-2 分钟把“为什么需要三级缓存”讲顺。
