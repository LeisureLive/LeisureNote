## 1. 这一块在回答什么问题

Week3 这些专题如果分开看，容易出现一个常见问题：每篇都懂一点，但一旦被问到“这些组件为什么会同时出现在一条数据链路里”，脑子里就只剩名词列表。

这份总摘要真正要回答的是下面这条连续问题：

1. 为什么数据平台和数据服务链路里，经常会同时出现 `Kafka`、`Flink`、`Spark`、`HDFS`、`Hive`、`Impala`、`Presto`、`Doris / Holo`、`ES` 这些组件。
2. 这些组件分别是在解决接入、计算、存储、元数据管理、交互查询、服务化分析和搜索查询中的哪一层问题。
3. 为什么它们经常不是互相替代关系，而是沿着同一条链路分层分工。
4. 如果题目从“讲组件”切到“讲链路”或“讲选型”，应该怎么把这些角色重新串起来。

所以这块的主线不是背“谁更快、谁更高级”，而是先把整条链路拆成几层：

- 数据进入系统的入口层
- 实时与离线计算层
- 存储与表语义层
- 交互式查询与分析服务层
- 面向搜索、服务读和平台治理的补充层

只有先把层次和职责分开，后面再谈选型、成本、边界和平台化能力才不会散。

## 2. 主线

1. 先看入口：数据通常先通过消息系统或日志采集进入平台，`Kafka` 这类组件解决的是解耦、缓冲和削峰，而不是最终分析本身。
2. 再看计算：`Flink` 更偏长期运行、状态化的实时处理，`Spark` 更偏批处理、离线计算和大规模重算。
3. 再看存储：`HDFS` 解决低成本、可容错的大规模留存，`Hive` 解决表语义、元数据和离线仓库组织。
4. 再看查询：`Impala / Presto` 更偏文件湖上的交互式 SQL，`Doris / Holo` 更偏低延迟、高并发、服务化分析。
5. 最后看补充能力：`ES` 解决的是全文检索、复杂倒排搜索和部分检索型聚合，不是事务主存储，也不是通用分析仓库。

## 3. 核心问题展开

### 3.1 为什么一条数据链路通常不会只靠一个引擎打到底

这一节在回答什么问题：

- 为什么现实里的数据链路总是多组件组合，而不是一个系统包办一切。
- 这些组件并存的根本原因是什么。

这个问题是怎么解决的：

一条完整的数据链路，通常要同时满足下面几类目标：

1. 数据进入系统时不能把上游和下游直接绑死。
2. 实时链路要尽快处理并输出结果。
3. 离线链路要能低成本保存大量历史数据，并支持反复重算。
4. 查询层既可能要交互式分析，也可能要低延迟服务读，还可能要全文搜索。
5. 平台层还要继续承担元数据、调度、监控、权限和多租户治理。

这些目标天然不是同一种成本结构：

- 有的更关注削峰和解耦。
- 有的更关注状态化实时计算。
- 有的更关注大规模留存和重算。
- 有的更关注交互式分析。
- 有的更关注低延迟高并发查询。
- 有的更关注文本检索和倒排索引。

所以多组件并存不是因为“技术栈堆得多”，而是因为它们各自对准的是不同问题。如果硬让一个系统同时把这些问题全接住，要么成本太高，要么边界很快失控。

### 3.2 `Kafka`、`Flink`、`Spark` 分别落在哪一层，它们为什么经常会连在一起

这一节在回答什么问题：

- 这三个高频组件分别在解决什么问题。
- 为什么它们经常会顺着同一条链路连续出现。

这个问题是怎么解决的：

这三者最容易被混成一句“都是大数据组件”，但它们的职责完全不同。

1. `Kafka`
   - 解决的是数据入口层的解耦、缓冲、削峰和异步分发。
   - 它回答的是“数据先怎么稳定进来、怎么让上下游不直接绑死”。
   - 它不是最终计算引擎，也不负责复杂分析。
2. `Flink`
   - 解决的是实时、状态化、长期运行的流处理。
   - 它回答的是“数据进来之后，如何按事件流持续加工、聚合、关联、补偿和输出”。
   - 它更适合需要分钟级甚至秒级更新、并且要保留中间上下文的场景。
3. `Spark`
   - 解决的是批处理、离线计算、大规模重算和一部分 SQL 分析执行。
   - 它回答的是“面对大批量历史数据，如何做更重的离线加工、宽表构建、回溯重跑和复杂聚合”。
   - 它不强调像 Flink 一样长期驻留和细粒度事件时间控制。

它们为什么经常连在一起？因为一条数据链路常常就是：

- 先用 `Kafka` 把数据接进来并缓冲。
- 再用 `Flink` 做实时加工和近实时产出。
- 再把数据沉淀到存储层，配合 `Spark` 做离线宽表、批处理、补数和重算。

所以三者不是并列替代关系，而是顺着“入口 -> 实时 -> 离线”逐层分工。

### 3.3 `HDFS` 和 `Hive` 为什么必须放在一起理解

这一节在回答什么问题：

- 为什么只说“HDFS 存数据”还远远不够。
- 为什么数据进了文件系统之后，还需要 Hive 这一层。

这个问题是怎么解决的：

`HDFS` 解决的是大规模、低成本、可容错的数据留存问题。它更像分布式存储基础层，适合大文件、顺序读写和离线计算共用数据。它不擅长：

- 高频随机更新
- 海量小文件管理
- 直接提供表语义和统一 SQL 入口

`Hive` 补的正是 `HDFS` 之上的组织层。它把文件组织成：

- 表
- 分区
- 字段定义
- 仓库语义
- 元数据目录

这样做的意义是：

- 数据不再只是目录和文件，而是可以被统一管理的数据集。
- 上层引擎不必各自维护一套表结构定义。
- 批处理、数仓分层、分区表治理和元数据管理才有统一入口。

所以 `HDFS` 和 `Hive` 必须放在一起理解，因为它们分别在回答：

- 数据怎么可靠存下来。
- 存下来的数据怎么被组织成可管理、可查询、可演化的表。

### 3.4 `Impala / Presto`、`Doris / Holo`、`ES` 为什么不能混成“查询引擎”

这一节在回答什么问题：

- 这些系统都能查数据，为什么仍然要继续区分。
- 它们各自适合的查询问题到底是什么。

这个问题是怎么解决的：

这些系统确实都能“查”，但它们对准的查询问题并不一样。

1. `Impala / Presto`
   - 更偏文件湖或共享表上的交互式 SQL 查询。
   - 常常直接复用 `HDFS / Hive Metastore` 上的数据和表定义。
   - 它们解决的是“已经有很多离线沉淀数据了，怎么更快地交互式查出来”。
   - 代价是它们通常仍然要面对文件组织、扫描范围和共享存储的约束。
2. `Doris / Holo`
   - 更偏服务化的分析查询、低延迟响应和更高并发承载。
   - 它们解决的是“分析查询不只是给人临时查，而是已经逐渐接近在线服务读能力”。
   - 相比直接查文件湖，它们更强调：
     - 查询延迟
     - 并发能力
     - 列式存储和分析型服务能力
3. `ES`
   - 更偏全文检索、倒排索引、多条件搜索和部分检索型聚合。
   - 它解决的是“按关键词、词条、文本字段、复杂过滤快速检索”。
   - 它不适合承担强事务主存储，也不是通用替代型 OLAP 仓库。

所以它们虽然都属于“查询层”，但回答的问题不同：

- `Impala / Presto`：共享表上的交互式 SQL。
- `Doris / Holo`：更服务化、更低延迟的分析查询。
- `ES`：搜索和倒排检索。

### 3.5 把这些组件放回一条典型链路里时，应该怎么理解它们的先后关系

这一节在回答什么问题：

- 如果题目变成“讲一条典型数据链路”，这些组件应该怎么串起来。
- 哪些组件更像前置能力，哪些更像后置能力。

这个问题是怎么解决的：

一条典型的数据平台链路，通常可以压成下面这条顺序：

1. `Kafka`
   - 数据入口和缓冲层。
2. `Flink`
   - 实时处理、近实时指标、流式加工、实时订阅和部分实时入库。
3. `HDFS`
   - 历史明细和离线数据留存层。
4. `Hive`
   - 表语义、元数据和仓库组织层。
5. `Spark`
   - 离线加工、宽表构建、批处理、补数重算。
6. `Impala / Presto`
   - 面向共享表和文件湖的交互式分析查询。
7. `Doris / Holo`
   - 面向更低延迟、更高并发的分析服务层。
8. `ES`
   - 面向搜索、复杂过滤和全文检索场景的补充查询层。

这里最重要的不是把顺序背下来，而是理解为什么它们会分层：

- 前面几层更偏“把数据接进来、算出来、存下来、组织好”。
- 后面几层更偏“怎么把已经存在的数据按不同查询模式提供出去”。

一旦把这个顺序理解成“先生产数据能力，再提供查询能力”，很多选型问题就不会再混。

### 3.6 面对实时订阅、实时查询和数据平台题时，这些组件各自最容易落在哪

这一节在回答什么问题：

- 同样一组组件，放到不同题型里，为什么重点会变化。
- 面试准备时最应该优先抓哪些角色映射。

这个问题是怎么解决的：

如果题型不同，组件角色的落点也会变。

1. 实时订阅系统
   - 重点更容易落在：
     - `Kafka` 作为入口和缓冲
     - `Flink` 或实时处理模块作为加工层
     - 下游 Kafka / HTTP / gRPC 等投递目标
   - 这时重点不是 `Hive` 或 `Impala` 怎么查，而是实时链路的吞吐、幂等、重试、隔离和扩展。
2. 实时查询系统
   - 重点更容易落在：
     - `Redis` 做热点层
     - `MySQL` 做事务型事实层
     - `ES` 做搜索型查询
     - `Doris / Impala / Presto` 做分析型查询
   - 这时真正要回答的是“不同查询模式为什么要落到不同引擎”。
3. 数据平台题
   - 重点更容易落在：
     - `Kafka / Flink / Spark / HDFS / Hive` 组成生产和加工链
     - `Impala / Presto / Doris / ES` 组成查询和服务能力层
     - 元数据、调度、监控、权限、多租户把这些引擎长期组织起来
   - 这时组件本身不是重点，重点是这些组件如何被平台化治理。

所以同样一组组件，在不同题型里不是平均展开，而是要看题目到底更偏：

- 数据进入和实时处理
- 查询服务和引擎选型
- 平台治理和能力抽象

### 3.7 为什么最终要把这些组件上升成“能力分层”，而不是停留在“技术栈列表”

这一节在回答什么问题：

- 为什么只会列组件，通常还不算真正理解了这条链路。
- 这些组件最后应该沉淀成什么样的认知结构。

这个问题是怎么解决的：

如果最终脑子里留下的还是：

- 我会 Kafka
- 我会 Flink
- 我会 Spark
- 我会 Hive

那这条 Week3 主线其实还没有真正闭环。更稳定的理解方式应该是把它们上升成下面几层能力：

1. `接入与削峰能力`
   - Kafka
2. `实时计算能力`
   - Flink
3. `离线计算与重算能力`
   - Spark
4. `大规模留存与表组织能力`
   - HDFS + Hive
5. `交互分析与服务化查询能力`
   - Impala / Presto / Doris / Holo
6. `搜索与检索能力`
   - ES
7. `平台治理能力`
   - 元数据、调度、监控、权限、多租户

只有把这些组件还原成能力分层，后面再讲系统设计、数据平台和项目升级时，你讲的才不是“用了哪些技术”，而是“系统为什么需要这些能力，以及这些能力分别落在哪一层”。

## 4. 边界与误区

- `Kafka` 是入口和缓冲层，不是复杂分析引擎。
- `Flink` 和 `Spark` 都能算数据，但前者更偏长期运行、状态化实时处理，后者更偏批处理和离线重算。
- `HDFS` 不是数据库，它解决的是低成本留存和容错，不负责高频随机更新。
- `Hive` 不是查询性能引擎，它解决的是表语义和元数据组织。
- `Impala / Presto` 快，并不等于它们天然适合所有在线查询。
- `Doris / Holo` 更服务化，并不等于可以替代所有事务型存储。
- `ES` 适合搜索和检索，不适合承担强事务主存储。
- 组件越多不代表架构越好，关键是每一层是否真的在接住不同成本结构的问题。

- 是否能区分 `Kafka`、`Flink`、`Spark` 分别在入口、实时和离线层解决什么问题。
- 是否能说明 `HDFS` 和 `Hive` 为什么必须放在一起理解。
- 是否能区分 `Impala / Presto`、`Doris / Holo`、`ES` 各自更适合的查询问题。
- 是否能把一条典型链路讲成 `Kafka -> Flink -> HDFS / Hive -> Impala / Doris / ES` 这种分层结构。
- 是否能说明为什么同样一组组件，放到实时订阅、实时查询、数据平台题里，重点会不一样。
- 是否能把这些组件最终上升成“接入、实时、离线、留存、查询、搜索、治理”七层能力，而不是只停留在技术栈列表。
