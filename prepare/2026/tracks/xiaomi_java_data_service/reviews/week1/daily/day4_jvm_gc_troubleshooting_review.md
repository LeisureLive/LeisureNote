### 3.2 讲散了的点

- 可达性分析：
  - 容易说成“没有被引用的对象会被回收”
  - 更准确应该是：
    - 从 `GC Roots` 出发不可达的对象，才会进入可回收集合
- `Minor GC`、`Major GC`、`Full GC`：
  - `Major GC` 这个词说得有点过死
  - `Full GC` 的触发条件说得偏窄，容易漏掉：
    - 晋升失败 / 分配担保失败
    - 元空间不足
    - 并发回收失败退化
- `STW`：
  - 知道“要停业务线程”
  - 但“一致性快照”这层机制没有主动讲出来
- `Full GC` 排查路径：
  - 会先按“回不回落”做分叉
  - 但后面容易直接猜原因，没有稳定补出：
    - GC 日志
    - `jstat` / `jcmd`
    - heap dump
    - MAT
    - 业务指标
- 区分内存泄漏 / 大缓存 / 积压：
  - 知道要看“大头对象是什么”
  - 但“趋势 + dump + 引用链 + 业务指标”还没有形成固定答法
- 判断 RT 抖动是否由 GC 导致：
  - 目前只说到了“对齐 GC 时间和 RT 时间”
  - 还没自然补上：
    - SQL
    - 下游接口
    - CPU
    - 线程池
    这些排除路径

### 4.2 当前算法薄弱点

- 题型识别：
  - 已经知道“数组里有负数时，普通滑动窗口不一定适用”
  - 但“什么时候应该立刻转前缀和 + 哈希”还没完全固化
- 思路建立：
  - `560` 需要提示，说明“区间和 -> 前缀和之差”的转换还不够自动化
- 边界处理：
  - `525` 的空前缀位置容易写错
- 代码实现：
  - 主框架能写
  - 但 `map` 存“出现次数”还是“第一次出现的位置”，还需要更主动地区分
- 口头解释：
  - 还需要把下面几句话压成 1 分钟版本：
    - 为什么不能直接用滑动窗口
    - 为什么查 `prefixSum - k`
    - 为什么这题要存次数 / 最早位置

### 4.3 “前缀和 + 哈希”题型识别信号

- 题目是连续子数组 / 连续区间问题
- 目标通常和下面这些量有关：
  - 区间和
  - 某种平衡关系
  - 满足条件的个数
  - 满足条件的最长长度
- 暴力思路往往要枚举所有区间，时间复杂度会落到 `O(n^2)`
- 普通滑动窗口不适合：
  - 数组里可能有负数
  - 或者题目条件不是靠窗口单调性就能维护
- 题意通常可以改写成：
  - 两个前缀和之差等于 `target`
  - 或两个位置的前缀和相等
- 遍历到当前位置时，需要快速查询“历史上是否出现过某种前缀状态”
- 做题时先固定问自己两句：
  - 这题要求的是“个数”还是“最长长度”
  - `map` 里该存“出现次数”还是“第一次出现的位置”
- 当前这两题对应关系要记牢：
  - `560`：
    - 求个数
    - `map` 存前缀和出现次数
  - `525`：
    - 求最长长度
    - `map` 存前缀和第一次出现的位置
