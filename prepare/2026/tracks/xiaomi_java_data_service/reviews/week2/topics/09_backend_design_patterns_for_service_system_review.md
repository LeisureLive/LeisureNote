## 1. 这一块在解决什么问题

后端设计模式这块不是在考你背多少定义，而是在考你能不能把下面三件事讲清楚：

1. 为什么后端系统里会反复出现“分支过多、对象创建复杂、流程骨架稳定但细节多变”这些问题。
2. 策略、工厂、模板方法分别在解决什么工程问题。
3. 你能不能把模式从抽象概念落到实际服务系统、配置扩展和业务治理场景里。

把这些点串起来，主线其实很稳定：

- 后端系统长期演进后，最常见的问题不是语法写不出来，而是分支膨胀、创建逻辑散落、流程修改牵一发动全身。
- 设计模式的价值不是“让代码显得高级”，而是帮你把变化点和稳定点分开。
- 策略、工厂、模板方法之所以高频，是因为它们正好对应“规则切换、对象创建收口、流程骨架复用”三类高频问题。
- 面试里之所以重项目场景，是因为模式只有落到真实演进问题里，才有可验证性。

所以这块真正要建立的是三种能力：

- 能解释模式在解决什么工程问题。
- 能解释为什么这个问题适合用某种模式，而不是只背定义。
- 能把模式落到真实后端项目场景里表达出来。

## 3. 核心重点展开

### 3.1 为什么后端系统需要设计模式，而不是继续堆 `if-else` 和手工 `new`

这一节在回答什么问题：

- 模式在后端系统里的真实价值是什么。
- 为什么它不是课堂题，而是演进问题的解决方案。

这个问题是怎么解决的：

后端系统需要设计模式，是因为系统一旦长期演进，变化点会越来越多。如果遇到规则切换就堆 `if-else`，遇到对象创建就到处手工 `new`，遇到流程变种就复制粘贴主流程，代码很快会出现分支膨胀、创建逻辑分散、修改牵连范围过大的问题。设计模式的价值，就是把变化点和稳定点拆开，让系统更容易扩展和维护。

放回场景里更容易理解。比如一个数据服务里，不同来源的数据校验规则不同，不同任务类型要创建不同处理器，整个处理流程大体一致，但不同任务在某几个步骤上有差异。这些问题如果都硬写在一个类里，系统很快会变脆。也就是说，模式解决的不是“功能写不写得出来”，而是“变化如何可控”。

从机制上看，这件事可以拆成三层：

1. 先识别哪些地方是稳定主线，哪些地方会变化。
2. 再通过模式把变化点抽成接口、工厂或可覆盖步骤。
3. 这样后续扩展时就不必频繁修改原有稳定代码。

收益是扩展成本下降、修改牵连范围变小；边界是设计模式不是目的，本质仍然是服务于真实变化点。如果没有清晰变化点，强行上模式只会增加复杂度。

### 3.2 策略模式在后端系统里通常在解决什么问题

这一节在回答什么问题：

- 为什么策略模式在业务规则场景里高频。
- 它到底比一串 `if-else` 强在哪里。

这个问题是怎么解决的：

策略模式通常在解决“同一类业务行为有多种可替换实现”这个问题。它的价值不是把分支藏起来，而是把不同规则实现收口成统一接口，让调用方只依赖抽象，不依赖一长串条件判断。

放回场景里看，比如不同数据源的清洗规则不同、不同下发渠道的路由规则不同、不同计费模式的计算逻辑不同。如果全部写在一个方法里，很容易变成不断膨胀的条件分支。策略模式做的事情，就是把“会不断增加的规则”独立出来，让新增规则更多表现为“扩展一个实现”，而不是“继续改主干”。

从机制上可以这样理解：

1. 先把可变规则抽成统一接口。
2. 不同规则实现分别封装在独立策略类里。
3. 调用方通过上下文或选择器决定当前使用哪种策略。
4. 这样新增规则时，通常不需要去改老逻辑主干。

收益是规则切换更清晰、主干逻辑更稳定；边界是并不是所有 `if-else` 都值得抽策略，只有当规则会继续增加、替换、按条件切换时，这种收口才真正有价值。

### 3.3 工厂模式在后端系统里为什么高频，它在解决什么创建问题

这一节在回答什么问题：

- 为什么对象创建逻辑也需要收口。
- 工厂模式和简单手工 `new` 的差别是什么。

这个问题是怎么解决的：

工厂模式高频，是因为后端系统里很多对象不是“无脑 new 一下”就结束了，而是创建时带条件判断、依赖组装、默认参数或不同实现选择。如果这些创建逻辑散在业务代码里，调用方就会同时承担“做业务”和“决定怎么造对象”两件事。工厂模式的价值，就是把对象创建责任集中起来。

放回场景里更容易理解。比如根据任务类型创建不同处理器、根据存储类型创建不同客户端、根据配置创建不同下游连接器，这些都属于典型工厂场景。此时调用方真正关心的是“我需要什么能力”，而不是“这个能力应该怎么组装出来”。

从机制上可以这样拆：

1. 先把对象选择和创建逻辑从业务代码里抽出去。
2. 由工厂根据条件、配置或类型决定返回哪个实现。
3. 调用方只依赖抽象能力，不依赖具体构造细节。

收益是创建责任收口、实现切换更自然、组装细节集中治理；边界是如果对象创建本身非常简单、没有条件分支和组装复杂度，强行引入工厂可能反而增加抽象层。

### 3.4 模板方法为什么适合流程骨架稳定、步骤细节多变的场景

这一节在回答什么问题：

- 模板方法到底适合什么类型的问题。
- 为什么它和策略模式不是同一类东西。

这个问题是怎么解决的：

模板方法适合“主流程稳定，但某几个步骤需要按场景变化”的问题。它的价值，是把固定流程骨架留在父类或基类里，把可变步骤延迟给子类实现。这样既保住了流程顺序的一致性，又允许局部差异化扩展。

放回场景里看，比如一个数据导入任务总会经历参数校验、数据拉取、转换清洗、落库或下发、结果记录。不同任务类型可能只是在“怎么拉、怎么转、怎么落”上不同，但整体顺序不变。这种场景不是“规则切换”，而是“固定顺序下局部步骤可变”，所以更适合模板方法，而不是策略模式。

从机制上可以这样理解：

1. 先把稳定流程沉淀为统一模板。
2. 把可变步骤抽成可覆盖方法。
3. 子类只需要实现差异步骤，不必重复整套流程。

收益是流程一致性更强、重复代码更少；边界是如果流程本身经常整体变化，而不是局部步骤变化，模板方法反而会变得僵硬。

### 3.5 为什么设计模式最后一定要落到项目场景里讲，怎么讲才不像背定义

这一节在回答什么问题：

- 为什么面试官听完模式定义后，通常还会继续追问“你项目里怎么用过”。
- 怎么把策略、工厂、模板方法讲成真实工程选择，而不是抽象概念。

这个问题是怎么解决的：

设计模式最后一定要落到项目场景里讲，是因为面试官真正想确认的不是“你知不知道这个名字”，而是“你能不能识别代码里的变化点，并知道为什么这个变化点适合用某种模式来收口”。如果回答里只有定义，没有项目场景、触发问题、选型理由和改造收益，那就还是停留在书本层。

所以这类题真正稳定的讲法应该是：先说项目里原来遇到了什么演进问题，再说为什么普通写法开始失控，再说为什么选这个模式，而不是别的方式，最后补改完后的收益和边界。放回场景里看，比如你在一个数据服务系统里做任务处理框架：最初不同任务类型都写在一个大类里，通过一串 `if-else` 分支选择执行逻辑；后来任务类型越来越多，新增一种任务就要改主干逻辑，还容易影响老流程。这时如果你把不同任务规则抽成策略，把处理器实例创建收口到工厂，把统一执行骨架沉淀成模板方法，面试时就能把三种模式都落到同一条真实演进链上。

从表达机制上，可以把“项目化表达”压成一个固定骨架：

1. 场景是什么。
2. 原问题是什么。
3. 为什么选这个模式。
4. 模式落在哪里。
5. 收益和边界是什么。

收益是面试官听到的是工程判断而不是模式名词；边界是如果项目里并没有对应的变化点，却强行往模式上靠，反而会显得像背模板。

## 4. 边界、误区、常见追问

- 不要把设计模式答成定义背诵题，重点要回到工程问题。
- 策略模式重点是规则切换，不是所有 `if-else` 都要强行抽策略。
- 工厂模式重点是创建责任收口，不是所有对象都必须走工厂。
- 模板方法重点是流程骨架稳定，不适合流程本身经常大改的场景。

## 5. 自测清单

- 你能不能先不背定义，直接讲设计模式在后端系统里的价值。
- 你能不能用业务规则切换场景解释策略模式。
- 你能不能用对象创建复杂场景解释工厂模式。
- 你能不能用流程骨架稳定场景解释模板方法。
- 你能不能按“场景 -> 痛点 -> 为什么选这个模式 -> 落地方式 -> 收益”把其中至少一个模式讲成项目经验。
