## Context

当前 `prepare/2026/` 同时承载了三类内容：

- 跨年份、跨岗位都可能复用的协作框架与模板
- 2026 年这一轮面试准备的年度入口与执行记录
- 小米 Java 数据服务这一条具体岗位主线的计划、知识地图、算法和复习材料

这种结构在准备初期可以快速落地，但随着文档增多，`workspace/` 的边界开始模糊：一部分内容其实是长期复用框架，一部分是当前年度的执行记录，二者混在 `prepare/2026/` 下会削弱长期可复用性，也会让入口说明越来越难写清。另一方面，`prepare/2026/tracks/xiaomi_java_data_service/` 中的计划、知识地图、算法路线和复习输出是逐步生成的，虽然已经具备较完整材料，但缺少一次“按目标岗位要求反推”的系统性复核。

这次变更的直接使用者是仓库维护者本人，核心诉求不是发布给外部团队，而是让后续任何一次面试准备都更容易进入、复用、调整和复盘。

## Goals / Non-Goals

**Goals:**

- 明确 `prepare/` 目录中“长期复用层”“年度执行层”“岗位 track 层”的职责边界。
- 让通用框架不再只看起来服务于 `2026`，同时保留 2026 这一轮准备的实际执行上下文。
- 为 `xiaomi_java_data_service` 建立一套“岗位要求 -> 现有材料 -> 缺口与调整”的复核方法。
- 输出可执行的结构调整和文档调整清单，后续可以直接按任务逐步落地。

**Non-Goals:**

- 不在本次设计阶段一次性重写所有已有面试材料正文。
- 不追求把所有准备内容抽象成通用模板系统或复杂知识库。
- 不要求在本次变更中新增原始 JD 存档体系或完整招聘信息数据库。

## Decisions

### Decision 1: 将 `prepare` 拆成“通用框架层 + 年度层 + track 层”的三层模型

采用三层模型来治理目录，而不是继续把所有内容堆在 `prepare/2026/` 下：

- 通用框架层：
  - 存放跨年份复用的方法、模板、协作规则和长期入口说明
  - 目标是服务未来每次面试准备，而不是只服务 2026
- 年度层：
  - 保留某一轮面试准备的实际上下文、年度入口和执行记录
- track 层：
  - 聚焦某个岗位方向、公司或专项主线的计划与复习内容

为什么这样做：

- 能回应用户对 `workspace/` “不应只作用于 2026”的担忧
- 能保留现有按年份组织的实际执行上下文
- 能继续支持未来新增其他年份或其他岗位方向，而不必重新推倒结构

备选方案：

- 备选 A：维持现有 `prepare/2026/workspace` 不动，只补 README
  - 放弃原因：只能缓解说明问题，不能真正解决长期复用层和年度层混杂的问题
- 备选 B：完全改成按公司或岗位组织，不保留年度层
  - 放弃原因：不利于后续按时间阶段管理一轮完整准备过程，也不利于记录阶段性进展

### Decision 2: 结构调整优先做“边界梳理 + 入口重写 + 小范围迁移”，而不是大规模重命名

本次调整遵循最小可行改动原则：

- 先建立清晰的目录职责和入口文档
- 再把明显属于长期复用的内容从年度层中抽离
- 对保留在 2026 年度层的文档，重写其定位和使用顺序

为什么这样做：

- 当前文档已经较多，直接大规模迁移容易打断现有使用习惯
- 先通过 README、索引和少量结构调整建立秩序，成本更低、回滚更容易

备选方案：

- 备选 A：一次性整体迁移所有 `workspace/` 文档并统一改名
  - 放弃原因：风险高，且很容易在没有稳定规则前做出新的混乱结构

### Decision 3: `xiaomi_java_data_service` 的 review 采用“岗位要求映射矩阵”驱动

对目标岗位主线不只做主观整理，而是建立显式映射：

- 岗位要求类别
  - Java 基础 / 框架链 / 分布式 / 数据平台 / 算法 / 系统设计 / 排障 / 项目表达
- 当前对应材料
  - 岗位要求汇总、知识地图、月计划、算法路线、复习文档、项目档案
- 结论
  - 已覆盖 / 覆盖不足 / 尚未覆盖
- 调整动作
  - 应修改哪份文档、应新增什么内容、优先级如何

为什么这样做：

- 用户担心“准备计划和岗位要求对不上的情况”
- 当前已有材料较多，靠直觉 review 很容易遗漏
- 显式映射能把“结构整理”和“准备质量 review”串到一起

备选方案：

- 备选 A：只人工阅读 `01_1month_plan.md` 做经验性点评
  - 放弃原因：难以覆盖算法、项目表达和资深岗能力缺口，也不利于后续反复复核

### Decision 4: 算法充分度 review 不只看专题数量，还要看训练闭环

算法部分的 review 不以“专题文件够不够多”为核心，而重点关注：

- 是否覆盖高频专题
- 是否有模板题 / 高频题 / 验收题的层次
- 是否有复盘、去重、口头表达与验收要求
- 强度安排是否与目标岗位和当前状态匹配

为什么这样做：

- 当前算法专题数量已经不少，真正的风险更可能在训练强度、验收闭环和表达迁移能力不足
- 这更贴合“是否足够充分”的判断标准

## Risks / Trade-offs

- [Risk] 过早移动大量文件可能打断当前使用路径
  → Mitigation: 先完成入口文档和结构规则，再做小范围迁移；每次迁移都同步更新入口说明。

- [Risk] 长期复用层抽象过度，反而增加维护成本
  → Mitigation: 只抽离明确跨年份复用的内容，不把所有文档都模板化。

- [Risk] 岗位要求映射 review 依赖现有合并 JD 文档，如果源信息本身不完整，会影响结论准确度
  → Mitigation: 在 review 输出中显式记录信息边界，并把“缺少原始 JD 存档”作为已知限制。

- [Risk] 结构整理与计划 review 同时进行，任务范围容易膨胀
  → Mitigation: 先完成目录边界和 gap review 结论，再决定哪些计划文档必须立即调整，哪些延后。

## Migration Plan

1. 盘点 `prepare/` 下现有目录和文档职责，形成“通用框架 / 年度执行 / track 主线”的归类表。
2. 定义目标结构及入口说明，先调整根 README 和关键 README。
3. 抽离或重定位明显属于长期复用层的文档，并更新旧入口到新入口的说明。
4. 对 `xiaomi_java_data_service` 建立岗位要求映射和 gap review 结论。
5. 根据 gap review 调整月计划、知识导航、track 入口或相关索引文档。
6. 复核所有入口文档，确保迁移后仍然容易找到内容。

回滚策略：

- 本次主要是文档结构与内容调整，可以通过 Git 直接回滚单次迁移或单个 README 改动。
- 若新结构验证后不理想，可保留 review 结论，只回退目录迁移和入口调整。

## Open Questions

- 通用框架层最终命名应采用 `common/`、`framework/`、`shared/` 还是其他更贴合面试准备语义的名称。
- `workspace/` 中哪些文件应该抽离为长期复用框架，哪些应保留为 2026 年度执行记录。
- 是否需要为后续年份预留统一的年度目录约定，例如 `prepare/2027/` 的推荐结构。
- 是否需要在本次变更中补一份明确的“岗位要求映射 / gap review”独立文档，作为后续持续更新的入口。
