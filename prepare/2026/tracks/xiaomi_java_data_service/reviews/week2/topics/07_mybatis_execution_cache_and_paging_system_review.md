## 1. 这一块在解决什么问题

MyBatis 这块不是在考你背多少类名，而是在考你能不能把下面四件事讲清楚：

1. 为什么数据访问层通常不直接在业务代码里手写 JDBC。
2. MyBatis 执行一条 SQL 时，Mapper 接口、代理对象、`SqlSession`、执行器之间是怎么串起来的。
3. 动态 SQL、一级缓存、二级缓存、分页问题为什么都会挂在这条执行链上。
4. MyBatis 适合解决什么，不适合解决什么，为什么有时不能把性能问题简单归因于框架。

把这些点串起来，可以这样理解：

- 业务代码需要数据访问层，但不希望手工管理大量 JDBC 模板代码。
- MyBatis 用 Mapper 接口和 SQL 映射把“调用方法”和“执行 SQL”接了起来。
- 执行链一旦统一，动态 SQL、缓存、参数映射、结果映射都可以挂到同一条流程里。
- 但分页和缓存如果用得不当，也会反过来带来性能和一致性边界问题。

所以这块真正要建立的是三种能力：

- 能解释 MyBatis 执行一条 SQL 的主流程。
- 能解释缓存和分页为什么要放回执行链里一起看。
- 能解释 MyBatis 的工程价值和风险边界。

## 3. 核心重点展开

### 3.1 为什么需要 MyBatis，而不是在业务代码里直接手写 JDBC

这一节在整条主线里解决的是第一个问题：

- MyBatis 的工程价值是什么。
- 为什么它不是“多包了一层没必要的框架”。

#### 先直接回答这个问题

需要 MyBatis，是因为业务系统的数据访问不能一直靠手工写 JDBC 模板代码。手工 JDBC 的问题不是能不能查出数据，而是连接获取、参数绑定、结果映射、异常处理这些样板逻辑会大量重复。MyBatis 的价值，就是把“SQL 仍由开发者掌控”与“数据访问样板被框架接管”结合起来。

#### 再用场景把问题钉住

如果每个业务方法都要自己写获取连接、创建 `PreparedStatement`、绑定参数、遍历 `ResultSet`、关闭资源，代码会非常重。MyBatis 把这些重复工作抽出来，让业务层更聚焦在 SQL 语义本身。

#### 是什么

MyBatis 可以先理解成：

- 一个强调 SQL 显式可控的数据访问框架。

#### 为什么有它

因为企业项目既需要：

- 避免 JDBC 样板代码泛滥。
- 又不想完全失去对 SQL 的控制。

#### 怎么理解它的机制

第一层，业务层调用的是 Mapper 接口方法。

第二层，框架负责把方法调用转换成 SQL 执行请求。

第三层，参数绑定、结果映射、资源管理由框架统一处理。

#### 面试时怎么讲才不空

“MyBatis 的价值不在于把 SQL 隐藏掉，而是在保留 SQL 可控性的同时，把 JDBC 样板代码收口。这样业务代码不需要重复处理连接、参数和结果映射，数据访问层会清晰很多。”

### 3.2 MyBatis 执行一条 SQL 的主流程到底是怎么串起来的

这一节在整条主线里解决的是第二个问题：

- Mapper 方法为什么能执行到数据库。
- `Mapper` 代理、`SqlSession`、执行器在这条链上各干什么。

#### 先直接回答这个问题

MyBatis 执行一条 SQL 的主流程，本质上是“Mapper 接口方法调用 -> 代理对象接管 -> 找到对应 SQL 映射 -> 通过 `SqlSession` / 执行器执行 -> 把结果映射回 Java 对象”。重点不是记住类名，而是知道方法调用怎么被转成一次数据库访问。

#### 再用场景把问题钉住

比如你调用 `userMapper.selectById(id)`，并不是接口自己能查库，而是其背后的代理对象接住调用，再根据方法和映射关系找到要执行的 SQL，完成参数绑定和结果转换。

#### 是什么

可以先把主流程压成下面几步：

1. 调用 Mapper 接口方法。
2. 代理对象拦住调用。
3. 定位对应 SQL 映射。
4. 通过 `SqlSession` 和执行器执行 SQL。
5. 把查询结果映射回目标对象。

#### 为什么有它

因为框架需要把“面向接口编程”的调用方式和“底层 SQL 执行”的细节解耦。

#### 怎么理解它的机制

第一层，Mapper 接口本身只是调用入口，真正工作的是代理对象。

第二层，代理根据方法信息找到对应的 SQL 定义。

第三层，执行器负责和数据库交互，完成参数绑定、语句执行和结果处理。

第四层，这条统一执行链也给后面的缓存、插件、分页等能力提供了挂点。

#### 面试时怎么讲才不空

“我回答 MyBatis 主流程时会先讲 Mapper 接口背后其实是代理对象，然后再讲它如何定位 SQL、通过 `SqlSession` 和执行器去执行，最后再做结果映射。这样缓存和分页为什么能插进来，也更容易接上。”

### 3.3 动态 SQL、一级缓存、二级缓存为什么总和主流程一起问

这一节在整条主线里解决的是第三个问题：

- 为什么这些能力不能脱离执行链单独理解。
- 缓存为什么既有收益，也有一致性边界。

#### 先直接回答这个问题

动态 SQL、一级缓存、二级缓存总和主流程一起问，是因为它们都不是独立漂在外面的功能，而是挂在同一次 SQL 执行链上的。动态 SQL 影响“实际发出去的 SQL 长什么样”，缓存影响“这次调用要不要真的访问数据库”。所以不理解执行链，就很难把这些能力讲顺。

#### 再用场景把问题钉住

比如一个查询接口：

- 查询条件是可选的，要动态拼装 SQL。
- 同一个 `SqlSession` 内重复查询，可能命中一级缓存。
- 跨会话复用结果时，可能涉及二级缓存。

这三个点看起来分散，但其实都挂在同一条“如何执行查询”的主线上。

#### 是什么

- 动态 SQL：根据参数条件动态生成最终 SQL。
- 一级缓存：`SqlSession` 级缓存。
- 二级缓存：Mapper 维度的跨会话缓存。

#### 为什么有它

因为数据访问既要灵活表达查询条件，也想减少重复查库成本。

#### 怎么理解它的机制和边界

第一层，动态 SQL 让同一条 Mapper 语句能覆盖不同查询条件组合。

第二层，一级缓存更贴近一次会话上下文，作用范围小，一致性边界相对更好控制。

第三层，二级缓存复用范围更大，但一致性和失效管理复杂度也更高。

第四层，所以缓存题不能只讲“能提速”，还要继续补“什么范围内有效、更新后怎么失效、为什么很多项目会谨慎使用二级缓存”。

#### 面试时怎么讲才不空

“动态 SQL 和缓存都应该放回执行链里讲。动态 SQL 决定最终发什么 SQL，一级缓存和二级缓存决定要不要真的去数据库。一级缓存相对自然，二级缓存收益更大但一致性边界更复杂，所以实际项目里往往更谨慎。”

### 3.4 分页和性能问题为什么不能简单归因于 MyBatis 本身

这一节在整条主线里解决的是第四个问题：

- 为什么分页题最后常会追到 SQL 和数据库本身。
- MyBatis 的边界到底在哪里。

#### 先直接回答这个问题

分页和性能问题不能简单归因于 MyBatis 本身，因为 MyBatis 更多是在组织 SQL 执行流程，真正决定查询代价的往往还是 SQL 设计、索引情况和数据库执行计划。MyBatis 可以通过插件或分页工具影响分页方式，但它不能替你把深分页、慢 SQL、本身低效的查询自动优化掉。

#### 再用场景把问题钉住

比如一个列表接口做深分页，如果底层 SQL 需要扫描大量记录后再 `limit`，即使框架层分页插件接入得很优雅，数据库代价仍然会很高。问题不在 Mapper 是否优雅，而在 SQL 自身执行代价。

#### 是什么

- 分页问题：在查询结果量大时，如何控制返回窗口和代价。
- 性能边界：区分框架开销和数据库执行开销。

#### 为什么有它

因为数据访问层问题很容易被误判成“框架慢”，但很多时候真正慢的是 SQL、索引、深分页策略。

#### 怎么理解它的机制和边界

第一层，MyBatis 可以帮你组织分页 SQL 或接入分页插件。

第二层，但物理分页、逻辑分页、深分页代价，最终还是落到数据库执行层。

第三层，所以排查性能时，要先区分是框架执行链问题，还是底层 SQL / 数据库问题。

#### 面试时怎么讲才不空

“我回答 MyBatis 分页问题，不会简单说‘用分页插件就行’，而是会先区分框架层做了什么，再回到底层 SQL 和数据库执行代价。因为深分页、复杂排序这些问题，最终瓶颈通常不在 MyBatis，而在查询本身。”

## 4. 边界、误区、常见追问

- 不要把 MyBatis 讲成 ORM 全自动框架，它更强调 SQL 可控。
- Mapper 接口本身不会执行 SQL，真正接住调用的是代理对象。
- 一级缓存和二级缓存不能只讲“提速”，还要讲作用范围和一致性边界。
- 分页问题很多时候要回到 SQL 和数据库本身判断，不要都归因到框架。
