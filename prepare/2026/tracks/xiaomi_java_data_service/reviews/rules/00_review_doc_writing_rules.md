# 面试复习文档整理规范

适用范围：

- `prepare/2026/tracks/xiaomi_java_data_service/reviews/`
- 后续 Week2 及之后的复习重点总结文档

## 1. 为什么需要这份规范

Week1 已经暴露出两个稳定问题：

- 单个周文档过大，内容容易失衡：
  - 有些模块被展开得很细
  - 有些模块只停留在大纲和提示层
- 文档在写重点时，容易只写：
  - “要抓住什么”
  - “高频会问什么”
  - “重点理解什么”

但没有继续写清楚：

- 这个重点到底怎么理解
- 这个问题到底怎么回答
- 这个机制到底在解决什么问题

所以从 Week2 开始，复习文档必须同时解决两个目标：

1. 文档结构不能再靠一个超长总文件硬撑
2. 每个重点不能只列问题，必须给出可复习的解释

进一步看，已经写过的文档还暴露出第三个更关键的问题：

- 标题虽然写成了问题
- 但正文经常没有先回答这个问题
- 而是很快退回到“这个知识点是什么、有哪些概念、有哪些分类”

这会导致一种很典型的失败写法：

- 看起来有主线
- 看起来每节都有标题问题
- 但实际阅读体验仍然是罗列知识点

这份规范后面要重点解决的，就是这个问题。

## 2. 总体原则

### 2.1 周总纲只做导航，不再承载完整知识正文

周总纲文档的职责应该是：

- 说明本周目标
- 说明本周模块划分
- 给出各专题文档入口
- 给出每天与专题的映射关系
- 给出本周复盘方式

周总纲不应该承担的事情：

- 把并发、JVM、MySQL、Redis、Kafka 全部正文塞进一个文件
- 同时承担导航、知识正文、问答提纲、排障链路四种角色

一句话要求：

- 周总纲是“导航文档”
- 专题文档才是“知识正文”

### 2.2 一大块知识点，必须单独成文

从 Week2 开始，下面这种粒度必须拆成单独文档：

- Spring 核心
- Spring MVC
- Spring Boot
- MyBatis
- Dubbo
- 系统设计

如果某个主题继续往下拆后，仍然明显过大，也要继续拆。

例如：

- `Spring 核心`
  - 可以继续拆成：
    - IOC / BeanFactory / ApplicationContext
    - Bean 生命周期
    - AOP / 动态代理

### 2.3 重点不能只列出来，必须讲清楚怎么理解

以后文档里禁止只出现下面这类提示型写法：

- “重点抓住这个点”
- “这里容易被问”
- “要重点理解”
- “会答这个问题”

除非后面紧接着补出至少一层解释：

- 它在解决什么问题
- 它的核心机制是什么
- 为什么会这么设计
- 面试时怎么讲才不空

### 2.4 专题文档必须有一条可回忆的主线

专题文档不能写成：

- 每一节都在讲自己的内容
- 单点解释都对，但彼此之间没有连接
- 读者看完某一节后，不知道它为什么会出现在这里

专题文档必须做到：

- 先给出一条总主线，说明为什么先讲 A，再讲 B，再讲 C
- 这条主线最好能压缩成 5-7 句连续问题或连续因果
- 每个大节开头都要明确：
  - 这一节在整条主线里解决什么问题
- 大节之间要有自然过渡：
  - 让读者顺着当前问题自然进入下一个问题
  - 但不要把过渡本身写成“这一节和下一节怎么接”这种独立小节标题

这条规则的目标不是让文档更“像答案模板”，而是让读者：

- 先理解
- 再记忆
- 最后才能稳定表达

如果读者看完后仍然觉得“知识点都看了，但脑子里串不起来”，说明文档还没有达标。

### 2.5 专题文档必须按“问题驱动”展开，而不是“知识点驱动”展开

这是后续最容易跑偏、也最必须被明确写死的一条规则。

专题文档的大节如果标题写成了一个问题，例如：

- 为什么同样走索引，扫描行数还是会差很多
- 事务和隔离级别在解决什么问题
- InnoDB 具体怎么兼顾一致性和并发
- 线上 SQL 真慢时到底先看什么

那么正文必须先正面回答这个问题，再展开相关知识点。

正确顺序必须是：

1. 先回答标题里的问题：
   - 先给结论
   - 说明这个问题的核心原因或核心判断
2. 再用场景或例子把问题钉住：
   - 让读者知道这个问题为什么真实存在
3. 再引出相关知识点：
   - 这些知识点为什么会出现在这里
   - 它们分别在回答这个问题中的什么部分
4. 最后再补机制、边界和表达

禁止的顺序是：

1. 标题提了一个问题
2. 正文直接开始讲定义、分类、知识点
3. 直到后面很靠后的位置，才隐约回到标题问题

这种写法虽然“内容没错”，但本质上仍然是在罗列知识点，不合格。

一句话要求：

- 知识点只能作为回答问题的材料
- 不能反过来让“问题标题”沦为“知识点罗列”的包装

### 2.6 专题文档必须讲清知识点的作用和效果，不能只讲知识点本身

后续文档里，任何知识点都不能只写：

- 它是什么
- 它有哪些术语
- 它有哪些分类

而必须继续补出下面至少两层：

- 它在当前问题里起什么作用
- 它会带来什么效果或代价

例如：

- 讲联合索引，不能只讲最左前缀
  - 必须继续讲它为什么会影响扫描范围和排序代价
- 讲 MVCC，不能只讲 undo log 和 Read View
  - 必须继续讲它为什么能减少读写阻塞、但又为什么不能替代锁
- 讲回表，不能只讲“二级索引后再查聚簇索引”
  - 必须继续讲它为什么会增加随机读和访问次数

如果读者看完后只是“知道有这个知识点”，但仍然不知道：

- 它为什么会出现
- 它在解决什么问题
- 它会带来什么效果

那这段内容就不合格。

### 2.7 光有规则还不够，必须有防跑偏的写作流程

即使规则写清楚了，内容里仍然可能出现细节问题，常见原因有三个：

1. 写作者在落笔时，默认思维仍然是“我来讲这个知识点”：
   - 而不是“我来回答这个问题”
2. 写作者知道主线存在，但一展开细节就会重新滑回熟悉的知识点罗列方式
3. 写完之后没有做“标题问题是否真的被回答”的验收检查

所以后续不能只靠“记住规则”，必须增加固定的写作流程和门禁。

### 2.8 防跑偏的最小写作流程

每个专题大节在正式展开前，必须先写出下面 4 行骨架：

1. 这一节的标题问题是什么
2. 用 3 句话先把这个问题答出来
3. 为了回答这个问题，需要引入哪几个知识点
4. 每个知识点分别在这个问题里起什么作用、带来什么效果

只有这 4 行写出来之后，才允许继续展开正文。

如果第 4 行写不出来，就说明：

- 这个知识点只是“你知道它”
- 但它还没有真正被接到标题问题里

这种情况下，不允许直接开始写知识点正文。

## 3. 专题文档的强制结构

每个专题文档至少要包含下面六层。

### 3.1 这一块在解决什么问题

要求：

- 先交代这块知识的定位
- 说明它要解决的核心问题
- 把零散知识点先串成主线
- 最好能明确它和上一块、下一块的关系
- 如果标题本身是一个问题，正文开头必须先正面回答这个问题
- 上下文衔接要体现在正文过渡里，不要单独暴露成写作提示标题

禁止：

- 一上来直接列概念名词
- 标题在提问，正文却先开始讲知识点定义

### 3.2 知识树

要求：

- 给出本专题的模块划分
- 让读者先建立整体地图

例如：

- Spring IOC
- Bean 生命周期
- AOP
- 事务

### 3.3 核心重点逐点展开

这是最关键的层次。每个重点都至少要写清楚下面四件事：

1. 是什么
2. 为什么有它 / 它解决什么问题
3. 怎么理解它的机制或流程
4. 面试里怎么讲才不空

如果这块知识本身位于某条更大的主线里，还要补清楚：

5. 它和前后知识点是怎么连起来的
6. 它在当前标题问题里起什么作用、带来什么效果或代价

如果只写到“是什么”，不算达标。

### 3.4 每个大节的推荐强制模板

如果一个大节的标题是问题句，推荐固定按下面顺序写：

1. `先直接回答这个问题`
   - 用 3-6 句先把标题问题答出来
2. `再用一个场景或例子把问题钉住`
   - 让读者知道这个问题真实发生在什么场景
3. `再回到相关知识点`
   - 解释这些知识点为什么会出现在这里
4. `再讲机制`
   - 这些知识点具体怎么工作
5. `再讲作用、效果、代价`
   - 为什么它会让系统变快、变稳，或带来什么成本
6. `最后补边界和表达`

如果缺少第 1 步，文档极容易重新滑回“知识点罗列”。

补充要求：

- 大节之间的承接要自然写进段落，不要单独写：
  - `这一节和下一节怎么接`
  - `这一节怎么收整篇主线`
- 这些属于写作者内部控制项，不是读者需要看到的文档标签

### 3.5 写作时的两条硬约束

#### 3.5.1 不能连续两段只讲知识点，不回扣标题问题

如果正文已经连续两段都在讲：

- 定义
- 分类
- 机制细节

却没有回扣下面任一信息：

- 这和标题问题有什么关系
- 它在当前问题里起什么作用
- 它带来了什么效果或代价

那么这一节大概率已经重新滑回“知识点罗列”。

这时应该暂停补内容，先回写承接句。

#### 3.5.2 每引入一个知识点，都必须回答两个问题

每引入一个知识点时，至少要继续回答：

1. 为什么这里要讲它？
2. 它在当前问题里具体起什么作用 / 带来什么效果？

如果这两个问题答不上来，就不要硬塞进正文。

### 3.4 边界、误区、常见追问

要求：

- 补出容易混淆的边界
- 补出常见误区
- 补出高频追问方向

例如：

- `volatile` 不是线程安全万能解
- 双亲委派不是绝对铁律
- Full GC 不等于 OOM

### 3.5 口头表达建议

专题文档不是答题模板文档，但至少要告诉你：

- 这一块最稳的回答顺序是什么
- 先说结论，还是先说问题背景
- 回答时最容易丢掉哪一层

要求：

- 至少给出 1 个“推荐回答顺序”
- 不要求写成长篇标准答案
- 但必须告诉读者怎么组织表达

### 3.6 自测清单

要求：

- 用 5-10 个问题检查是否真的掌握
- 这些问题要覆盖：
  - 概念
  - 机制
  - 边界
  - 误区
  - 表达

## 4. 每日复盘文档规则

每日复盘文档和专题系统复习文档不是一类东西。

专题文档的核心任务是：

- 讲清知识
- 建立主线
- 解释机制

每日复盘文档的核心任务是：

- 记录当天真实表现
- 识别当天薄弱点
- 给出第二天的修正动作

所以每日复盘不能写成：

- 又一份知识大纲
- 又一份专题正文
- 又一份标准答案合集

### 4.1 每日复盘的输入来源

每日复盘必须基于当天真实发生过的内容来写，至少来自下面两类输入：

- 模拟面试问答：
  - 当天口头回答过哪些题
  - 哪些题讲顺了
  - 哪些题讲散了
  - 哪些题根本答不上来
- 算法题练习：
  - 做了哪道题
  - 是否独立做出
  - 卡在什么地方
  - 是思路问题、代码问题，还是表达问题

如果没有基于当天真实表现，而只是把当天专题重新抄一遍，这份每日复盘就不合格。

### 4.2 每日复盘的主目标

每日复盘最重要的不是“写了多少内容”，而是回答清楚下面四件事：

1. 今天实际表现怎么样
2. 今天最弱的点是什么
3. 这些薄弱点是知识问题、表达问题，还是算法问题
4. 明天应该怎么修

换句话说：

- 每日复盘是“表现诊断文档”
- 不是“知识补课正文”

### 4.3 每日复盘必须突出薄弱点

每日复盘可以写已完成项，但重点必须放在薄弱点上。

建议优先写：

- 哪个问题会答但讲不顺
- 哪个问题知道结论但说不出因果链
- 哪个问题一追问就乱
- 哪道算法题思路对但代码细节不稳
- 哪道算法题连题型识别都不准

不要把大量篇幅花在：

- 今天学了什么
- 今天看了哪些知识点
- 今天知识点有哪些定义

这些最多只做背景说明，不应该成为主体。

### 4.4 薄弱点必须落到“现象 -> 原因 -> 修正动作”

这是每日复盘最核心的质量要求。

每个薄弱点至少要写清三件事：

1. 现象：
   - 你当时具体卡在什么地方
2. 原因：
   - 是概念不清、机制没串起来、表达没层次，还是算法模板不稳
3. 修正动作：
   - 明天准备怎么补

例如，不合格写法：

- “JMM 还不熟”

合格写法应该更像：

- 现象：
  - 能说出 JMM 是内存模型，但解释不清它怎么回答可见性和有序性
- 原因：
  - 只记了概念，没有把主内存 / 工作内存和 `happens-before` 串起来
- 修正动作：
  - 明天先重讲一遍 JMM 的抽象模型，再做 2 轮 1 分钟口头回答

### 4.5 每日复盘的强制结构

每日复盘文档至少要包含下面几层：

#### 4.5.1 今天的输入

要求写清：

- 今天复习了什么专题
- 做了哪些模拟问答
- 做了哪些算法题

作用：

- 说明今天的复盘是基于什么输入产生的

#### 4.5.2 今日实际表现总结

要求写清：

- 今天整体完成情况
- 哪些完成了
- 哪些没完成
- 今天的总体表现结论

注意：

- 这里要写“实际表现”
- 不是再写一遍知识大纲

#### 4.5.3 模拟问答复盘

要求优先写：

- 哪些问题讲顺了
- 哪些问题讲散了
- 哪些问题会答结论但不会展开
- 哪些问题需要继续追问训练

最好能体现：

- 不是单纯“知道不知道”
- 而是“能不能讲顺、能不能承接追问”

#### 4.5.4 算法复盘

要求写清：

- 做了哪道题
- 结果怎样
- 卡点在哪
- 当前最不稳的是：
  - 题型识别
  - 思路建立
  - 边界处理
  - 代码实现
  - 口头解释

#### 4.5.5 今日薄弱点 Top 3

这是每日复盘的主体。

每个薄弱点都建议按下面格式写：

- 薄弱点：
  - 是哪个知识点 / 哪类题
- 表现：
  - 今天具体暴露出了什么问题
- 原因：
  - 为什么会这样
- 修正动作：
  - 明天怎么补

#### 4.5.6 明日修正动作

要求写成可以直接执行的动作，不要写空泛结论。

例如合格写法：

- 重新口头讲 2 遍“JMM 如何回答可见性 / 有序性”
- 把 `LongAdder` 和 `AtomicLong` 的选型边界重新整理成 1 分钟版本
- 重做今天没独立做出的算法题，并补一遍边界条件

不合格写法：

- 继续复习 JMM
- 再看看算法

### 4.6 每日复盘不应该怎么写

下面这些写法要避免：

- 只写今天学了什么，不写今天表现怎么样
- 只写知识点，不写答题时卡在哪里
- 只写“已完成 / 未完成”，不写为什么没完成
- 只写“这个点不熟”，不写怎么补
- 把每日复盘写成另一份专题知识正文

### 4.7 每日复盘的最低质量要求

一份合格的每日复盘，至少要满足：

- 能看出当天真实做了哪些问答和算法题
- 能看出当天最弱的 2-3 个点是什么
- 能区分知识问题、表达问题和算法问题
- 能看出明天具体准备怎么修

如果看完整份文档后，只能知道“今天复习了很多内容”，却看不出“今天到底哪里最弱”，那这份每日复盘不合格。

## 5. 写作时的最低质量要求

### 5.1 禁止“只讲问题，不讲答案”

例如下面这种写法不合格：

- “深挖点：TLAB、对象头、指针碰撞”

合格写法应该至少是：

- TLAB 在解决什么问题
- 指针碰撞和空闲列表分别适合什么场景

### 5.2 禁止“标题在提问，正文在罗列知识点”

这是后续专题文档最需要重点检查的一条。

不合格写法类似：

- 标题：
  - 为什么同样走索引，扫描行数还是会差很多
- 正文：
  - 联合索引是什么
  - 最左前缀是什么
  - 联合索引怎么设计

这类内容单看并不一定错，但它没有先回答标题问题，所以仍然不合格。

合格写法应该更像：

- 先回答：
  - 因为走索引不等于扫描范围足够小
  - 扫描量取决于前缀利用程度、等值 / 范围条件、排序分页是否顺着索引完成
- 再举例：
  - 两条都走索引的 SQL，为什么扫描行数差很多
- 最后再引出：
  - 联合索引
  - 最左前缀
  - 范围条件
  - 索引下推

### 5.3 写完后必须做“问题-内容一致性检查”

每个大节写完后，至少要自查下面 4 个问题：

1. 如果只看标题和第一屏正文，读者能不能立刻知道这节的答案是什么？
2. 这节里的每个知识点，是不是都在服务标题问题？
3. 如果删掉标题，正文是不是会退化成一段普通知识点罗列？
4. 读完这节后，读者能不能说出这些知识点分别起了什么作用、带来了什么效果？

只要其中有 1 个问题答不上来，这节就应该重写，而不是继续补更多知识点。

### 5.4 写完后必须做“删标签检查”

这是一个非常直接的防跑偏检查。

做法是：

- 暂时不看这一节里出现的术语名词
- 只看标题、每段首句和例子

检查自己是否仍然能看出：

- 这一节在回答什么问题
- 核心答案是什么
- 每个知识点分别在起什么作用

如果删掉术语标签后，正文只剩下一堆零散说明，说明这节仍然是“靠知识点撑着”，不是“靠问题主线撑着”。

### 5.5 写完后必须做“反向提问检查”

这是最终门禁。

假设读者看完这一节后追问下面任一问题：

- 所以你这节到底想回答什么？
- 你为什么在这里讲这个知识点？
- 这个知识点让系统发生了什么变化？
- 它解决了什么问题，又带来了什么代价？

如果正文里找不到直接答案，就说明这一节虽然写了很多字，但仍然没有写到位。

后续默认要求：

- 先修“问题有没有被回答”
- 再修“知识点讲得是否足够深入”

不能反过来先补更多知识点细节。
- 对象头包含什么运行时信息

### 4.2 禁止“只列提纲，不交代机制”

例如下面这种写法不合格：

- “JMM 主要回答三件事：可见性、有序性、原子性”

合格写法应该继续补出：

- JMM 是通过什么抽象来回答的
- `happens-before` 在里面扮演什么角色

### 4.3 禁止“只给结论，不给因果链”

例如下面这种写法不合格：

- “G1 更适合大堆”

合格写法至少要补：

- 为什么大堆场景下它更容易被提到
- 它靠什么机制支撑这种特点

### 4.4 如果一个重点不能用 3-6 行讲清，就说明还没写透

这条不是让文档冗长，而是提醒：

- 每个核心重点至少要写到“可被理解”
- 不能停在词语标签层

## 6. 文档拆分规则

### 5.1 周文档拆分规则

每周至少采用下面这种结构：

- `00_weekX_review_hub.md`
  - 周导航文档
- `01_xxx_system_review.md`
  - 专题 1
- `02_xxx_system_review.md`
  - 专题 2
- `03_xxx_system_review.md`
  - 专题 3

### 5.2 单文档超长时必须拆

出现下面任一情况，就必须拆文档：

- 文档超过 800 行
- 一个文档同时承载超过 3 个大主题
- 同一文档里已经出现明显的“前面很细、后面很粗”

补充说明：

- `600-800` 行之间如果仍然是单一主题，且结构稳定、没有明显失衡，可以保留为单文档
- 但只要已经出现“前细后粗”或多个主题混装，即使没到阈值也应该拆

### 5.3 允许“总纲短、专题长”，不允许“总纲长、专题缺位”

这是后续所有复习文档的默认策略。

## 7. Week2 开始的推荐结构

- `reviews/week2/00_week2_review_hub.md`
- `reviews/week2/01_spring_core_system_review.md`
- `reviews/week2/02_spring_aop_lifecycle_review.md`

- `reviews/week2/04_mybatis_review.md`
- `reviews/week2/05_dubbo_review.md`

如果后续发现其中某一块仍然过大，再继续拆。

## 8. 生成前自检清单

每次产出复习文档前，至少自查下面这些问题：

- 这份文档是导航，还是正文？角色是否混乱？
- 如果是每日复盘，是否真的基于当天实际表现，而不是重写知识大纲？
- 这个主题是否已经大到应该单独成文？
- 每个重点是否都写清了：
  - 是什么
  - 为什么
  - 怎么理解
  - 怎么讲
- 是否出现了“重点抓住”“重点理解”“高频会问”但没有展开解释的地方？
- 是否出现了“前面很细、后面越来越粗”的结构失衡？
- 是否明确写出了当天最弱的点，以及对应修正动作？
- 自测问题是否真的能检查理解，而不只是检查名词记忆？

## 9. 一句话记忆

从 Week2 开始，复习文档默认遵循下面这条规则：

- 周总纲只做导航
- 大块知识单独成文
- 每个重点必须同时写清“是什么、为什么、怎么理解、怎么讲”
- 每日复盘必须基于当天真实表现，并突出薄弱点和修正动作
