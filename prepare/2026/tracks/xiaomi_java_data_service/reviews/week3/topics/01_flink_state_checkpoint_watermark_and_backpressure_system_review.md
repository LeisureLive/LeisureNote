## 1. 这一块在回答什么问题

Flink 这块真正要回答的，不是“它是不是一个流处理框架”，而是下面这条连续问题：

1. 数据流是持续不断进入的，系统怎么把一条无界数据流稳定地拆成可执行的计算链路。
2. 业务计算经常不是看完一条数据就结束，系统怎么记住中间上下文。
3. 任务运行几小时、几天甚至更久，进程挂掉后怎么恢复到一致状态，而不是从头重算或产生更多重复结果。
4. 流里的事件经常乱序到达，系统怎么在“尽快出结果”和“尽量保证正确”之间取得平衡。
5. 一条链路里只要某个环节处理不过来，为什么压力会沿着整条链反推回去。

所以这块的主线不是术语列表，而是：

- Flink 先把无界流组织成一条长期运行的算子链。
- 算子要持续计算，就必须管理状态。
- 状态要可靠，就必须有一致性的快照机制。
- 结果要尽量按业务时间正确输出，就必须处理乱序事件。
- 链路要稳定运行，就必须识别和控制背压。

如果这条主线抓住了，`state`、`checkpoint`、`watermark`、`backpressure` 这些词就不会再是分散名词。

## 2. 主线

1. 先看运行模型：Flink 不是批次执行完就退出，而是把作业编译成长期运行的数据流图。
2. 再看状态：流计算要持续记住每个 key、每个窗口、每段聚合的中间结果。
3. 再看容错：状态一旦重要，系统就必须定期拍出一致性快照，故障后才能回到一个可信位置。
4. 再看时间语义：流里数据到达顺序不可靠，系统不能只看机器收到的时间。
5. 最后看背压：当下游跟不上时，问题不会只停在一个节点，而会沿链路向上游扩散。

## 3. 核心问题展开

### 3.1 Flink 为什么适合长期运行的实时处理，而不是只把消息一条条消费掉

这一节在回答什么问题：

- Flink 的运行模型到底解决了什么问题。
- 为什么它不只是“Kafka 消费后写一段业务逻辑”这么简单。

这个问题是怎么解决的：

Flink 适合长期运行的实时处理，是因为它把计算任务组织成一张持续运行的数据流图，而不是把每批数据临时拉起来处理一次。作业提交后，Flink 会把逻辑上的 source、transform、sink 编译成物理执行图，再分发给各个并行子任务长期运行。这样系统处理的不是“一批数据”，而是“持续流入的数据”。

这件事的关键不在于“它能消费消息”，而在于它把实时处理里的几个长期问题统一收口了：

1. 数据是持续进入的，任务必须长期驻留，而不是反复启动。
2. 计算链路通常由多个算子组成，系统需要负责并行切分、调度和数据交换。
3. 上下游吞吐不一致时，系统需要能识别链路压力，而不是让业务代码自己猜。
4. 任务异常后，系统要知道怎么回到之前的正确进度。

放到运行时看，Flink 一般会把作业交给 JobManager 和 TaskManager 这两类角色协作完成。JobManager 更偏控制面，负责作业图、调度、checkpoint 协调等；TaskManager 更偏执行面，真正运行并行子任务，处理数据和状态。这个划分的价值是把“怎么调度、怎么恢复”和“怎么执行数据计算”拆开，不让每个业务算子自己承担整套运行时职责。

再往下看，Flink 并不是每条数据都完整走一遍独立作业，而是让每个并行子任务持续处理属于自己的数据分片。这样做的收益是：

- 避免频繁起任务带来的调度成本。
- 能把状态留在任务本地附近，减少每次计算都从外部系统回查。
- 更容易在整条链路上做统一的容错和流控。

代价也很明确：

- 这类任务不是“跑完即走”，而是持续占用资源。
- 一旦状态变大、并行度配置不合理或下游依赖变慢，问题会被长期放大。
- 运维关注点会从“任务成功结束没有”变成“任务是否稳定地长期运行”。

所以 Flink 的核心价值不是“流处理”这四个字，而是它提供了一套长期运行、可状态化、可恢复、可观测的实时计算运行时。

### 3.2 为什么状态是 Flink 里最核心的一层，它到底在保存什么

这一节在回答什么问题：

- Flink 为什么总在讲状态。
- 状态在实时计算里到底承担什么作用。

这个问题是怎么解决的：

状态之所以是 Flink 里最核心的一层，是因为大多数有业务价值的实时计算都不是无状态映射。只要你做过聚合、去重、窗口统计、会话分析、关联、去重补偿、规则命中，这些逻辑都在依赖“之前处理过什么”。Flink 把这些“之前发生过什么”沉淀成状态，让计算不需要每次都回源外部存储。

状态常见地可以分成两类：

1. `Keyed State`
   - 针对某个 key 保存局部上下文。
   - 例如按用户、设备、订单维度累加、去重、记最后一次行为。
2. `Operator State`
   - 针对算子实例保存状态。
   - 更常见在 source 进度、分片分配、某些算子级别缓存里使用。

为什么要这么区分？因为它们解决的是两种不同的问题。`Keyed State` 的目标是把同一个 key 的数据固定落到同一个并行子任务上，这样这个 key 的上下文就能稳定累积。`Operator State` 则更像某个算子实例自己的运行现场，它不一定围绕业务 key。

这里最容易空掉的一点是：状态不是“额外附带的缓存”，而是计算正确性的一部分。比如窗口聚合如果没有状态，就无法记住当前窗口已经累计了多少；去重如果没有状态，就不知道某个 id 是否已经见过；流 join 如果没有状态，就留不住等待匹配的数据。

Flink 用状态带来的收益主要有三类：

1. 把大量中间上下文留在计算链路里，避免频繁访问外部存储。
2. 让每个并行子任务只维护自己负责的数据局部，提高并行处理效率。
3. 让后续的 checkpoint 有东西可拍，容错不再只是“重启进程”，而是“恢复计算现场”。

但状态也直接带来了成本：

- 状态越大，checkpoint 成本越高。
- 热 key 会把部分状态压得特别重，造成局部热点。
- 状态设计不合理会导致 TTL、清理策略、序列化成本、恢复时长都失控。

所以状态不是可选配件，而是 Flink 能做复杂实时计算的前提；同时，它也是实时链路里最常见的性能和稳定性压力来源之一。

### 3.3 状态后端、TTL 和恢复成本为什么会直接决定链路稳定性

这一节在回答什么问题：

- 状态除了“逻辑上存在”，在运行时到底放在哪里。
- 为什么状态位置、大小和生命周期会直接影响吞吐、checkpoint 和恢复速度。

这个问题是怎么解决的：

只知道“Flink 有状态”还不够，因为状态一旦进入真实运行环境，系统还要继续回答三个问题：

1. 状态放在哪里。
2. 状态保留多久。
3. 故障恢复时，要花多大代价把它拉回来。

这里最容易混淆的一点是：运行时状态放在哪里，和 checkpoint 最终落到哪里，不完全是一回事。更准确地说，要拆成两层看：

1. `State Backend`
   - 更偏“运行中的状态怎么组织、主要放哪里、读写代价怎样”。
   - 它影响的是任务平时读写状态时的延迟、可承载规模和本地资源占用。
2. `Checkpoint Storage`
   - 更偏“做 checkpoint 时，一致性快照最终写到哪里”。
   - 它影响的是快照落盘位置、远端存储体积和故障恢复时的回放来源。

所以很多人会把“状态后端”和“状态持久化位置”混成一个概念，但面试里最好主动拆开讲。你可以把它理解成：

- `State Backend` 决定平时怎么用状态。
- `Checkpoint Storage` 决定拍快照时把状态存到哪。

这也是为什么状态后端会成为运行时核心配置的一部分。更本质地看，它要解决的不是“有没有地方存状态”，而是要在下面几件事之间做平衡：

- 读写延迟
- 可承载的状态规模
- checkpoint 成本
- 恢复时长

如果把这件事压到工程层，常见取舍一般是：

1. 运行时状态更偏本地内存
   - 访问更快。
   - 适合状态不大、低延迟要求更高的场景。
   - 代价是可承载规模有限，状态一大就会把内存和 checkpoint 压得很重。
2. 运行时状态更偏本地磁盘或嵌入式 KV 存储
   - 单次访问可能更慢。
   - 但更能承受大状态，checkpoint 也更容易和增量快照配合。
   - 代价是序列化、反序列化、磁盘 IO 和恢复链路会更复杂。

放到常见实现上，可以记成下面两组典型搭配：

1. `HashMapStateBackend` + 文件系统 checkpoint
   - 运行时状态主要在 TaskManager 进程内存里。
   - checkpoint 时，再把快照写到文件系统、HDFS 或对象存储。
   - 这类方案访问快，但状态规模上来后，内存压力和 checkpoint 压力会明显上升。
2. `EmbeddedRocksDBStateBackend` + 文件系统 checkpoint
   - 运行时状态主要在本地 RocksDB。
   - checkpoint 时，把快照持久化到远端存储。
   - 它更能承受大状态，但平时读写、序列化和恢复链路一般更重。

所以如果你看到“状态存到 filesystem”这种说法，要先追问它说的是哪一层：

- 如果说的是旧概念里的 `FsStateBackend`，更接近“运行时状态偏内存，checkpoint 写到文件系统”。
- 如果说的是新版配置里的 `FileSystemCheckpointStorage`，那更明确是在说“checkpoint 落盘位置”，不等于运行时状态一直在文件系统里。

这也是为什么很多实时任务刚开始用得很轻松，数据量一上来问题就会集中爆发。因为状态不只是“逻辑变量”，它最终会变成：

- 内存占用
- 本地磁盘占用
- 远端 checkpoint 存储体积
- 恢复时需要重新加载的数据量

TTL 也是同一条主线上的问题。状态如果只增不减，系统迟早会被自己保存下来的历史上下文拖慢。TTL 的价值不是“顺手清理旧数据”，而是主动控制状态生命周期，避免已经没有业务价值的数据继续消耗下面这些成本：

- checkpoint 体积
- 恢复时间
- 热 key 状态膨胀
- 序列化和反序列化开销

但 TTL 也不是随便配一个时间就结束。TTL 配得太短，会把业务上仍然有效的上下文提前清掉，结果正确性受影响；TTL 配得太长，又会把无效状态继续保留下来，最终把性能问题推迟成更大的稳定性问题。

所以状态设计真正要看的，不只是“这段逻辑要不要状态”，还包括：

- 这个状态按 key 还是按算子维度维护。
- 这个状态会增长到什么规模。
- 这个状态多久还能产生业务价值。
- checkpoint 和恢复是否还承受得住它的体积。

这也是为什么真实项目里，状态后端、TTL、序列化方式和 key 设计，经常会直接决定一条 Flink 链路到底能不能稳定跑长时间。

### 3.4 Checkpoint、Savepoint 和 Exactly-Once 为什么总被放在一起讲

这一节在回答什么问题：

- Flink 是怎么在失败后恢复一致状态的。
- 为什么 `checkpoint`、`savepoint`、`exactly-once` 这几个词总会一起出现。

这个问题是怎么解决的：

状态一旦重要，系统就必须回答一个问题：任务挂了之后，到底从哪里恢复，才能既不把状态丢掉，也不把结果重复放大。Flink 的答案是定期做一致性快照，也就是 `checkpoint`。它不是简单把某个节点的内存复制出来，而是要在整条数据流图上切出一个一致的状态截面。

核心机制可以压成下面这条链：

1. `checkpoint coordinator` 周期性触发快照。
2. `barrier` 沿着数据流从 source 往下游传播。
3. 各个算子在同一个 checkpoint 边界上把当前状态持久化。
4. source 的消费进度、算子状态、必要时 sink 的提交边界一起被纳入一致性视图。
5. 故障恢复时，任务回到最近一次成功 checkpoint，再从那个位置继续处理。

这里的关键不是“有快照”这么简单，而是“快照必须一致”。如果 source 进度已经向前推进，但某个关键算子的状态没有对应保存下来，恢复后就会出现状态和输入位置不匹配。Flink 用 barrier 协调，就是为了让整条链对“我已经处理到了哪里”形成统一截面。

`savepoint` 和 `checkpoint` 会一起出现，是因为它们都在保存状态，但用途不同：

- `checkpoint`
  - 更偏运行时自动容错。
  - 周期性触发，目标是故障恢复。
- `savepoint`
  - 更偏人工控制的状态留存。
  - 常用于版本升级、作业迁移、手工切换。

`exactly-once` 则是在更上层回答“每条输入数据对最终可见结果的影响，是否只生效一次”。这里最容易误解的地方是：它不是在承诺一条数据只会被消费一次、计算一次、发送一次，而是在承诺即使发生重放、重算、重发，最终对下游真正可见并生效的结果仍然只有一次。

放到 `KafkaSource + KafkaSink` 这种典型场景里，更容易看清它到底在保护什么。真正需要一起对齐的不是单个算子里的某个变量，而是下面三样东西：

1. source 当前已经读到哪里，也就是消费进度。
2. operator 当前状态长什么样，也就是计算现场。
3. sink 哪一批输出已经可以正式对外生效，也就是提交边界。

如果只保存前两样，不控制第三样，就会出现一种典型错误：任务恢复后内部状态回去了，但外部结果早就已经写出去了，于是重放后对下游看起来就是重复结果。所以端到端 `exactly-once` 的难点不在 source，而在 sink 能不能把“结果什么时候正式生效”也纳入一致恢复语义。

以 Kafka 事务性 sink 为例，可以把主线压成下面两种常见情况：

1. 如果任务失败发生在某次 checkpoint 成功之前：
   - `KafkaSource` 会回退到最近一次成功 checkpoint 的消费位置。
   - 这段时间里 `KafkaSink` 虽然可能已经把结果写进了事务，但事务还没有随 checkpoint 一起确认。
   - 恢复后这批数据会被重新处理、重新发送，但旧事务不会成为成功提交事务；最终对 `read_committed` 下游可见的，只会是恢复后那笔真正提交成功的事务结果。
2. 如果 checkpoint 已经成功，但提交事务时发生异常：
   - 系统恢复时不能简单把这批数据当成“彻底失败然后重做”，而是要基于 checkpoint 中保存的 pending transaction 信息，对这笔事务继续做收尾。
   - 如果 broker 端其实已经提交成功，只是客户端没拿到确认，那么恢复逻辑要识别或等待这笔事务完成，而不是再让同一批结果以新事务重复生效。
   - 如果事务还没提交成功，则继续补提交或安全结束；真正危险的是恢复时间过长，导致 Kafka 事务超时失效。

所以这里可以把结论记成一句话：

- `exactly-once` 保证的不是“处理过程没有重复”，而是“最终提交给下游的业务效果没有重复”。

这也是它的边界：

- 只靠 Flink 自己，不能替一个不支持幂等或事务的外部 sink 自动兜底。
- `read_committed` 这类事务隔离语义，是 Kafka 端对下游“只看到已成功提交结果”的关键前提。
- checkpoint 间隔不是越短越好，太短会抬高运行时开销；但太长又会拉大恢复重放窗口和结果可见延迟。
- Kafka 事务超时必须大于 checkpoint 完成时间和故障恢复时间，否则 pending transaction 可能在恢复前就已经失效。
- `savepoint` 也不是备份系统替代品，它更像“可控的状态切换点”。

所以这几个词之所以总被放在一起，是因为它们共同回答的是“状态化实时任务如何在失败后继续保持正确性”。

### 3.5 Barrier 对齐、Checkpoint 时长和背压为什么会互相影响

这一节在回答什么问题：

- 为什么有些 Flink 任务平时还能跑，但 checkpoint 会越来越慢、越来越容易超时。
- 为什么背压和 checkpoint 往往不是两个独立问题。

这个问题是怎么解决的：

只理解“checkpoint 要做一致快照”还不够，因为真实系统里最容易出问题的往往不是概念，而是快照推进速度。Flink 要做一致快照，就意味着同一个 checkpoint 边界必须在整条链路上形成一致截面。这里 barrier 不是一个装饰性标记，它是在告诉算子：

- 当前这个 checkpoint 的边界已经推进到这里。
- 这个边界之前的数据和状态，需要一起纳入同一份快照。

问题在于，数据流图里不同输入分支的处理速度经常不一致。某个输入已经把 barrier 推过来了，另一个输入还卡在前面，算子就没法马上在这个 checkpoint 上完成一致切面。这就是 barrier 对齐要解决的问题：等待同一个 checkpoint 的各路输入在逻辑上对齐。

一旦链路已经出现背压，这件事就会被明显放大：

1. 下游处理慢，数据堆积。
2. barrier 也跟着数据流推进，所以它会被堵在拥塞路径上。
3. barrier 到不了位，checkpoint 完成时间就被拉长。
4. checkpoint 越慢，状态持久化和资源占用时间越长。
5. 如果状态又很大，链路就会进一步变慢，形成相互放大。

所以很多线上现象其实是一条链：

- 慢 sink 或热 key
- 引发背压
- barrier 推进变慢
- checkpoint 时长上升甚至超时
- 恢复窗口变大、稳定性继续下降

这也是为什么 checkpoint 时长不能只理解成“存储慢”：

- 有可能是远端状态存储慢。
- 有可能是状态本身体积太大。
- 有可能是 barrier 对齐被背压拖住了。
- 有可能是某个输入分支明显更慢，导致多输入算子一直等不齐。

`unaligned checkpoint` 之所以会被拿出来单独讲，就是因为它在解决这个边界问题。正常对齐式 checkpoint 追求的是“边界对齐后再拍快照”，而 unaligned checkpoint 更像是在高背压场景下减少等待对齐的时间，把一部分通道中的在途数据也纳入快照，优先保证 checkpoint 能更快完成。它的收益是重背压时更不容易因为长时间对齐而超时，代价则是快照体积和恢复复杂度可能上升。

所以 checkpoint、barrier、背压其实不是三件分离的小事，而是在共同回答一个问题：这条长期运行的实时链路，能不能在吞吐波动和局部拥塞下，仍然维持可恢复的一致性。

### 3.6 Event Time、Watermark、Window 为什么总是成组出现

这一节在回答什么问题：

- Flink 为什么不能只按机器收到数据的时间来算。
- `event time`、`watermark`、`window` 这几个概念分别在解决什么问题。

这个问题是怎么解决的：

这组概念真正先要回答的，不是术语定义，而是一个工程选择：这条链路到底应该按“事件真实发生的时间”算，还是按“系统处理到它的时间”算。

如果业务只关心机器当前处理到了哪里，例如某些内部监控、粗粒度告警、对时间口径不敏感的近实时指标，那么按 `processing time` 算会更直接。它的好处是链路简单、延迟低，不需要额外处理 watermark 和迟到数据；代价是结果会受到网络抖动、消息堆积、补数回放的影响，前后两次跑出来的统计口径可能并不一致。

但很多实时业务并不接受这种“谁先到就按谁算”的口径。行为日志、订单漏斗、实时看板、会话分析这类场景，真正关心的是事件在业务上什么时候发生，而不是 TaskManager 在几点几分收到它。流里的数据经常乱序到达：网络抖动、上游批量发送、日志采集延迟、跨机房传输，都会导致“先发生的事件后到达，后发生的事件先到达”。如果系统只看处理时刻，统计结果就会被链路延迟污染。于是 Flink 引入 `event time`：尽量按事件真正发生的业务时间来算。

所以这里可以先把两种时间语义的使用边界压成一句话：

- `event time`
  - 更适合结果必须按业务事实时间定义，且要容忍乱序、延迟、回放的场景。
- `processing time`
  - 更适合业务只关心当前处理进度，优先追求简单和低延迟的场景。

一旦选择了 `event time`，系统又会立刻遇到第二个问题：什么时候可以认为某个时间范围的数据已经差不多到齐，可以开始算结果了。`window` 和 `watermark` 就是在继续回答这个问题，但它们分工不同。

1. `window`
   - 负责定义“哪些事件属于同一段计算范围”。
   - 它的本质不是简单攒一批数据，而是把无界流切成有限的计算边界，方便做聚合、统计、触发输出。
   - 例如滚动窗口是在回答“每 5 分钟各算一次”，滑动窗口是在回答“每 1 分钟看最近 10 分钟”，会话窗口是在回答“用户多长时间没动作就视为一次会话结束”。
2. `watermark`
   - 负责定义“事件时间已经推进到哪里了”。
   - 它不是说“这个时间点之前的数据绝对不会再来”，而是系统当前对数据完整度的一个进度判断。
   - 更口语一点看，它是在告诉窗口：你现在大致可以关到哪里、出到哪里。

所以这条链最好按下面顺序记：

1. `event time`
   - 决定按什么时间定义业务事实。
2. `window`
   - 决定按什么规则把无界流切成有限范围。
3. `watermark`
   - 决定系统认为什么时候可以推动这些范围出结果。

watermark 真正难的地方，不在于知道它是“进度线”，而在于它一定带着取舍。最常见的策略是：系统观察到一个越来越大的最大事件时间，然后按“最大事件时间减去一段可容忍乱序时间”去推进 watermark。这里的那段容忍范围，就是工程上的关键权衡。

1. watermark 更保守
   - 意味着系统愿意等更久，容忍更大的乱序和延迟。
   - 收益是迟到数据更少，结果更完整。
   - 代价是窗口触发更晚、状态保留更久、延迟和 checkpoint 压力通常更高。
2. watermark 更激进
   - 意味着系统只愿意等更短时间，希望尽快出结果。
   - 收益是结果更早输出、端到端延迟更低。
   - 代价是更多正常晚到的数据会被判成 late data，结果完整性下降，后续可能需要补算、侧输出或修正结果。

所以“保守”与“激进”并不是主观评价，而是在回答同一个问题：这条链路更优先要时效性，还是更优先要完整性。

把三者放到一起，实时任务里真正协调的是下面三件互相拉扯的事：

- 结果尽快出来。
- 结果尽量按业务时间正确。
- 状态和等待成本不能无限膨胀。

只要把这三件事放在一起看，为什么实时任务里总会同时出现 `event time`、`window`、`watermark`，以及为什么面试里总会继续追问“什么时候用 event time、什么时候用 processing time、watermark 为什么会过保守或过激进”，就自然了。

### 3.7 迟到数据、Allowed Lateness 和 Idle Source 为什么也必须一起理解

这一节在回答什么问题：

- 为什么已经有了 watermark，窗口计算里仍然还会不断遇到迟到数据问题。
- 为什么有些任务明明没有报错，但窗口结果就是迟迟不出。

这个问题是怎么解决的：

watermark 解决的是“系统如何推进事件时间”，但它并没有消灭乱序和延迟本身。只要上游仍然可能晚到，就一定会遇到一个现实问题：某条数据到来时，它所属的窗口按当前 watermark 判断已经结束了，这条数据就成了 late data。

这时系统不能只回答“迟到了”，还要继续回答：

1. 迟到数据是直接丢掉。
2. 迟到后是否还允许有限度地修正窗口结果。
3. 如果不想混进主结果流，是否要把它单独侧输出给其他链路处理。

`allowed lateness` 解决的就是第二个问题：窗口在主触发之后，是否还愿意为一段有限迟到时间继续保留状态，等待迟到数据补进来。它带来的收益是结果更完整，尤其适合延迟分布比较稳定、但确实存在一定晚到的业务场景；代价则是窗口状态保留更久，状态体积、清理时机和结果重复修正成本都会上升。

side output 解决的则是第三个问题：当某些迟到数据已经不适合再回写主结果流时，系统至少还能把它们分流出来，留给补数、审计或离线校正链路处理，而不是直接在主链里悄悄丢掉。

除了迟到数据，还有一个非常容易被忽略、但线上非常常见的问题：idle source 或空闲分区。如果 watermark 是按多个输入分区的最小进度推进，而其中某个分区长时间没有新数据，这个分区的 watermark 可能就停在原地。结果是：

- 其他分区的数据其实已经到了更晚的业务时间。
- 但全局 watermark 仍然被最慢、最空闲的分区卡住。
- 窗口迟迟无法触发输出，看起来像“任务没问题，但结果一直不出来”。

所以 idle source 配置的价值不是性能优化，而是告诉系统：某个输入分区当前暂时没有活跃数据，不应该继续拿它的旧 watermark 卡住全局事件时间推进。

把这几个点放到一起看，时间语义这一块真正要解决的是三层问题：

1. 按什么时间理解事件。
2. 系统如何判断一个时间点之前的数据大体到齐。
3. 对那些仍然晚到、缺席或者暂时空闲的输入，到底怎么处理边界。

只有把迟到数据、allowed lateness、side output、idle source 这几个点也接上，时间语义这条线才算真正落到工程层。

把这套用法压成一个最小案例，可以这样理解：假设要统计“每个商品每分钟点击数”，事件里自带业务时间 `eventTime`。这类统计如果按机器处理时间算，结果会被采集延迟、网络抖动、Kafka 积压污染，所以更适合用 `event time`。窗口用 1 分钟滚动窗口，是在定义“哪些事件属于同一次分钟统计”；watermark 则是在定义“系统愿意为乱序最多等多久”。

例如：

- 窗口：1 分钟滚动窗口。
- watermark 策略：最多容忍 10 秒乱序。
- `allowed lateness`：额外接受 30 秒迟到数据。
- 超过这 30 秒的超晚数据：走 side output，交给补数或审计链路。

这套配置在工程上对应的是下面这条判断链：

1. 事件先按 `eventTime` 进入 `[10:00:00, 10:01:00)` 这样的窗口。
2. 系统持续观察当前看到的最大事件时间，并按“最大事件时间减去 10 秒”推进 watermark。
3. 当 watermark 推进到 `10:01:00` 之后，窗口会第一次触发输出主结果。
4. 如果这时又来了一条属于 `[10:00:00, 10:01:00)` 的数据，它已经是 late data。
5. 但只要当前 watermark 还没超过 `10:01:30`，这条数据仍然可以进入窗口，触发一次修正结果。
6. 如果当前 watermark 已经超过 `10:01:30`，这条数据就不再进入主窗口，而是进 side output。

把这个例子继续压缩，watermark 的作用其实就是一句话：

- 它不是保证“这个时间点之前绝对没有数据再来”，而是在说“系统现在愿意按这个时间进度先出结果”。

对应到取舍上：

- watermark 更保守
  - 结果出得更晚，但迟到数据更少，结果更完整。
- watermark 更激进
  - 结果出得更快，但更多正常晚到数据会变成 late data。

这类窗口结果还有一个很容易在面试里漏讲的边界：只要配置了 `allowed lateness`，同一个窗口就可能不只输出一次。第一次是主触发结果，后面迟到但仍在容忍范围内的数据进来后，还可能继续触发修正结果。所以下游如果要接这类窗口结果，通常不能假设“一窗只出一条最终结果”，而要考虑幂等更新、覆盖写，或者明确接受多次修正。

如果 source 存在多个分区，还要再补一个运行时判断：某个分区长时间没数据，不一定代表整体时间没推进，有可能只是 idle source 把全局 watermark 卡住了。这时要结合空闲分区识别一起看，否则很容易误判成“窗口策略有问题”。

### 3.8 背压为什么会出现，它暴露的到底是什么问题

这一节在回答什么问题：

- Flink 链路里的背压是怎么形成的。
- 背压暴露的是吞吐问题、状态问题，还是下游依赖问题。

这个问题是怎么解决的：

背压本质上不是一个独立故障，而是“下游处理不过来，导致上游也被迫慢下来”的连锁现象。Flink 是一条连续的数据流链路，只要某个环节处理速度明显低于上游产出速度，下游输入缓冲区会逐渐变满，上游写不进去，就只能阻塞或降速，这种压力会沿链路向前传播。

所以背压不是偶然现象，而是流系统的自然反馈机制。它至少可能来自下面几类原因：

1. `sink` 太慢
   - 例如外部数据库、消息系统、存储系统写入跟不上。
2. 某个算子计算太重
   - 例如复杂序列化、大状态计算、窗口聚合过重。
3. 数据分布不均
   - 某些 key 特别热，导致部分并行子任务成为瓶颈。
4. 资源配置不合理
   - 并行度太低、slot 不足、网络或磁盘瓶颈。
5. checkpoint 或状态管理成本过高
   - 大状态、频繁快照、后端存储慢都会拖慢处理。

背压的价值在于它不是简单“报错”，而是直接暴露系统在哪个环节跟不上吞吐。也正因为如此，处理背压不能只靠“加机器”或“调并行度”，而要先分清瓶颈到底在哪里：

- 如果是 sink 慢，重点是异步写、批量写、限流和下游承载能力。
- 如果是热 key，重点是 key 打散、聚合重构、链路拆分。
- 如果是状态太大，重点是状态设计、TTL、序列化和 checkpoint 策略。
- 如果是 CPU 重算，重点是算子逻辑、对象创建、序列化、数据格式。

背压能带来的一个正向价值是：系统不会在吞吐超过极限时悄悄丢数据，而是通过流控把压力显性化。但它也说明你的链路已经失衡，继续忽视就会演化成消费延迟、状态膨胀、checkpoint 超时甚至任务失败。

所以背压真正暴露的不是“Flink 有问题”，而是整条链路里存在吞吐失衡点，必须回到算子、状态、数据分布和外部依赖去定位。

## 4. 边界与误区

- `checkpoint` 不是普通备份；它的关键是整条流图的一致性截面，而不是单机状态转储。
- `savepoint` 不是定时容错主机制；它更偏人工控制的升级和迁移节点。
- `exactly-once` 不是只要开了 checkpoint 就天然成立；source、state、sink 三端都要配合。
- 状态不是“算子里顺手放点缓存”；它最终会转化成内存、磁盘、checkpoint 体积和恢复时长。
- 状态 TTL 也不是越长越稳；TTL 太长会拖大状态和快照成本，TTL 太短又可能提前破坏业务上下文。
- `watermark` 不是“绝对不会再有更早数据”的保证，它只是系统推进事件时间的判断依据。
- `allowed lateness` 不是免费修正机制；它用更长的状态保留时间换取更高结果完整性。
- 多输入链路里某个分区长时间空闲，可能会把全局 watermark 卡住，导致窗口迟迟不触发。
- 背压不是纯坏事，它首先是一种流控反馈；真正要找的是链路失衡点。
- checkpoint 变慢不一定只是存储慢，也可能是 barrier 对齐被背压、热 key 或慢分支拖住了。
- 并行度调大也不是万能解，如果瓶颈在热 key、慢 sink 或大状态，盲目加并行度会把问题换个位置继续放大。
