## 1. 这一块在回答什么问题

实时查询系统这题真正要弄懂的，不是“怎么把数据查出来”，而是下面这条连续问题：

1. 为什么业务方会需要一套独立的实时查询系统，而不是直接查事务库、日志链路或离线数仓。
2. 查询系统的复杂度为什么不只在“提供一个查询接口”，而在查询模型、数据组织、索引、缓存、时延和一致性会互相牵制。
3. 为什么一旦查询场景从点查走向范围查、筛选、排序、聚合、多维分析，数据库和引擎选型就会直接进入主线。
4. 为什么热点查询、长尾查询、历史数据、低延迟要求和高并发要求，会逼系统做缓存、预计算、冷热分层和多引擎协同。
5. 当量级上涨、热点变重、重查询变多、多租户并发增加时，瓶颈会先暴露在哪里，系统应该先改哪一层。

所以这篇文档的主线不是从数据库产品开始，而是从查询问题本身开始：

- 先定义系统到底要支持什么查询，服务谁，对时延和可见性要求是什么。
- 再做最小规模估算，判断主压力在热点命中、范围扫描、聚合计算还是历史数据访问。
- 再把链路拆成入口、路由、缓存、主存储、搜索 / 分析引擎、结果治理几个阶段。
- 再把 MySQL、Redis、ES、Doris / Impala / Presto 放回不同查询模式里。
- 最后补高可用、限流、隔离、监控和扩容，形成设计闭环。

## 2. 主线

1. 先明确需求和约束：查的是明细、状态、窗口结果还是分析结果，时延目标和一致性窗口分别是什么。
2. 再做规模估算：查询 QPS、峰值倍数、热点分布、结果集大小、扫描量和历史数据规模大概是多少。
3. 再定义查询模型：是点查、条件筛选、分页、全文检索、聚合统计还是混合查询。
4. 再拆查询链路：入口层、路由层、缓存层、主存储层、搜索 / 分析层分别承担什么职责。
5. 再做数据组织：索引、宽表、预计算、冷热分层、多引擎协同分别在控制什么成本。
6. 最后看治理与演化：热点、慢查询、多租户、高并发和量级增长时系统如何继续稳定工作。

## 3. 核心问题展开

### 3.1 实时查询系统真正要先定义的是什么，为什么不先定义就没法选存储

这一节在回答什么问题：

- 实时查询系统设计的第一步到底是什么。
- 为什么不先定义查询模式，后面的存储选型都没有依据。

这个问题是怎么解决的：

实时查询系统设计的第一步，不是看自己熟悉哪种数据库，而是先定义“查询到底长什么样”。因为不同查询模式，对底层数据组织方式的要求完全不同。

常见查询模式至少可以先分成下面几类：

1. 单 key 精确查询
   - 更偏主键或高选择性索引命中。
   - 典型目标是低延迟、低扫描成本。
2. 条件筛选和分页
   - 更偏联合索引、排序字段、分页代价控制。
3. 范围查询或最近窗口查询
   - 更依赖时间维度组织、冷热分层和写入可见性。
4. 全文检索或多字段搜索
   - 更依赖倒排索引和搜索型引擎。
5. 聚合、多维统计、分析查询
   - 更依赖列式存储、预聚合和 OLAP 引擎。

如果这一步不先定义，很容易掉进一个很常见的误区：先把数据放进自己熟悉的某个库，再试图通过加索引、加缓存或堆机器去“补救”查询能力。这样就很容易出现：

- 用 MySQL 扛大扫描聚合
- 用 Redis 扛复杂筛选
- 用 ES 扛强事务状态
- 用 OLAP 引擎扛高频点更新

这些问题并不是组件本身不行，而是查询模式和引擎假设不匹配。

所以更稳的设计起点应该先回答：

- 查询入口面向谁，是接口服务、内部平台，还是分析工具
- 查询条件有哪些维度，是否固定，是否经常变化
- 返回结果是明细还是聚合
- 延迟目标和并发目标分别是多少
- 数据更新后多快必须可见

只有这些问题明确了，后面谈索引、缓存、分层和引擎选型才真正有判断依据。

### 3.2 为什么实时查询系统必须先做规模估算，真正该估哪些量

这一节在回答什么问题：

- 查询系统最关键的规模量是什么。
- 为什么这些量会直接改变架构。

这个问题是怎么解决的：

查询系统的瓶颈往往不在“有多少数据”这一件事上，而在：

- 查询总量和峰值有多大
- 热点是否集中
- 单次查询扫描量有多大
- 聚合和排序代价有多高
- 历史数据是否必须实时可查

如果不先估这些量，就很容易出现两种错误：

- 数据量其实不大，却过度引入多引擎和复杂分层。
- 主问题已经是热点和重查询，却还在只盯存储容量。

查询系统至少要先估下面几类量：

1. 查询 QPS
   - 平均 QPS、峰值 QPS、峰谷倍数。
2. 热点集中度
   - 是否存在极少数 key、极少数租户或极少数报表占掉大部分流量。
3. 单次扫描量
   - 一次查询平均会扫多少行、多少列、多少分区。
4. 结果集大小
   - 是几十条、几百条，还是上万条。
5. 历史数据规模
   - 近实时热数据和长期历史数据分别多大。
6. 可见性要求
   - 数据写入后多久必须能被查到。

一些最小估算可以先粗到这个程度：

- 峰值查询压力 = 平均 QPS x 峰谷倍数
- 主存储读压力 = 未命中缓存的查询数 x 平均回源代价
- 热点穿透风险 = 热点 key 请求数 x 单次回源成本
- 聚合查询成本 = 单次扫描量 x 查询并发数
- 历史查询代价 = 历史扫描范围 x 数据分布层级

这些数字为什么重要：

- 当热点极端集中时，主问题通常先是缓存和隔离，而不是主库容量。
- 当扫描量远大于结果集时，主问题通常先是数据组织和引擎匹配，而不是接口代码。
- 当历史数据规模很大但热数据占比很小，主问题通常先是冷热分层，而不是单一引擎继续硬扛。

所以规模估算的价值，不是为了得出非常精确的数字，而是为了先判断：

- 查询系统到底是在“命中型压力”下工作，还是在“扫描型压力”下工作
- 真正贵的是热点回源、复杂筛选，还是大范围聚合

### 3.3 一条实时查询链路到底该怎么拆，缓存、路由、主存储和分析引擎分别落在哪

这一节在回答什么问题：

- 实时查询系统应该按什么主线拆链路。
- 为什么缓存、索引、主存储和分析引擎不能混成一个笼统的“查询层”。

这个问题是怎么解决的：

实时查询链路要拆得稳定，至少要先分清下面几层：

1. 查询入口层
   - API、鉴权、租户校验、参数检查、分页保护。
   - 主要风险是入口放大、参数失控、权限越界。
2. 查询路由层
   - 根据查询类型决定走缓存、事务库、搜索引擎还是分析引擎。
   - 主要风险是路由错误、重查询误入低延迟链路。
3. 缓存层
   - 承接热点 key、热点结果和部分预计算结果。
   - 主要风险是命中率下降、击穿、缓存不一致。
4. 主存储层
   - 承接强一致状态、点查基线和部分准实时查询。
   - 主要风险是索引不匹配、回源压力过大、写读竞争。
5. 搜索 / 分析层
   - 承接复杂筛选、全文检索、聚合、多维分析、大范围扫描。
   - 主要风险是重查询抢资源、代价失控、结果延迟过大。
6. 结果治理层
   - 承接超时、限流、降级、结果截断、审计和监控。
   - 主要风险是重查询没有边界、异常请求拖垮公共资源。

这类拆法的价值在于，很多能力会自然找到准确位置：

- 缓存控制的是热点回源成本，而不是所有查询都该先进缓存。
- 主存储控制的是事实状态基线，而不是所有复杂筛选都该压给它。
- 搜索 / 分析引擎控制的是扫描、筛选、聚合成本，而不是所有在线查询都该直接丢过去。
- 结果治理控制的是资源边界，而不是最后再补一个超时参数。

所以实时查询系统真正稳定的讲法，不是“查询服务连数据库和缓存”，而是先把读路径拆出来，再把每层职责和代价钉住。

### 3.4 为什么实时查询题最后总会落到多引擎选型，MySQL、Redis、ES、Doris / Impala / Presto 分别适合什么问题

这一节在回答什么问题：

- 为什么实时查询系统几乎一定会被追问数据库或引擎选型。
- 常见引擎到底各自适合什么查询模式。

这个问题是怎么解决的：

实时查询题最终会落到选型，是因为“实时”只回答了数据可见性的时效，“查询”真正决定成本的是读模式。不同引擎擅长的，是不同类型的读。

#### MySQL：控制事务状态和索引型在线读

MySQL 更适合：

- 事务状态管理
- 主键查询和高选择性索引查询
- 相对强一致的在线读
- 记录级更新频繁的场景

它为什么有效：

- 行存储和索引结构更适合点查和小范围范围查。
- 事务和一致性模型相对成熟。

它的边界：

- 大扫描和复杂聚合代价高。
- 为分析型读不断加宽表、加索引，收益会快速下降。

#### Redis：控制热点结果和低延迟命中

Redis 更适合：

- 热点 key
- 热点结果缓存
- 轻状态或预计算结果

它为什么有效：

- 内存访问延迟低，适合拦住重复热点请求。

它的边界：

- 它不是强一致事实来源。
- 不适合复杂筛选和大范围扫描。

#### ES：控制搜索型查询成本

ES 更适合：

- 多字段搜索
- 全文检索
- 复杂筛选和倒排索引场景

它为什么有效：

- 倒排索引更适合搜索型条件组合。

它的边界：

- 不适合承担强事务状态基线。
- 数据写入和搜索可见性通常会带来一定时延窗口。

#### Doris / Impala / Presto：控制分析型查询成本

这类引擎更适合：

- 大扫描
- 聚合统计
- 多维分析
- 宽表分析

它们为什么有效：

- 更偏列式存储、MPP 执行和分析型读模型。

它们的边界：

- 不天然适合高频点更新。
- 并不等于所有在线查询都该直接压给分析引擎。

所以多引擎并不是“炫技”，而是把不同类型的查询成本交给更适合的系统承接。真正的判断关键不是产品名，而是：

- 查询条件复杂度
- 扫描量
- 实时可见性要求
- 更新模式
- 并发和成本目标

### 3.5 为什么实时查询系统经常要做结果缓存、预计算和冷热分层，它们分别在削减什么成本

这一节在回答什么问题：

- 为什么很多查询系统只靠单层存储扛不住。
- 结果缓存、预计算和冷热分层分别在控制什么成本。

这个问题是怎么解决的：

实时查询系统很容易同时面对两类压力：

1. 热点请求反复打在同一批数据上。
2. 长尾查询偶尔扫很大范围，但单次代价很高。

如果所有查询都直接打主存储，很快会出现两个问题：

- 热点读把主链路回源压力打高。
- 重查询拖慢正常低延迟查询。

所以查询系统常见地会同时引入下面几种手段：

#### 结果缓存：削减热点回源成本

结果缓存适合解决：

- 相同请求被反复打
- 热点查询集中在极少数 key 或条件组合上

它的收益：

- 把热点重复访问从主存储前面拦住。

它的代价：

- 一致性窗口
- 失效策略复杂度
- 热 key 风险

#### 预计算：削减现算成本

预计算适合解决：

- 某些聚合、统计、宽表查询每次现算太贵
- 查询结果可以接受通过异步链路提前准备

它的收益：

- 把一部分查询代价前移到写路径或异步计算链路。

它的代价：

- 链路更复杂
- 结果时效边界更明显
- 预计算结果的维护成本更高

#### 冷热分层：削减不同时间层级数据共用同一成本结构的问题

冷热分层适合解决：

- 近实时热数据和长期历史数据访问模式完全不同
- 热数据要求低延迟，冷数据要求低成本留存

它的收益：

- 让高频低延迟查询和低频大扫描查询不共用同一套资源结构。

它的代价：

- 路由复杂度上升
- 数据管理和一致性边界更复杂

所以正确的讲法不是“为了性能我们会上缓存、预计算、冷热分层”，而是要说清：

- 缓存在削热点回源成本
- 预计算在削复杂结果现算成本
- 冷热分层在削不同访问模式共用同一引擎的结构性成本

### 3.6 查询系统怎样兼顾高并发、可用性和多租户，而不是一出热点就整体抖动

这一节在回答什么问题：

- 查询系统高并发下最容易先出什么问题。
- 为什么限流、降级、隔离和多租户治理必须进入设计主线。

这个问题是怎么解决的：

查询系统高并发下，最容易先暴露的问题通常不是“库直接挂掉”，而是：

- 某类热点查询把缓存击穿到主存储
- 某个重型查询拖慢共享线程池或查询集群
- 某个租户的高代价查询把其他租户挤压掉

所以查询系统如果只停在“查得出来”，没有继续回答“高峰下如何继续稳定工作”，就不算设计闭环。

常见治理手段通常包括：

1. `限流`
   - 控制入口流量，防止热点或异常查询直接打穿下游。
2. `降级`
   - 当分析型或次要查询代价过高时，优先保证核心查询。
3. `隔离`
   - 线程池隔离、租户隔离、查询类型隔离，避免互相拖累。
4. `超时与取消`
   - 长查询不能无限占资源，必须有边界。
5. `分页与结果保护`
   - 避免大结果集直接拉垮系统。

多租户场景下，至少要考虑：

- 是否有按租户配额
- 是否有按租户观察面
- 是否允许某租户重查询长期吃满公共资源

所以查询系统的可用性问题，本质上经常先表现为资源竞争问题，而不是单点故障问题。

### 3.7 这个系统应该怎么监控、止损和演化，才能形成设计闭环

这一节在回答什么问题：

- 查询系统怎样通过监控和止损形成可运营闭环。
- 量级涨 10 倍时，应该沿什么顺序演化。

这个问题是怎么解决的：

查询系统上线后，真正重要的不只是“接口通了”，而是：

- 问题什么时候能被发现
- 出现热点或重查询时能否止损
- 量级上涨时是否知道先改哪一层

监控至少要覆盖下面几类指标：

1. 入口指标
   - QPS、峰值倍数、错误率、超时率、被限流次数。
2. 缓存指标
   - 命中率、回源率、热 key、失效率。
3. 主存储指标
   - 查询 RT、慢查询数、扫描行数、索引命中情况、连接池压力。
4. 搜索 / 分析层指标
   - 查询耗时、扫描量、失败率、资源占用、重查询占比。
5. 隔离指标
   - 按租户、按查询类型、按资源池的吞吐、延迟和资源占用。

止损动作通常要和这些指标绑定：

- 缓存命中率骤降，先检查失效策略和回源压力。
- 某类重查询大量积压，先限流、取消或降级这类查询。
- 某租户长期占满资源，先启用配额和隔离。
- 主库慢查询上升，先判断是不是重查询误打到了错误链路。

如果量级涨 10 倍，演化顺序通常不是“直接上更多数据库”，而是：

1. 先判断瓶颈在热点回源、复杂筛选、聚合计算还是历史查询。
2. 先补缓存、限流、隔离和路由边界。
3. 再把不适合当前主存储的查询拆到更合适的引擎。
4. 再把热数据、冷数据和分析数据做更细粒度分层。
5. 最后再考虑更大的架构重构。

所以设计闭环的关键是：系统不仅要能查，还要能发现问题、约束问题并逐步演化。

## 4. 边界、误区、常见混淆

- 实时查询系统不是简单加一个查询接口，真正难的是查询模型、数据组织和成本控制。
- 不要先选数据库，再反推场景；应该先定义查询类型，再决定引擎和索引组织。
- Redis 是热点层，不是强一致事实来源；ES 是搜索引擎，不是事务主存储。
- MySQL 扛在线读没问题，但一旦进入大扫描和复杂聚合，继续堆索引通常不是长期解。
- 结果缓存、预计算、冷热分层都不是默认必须加的动作，它们分别针对不同成本项。
- 查询系统的高可用问题经常首先表现为资源竞争和热点拖垮，而不是数据库彻底挂掉。
- 多引擎不代表越多越好，关键在于是否真的在用不同系统承接不同成本结构。

## 5. 自测清单

- 你能不能先定义一个实时查询系统到底面对哪类查询，而不是先说用什么数据库。
- 你能不能做出查询系统的最小规模估算，并解释热点、扫描量和历史数据规模分别会把压力推向哪一层。
- 你能不能把查询链路拆成入口、路由、缓存、主存储、搜索 / 分析层、结果治理六层。
- 你能不能说明 MySQL、Redis、ES、Doris / Impala / Presto 分别更适合什么查询模式。
- 你能不能解释为什么结果缓存、预计算、冷热分层会一起出现，以及它们分别在削减什么成本。
- 你能不能说明热点查询、重型查询、多租户为什么会把限流、降级和隔离推到主线里。
- 你能不能结合自己的项目背景说明为什么某类查询继续放 MySQL 不合适，应该拆到更适合的引擎。
- 你能不能面对“量级涨 10 倍怎么办”时，先讲瓶颈位置、隔离和演化顺序，而不是直接堆组件。
