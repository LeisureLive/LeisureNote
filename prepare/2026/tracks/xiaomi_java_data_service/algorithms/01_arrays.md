# 算法专题 01：数组

适用目标：

- 系统理解数组题的核心特点
- 能识别数组题常见模式
- 为后续刷题和面试表达提供模板

## 1. 数组的核心特点

- 内存连续
- 支持按下标 `O(1)` 访问
- 尾部追加通常较高效
- 中间插入 / 删除通常需要搬移元素，代价较高
- 很多题的本质是“如何在遍历过程中维护一段有意义的区间”

## 2. 什么题应该优先想到数组

如果题目有下面这些特征，通常优先考虑数组思维：

- 输入本身就是数组，要求原地修改
- 需要找连续区间、子数组、窗口
- 需要按位置关系处理：左边、右边、前缀、后缀
- 需要通过下标做状态表达
- 需要频繁访问相邻元素

## 3. 数组题常考模式

建议你看这一节时，不要只记“题型名字”，而要固定追问三件事：

- 这题为什么会落到这个模式
- 这个模式的核心状态是什么
- 代码里先做什么、后做什么

### 3.1 遍历与模拟

遍历与模拟的核心不是套固定算法，而是把题意翻译成一串可执行的“机械动作”。

这类题通常不难在思路命名，而难在：

- 访问顺序有没有写对
- 状态更新顺序会不会打架
- 边界、转向、停止条件会不会漏掉

#### 1. 使用信号

如果题目出现下面这些特征，优先考虑遍历与模拟：

- 题目规则很明确，答案可以通过“按题意一步一步执行”得到
- 重点不在最优决策，而在过程控制
- 需要你明确描述：
  - 当前处理到哪里
  - 下一步往哪走
  - 什么时候转向
  - 什么时候停止
- 输出往往是：
  - 一个按规则重排后的数组
  - 一个按顺序读取出来的结果列表
  - 一个按指定路径填充出的矩阵
- 暴力写法通常不是“不会做”，而是容易写出：
  - 重复访问
  - 漏访问
  - 越界
  - 转向时机错误

一句话判断：

- 如果题目的难点是“把过程稳定写对”，而不是“选择最优策略”，就优先考虑遍历与模拟。

#### 2. 常见题型与对应套路

##### 2.1 线性模拟：按规则合并或搬运元素

特征：

- 输入通常是一个或多个数组
- 需要按比较规则把元素依次放到目标位置
- 难点在于写入时不能破坏还没处理的数据

代表题：

- `88. 合并两个有序数组`

通用套路：

1. 先确定结果写入位置从哪里开始最安全。
2. 再定义每个指针各自指向哪一段未处理区域。
3. 每轮根据题意比较当前候选值，决定本轮写谁。
4. 写入后推进对应指针，直到某一侧处理完。
5. 最后把剩余未处理部分补完。

核心点：

- `88` 的关键不是“合并有序数组”这几个字，而是“目标数组本身也是输入的一部分”
- 因此前向写入会覆盖 `nums1` 里还没比较的值，正确做法是从后往前填

##### 2.2 方向数组模拟：按既定路径遍历矩阵

特征：

- 当前位置和前进方向都很重要
- 下一步是否继续直走，取决于：
  - 会不会越界
  - 会不会走到已经访问过的位置
- 题目本质是“沿着一条规则路径把格子按顺序读出来”

代表题：

- `54. 螺旋矩阵`

通用套路：

1. 用 `row`、`col` 表示当前位置。
2. 用方向数组维护“右、下、左、上”这类固定行进顺序。
3. 每次先处理当前格子，再尝试看下一步能不能继续直走。
4. 如果下一步越界或已经访问过，就切换方向。
5. 重复直到所有位置都访问完成。

核心点：

- `54` 的重点不是背四个方向，而是能稳定处理“什么时候该转向”
- 只要“当前位置、方向、访问状态、结束条件”这 4 个状态定义清楚，代码就会很稳

##### 2.3 方向数组模拟：按既定路径填充矩阵

特征：

- 和矩阵遍历几乎是同一套过程控制
- 区别只是“读取当前位置的值”变成了“向当前位置写入值”
- 你要同时关心：
  - 下一个位置是否合法
  - 当前写入源是否还有值

代表题：

- `59. 螺旋矩阵 II`
- `2326. 螺旋矩阵 IV`

通用套路：

1. 先初始化结果矩阵，以及“未填充位置”的表示方式。
2. 用方向数组控制行进顺序。
3. 每轮先把当前值写到当前位置，再判断下一步是否需要转向。
4. 如果下一步越界，或者下一个位置已经填过，就顺时针切换方向。
5. 重复直到：
   - 所有位置填满
   - 或数据源提前耗尽

核心点：

- `59` 和 `2326` 的过程几乎一样，区别只在于：
  - `59` 的数据源是连续递增的数字
  - `2326` 的数据源是链表，可能提前结束
- 所以面试表达时要主动先说清：这题的主干不是链表，而是“螺旋顺序填充 + 转向判断”

#### 3. 通用骨架

##### 3.1 线性模拟：多来源按规则写入结果

```java
while (来源 A 和来源 B 都还有元素) {
    if (当前应该取 A) {
        write(A);
        moveA();
    } else {
        write(B);
        moveB();
    }
    moveWrite();
}

while (某一侧还有剩余) {
    write(剩余元素);
    movePointer();
    moveWrite();
}
```

如果目标数组本身也是输入的一部分，先优先考虑：

- 能不能从后往前写，避免覆盖未处理数据

##### 3.2 方向数组模拟：遍历 / 填充矩阵

```java
int[][] dirs = {{0, 1}, {1, 0}, {0, -1}, {-1, 0}};
int dirIndex = 0;
int row = 0;
int col = 0;

while (还没有结束) {
    process(row, col);

    int nextRow = row + dirs[dirIndex][0];
    int nextCol = col + dirs[dirIndex][1];

    if (nextRow 越界 || nextCol 越界 || nextPosition 已处理) {
        dirIndex = (dirIndex + 1) % 4;
    }

    row += dirs[dirIndex][0];
    col += dirs[dirIndex][1];
}
```

这类骨架真正要先讲清 4 件事：

- 当前坐标怎么表示
- 方向顺序怎么定义
- 什么情况下转向
- 什么情况下结束

### 3.2 双指针

双指针本质上不是“写两个下标”。

它真正解决的是两类问题：

- 一个指针负责读，一个指针负责写，用 `O(1)` 额外空间原地整理数组
- 两个指针从两端或同向推进，利用有序性 / 单调性 / 边界信息缩小搜索范围

#### 1. 使用信号

如果题目出现下面这些特征，优先考虑双指针：

- 题目要求原地修改数组，且额外空间要控制在 `O(1)`
- 题目要求：
  - 原地去重
  - 原地删除
  - 保留某些元素并写回前缀
  - 移动某类元素到末尾，同时尽量保持顺序
- 输入本身有序，重复元素天然聚在一起，可以边扫边压缩结果
- 题目答案依赖左右两侧的信息，且移动一边后，另一边的判断仍然成立
- 暴力写法会自然变成：
  - 删一个元素就整体搬移一次
  - 对每个位置都再向左 / 向右找一遍
  - 两层循环反复比较两端关系

一句话判断：

- 如果题目核心是“两个位置分工推进”，而不是“维护一个连续窗口的合法性”，就优先考虑双指针。

#### 2. 常见题型与对应套路

##### 2.1 快慢指针：原地覆盖 / 原地过滤

特征：

- 结果仍然写回原数组前缀
- 被删掉或被覆盖的尾部内容通常不重要
- 一个指针负责扫描输入，另一个指针负责维护“有效结果写到哪里”

代表题：

- `26. 删除有序数组中的重复项`
- `27. 移除元素`
- `283. 移动零`

通用套路：

1. `fast` 从左到右扫描，判断当前元素是否应该保留。
2. `slow` 维护“下一个有效元素应该写入的位置”。
3. 如果当前元素需要保留，就写到 `nums[slow]`，然后 `slow++`。
4. 扫描结束后，`[0, slow)` 就是整理后的有效区间。
5. 如果题目还有额外收尾动作，再单独补上：
   - `283` 需要把 `slow` 之后统一补 `0`

核心点：

- `26` 依赖有序性，只需要比较“当前值”和“结果尾部”是否相同
- `27` 不依赖有序性，本质是“按条件过滤”
- `283` 和 `27` 很像，但多了“保持非零元素相对顺序”的约束，所以不能随便交换

##### 2.2 快慢指针：有序数组限次保留

特征：

- 输入有序
- 不是简单“保留 / 删除”，而是“每个值最多保留几次”
- 题目关键不在找新位置，而在维护“当前这段重复值已经保留了多少次”

代表题：

- `80. 删除有序数组中的重复项 II`

通用套路：

1. 用扫描指针遍历整个有序数组。
2. 用写指针维护当前结果尾部。
3. 再补一个状态记录“当前值已经保留了几次”。
4. 遇到新值时：
   - 写入结果
   - 重置计数
5. 遇到重复值时：
   - 如果还没达到保留上限，就继续写入
   - 否则直接跳过

核心点：

- 这类题和 `26` 的区别不是“是否双指针”，而是“保留规则更细”
- 面试表达里要主动点明：`26` 是每个值保留 `1` 次，`80` 是每个值保留 `2` 次，本质是同一模板的变体

##### 2.3 左右夹逼：两端信息决定当前结算或收缩方向

特征：

- 左右两端都有信息，当前应该先处理哪一边，取决于两边的比较结果
- 每次移动一边后，都能排除一部分不必再看的状态
- 关键难点不是“怎么写循环”，而是“为什么只动这一边不会漏解”

代表题：

- `42. 接雨水` 的双指针解法

延伸题：

- `167. 两数之和 II - 输入有序数组`
- `11. 盛最多水的容器`

通用套路：

1. `left`、`right` 分别指向当前尚未处理区间的两端。
2. 维护两侧当前决策所需的边界信息。
   - 例如 `42` 里是 `leftMax`、`rightMax`
3. 每一轮先判断“哪一侧的信息已经足够确定答案 / 哪一侧更弱”。
4. 先结算这一侧当前位置的答案，然后把这一侧指针向中间推进。
5. 重复直到两指针相遇。

核心点：

- `42` 的关键不是背公式，而是能解释：
  - 为什么 `leftMax <= rightMax` 时，可以先结算左侧
  - 为什么这样不会漏掉右侧未来更高挡板带来的影响
- 这类题的本质是：当前较弱一侧已经决定了这一侧位置的上界，所以可以安全推进

#### 3. 通用骨架

##### 3.1 快慢指针：原地覆盖 / 过滤

```java
int slow = 0;
for (int fast = 0; fast < n; fast++) {
    if (当前元素应该保留) {
        nums[slow] = nums[fast];
        slow++;
    }
}

// [0, slow) 是有效结果区间
return slow;
```

如果题目要求稳定搬移后的收尾处理，例如 `283`：

```java
for (int i = slow; i < n; i++) {
    nums[i] = 0;
}
```

##### 3.2 快慢指针：有序数组限次保留

```java
int write = 0;
int count = 0;

for (int read = 0; read < n; read++) {
    if (read == 0 || nums[read] != nums[read - 1]) {
        nums[write++] = nums[read];
        count = 1;
    } else if (count < limit) {
        nums[write++] = nums[read];
        count++;
    }
}

return write;
```

这个骨架适合表达 `80` 这种“每个值最多保留 `limit` 次”的题。

##### 3.3 左右夹逼：按较弱一侧推进

```java
int left = 0;
int right = n - 1;

while (left < right) {
    updateLeftState();
    updateRightState();

    if (左侧信息已经足够确定答案) {
        settleLeft();
        left++;
    } else {
        settleRight();
        right--;
    }
}
```

这一类骨架不要求你死记代码，要求你先说清 3 件事：

- 两个指针各自代表什么
- 额外维护了什么边界状态
- 为什么当前只推进这一边是安全的

### 3.3 滑动窗口

适用场景：

- 连续子数组 / 连续子串问题
- 窗口内需要动态维护某种状态
- 目标是求最长、最短，或统计满足条件的连续区间

#### 1. 使用信号

如果题目出现下面这些特征，优先考虑滑动窗口：

- 题目对象是连续子数组 / 连续子串
- 题目要求窗口内满足某种约束，例如：
  - 无重复
  - 至多 `k` 个不合法元素
  - 覆盖某些字符
  - 和达到某个阈值
- 目标通常是：
  - 最长合法区间
  - 最短满足条件区间
  - 统计满足条件的区间
  - 找固定长度窗口中的最优结果
- 暴力写法会自然变成：
  - 枚举左端点，再不断向右扩
  - 或枚举所有子串 / 子数组

一句话判断：

- 只要题目是“连续区间 + 区间状态可维护 + 左边界右移后旧状态可删除”，就优先考虑滑动窗口。

#### 2. 算法定义

滑动窗口本质上是在维护一个连续区间，以及这个区间对应的状态。

核心只有 3 个部分：

- 窗口边界：
  - `left`
  - `right`
- 窗口状态：
  - 频次数组 / 哈希表
  - 当前和
  - 当前种类数
  - 坏元素个数
  - 已满足条件的计数
- 窗口不变量：
  - 当前窗口是否合法
  - 当前窗口是否已经满足目标条件

滑动窗口真正要做的事情，不是机械移动双指针，而是：

- **把右侧新元素纳入窗口状态**
- 在需要时移动左边界并删除旧状态
- 在“合法 / 不合法 / 满足 / 不满足”之间切换
- 在合适时机更新答案

#### 3. 区间写法怎么选

滑动窗口常见有两种区间写法。

##### 3.1 左闭右闭 `[left, right]`

含义：

- `right` 指向当前已经在窗口里的最后一个元素

常见特点：

- 窗口长度通常写成 `right - left + 1`
- 常见写法是 `for (int right = 0; right < n; right++)`

##### 3.2 左闭右开 `[left, right)`

含义：

- `right` 指向下一个准备进入窗口的位置

常见特点：

- 窗口长度通常写成 `right - left`
- 外层循环条件常见写法是 `while (right < n)`
- 进入循环后，先处理 `nums[right]` 入窗，再让 `right++`

##### 3.3 怎么选

结论：

- 这几类滑动窗口题型都不强绑定某一种区间写法
- 关键不是题型，而是整段代码的语义必须统一

默认建议：

- 专题和模板优先统一成 `[left, right)`
- 因为：
  - 长度公式统一是 `right - left`
  - `right` 更自然地表示“下一个待进入位置”
  - 更适合写成“先入窗，再修复，再更新答案”

什么时候用 `[left, right]` 也可以：

- 你更习惯把 `right` 理解成“当前窗口最后一个元素”
- 你更习惯在窗口长度里写 `right - left + 1`

一句话要求：

- 选一种写法后，窗口语义、长度公式、推进顺序、答案更新时机必须保持一致。

#### 4. 常见题型与对应套路

##### 4.1 固定长度窗口

特征：

- 窗口长度固定不变
- 每次只需要维护“进一个、出一个”

代表题：

- `438. 找到字符串中所有字母异位词`

通用套路：

1. 把右侧新元素纳入窗口状态
2. 如果窗口长度超过固定值，就移动左边界并删除旧状态
3. 当窗口长度恰好等于目标长度时，判断是否满足题意

核心点：

- 重点不是“合法 / 不合法”
- 而是固定长度下的状态维护

##### 4.2 变长窗口：求最长合法区间

特征：

- 窗口内有合法性约束
- 目标是求最长

代表题：

- `3. 无重复字符的最长子串`
- `485. 最大连续 1 的个数`
- `1004. Max Consecutive Ones III`

通用套路：

1. 把右侧新元素纳入窗口状态
2. 如果窗口不合法，就不断移动左边界修复窗口
3. 窗口重新合法后，更新最长答案

核心点：

- 不合法就缩
- 合法时更新答案

##### 4.3 变长窗口：求最短满足区间

特征：

- 窗口需要先满足条件
- 目标是求最短

代表题：

- `209. 长度最小的子数组`
- `76. 最小覆盖子串`

通用套路：

1. 把右侧新元素纳入窗口状态
2. 一旦窗口满足条件，就不断移动左边界压缩窗口
3. 在压缩过程中更新最短答案
4. 直到窗口重新不满足，再继续处理后续元素

核心点：

- 满足了就尽量缩
- 和“最长合法窗口”正好相反

##### 4.4 容错型窗口

特征：

- 窗口内允许最多 `k` 个不满足条件的元素
- 常见表述是：
  - 最多替换 `k` 次
  - 最多包含 `k` 个 `0`
  - 最多删除 / 修改 `k` 个字符

代表题：

- `1004. Max Consecutive Ones III`

通用套路：

1. 维护一个“坏元素计数”
2. 把右侧新元素纳入窗口状态时更新坏元素数
3. 如果坏元素数超过 `k`，就移动左边界修复
4. 合法时更新答案

核心点：

- 本质上还是“最长合法窗口”
- 只是合法条件变成了“坏元素数不超过 `k`”

#### 5. 通用骨架

下面默认按 `[left, right)` 写。

##### 5.1 最长合法窗口

```java
while (right < n) {
    // 1. 把 nums[right] 纳入窗口状态
    add(nums[right]);
    right++;

    // 2. 如果窗口不合法，就移动 left 修复
    while (窗口不合法) {
        remove(nums[left]);
        left++;
    }

    // 3. 窗口合法时更新答案
    update(right - left);
}
```

##### 5.2 最短满足窗口

```java
while (right < n) {
    // 1. 把 nums[right] 纳入窗口状态
    add(nums[right]);
    right++;

    // 2. 只要窗口满足条件，就尽量收缩
    while (窗口满足条件) {
        update(right - left);
        remove(nums[left]);
        left++;
    }
}
```

##### 5.3 固定长度窗口

```java
while (right < n) {
    // 1. 把 nums[right] 纳入窗口状态
    add(nums[right]);
    right++;

    // 2. 如果窗口长度超过固定值，就收缩
    if (right - left > fixedLen) {
        remove(nums[left]);
        left++;
    }

    // 3. 当窗口长度刚好等于固定值时，判断答案
    if (right - left == fixedLen) {
        update();
    }
}
```

如果采用 `[left, right]`，则更常见写法是：

```java
for (int right = 0; right < n; right++) {
    add(nums[right]);

    while (窗口不合法) {
        remove(nums[left]);
        left++;
    }

    update(right - left + 1);
}
```

### 3.4 前缀和

适用场景：

- 频繁查询某个区间和
- 子数组求和类问题
- 需要把“区间信息”转成“两个前缀的差”

典型题：

- `303. 区域和检索 - 数组不可变`
- `724. 寻找数组的中心下标`
- `560. 和为 K 的子数组`（与哈希表结合）

通用套路：

1. 先把“区间信息”改写成“两个前缀的差”。
2. 明确定义：
   - `prefix[i]` 到底表示前 `i` 个元素之和
   - 还是表示到下标 `i` 为止的和
3. 如果是离线多次查询区间和，通常先建前缀和数组。
4. 如果是边遍历边统计答案，通常用“滚动前缀和 + 其他结构”。
5. 写代码时一定把下标和定义对齐，不然很容易差一位。

你真正要会的是：

- 看到区间和先想到“能不能转前缀差”
- 能把数学等式稳定翻译成代码，而不是凭感觉写

### 3.5 前后缀分解

#### 1. 怎么判断是不是前后缀分解

前后缀分解最常见的信号，不是“出现了前缀”两个字，而是：

- 题目在问“每个位置的答案”
- `ans[i]` 同时依赖 `i` 左边的信息和右边的信息
- 如果直接写，暴力会变成：
  - 对每个 `i` 向左扫一遍
  - 再向右扫一遍
- 左右两边的信息彼此独立，可以先分别算好，再在当前位置合并

常见表述：

- 除自身以外
- 左边最大 / 右边最大
- 每个位置都要结合左右两边的信息

一句话判断：

- 只要题目是“求每个位置的答案，而且答案依赖左右两侧信息”，就优先考虑前后缀分解。

#### 2. 算法定义

前后缀分解本质上是在做：

- 先定义每个位置需要的左侧信息
- 再定义每个位置需要的右侧信息
- 最后把左右信息合并成答案

核心只有 3 个部分：

- 左侧状态：
  - `left[i]`
- 右侧状态：
  - `right[i]`
- 合并公式：
  - `ans[i] = merge(left[i], right[i], nums[i])`

这里的左信息 / 右信息不一定是和或乘积，也可以是：

- 最大值
- 最小值
- 计数
- 其他可以独立维护的信息

前后缀分解真正要做的事情，不是机械开两个数组，而是：

- **先定义清楚每个位置到底需要什么左信息**
- **再定义清楚每个位置到底需要什么右信息**
- 明确这些信息是否包含当前位置自己
- 最后按题目关系把它们合并

#### 3. 状态定义怎么选

前后缀分解最容易错的地方，不是复杂度，而是定义不清。

##### 3.1 是否包含自己

常见有两种定义方式。

第一种：不包含自己

- `left[i]` 表示 `i` 左边所有元素的信息
- `right[i]` 表示 `i` 右边所有元素的信息

代表题：

- `238. 除自身以外数组的乘积`

例如：

- `left[i]` 表示 `nums[0..i-1]` 的乘积
- `right[i]` 表示 `nums[i+1..n-1]` 的乘积
- `ans[i] = left[i] * right[i]`

第二种：包含自己

- `left[i]` 表示从左到 `i` 为止的信息
- `right[i]` 表示从右到 `i` 为止的信息

代表题：

- `42. 接雨水`

例如：

- `leftMax[i]` 表示 `0..i` 的最大值
- `rightMax[i]` 表示 `i..n-1` 的最大值
- `water[i] = min(leftMax[i], rightMax[i]) - height[i]`

一句话要求：

- 必须先定死 `left[i]` / `right[i]` 是否包含 `i` 自己，再写代码。

##### 3.2 辅助数组长度怎么选

结论：

- 前后缀分解里，最常见是长度为 `n` 的数组
- 但这不是硬规则，关键是定义和边界是否统一

为什么经常是 `n`：

- 因为它通常是在描述“每个位置 `i` 对应的左信息 / 右信息”
- 所以天然就是一位置配一个状态

和前缀和的区别：

- 前缀和更常见开 `n + 1`
- 因为它常用来统一表达“前 0 个元素”的状态
- 这样区间和可以稳定写成两个前缀的差

一句话区分：

- 前缀和重点是“区间”
- 前后缀分解重点是“位置”

##### 3.3 是否一定要两个数组

不一定。

最常见有两种写法：

- 两个辅助数组：一个存左信息，一个存右信息
- 一个结果数组 + 一个滚动变量：先存一边，再边遍历边补另一边

例如：

- `238` 可以先让答案数组存左乘积
- 再用一个 `suffixProduct` 从右向左补右乘积

一句话判断：

- 第一次做题时，先用双数组把定义讲清楚
- 能稳定写对后，再考虑空间压缩

#### 4. 常见题型与对应套路

##### 4.1 左右乘积 / 左右和

特征：

- 每个位置的答案等于“左边贡献”和“右边贡献”的组合
- 通常不适合直接用总量减自身，或者题目禁止这样做

代表题：

- `238. 除自身以外数组的乘积`

通用套路：

1. 定义左侧贡献
2. 定义右侧贡献
3. 在当前位置把两边贡献相乘 / 相加

核心点：

- 要先想清楚左右信息是否包含自己

##### 4.2 左右最大值 / 最小值

特征：

- 每个位置的答案取决于左边最强约束和右边最强约束
- 常见是“当前位置能达到的上界 / 下界由左右两边共同决定”

代表题：

- `42. 接雨水`

通用套路：

1. 从左到右预处理左侧最值
2. 从右到左预处理右侧最值
3. 在当前位置按公式合并

核心点：

- 重点不是记公式
- 而是先看出“每个位置的答案由左右两侧边界共同决定”

##### 4.3 双数组可以压缩成滚动变量

特征：

- 左信息可以直接存进答案数组
- 右信息可以边走边维护，不需要完整数组

代表题：

- `238. 除自身以外数组的乘积`

通用套路：

1. 先把一侧信息写进结果数组
2. 再用滚动变量维护另一侧信息
3. 边遍历边补全答案

核心点：

- 优化空间之前，先保证状态定义本身是对的

#### 5. 通用骨架

##### 5.1 双数组写法

```java
int[] left = new int[n];
int[] right = new int[n];
int[] ans = new int[n];

// 1. 预处理左信息
for (int i = 0; i < n; i++) {
    left[i] = ...;
}

// 2. 预处理右信息
for (int i = n - 1; i >= 0; i--) {
    right[i] = ...;
}

// 3. 合并答案
for (int i = 0; i < n; i++) {
    ans[i] = merge(left[i], right[i], nums[i]);
}
```

##### 5.2 一个数组 + 滚动变量

```java
int[] ans = new int[n];

// 1. 先把左信息写进 ans
for (int i = 0; i < n; i++) {
    ans[i] = ...;
}

// 2. 用滚动变量补右信息
int suffixState = ...;
for (int i = n - 1; i >= 0; i--) {
    ans[i] = merge(ans[i], suffixState, nums[i]);
    update(suffixState, nums[i]);
}
```

你真正要会的是：

- 先判断这题是不是“求每个位置的答案”
- 再判断答案是否同时依赖左右两边
- 再把 `left[i]` / `right[i]` 的定义写清楚
- 最后再决定是双数组，还是压缩空间

### 3.6 原地交换 / 原地标记

适用场景：

- 要求 `O(1)` 额外空间
- 元素值和下标之间存在天然映射关系

典型题：

- `41. 缺失的第一个正数`
- `448. 找到所有数组中消失的数字`

通用套路：

1. 先判断元素值和下标之间是否存在天然映射关系。
2. 如果题目要求 `O(1)` 额外空间，经常要考虑：
   - 把值放回它“应该去的位置”
   - 或直接在原数组上做标记
3. 原地交换题常见流程是：
   - 当前值合法且不在目标位置，就持续交换
   - 直到当前位置无法继续交换
4. 原地标记题常见流程是：
   - 用正负号或特殊值记录“这个位置代表的数字见过没有”
5. 这类题最怕死循环和覆盖错误，写之前一定先讲清交换条件和停止条件。

你真正要会的是：

- 先看出“值和位置能不能绑定”
- 再决定是交换归位还是原地标记

### 3.7 二分查找

适用场景：

- 数组有序
- 答案空间具有单调性

典型题：

- `704. 二分查找`
- `34. 在排序数组中查找元素的第一个和最后一个位置`
- `875. 爱吃香蕉的珂珂`

通用套路：

1. 先分清这是哪一类二分：
   - 有序数组找目标
   - 找左边界 / 右边界
   - 对答案空间做二分
2. 先定义搜索区间，再统一边界风格：
   - 左闭右闭
   - 或左闭右开
3. 每次取 `mid` 后，要明确当前是在判断：
   - 值相等关系
   - 还是某个单调条件是否成立
4. 如果是答案空间二分，核心不是数组有序，而是“可行性随答案单调变化”。
5. 二分写不稳，通常不是代码能力问题，而是：
   - 区间定义没定死
   - 更新规则和区间定义不一致

你真正要会的是：

- 先讲清单调性或边界目标
- 再套统一模板，而不是每题临场重写一套二分

## 4. 数组题的高频坑点

- 下标越界
- 左闭右闭、左闭右开不统一
- 原地修改时覆盖掉后面还要用的数据
- 忘记处理空数组、长度为 `1` 的数组
- 写出了正确思路，但“定义”和“代码实现”不一致
- 复杂度判断不严谨，以为用了数组就默认 `O(n)`

## 5. 数组题的面试表达模板

面试里可以优先用这个结构表达：

1. 先说输入特征
2. 再说为什么想到某种数组技巧
3. 再说核心状态定义
4. 再说遍历顺序
5. 最后给复杂度和边界

例子：

```text
这题输入是数组，而且每个位置的答案依赖左边和右边的信息，所以我会考虑前后缀分解。
我定义 prefix[i] 表示 i 左边元素的乘积，suffix[i] 表示 i 右边元素的乘积，
那么答案就是 prefix[i] * suffix[i]。
这样只需要线性遍历两到三次，时间复杂度 O(n)。
```

### 6.1 `238. 除自身以外数组的乘积`

本质：

- 数组题

- 不需要哈希表

你需要掌握：

- 如何优化到 `O(1)` 额外空间

### 6.2 `167. 两数之和 II - 输入有序数组`

本质：

- 数组题
- 重点是双指针
- 关键信号是“有序数组 + 找两数之和”

题目约束下要先想到：

- 不要先写暴力 `O(n^2)`
- 目标是把时间复杂度降到 `O(n)`
- 额外空间应控制在 `O(1)`

为什么能用双指针：

- 数组已经有序
- 左指针指向较小值，右指针指向较大值
- 当前和如果偏大，就只能让右指针左移
- 当前和如果偏小，就只能让左指针右移

这题真正要掌握的不是“会写”，而是：

- 为什么看到“有序数组”会想到双指针
- 为什么 `sum > target` 时可以放心移动右指针
- 为什么 `sum < target` 时可以放心移动左指针
- 为什么每次移动都不会漏解

面试里建议按这个顺序回答：

1. 先说输入特征：有序数组，找两个数之和
2. 再说为什么不用暴力：`O(n^2)` 不必要
3. 再说双指针定义：一个从左，一个从右
4. 再说移动规则：
   - `sum > target`：右指针左移
   - `sum < target`：左指针右移
   - `sum == target`：直接返回
5. 最后补复杂度和边界

你需要掌握：

- 双指针为什么能把复杂度降到 `O(n)`
- 内部更推荐用 `0-based` 指针，返回结果时再转成 `1-based`
- 如果题目不保证一定有解，循环条件不要直接写 `while (true)`
- 口头表达时，要把“移动方向不会漏解”的理由说清楚
