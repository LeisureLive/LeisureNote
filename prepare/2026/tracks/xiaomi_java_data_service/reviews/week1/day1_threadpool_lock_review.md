## 2. Day1 必须掌握的知识框架

### 2.1 线程池

#### 1）为什么要用线程池

- 减少线程频繁创建和销毁的开销
- 复用线程，提高整体吞吐
- 控制并发度，避免任务高峰把机器资源打满
- 统一管理线程命名、异常、监控、拒绝策略

#### 2）线程池接收任务的执行流程

- 当前线程数小于 `corePoolSize`：直接创建核心线程执行任务
- 否则：尝试把任务放入 `workQueue`
- 队列满了且当前线程数小于 `maximumPoolSize`：创建非核心线程执行任务
- 队列满且线程数已到 `maximumPoolSize`：执行拒绝策略

一句话总结：

- 核心线程扛常态流量
- 队列负责削峰
- 最大线程数负责短时兜底
- 拒绝策略负责过载保护

#### 3）`ThreadPoolExecutor` 必须记住的参数

- `corePoolSize`：常驻并发能力
- `maximumPoolSize`：高峰期兜底能力
- `keepAliveTime` + `unit`：非核心线程空闲回收时间
- `workQueue`：任务排队和削峰缓冲
- `threadFactory`：线程名、优先级、异常处理等定制入口
- `RejectedExecutionHandler`：线程池满载时的处理方式

#### 4）拒绝策略要会背，也要会说场景

- `AbortPolicy`：直接抛异常，适合必须显式感知失败的场景
- `CallerRunsPolicy`：由提交线程执行任务，适合需要反压且调用方线程可被占用的场景
- `DiscardPolicy`：直接丢弃，只有任务允许丢失时才可用
- `DiscardOldestPolicy`：丢最旧任务，适合更关注新任务的场景，但要谨慎

线上更常见的做法：

- 自定义拒绝策略，补日志、监控、告警、降级或补偿逻辑

#### 5）为什么不建议直接使用 `Executors`

- `newFixedThreadPool` / `newSingleThreadExecutor`：
  - 主要风险是无界队列，任务堆积后容易造成内存占用持续升高，延迟失控，严重时可能 OOM
- `newCachedThreadPool`：
  - 主要风险是线程数可能快速膨胀，带来 CPU 抖动、上下文切换增加，严重时拖垮整个服务

结论：

- 生产环境更推荐显式创建 `ThreadPoolExecutor`
- 把线程数、队列、拒绝策略都控制在可预期范围内

#### 6）线程池参数如何配置

先记方法，再记数字。

配置思路：

1. 先判断任务是 CPU 密集型还是 IO 密集型
2. 再看接口 `TPS`、平均 `RT`、超时、机器核数和内存
3. 再看下游依赖的稳定承载能力
4. 决定 `corePoolSize`、`maximumPoolSize`、`workQueue`、拒绝策略
5. 上线后通过监控动态调参，而不是一次拍死

常用估算抓手：

- 平均并发约等于：`TPS x RT`
- CPU 核需求约等于：`TPS x 单请求 CPU 时间`
- 线程数初估可参考：`CPU 核数 x (1 + 等待时间 / 计算时间)`

注意：

- 不要只背 `N+1` 或 `2N`
- 资深岗更关注你是否会结合业务场景做容量判断

#### 6.1）实际场景下怎么给线程池初始配置

场景示例：

- 机器配置：`8 核 16G`
- 接口类型：在线查询接口
- 峰值 `TPS`：`300`
- 平均 `RT`：`200ms`
- 单请求耗时拆分：
  - CPU 计算：`50ms`
  - 查 Redis / 调下游 RPC：`150ms`
- 下游稳定承载并发：`80`

第一步先算平均并发：

- 平均并发约等于：`300 x 0.2 = 60`

说明在峰值时，平均大约有 `60` 个任务同时在跑。

第二步看 CPU 是否扛得住：

- CPU 核需求约等于：`300 x 0.05 = 15 核`

这里要立刻意识到一个关键信号：

- 如果这是单机指标，那 `8` 核机器其实扛不住 `300 TPS`
- 这时不能指望“把线程池调大”解决问题
- 更合理的动作是：
  - 先确认这是单机流量还是集群总流量
  - 如果确实是单机目标，就要优先扩容、优化 CPU 计算、限流或降级

第三步如果面试官说“先别纠结，给我一个保护性的初始配置”：

- `corePoolSize = 32`
- `maximumPoolSize = 48`
- `workQueue = ArrayBlockingQueue(100)`
- 拒绝策略：优先自定义拒绝策略，次选 `AbortPolicy`

为什么这样配：

- `corePoolSize = 32`
  - 可参考公式：`8 x (1 + 150 / 50) = 32`
  - 因为这个任务里等待时间明显高于计算时间，线程数可以比 CPU 核数大
- `maximumPoolSize = 48`
  - 用来承接短时波峰
  - 但不会一味往下游稳定承载并发 `80` 靠近，避免把压力直接放大给下游
- `ArrayBlockingQueue(100)`
  - 查询接口更关注延迟，队列只做短时削峰，不应该无限堆积任务
  - 有界队列能更早暴露过载，而不是把问题拖成长尾延迟和内存风险
- 自定义拒绝策略 / `AbortPolicy`
  - 这类在线查询场景更适合快速失败、记录监控并做降级
  - 不优先选 `CallerRunsPolicy`，因为调用方往往是请求线程，容易把接口整体 RT 拖长

这个配置要怎么理解：

- 这是“系统保护型初始配置”
- 它的目标是让系统在高峰期可控，不是承诺单机一定能吃下这组流量
- 如果 `300 TPS` 真的要单机稳定承接，线程池不是核心问题，核心问题是机器算力和整体容量设计

第四步上线后重点看什么：

- 活跃线程数是否长期贴近 `corePoolSize` 或 `maximumPoolSize`
- 队列长度是否持续增长，而不是偶发尖峰
- 拒绝次数是否开始出现
- 任务执行耗时和超时率是否抬升
- 下游 Redis / RPC 的 `RT`、错误率、并发是否恶化
- 服务 CPU、Load、GC 是否异常

第五步怎么调：

- 如果队列偶发上涨，但 CPU 和下游都还有明显余量，可以小步上调 `maximumPoolSize`
- 如果队列持续上涨，同时 CPU 打满或下游 RT 明显恶化，不要继续放大线程池，优先扩容、限流、降级或优化慢点
- 如果队列几乎不涨、活跃线程长期明显低于核心线程数，说明当前配置偏保守，可以保持不动

你要记住这类题的回答节奏：

1. 先算平均并发
2. 再算 CPU 是否够用
3. 再看下游承载上限
4. 然后给一个保护性的初始配置
5. 最后补上线后的监控和调参路径

#### 6.2）三类典型场景怎么配线程池

这部分不是让你死背数字，而是建立“不同任务类型，为什么配置思路不同”的感觉。

##### 场景 A：CPU 密集型任务

典型例子：

- 复杂规则计算
- 批量加解密
- JSON / Protobuf 大量编解码
- 本地排序、聚合、打分计算

场景特征：

- 线程大部分时间都在占用 CPU
- 等待外部资源的时间很少
- 线程开太多会带来严重的上下文切换

配置原则：

- `corePoolSize` 不要配太大，通常以 CPU 核数附近为起点
- `maximumPoolSize` 和 `corePoolSize` 差距不要过大
- 队列可以适当保守，但不能无界
- 优先保护 CPU，不要让线程数膨胀

示例：

- 机器：`8 核`
- 任务：纯计算型，外部等待很少
- 可给初始配置：
  - `corePoolSize = 8`
  - `maximumPoolSize = 10`
  - `workQueue = ArrayBlockingQueue(200)`
  - 拒绝策略：`AbortPolicy` 或自定义拒绝

为什么这样配：

- 核心线程数贴近 CPU 核数，减少无效切换
- 最大线程数只比核心线程多一点，用来承接小波峰
- 有界队列让系统在高峰时可控，不会无限积压

最容易犯的错：

- 看到任务变慢就盲目加线程
- 实际上 CPU 密集型任务越加线程越可能更慢

##### 场景 B：IO 密集型任务

典型例子：

- 查询 Redis / MySQL
- 调用多个下游 RPC
- 远程文件 / 对象存储访问
- 需要等待网络或磁盘响应的任务

场景特征：

- 线程经常处于等待状态
- CPU 真正执行的时间占比不高
- 线程数通常可以大于 CPU 核数

配置原则：

- `corePoolSize` 可以明显大于 CPU 核数
- `maximumPoolSize` 要受下游承载能力约束
- 队列只做削峰，不做无限兜底
- 拒绝策略要结合业务是否允许失败、是否能降级

示例：

- 机器：`8 核`
- 任务：请求平均 `200ms`，其中 `40ms` CPU，`160ms` 等待外部 IO
- 线程数初估：`8 x (1 + 160 / 40) = 40`
- 可给初始配置：
  - `corePoolSize = 32`
  - `maximumPoolSize = 48`
  - `workQueue = ArrayBlockingQueue(100)`
  - 拒绝策略：在线接口优先自定义拒绝或 `AbortPolicy`

为什么这样配：

- 等待时间明显高于计算时间，线程可以比 CPU 核数多
- 但也不能无限放大，否则会把压力推给下游系统
- 队列只负责削峰，避免查询任务长时间排队变成长尾

最容易犯的错：

- 只看本机有余量，不看 Redis、MySQL、RPC 下游能不能扛住
- 只要线程池还能扩，就持续加 `maximumPoolSize`

##### 场景 C：批处理 / 离线异步任务

典型例子：

- MQ 消费后做异步处理
- 离线补数任务
- 定时扫描、批量修复、报表生成
- 不直接挂在用户请求主链路上的后台作业

场景特征：

- 更关注吞吐，而不是单次低延迟
- 有些任务允许慢一点，但不能把线上主链路拖死
- 经常需要和数据库、消息队列、外部服务长期交互

配置原则：

- 线程池配置要和主链路隔离，不能共用
- 队列可以比在线接口稍大，但仍然建议有界
- 拒绝策略可以更偏向“反压”或“延后处理”
- 如果任务天然可重试、可补偿，策略空间更大

示例：

- 机器：`8 核`
- 任务：后台批处理，单任务 `500ms`，其中 `100ms` CPU，`400ms` IO
- 线程数初估：`8 x (1 + 400 / 100) = 40`
- 可给初始配置：
  - `corePoolSize = 16`
  - `maximumPoolSize = 32`
  - `workQueue = ArrayBlockingQueue(500)`
  - 拒绝策略：可考虑自定义拒绝或 `CallerRunsPolicy`

为什么这里和在线接口不同：

- 批处理更关注总吞吐，可以接受一定排队
- 如果提交方不是主链路请求线程，`CallerRunsPolicy` 的副作用会小很多
- 但前提是调用方线程被占用不会影响核心业务

最容易犯的错：

- 后台任务和在线请求共用线程池
- 批处理任务积压时，直接把线程池开得很大，最后把数据库或下游服务打挂

##### 一张表记住三类任务的差别

| 场景 | 主要目标 | 线程数倾向 | 队列策略 | 拒绝策略倾向 |
| --- | --- | --- | --- | --- |
| CPU 密集型 | 保护 CPU、稳定执行 | 接近 CPU 核数 | 有界，小心积压 | `AbortPolicy` / 自定义 |
| IO 密集型 | 平衡等待时间和下游承载 | 可大于 CPU 核数 | 有界，只做削峰 | 在线接口优先快速失败 |
| 批处理 / 离线任务 | 吞吐优先、与主链路隔离 | 视资源和下游而定 | 可比在线任务稍大，但仍建议有界 | 自定义 / 部分场景可 `CallerRunsPolicy` |

一句话总结：

- CPU 密集型：别开太多线程
- IO 密集型：线程可以放大，但要受下游约束
- 批处理任务：重隔离、重吞吐，但不能拖垮主链路

#### 7）线程池上线后重点看什么

- 活跃线程数
- 当前线程池大小
- 队列长度
- 拒绝次数
- 任务执行耗时和超时率
- 下游 Redis / RPC 的 RT、错误率和并发
- 服务 CPU、Load、Full GC 等系统指标

### 2.2 锁：`synchronized` 和 `ReentrantLock`

#### 1）共同点

- 都能实现互斥同步
- 都可重入
- 都能保证可见性
- 都能解决竞态条件

#### 2）核心差异

- `synchronized` 是 JVM 层面的内置锁，语法简单
- `ReentrantLock` 是 JUC 显式锁，需要手动 `lock` / `unlock`
- `synchronized` 异常退出时会自动释放锁
- `ReentrantLock` 必须放在 `try/finally` 中释放，否则有死锁风险
- `ReentrantLock` 支持：
  - `tryLock`
  - 可中断获取锁
  - 公平锁
  - 多个 `Condition`
- `synchronized` 在这些能力上不如 `ReentrantLock` 灵活

补充理解：

- “可中断获取锁”指的是线程在等待锁的过程中，如果收到中断信号，可以放弃继续等待
- `ReentrantLock` 可以通过 `lockInterruptibly()` 支持这一点
- `synchronized` 在线程等待进入监视器锁时，不能因为 `interrupt` 立即退出等待
- 这个能力适合做超时控制、任务取消和避免线程长时间卡死
- 要和“等待条件”区分开：
  - `wait()` 和 `Condition.await()` 都发生在已经拿到锁之后
  - 两者都会释放当前锁，进入 `WAITING`，并且都能响应中断
  - 真正的差异点在“获取锁阶段”：
    - `ReentrantLock` 可中断获取锁
    - `synchronized` 不支持在等锁阶段因为中断立即退出

#### 3）选型原则

- 简单临界区优先 `synchronized`
- 需要超时、中断、公平性、多条件队列时，优先 `ReentrantLock`
- 不要用“谁一定更快”作为主要选型标准
- 在现代 JDK 下，性能通常不是两者的主要分界线

#### 4）项目中常见的 `ReentrantLock` 使用场景

- 需要 `tryLock(timeout)` 做快速失败或避免死锁
- 需要 `lockInterruptibly()` 响应中断, 指的是等待获取锁时就可以响应中断
- 需要公平锁减少线程饥饿
- 生产者消费者模型里，使用多个 `Condition` 区分“队列未满”和“队列非空”

## 3. 高频面试题标准答法

### 3.1 线程池接收一个任务后的执行流程是什么

标准答法：

```text
线程池接收一个任务后，先看当前运行线程数是不是小于 corePoolSize，如果小于，就直接创建核心线程执行任务；
如果已经达到 corePoolSize，就尝试把任务放进阻塞队列；
如果队列也满了，再看当前线程数是不是小于 maximumPoolSize，如果小于，就创建非核心线程处理；
如果线程数已经到 maximumPoolSize 且队列也满了，就执行拒绝策略。

所以线程池的本质是：核心线程扛常态、队列做削峰、最大线程数兜短时高峰、拒绝策略做系统保护。
```

### 3.2 为什么不建议直接使用 `Executors` 创建线程池

标准答法：

```text
我一般不建议直接用 Executors 提供的默认线程池工厂，主要原因是默认配置在生产环境里可能带来资源失控风险。

比如 newFixedThreadPool 和 newSingleThreadExecutor，底层通常是无界队列。任务处理不过来时会不断堆积，最终可能导致内存占用持续升高，甚至 OOM。
而 newCachedThreadPool 的问题是线程数可能快速膨胀，在流量突增或者下游变慢时，容易带来 CPU 抖动和上下文切换增加。

所以生产中我更倾向显式使用 ThreadPoolExecutor，把核心线程数、最大线程数、队列和拒绝策略都明确控制住。
```

### 3.3 线程池参数如何配置

标准答法：

```text
线程池参数配置我一般不会死背公式，而是按任务类型、流量、机器资源和下游承载能力来定。

第一步先判断任务是 CPU 密集型还是 IO 密集型。
第二步结合 TPS、平均 RT、超时设置、机器核数和内存，估算核心线程数和队列长度。
第三步看下游系统能承受的并发上限，避免把 maximumPoolSize 一味放大，把压力放大到下游。
第四步根据业务是否允许失败，选择合适的拒绝策略。
最后通过线程池活跃线程数、队列长度、拒绝次数、任务耗时和下游 RT 等指标做动态调参。

生产上我通常会显式创建 ThreadPoolExecutor，而不是直接用 Executors，确保参数可控、行为可观测。
```

### 3.3.1 面试追问：给你一个真实场景，你怎么给线程池初始配置

标准答法：

```text
比如一个在线查询接口部署在 8 核 16G 的机器上，峰值 TPS 300，平均 RT 200ms，其中大约 50ms 是 CPU 计算，150ms 在查 Redis 和调下游 RPC，下游稳定承载并发大约是 80。

我会先算平均并发，300 x 0.2，大约是 60；再算 CPU 需求，300 x 0.05，大约需要 15 核。
如果这是单机指标，我会先指出这台 8 核机器其实扛不住，线程池调再大也解决不了算力问题，应该先考虑扩容、优化 CPU 计算或者做限流降级。

如果面试官要求我先给一个保护性的初始线程池配置，我会先给 corePoolSize 32，maximumPoolSize 48，workQueue 用 ArrayBlockingQueue(100)，拒绝策略优先自定义拒绝，次选 AbortPolicy。
32 可以参考 8 x (1 + 150/50) 这个估算；48 用来兜短时高峰，但不会无限接近下游并发上限；100 的有界队列只做短时削峰，不让查询任务无限堆积。

上线后我会重点看活跃线程数、队列长度、拒绝次数、任务耗时、CPU 和下游 RT。如果队列持续上涨但 CPU 和下游仍有余量，再考虑小步调整；如果 CPU 或下游已经成为瓶颈，就不继续放大线程池，而是优先扩容和降级。
```

### 3.3.2 面试追问：CPU 密集型、IO 密集型、批处理任务的线程池配置有什么差别

标准答法：

```text
我一般不会用一套固定参数去配所有线程池，而是先看任务类型。

如果是 CPU 密集型任务，比如复杂计算、加解密、本地聚合，我会把线程数控制在 CPU 核数附近，因为线程大部分时间都在真正占用 CPU，线程开太多只会带来上下文切换，通常 corePoolSize 贴近核数，maximumPoolSize 只比它略大一点。

如果是 IO 密集型任务，比如查 Redis、查库、调 RPC，线程经常在等外部资源，这时线程数可以比 CPU 核数大一些。常见做法是结合等待时间和计算时间估一个初值，但 maximumPoolSize 不能只看本机，要同时受下游承载能力约束，队列也要保持有界，避免把请求拖成长尾。

如果是批处理或离线异步任务，我会更关注吞吐，并且强调和在线主链路隔离。它的队列可以比在线接口稍大，部分场景也可以考虑 CallerRunsPolicy 做反压，但前提是提交方线程不是主链路请求线程，不会影响核心服务。

所以总结下来就是：CPU 密集型重点保护 CPU，IO 密集型重点平衡等待时间和下游承载，批处理任务重点做隔离和吞吐控制。
```

### 3.4 为什么在线查询场景里更倾向 `AbortPolicy` 或自定义拒绝，而不是 `CallerRunsPolicy`

标准答法：

```text
CallerRunsPolicy 的优点是有天然反压，线程池满了以后，提交方线程自己执行任务，能够减缓任务继续堆积。

但在在线查询接口里，提交任务的往往就是 Web 请求线程。如果线程池满了，请求线程自己去执行异步任务，就会直接拉长接口 RT，占住请求处理线程，进一步导致更多请求排队、超时，把局部线程池过载放大成整个服务吞吐下降。

所以对在线查询接口，我更倾向 AbortPolicy 或自定义拒绝策略。这样能显式失败，便于上层做降级、限流、告警或兜底处理。
如果调用方不是主链路线程，而是后台任务线程，CallerRunsPolicy 才更适合。
```

### 3.5 `synchronized` 和 `ReentrantLock` 的核心区别，1 分钟怎么讲

标准答法：

```text
synchronized 和 ReentrantLock 都能实现互斥同步，也都可重入，并且都能保证可见性。

核心区别主要有三点。
第一，synchronized 是 JVM 层面的内置锁，语法简单；ReentrantLock 是显式锁，要手动 lock 和 unlock。
第二，synchronized 在异常退出时会自动释放锁；ReentrantLock 必须放在 try/finally 里释放，否则有死锁风险。
第三，ReentrantLock 的功能更强，支持 tryLock、可中断获取锁、公平锁和多个 Condition。

所以我的原则一般是：简单临界区优先 synchronized；需要更强控制能力时，用 ReentrantLock。
```

### 3.6 项目里怎么选 `synchronized` 和 `ReentrantLock`

标准答法：

```text
如果只是保护一小段简单临界区逻辑，比如更新内存状态、本地缓存读写保护，我会优先用 synchronized。它语法简单，异常时会自动释放锁，代码更稳。

如果场景对锁控制有更强需求，我会选 ReentrantLock。比如需要超时获取锁、响应中断、使用公平锁，或者需要多个 Condition 做条件等待和唤醒时，ReentrantLock 更适合。

所以我的原则是：简单互斥用 synchronized，复杂控制用 ReentrantLock。性能不是我主要的选型依据，场景匹配和可维护性更重要。
```
