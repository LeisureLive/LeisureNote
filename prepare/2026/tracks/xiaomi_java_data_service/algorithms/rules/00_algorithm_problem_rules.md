# 算法题出题规则、专题规划与答题框架

适用目标：

- 后续算法练习时，题目描述更清楚，减少来回补充约束
- 保证每道题都能自然进入“先识别题型，再写代码”的训练节奏
- 让题目、样例、限制条件、答题要求保持统一风格

## 1. 为什么需要这份规则

算法练习里最容易出现两个问题：

- 题目描述不完整：
  - 没说清输入输出
  - 没说清是否允许额外空间
  - 没说清时间复杂度要求
  - 没给样例，导致理解偏差
- 直接进入写代码：
  - 还没确认题型
  - 还没明确窗口、指针、状态定义
  - 还没说复杂度和边界

所以后续出题时，要先把题面描述清楚，再固定进入答题框架。

## 2. 强制出题结构

每道算法题至少要包含下面六部分。

### 2.1 题目标题

建议格式：

- `题号. 题目名`
- 如有必要，可补充题型标签

示例：

- `3. 无重复字符的最长子串`
- `209. 长度最小的子数组`

说明：

- 默认不要只说“做一道滑动窗口题”
- 题号和题名尽量一起给，方便后续复盘和检索

### 2.2 题目描述

要求：

- 说明输入是什么
- 说明输出是什么
- 说明目标是什么
- 描述尽量避免歧义

合格写法：

- 给定一个字符串 `s`，请你找出其中不含有重复字符的最长子串的长度。

不合格写法：

- 给你一个字符串，求最长不重复子串。

问题在于：

- 没明确输出的是“子串本身”还是“长度”

### 2.3 输入输出与方法签名

如果是 Java 练习，建议同步给出方法签名。

示例：

```java
public int lengthOfLongestSubstring(String s)
```

要求：

- 输入参数类型要明确
- 返回值类型要明确
- 如果题目要求原地修改数组，要明确返回什么

例如：

- 返回新长度
- 返回结果数组
- 直接修改原数组，不额外返回

补充要求：

- 出题模式下，默认只给方法骨架，不给具体实现
- 默认同时给两种形式：
  - 对话里提供可直接提交到 LeetCode 的 Java 方法骨架
  - 在本地目录 `code/algorithm/src/main/java/com/leisure/note/algorithm/weekN/dayN/` 初始化对应练习文件
- 本地练习文件需要和当天 week / day 对齐，便于你直接在仓库里继续写
- 如果当天目录不存在，先创建对应目录再初始化题目文件
- 题目文件里可以保留：
  - 类名
  - 方法签名
  - 题目描述
  - 样例
  - 复杂度要求
  - 答题框架
- 题目文件里默认不直接写完整解法
- 如果后续需要标准答案，单独在问答中提供，或放到单独的题解文件中

本地练习文件默认要求：

- 使用对应的 `package`
- 保留题目描述、样例、复杂度要求和答题框架
- 初始化可直接继续作答的方法骨架
- 如有必要，补最小 `main` 方法用于本地验证
- 以“可本地练习 + 可平移到 LeetCode”为目标，但默认不直接写完整解法

### 2.4 输入输出样例

要求：

- 视情况尽量提供 `1-2` 个具体样例
- 样例尽量覆盖典型情况，而不是只给最简单情况
- 如有必要，补一句样例解释

推荐格式：

```text
示例 1：
输入：s = "abcabcbb"
输出：3
解释：最长不含重复字符的子串是 "abc"，长度为 3。

示例 2：
输入：s = "bbbbb"
输出：1
解释：最长不含重复字符的子串是 "b"，长度为 1。
```

### 2.5 额外要求

这是最容易缺失、但最应该提前说明的部分。

至少要明确下面几项中的相关内容：

- 时间复杂度要求：
  - 如目标为 `O(n)`、`O(n log n)`，要明确写出
- 空间复杂度要求：
  - 如要求 `O(1)` 额外空间，要明确写出
- 是否允许使用额外空间：
  - 如允许哈希表、辅助数组、队列等
- 是否要求原地修改：
  - 如数组必须原地处理
- 是否允许暴力解：
  - 如不能接受 `O(n^2)` 枚举
- 是否保证有解：
  - 如默认至少存在一个解
- 下标约定：
  - 返回 `0-based` 还是 `1-based`
- 结果是否唯一：
  - 如果不唯一，返回任意一个还是全部返回

推荐写法：

```text
额外要求：
- 目标时间复杂度为 O(n)
- 允许使用额外空间
- 不接受暴力枚举所有子串的 O(n^2) 解法
```

### 2.6 答题顺序要求

出题时默认要说明答题顺序，避免一上来直接写代码。

默认流程：

1. 先复述题目要求和限制条件
2. 先判断题型
3. 先口头说思路
4. 再写代码
5. 最后补复杂度、边界和易错点

## 3. 默认要补清楚的约束清单

下面这些点，如果题目里有歧义，默认必须补清：

- 输入规模：
  - 是否需要按大输入规模选择线性或对数复杂度解法
- 输入元素范围：
  - 是否可能有负数、重复值、空字符串、空数组
- 字符范围：
  - 只包含小写字母，还是任意 ASCII / Unicode
- 返回结果形式：
  - 返回长度、下标、区间，还是实际内容
- 多解场景：
  - 返回任意一个，还是全部返回
- 有序性：
  - 输入是否已经有序
- 稳定性要求：
  - 结果是否要保持原相对顺序

## 4. 不建议的出题写法

后续出题时，尽量避免下面这些问题。

### 4.1 只给题型，不给题面

不建议：

- 做一道滑动窗口题

更合理：

- `3. 无重复字符的最长子串`
- 给定字符串 `s`，求不含重复字符的最长子串长度

### 4.2 不给样例

不建议：

- 请实现最优解

问题：

- 容易误解输出形式
- 不方便快速校验边界

### 4.3 不写复杂度要求

不建议：

- 求解这道题

问题：

- 容易写成暴力解
- 没法训练“看到题就先判断复杂度边界”的习惯

### 4.4 不写能否使用额外空间

问题：

- 同一题可能有：
  - 哈希表解法
  - 原地解法
  - 排序解法
- 如果不提前说明，很容易在答题中途反复调整方案

## 5. 默认答题框架

每道题默认按下面结构回答。

### 5.1 先确认题目要求

至少先确认：

- 输入是什么
- 输出是什么
- 复杂度要求是什么
- 是否允许额外空间

### 5.2 先识别题型

默认先说一句：

- 这题为什么更像滑动窗口 / 双指针 / 哈希 / 前缀和 / 二分

### 5.3 再说核心状态定义

根据题型说明：

- 指针怎么定义
- 窗口怎么定义
- 哈希表存什么
- 前缀和表示什么
- 队列 / 栈维护什么

### 5.4 再说转移或移动规则

例如：

- 什么时候扩张窗口
- 什么时候收缩窗口
- 什么时候更新答案
- 为什么这样移动不会漏解

### 5.5 再写代码

要求：

- 先保证思路和代码一致
- 再关注边界条件

### 5.6 最后补三件事

- 时间复杂度
- 空间复杂度
- 边界条件和易错点

## 6. 推荐出题模板

后续算法题默认优先按下面模板出。

```text
题目：
`3. 无重复字符的最长子串`

题目描述：
给定一个字符串 s，请你找出其中不含有重复字符的最长子串的长度。

方法签名：
public int lengthOfLongestSubstring(String s)

示例 1：
输入：s = "abcabcbb"
输出：3
解释：最长不含重复字符的子串是 "abc"，长度为 3。

示例 2：
输入：s = "bbbbb"
输出：1
解释：最长不含重复字符的子串是 "b"，长度为 1。

额外要求：
- 目标时间复杂度为 O(n)
- 允许使用额外空间
- 不接受 O(n^2) 的暴力枚举所有子串解法

答题顺序：
1. 先说这题为什么是滑动窗口
2. 再说窗口里维护什么状态
3. 再说什么时候扩张、什么时候收缩
4. 再写代码
5. 最后补复杂度、边界和易错点
```

## 7. 对当前训练节奏的建议

结合当前仓库和练习节奏，后续默认这样执行：

- 工作日每题都给清楚题面、样例和复杂度要求
- 默认先口头答，再写代码
- 如果当天目标是恢复手感，优先 1 道题讲清，不追求题量
- 如果题目本身就是模板题，优先训练：
  - 题型识别
  - 状态定义
  - 移动规则
  - 复杂度表达
- 如果你答题明显卡在题面不清楚，就先补出题规则，不继续硬做

### 7.1 解后注释沉淀规则

从现在开始，算法题在“写完代码并完成检查 / 复盘”后，默认还要补一层“解后注释”。

目的不是为了把题解写成长文，而是为了保证你以后回看本地代码时，能一眼看出：

- 这题属于哪个专题、哪种模式
- 当时为什么想到这个解法
- 这个解法的成立前提是什么
- 这题如果换成变体，应该往哪边调整

一句话要求：

- 代码写完只是“做出来”
- 补完解后注释，才算“能回顾、能复盘、能再讲”

### 7.2 触发时机

默认在下面这个时机补：

1. 题目已经完成实现
2. 已经过了最小验证：
   - 至少跑过 `main`、样例或最小编译检查
3. 已经做过一次检查 / 复盘：
   - 确认解法方向没问题
   - 确认自己知道这题属于什么模式

不要在下面这些时机急着补：

- 题目还没做出来
- 当前写法自己都不确定是否正确
- 还没想清楚这题到底属于什么模式

### 7.3 默认必须补的注释结构

后续本地 Java 练习文件里的已完成题，默认至少补下面这些块。

#### A. 本题在专题中的定位

至少写清：

- 属于哪个专题：
  - 数组 / 哈希 / 链表 / 栈队列 / 字符串 / ...
- 属于这个专题里的哪种模式：
  - 双指针 / 滑动窗口 / 前缀和 + 哈希 / 分组归类 / 快慢指针 / 单调栈 / ...

#### B. 题型特征

至少写清：

- 看到什么信号会想到这类题
- 这题和相邻模式的区别是什么

例如：

- 连续子数组 + 最短 + 全正数 -> 滑动窗口
- 求个数 + 区间和 + 可含负数 -> 前缀和 + 哈希计数
- 分组题 + 相同特征归类 -> 稳定 key + 分桶

#### C. 解题思路

至少写清：

- 核心状态是什么
- 主流程先做什么、后做什么
- 为什么复杂度能降下来

要求：

- 用“步骤 + 因果”写法
- 不要只写一句“使用哈希表求解”

#### D. 优点

至少写清：

- 这版解法为什么值得记
- 它在复杂度、表达直观性、模板价值里强在哪里

#### E. 缺点 / 适用前提

至少写清：

- 它依赖什么前提
- 它在哪些情况下不能直接套
- 有没有更优 / 更稳 / 更直观的替代方案

#### F. 变体应对 / 注意事项

至少写清一项：

- 如果题目条件变化，该往哪种模式调整
- 这题最容易错的边界是什么
- 这题和某道相似题最关键的差异是什么

### 7.4 多解法题的额外要求

如果同一题保留两种及以上解法，例如：

- 排序 key
- 频次数组签名 key

那默认要求：

- 每种解法单独写自己的“思路 + 优缺点 + 适用场景”
- 不要只在类注释里混着写一句“还有另一种做法”

这样后续你回看时，才能真正形成：

- 第一版怎么讲
- 追问优化时怎么讲
- 两种方案什么时候选谁

### 7.5 注释落点要求

默认放置位置：

- 单题单方法：
  - 放在方法注释上方
- 一个类里有多题：
  - 每个题对应的方法各写自己的注释
- 辅助方法：
  - 只补必要说明，例如 key 构造方式、状态含义

默认不建议：

- 只在类顶部写一段大而空的总结
- 方法里塞大量零碎行内注释代替结构化题解
- 注释内容和当前实现不一致

### 7.6 解后注释的最小完成标准

以后你让我“检查 / 复盘一题”时，除了代码本身，默认也要顺手检查这层注释是否达标。

一份合格的解后注释，至少应该让你以后回看代码时能快速回答：

1. 这题是什么模式
2. 为什么想到这个解法
3. 这版解法强在哪、弱在哪
4. 如果题目换条件，我应该往哪种模式切

## 9. 算法专题规划规则

后续整理算法专题时，默认不按“多刷题、多堆题”推进，而按“专题吃透”推进。

### 9.1 禁止平均刷题

每个专题的目标都不是：

- 机械累计题量
- 同一类题连续刷很多道相似题
- 看到专题就把常见题全列一遍

而应该是：

- 先抓这个专题最核心的 2-4 个模式
- 先抓最常见、最面试高频的题
- 让题目组合能覆盖这个专题的主线能力

### 9.2 专题正文默认写法

以后写算法专题文档时，不要再写成：

- 先堆很多零散知识点
- 再给一个过于抽象、几乎不能直接落地的“通用套路”
- 最后再列一堆题号

更推荐的方式是：

- 先把这一大类算法真正拆清
- 每个大类型只抓最关键的几个点
- 让读文档时可以直接回答：
  - 我什么时候该想到这个算法
  - 这个算法到底在维护什么
  - 这类题常见会分成哪几种
  - 每一种分别该怎么写

一句话要求：

- 专题文档不是算法百科，也不是题单罗列，而是“帮助识别题型并稳定落到套路”的训练文档。

### 9.3 每个大类型默认只写 3 件事

以后专题文档里，每个大类型默认优先按下面结构写：

1. 使用信号
2. 算法定义
3. 常见题型与对应套路

#### 9.3.1 使用信号

这里要回答的是：

- 什么题面特征会让我想到这一类算法
- 什么情况下它比暴力、普通双指针、普通哈希更自然

写法要求：

- 用 2-5 条最关键的题目信号说明即可
- 优先写“题目里会怎么问”
- 不要写成泛泛的定义介绍

例如：

- 单调栈：
  - 每个位置都要找左边 / 右边第一个更大 / 更小元素
  - 不想写暴力双重循环
- 滑动窗口：
  - 连续子数组 / 子串
  - 窗口内有合法性约束
- 二分：
  - 答案具有单调性
  - 题目不是直接找元素，而是在找最小可行值 / 最大可行值

#### 9.3.2 算法定义

这里要回答的是：

- 这个算法本质在维护什么
- 这个算法里的“状态 / 结构 / 不变量”到底是什么

写法要求：

- 只解释最核心的抽象，不要展开成大段泛理论
- 默认优先解释：
  - 指针 / 窗口 / 栈 / 队列 / 哈希表 / dp 数组 / 图状态
  - 它们各自存什么
  - 为什么这样维护

一句话要求：

- 定义不是为了下术语，而是为了让后面的套路有落点。

#### 9.3.3 常见题型与对应套路

这是后续专题文档最必须写清楚的一层。

要求：

- 不能只给一个过于抽象的“通用套路”
- 必须先把这一大类算法再抽成 2-4 个常见题型
- 然后针对每个题型分别给对应套路

例如：

- 单调栈不能只写：
  - 遍历当前元素，打破单调性就弹栈
- 更合理的写法应该拆成：
  - 左边第一个更大
  - 左边第一个更小
  - 右边第一个更大
  - 右边第一个更小
  - 分别说明：
    - 怎么遍历
    - 用什么单调栈
    - `while` 条件是什么

再例如：

- 滑动窗口不能只写：
  - 先扩张窗口，再收缩窗口
- 更合理的写法应该拆成：
  - 固定长度窗口
  - 变长窗口
  - 满足条件后求最长
  - 满足条件后求最短
  - 分别说明：
    - 什么时候扩
    - 什么时候缩
    - 什么时候更新答案

一句话要求：

- 专题文档要抽象题型，但不能停在抽象层，必须继续落到对应套路。

### 9.4 专题正文的默认粒度

每个大类型的内容，默认写到“能直接指导做题”即可，不继续无限展开。

推荐粒度：

- 使用信号：2-5 条
- 算法定义：1-2 段
- 常见题型：2-4 类
- 每类套路：说明
  - 怎么遍历
  - 维护什么状态 / 结构
  - 核心判断条件是什么

不建议：

- 一个大类型下再铺很多层级
- 先讲很长的理论，再到最后才给做题套路
- 明明题型差异很大，却只用一句统一模板带过

### 9.5 每个专题都必须有训练层次

默认至少拆成三层：

- 入门模板题：
  - 1-2 题
  - 目标是建立题型识别和基础模板
- 高频主战题：
  - 2-3 题
  - 目标是覆盖面试里最常出现、最值得讲透的问题
- 变体 / 验收题：
  - 1-2 题
  - 目标是验证你是否真的理解，而不是只会背模板

### 9.6 每道题都必须说明“为什么选它”

后续在专题文档里规划题目时，不能只列题号，至少要说明：

- 这题训练哪个模式
- 这题在专题里的角色是什么：
  - 模板题 / 高频题 / 验收题
- 做完它以后，应该额外掌握什么

一句话要求：

- 题目不是为了凑数量，而是为了覆盖专题能力

### 9.7 高频优先于全面

默认优先级：

1. 面试最高频
2. 最能代表专题主线
3. 最能训练表达和复杂度判断
4. 再考虑更偏技巧或更偏竞赛化的题

如果一个专题里有很多题思路高度相似，优先保留：

- 更高频的
- 更典型的
- 更容易作为面试模板讲清的

### 9.8 同一专题的题目不能思路过于雷同

同一专题里可以有“模板题 -> 变体题”的递进，但不能：

- 连续安排 2-3 道几乎同一个解法骨架的题
- 只换题面，不换核心思路
- 刷完以后还是只会一个套路

更合理的专题题目组合应尽量覆盖：

- 一个最基础模板
- 一个高频主战场景
- 一个容易出边界或变体的问题

### 9.9 每个专题都必须有验收标准

专题文档最后默认要有“验收标准”，至少检查这几件事：

- 我能不能识别这类题为什么属于这个专题
- 我能不能说清核心状态定义
- 我能不能解释复杂度为什么成立
- 我能不能讲出 1-2 个高频坑点
- 我能不能用 1 分钟到 3 分钟讲清代表题

如果以上几项做不到，就说明这个专题还没吃透，不应该急着扩展新题。

## 10. 出题前查重与相似度检查

为了避免后续重复出题、或者连续出到解法过于相似的题，默认增加下面这条门禁。

### 10.1 出题前必须先查三处

1. 题目台账与去重索引：
   - `prepare/2026/tracks/xiaomi_java_data_service/algorithms/problem_ledger_and_dedup_index.md`
2. 本地已有练习代码：
   - `code/algorithm/src/main/java/com/leisure/note/algorithm/week*/day*/`
3. 之前对话里刚刚安排过、刚刚做过的题

### 10.2 默认避免两类重复

- 完全重复：
  - 同一题号重复安排
- 强相似重复：
  - 题号不同，但核心思路、状态定义、移动规则几乎一样
  - 且放在相邻练习里没有明确的对比训练目的

### 10.3 什么时候允许“相似题”

只有下面两种情况允许刻意安排相似题：

- 这是“模板题 -> 变体题”的对照训练
- 这是间隔一段时间后的复盘回刷

但即使允许，也要明确说出：

- 这次为什么还要练它
- 它和上一题的区别是什么
- 这次重点比较哪个思路差异
