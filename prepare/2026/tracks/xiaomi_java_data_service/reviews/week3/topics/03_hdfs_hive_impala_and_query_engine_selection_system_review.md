# Week3 HDFS、Hive、Impala 与查询链路选型专题系统复习

对应计划：

## 1. 这一块在回答什么问题

这块真正要弄懂的，不是“HDFS、Hive、Impala、Doris 分别是什么”，而是下面这条连续问题：

1. 当数据规模、扫描范围和分析需求持续变大时，为什么不能继续把所有压力都压在 MySQL 这类 OLTP 存储上。
2. 如果海量数据要走分布式文件存储，HDFS 到底解决了什么问题，又明确没有解决什么问题。
3. 数据落到 HDFS 之后，为什么还必须有 Hive 这种表语义和元数据层，而不是直接对目录和文件做约定。
4. 既然已经有 Hive，为什么还会继续出现 Impala、Presto 这类查询执行引擎。
5. 如果业务进一步要求低延迟、高并发、服务化分析，为什么又会引入 Doris、Holo 这类一体化 OLAP 引擎。
6. 一条典型的数据存储与查询链路里，存储层、元数据层、查询层和分析服务层到底应该怎么分工。

所以这篇文档的主线不是做产品介绍，而是把一条典型查询链路拆清楚：

- HDFS 先解决海量数据低成本、可扩展、可容错地存下来。
- Hive 再把文件组织成表、分区和可治理的元数据体系。
- Impala / Presto 解决“这些表如何被交互式查询得更快”。
- Doris / Holo 解决“分析查询如何进一步走向低延迟、高并发和服务化”。
- 最后所有选型都回到数据规模、查询类型、更新方式、时延目标和成本结构。

## 2. 主线

1. 先看为什么 OLTP 系统扛不住大规模分析：问题不只是容量，更是扫描方式、并发模型和成本模型不匹配。
2. 再看 HDFS：它解决的是低成本、顺序读写、容错和并行处理的数据底座。
3. 再看 Hive：有了文件还不够，还需要表、分区、格式、元数据和仓库语义。
4. 再看 Impala / Presto：有了表之后，还需要更适合交互式查询的 MPP 执行引擎。
5. 再看 Doris / Holo：如果查询延迟和并发要求进一步提高，文件湖上的查询引擎未必是最终答案。
6. 最后看完整链路和选型边界：数据应该落在哪一层，查询应该走哪一层，为什么。

## 3. 核心问题展开

### 3.1 为什么大数据平台不能继续把所有分析压力都压在 MySQL 这类 OLTP 系统上

这一节在回答什么问题：

- MySQL 这类 OLTP 系统为什么不适合承接所有分析需求。
- 问题真正卡在什么地方。

这个问题是怎么解决的：

OLTP 系统的设计目标是高频更新、事务一致性、索引型点查和中小范围范围查询。它假设的数据访问模式通常是：

- 单条或少量记录更新
- 基于索引的过滤
- 较强一致性
- 较小结果集快速返回

而分析型查询的访问模式往往完全不同：

- 需要扫大量历史数据
- 经常伴随聚合、排序、多维过滤
- 单次查询会读取很多列或很多分区
- 结果虽然不一定大，但中间扫描量很大

这意味着两类系统的主矛盾不同：

1. OLTP 更在意
   - 单次事务延迟
   - 锁冲突
   - 索引命中
   - 更新一致性
2. OLAP / 数仓场景更在意
   - 大范围扫描成本
   - 聚合代价
   - 存储压缩率
   - 大批量并行处理能力

如果继续把所有分析压力压在 MySQL 上，常见问题通常会先暴露在下面几层：

1. 存储层
   - 历史数据越来越大，单机磁盘和单库维护成本不断上升。
2. 查询层
   - 复杂分析查询需要扫描大量行，索引型设计难以覆盖。
3. 资源层
   - 分析型大查询会和在线事务争抢 CPU、IO、Buffer Pool 和连接资源。
4. 架构层
   - 为了支持分析而不断加宽表、加索引、做复杂 ETL，会让 OLTP 系统越来越臃肿。

所以问题并不只是“数据太多了”，而是：

- 读写模型不匹配
- 存储组织方式不匹配
- 查询执行方式不匹配
- 成本结构不匹配

这也是为什么数据平台通常会把事务处理和分析处理分层，而不是试图让一个系统同时在所有维度都最优。

### 3.2 HDFS 到底解决了什么问题，为什么它适合做数据底座却不适合直接做查询系统

这一节在回答什么问题：

- HDFS 在整条链路里的真实定位是什么。
- 它为什么能存海量数据，却不能直接等同于数据库或查询系统。

这个问题是怎么解决的：

HDFS 解决的是海量数据在分布式环境下的存储、容错和顺序访问问题。它的核心目标不是支持细粒度随机更新，而是让大量数据能低成本地存下来，并且方便后续并行处理。

这套能力主要由下面几层机制支撑：

1. `NameNode`
   - 维护文件系统元数据。
   - 它知道目录结构、文件与 block 的映射关系、block 位于哪些 DataNode。
2. `DataNode`
   - 真正存储 block 数据。
3. `block`
   - 文件会被切分为较大的 block，分散存放在多个节点。
4. `replica`
   - block 会保留多副本，保证单节点损坏后仍然可读。

这套设计为什么适合大数据底座：

- 容量可以通过增加节点水平扩展。
- 计算任务可以尽量靠近数据，减少跨节点搬运。
- 单节点损坏不会导致整份数据不可用。
- 顺序读写和大文件场景下，吞吐通常更容易做高。

但 HDFS 明确没有解决下面几件事：

1. 高频随机更新
   - HDFS 更适合写入后多次读，不适合像数据库那样高频点更新。
2. 表语义和元数据治理
   - 它只知道文件和 block，不知道业务字段、分区规则和表结构。
3. 高效交互式查询
   - 它负责把数据存好，不负责把 SQL 查快。

理解 HDFS，最好顺着一次写入和一次读取来看：

#### 写入时它在做什么

- 客户端先向 `NameNode` 申请写文件。
- `NameNode` 返回 block 放置方案。
- 客户端把数据按 block 切分，写入多个 `DataNode`。
- 多副本链路保证某个节点写失败时仍有冗余。

这解决的是：

- 数据太大单机放不下
- 单点损坏会丢数据
- 后续并行计算需要把数据切散

#### 读取时它在做什么

- 客户端先向 `NameNode` 查询 block 位置。
- 再从对应 `DataNode` 读取 block。
- 如果某个副本不可用，可以切到其他副本。

这解决的是：

- 大文件如何在分布式环境下被稳定读出来
- 节点故障时如何继续读取

小文件问题也正是从这里来的：

- 每个文件、每个 block 都需要 `NameNode` 维护元数据。
- 小文件越多，元数据项越多，`NameNode` 内存和调度开销越大。
- 小文件越多，单次任务的文件打开、关闭、调度成本越容易超过真正的数据扫描成本。

所以 HDFS 不是“更大的硬盘”，而是围绕大文件、顺序 IO、容错和并行处理优化的分布式存储层。

### 3.3 数据已经落到 HDFS 了，为什么还必须有 Hive

这一节在回答什么问题：

- 为什么“数据存下来了”不等于“数据已经能被高效管理和使用”。
- Hive 在这条链路里补的是哪一层能力。

这个问题是怎么解决的：

如果只有 HDFS，系统里仍然只有目录和文件。业务方或查询引擎不知道：

- 这份数据有哪些字段
- 字段类型是什么
- 数据按什么维度切分
- 文件格式是什么
- 哪些目录对应哪张表、哪个分区、哪个业务口径

Hive 补的正是这层“文件之上的表语义和元数据组织”。

Hive 的关键能力通常有四层：

1. `Metastore`
   - 统一维护表结构、分区、存储位置、文件格式等元数据。
   - 上层查询引擎和任务系统可以围绕同一套元数据工作。
2. `Schema on read`
   - 数据写入时可以先保持底层文件灵活，读取时再按表定义解释。
   - 它让底层文件存储和上层表语义解耦，但也把一部分解释成本放到了读侧。
   - 可以把它和传统数据库常见的 `schema on write` 对照着理解：
     - 前者是先把数据落成文件，查询时再按 Hive 表结构解释。
     - 后者是写入时就先按表结构校验和落库。
   - 它的价值在于：
     - 文件存储和表语义可以解耦。
     - 上游写入链路更灵活。
     - schema 演进时，不一定每次都要重写历史文件。
   - 但它的代价也很明确：
     - 读侧要承担更多解释和兼容成本。
     - 真正的兼容边界还取决于文件格式、查询引擎和 metastore 定义。
3. `Partition`
   - 按日期、业务线、租户、事件类型等维度切目录。
   - 它的核心价值不是“目录好看”，而是减少不必要扫描。
4. `File format`
   - 常配合 ORC、Parquet 这类列式格式。
   - 列式格式更适合分析查询，因为它更容易做列裁剪、压缩和向量化读取。
   - ORC、Parquet 不是“把普通文件再压缩一下”，而是带有 schema、统计信息、编码和压缩能力的列式文件格式。
   - 它们的核心不是按“整行”存，而是按“列”分块存：
     - 行式更像一行一行连续存放。
     - 列式更像把 `user_id`、`city`、`age` 这类列分别组织和压缩。
   - 这类格式通常还会把文件继续分块，方便查询时跳过不需要的数据：
     - Parquet 常见结构是 `row group -> column chunk -> page`。
     - ORC 常见结构是 `stripe -> column streams / index / footer`。
   - 列式格式更适合分析查询，主要是因为它同时利用了三类能力：
     - 列裁剪：
       - 查询只读取真正需要的列，不必把整行所有字段都扫出来。
     - 压缩与编码：
       - 同一列的数据类型更一致、分布更相似，更容易做字典编码、RLE、delta 等优化，再配合压缩减少磁盘和网络开销。
     - 向量化读取：
       - 查询引擎不是一行一行解码和计算，而是按 batch 批量处理一列或多列数据，减少对象创建、函数调用和 CPU 开销。
   - 它们对嵌套数据也有支持：
     - 常见嵌套类型是 `struct`、`array`、`map`。
     - Parquet 不会简单把这类字段当成 JSON 字符串整块存进去，而是把叶子字段按列展开，再配合 `definition level` 和 `repetition level` 还原层级关系。
     - 其中 `definition level` 用来表示某个叶子值沿路径定义到了哪一层，因此可以区分“叶子字段自己是 null”还是“更上层的 struct / array 从某一层开始就没定义”。
     - `repetition level` 用来表示当前值是不是同一个重复字段的继续，因此主要用来还原数组或重复结构的边界。
   - ORC 和 Parquet 的选型通常不在于谁“绝对更快”，而在于：
     - Hive 数仓链路更重时，ORC 很常见。
     - Spark 或多引擎共享读取更常见时，Parquet 往往更通用。

Hive 为什么有价值，本质上是因为它把“文件存储”升级成“可治理的数仓表”：

- 调度系统可以围绕表和分区编排任务，而不是围绕路径字符串写脚本。
- 查询引擎可以统一读取表结构，而不是每个引擎各自猜字段。
- 数据治理可以围绕元数据展开，而不是只能围绕底层文件做弱约束。

Hive 的边界也必须说清：

1. Hive 不是天然的低延迟交互式查询引擎
   - 它解决的是仓库语义和元数据组织，不是直接把查询做到最快。
2. 分区不是越细越好
   - 分区切得过碎，会放大元数据管理成本和小文件问题。
3. 列式格式也不是对所有场景都最优
   - 它更适合大扫描和聚合，不是为高频小更新准备的。

所以 Hive 的价值不在于“能写 SQL”，而在于它把文件、表、分区、格式和元数据组织成同一套数仓基础层。

### 3.4 既然已经有 Hive，为什么还会继续出现 Impala 和 Presto

这一节在回答什么问题：

- Hive、Impala、Presto 为什么经常一起出现。
- 它们之间到底是替代关系还是分工关系。

这个问题是怎么解决的：

Hive 更偏“数据怎么组织成可治理的表”，Impala / Presto 更偏“这些表如何被更快地查出来”。它们之所以会一起出现，是因为表抽象和查询执行本来就是两层能力。

先把 Hive 自己在查询时的定位看清，会更容易理解为什么后面还会有 Impala / Presto。

Hive 虽然也能执行 SQL，但它传统上更像“把 SQL 翻译成批式或半批式执行任务”的入口：

- 早期最典型的是把 SQL 拆成一个或多个 MapReduce 任务。
- 后来底层执行引擎可以换成 Tez 或 Spark，不再一定是 MapReduce。
- 但它整体仍然更偏数仓、ETL、大扫描、大聚合，而不是天然为低延迟交互式查询设计。

这意味着 Hive 查 SQL 时，常见代价不是“不会并行”，而是：

- 查询启动和调度开销更重。
- 中间结果更容易沿着批处理路径被物化。
- 执行链更像任务 DAG，而不是一套常驻的交互式查询引擎。

当数据已经在 HDFS 或对象存储里，表结构也已经由 Hive Metastore 管理后，系统接下来要面对的问题就变成：

- 能不能更快完成交互式查询
- 能不能把查询执行做成 MPP 并行模式
- 能不能避免每次查询都沿着更偏离线的执行路径走

Impala / Presto 这类引擎的共同点是：

- 复用 Hive Metastore
- 复用底层文件存储
- 通过更偏查询优化和并行执行的引擎，提升交互式分析体验

它们为什么会有效，本质上是因为查询执行模型变了：

1. 查询被拆成多个并行阶段
   - 多个节点同时扫描和计算
2. 过滤、投影、聚合尽量前推
   - 尽早减少需要传输和聚合的数据量
3. 更强调内存和并行执行效率
   - 尽量缩短“从表到结果”的路径

可以先把它们抽象理解成 MPP 查询引擎：

- Coordinator 负责任务拆分和调度
- Worker 负责扫描、过滤、聚合和数据交换

这个模型为什么比传统离线执行更适合交互式查询：

- 多节点可以并行扫数
- 中间结果可以尽早裁剪
- 查询不必总依赖更重的批处理执行路径

如果继续往下压，Impala 查得更快，通常不只是因为“也会并行”，而是因为它把几类额外开销继续压下去了：

1. 常驻进程执行
   - `impalad` 本来就在集群里常驻，不需要像批处理引擎那样每次查询都重新走一轮重型任务启动路径。
2. 更偏内存和流水线执行
   - 扫描、过滤、聚合、Join 更强调边处理边传递，尽量减少中间结果落盘和反复物化。
3. 对列式格式的利用更直接
   - 能更充分利用列裁剪、分区裁剪、谓词下推、统计信息跳块、向量化读取这些能力。
4. 运行时优化更激进
   - 例如更积极地做 Join 策略选择、运行时过滤、基于统计信息的计划优化。

所以 Hive 和 Impala 的差异，不是“一个会执行 SQL，一个不会”，而是：

- Hive 更像“表语义 + 数仓执行入口”。
- Impala 更像“直接查这些表的低延迟 MPP 查询引擎”。

Impala 和 Presto 的侧重点又略有不同：

1. `Impala`
   - 更偏 Hadoop 生态内部的低延迟交互式查询。
   - 对 HDFS / Hive 表的结合通常更直接。
2. `Presto`
   - 更偏通用分布式 SQL 和联邦查询。
   - 除了查 Hive 表，也擅长把多个异构数据源统一成一个 SQL 入口。

这里还有一个非常容易被问混的点：

- “Impala 复用 Hive Metastore”并不等于“必须部署完整的 Hive 查询服务”。
- 很多生产环境里，看得到 Impala，但未必看得到 HiveServer2 或 Hive CLI。

更准确地说，Hive Metastore 这一层本质上承担的是：

- 表、字段、分区、存储位置、文件格式等元数据的权威管理。
- 底层元数据通常还要落到 MySQL / PostgreSQL 这类数据库里。

而 Impala 自己还会有一层查询侧的 catalog 能力：

- `catalogd` 负责加载、缓存、分发和失效同步元数据。
- `impalad` 更多是消费这份 catalog，并执行查询。
- `statestored` 更像集群里的状态总线：
  - 负责维护哪些 `impalad` / `catalogd` 还存活。
  - 负责把集群成员变化和部分 catalog 状态变化广播给各节点。

这三者如果放在一起理解，会更稳定：

- `impalad`
  - 执行 SQL、扫描数据、做 Join 和聚合。
- `catalogd`
  - 管理查询侧的 catalog，负责把元数据加载进来并分发出去。
- `statestored`
  - 负责维护集群视图和状态传播，让各个节点对“谁还活着、catalog 是否有变化”形成一致认知。

因此 `statestored` 不负责：

- 表结构的权威存储
- SQL 执行
- 替代 Hive Metastore 或 `catalogd`

它更像 Impala 集群里的协调中心，而不是元数据权威来源。

因此在很多平台化部署里，用户感知到的会是：

- 没有完整暴露 Hive 查询服务。
- 但底层元数据仍然是 HMS 兼容体系。
- Impala 通过 `catalogd` 把这层依赖封装起来，对查询侧提供更稳定的 metadata 服务。

所以更稳的理解不是“Impala 已经完全不需要 Hive Metastore”，而是：

- Impala 有自己的 catalog 管理能力。
- 但底层权威元数据来源，很多时候仍然是 Hive Metastore 体系或其兼容实现。

它们的边界同样很重要：

- 这类引擎通常更吃内存、网络和集群资源。
- 小文件过多时，查询调度和文件打开成本会非常明显。
- 它们更适合交互式分析，不一定适合超高并发、极低延迟的服务化查询。

所以 Hive 和 Impala / Presto 的关系不是“谁替掉谁”，而是：

- Hive 负责把数据组织成可治理的表
- Impala / Presto 负责更高效地执行查询

### 3.5 为什么有些场景最后还是会走到 Doris / Holo，而不是只停在 HDFS + Hive + Impala

这一节在回答什么问题：

- 文件湖上的查询引擎为什么还不够覆盖所有分析场景。
- Doris / Holo 这类一体化 OLAP 引擎到底补了什么能力。

这个问题是怎么解决的：

当业务的分析需求继续往下面走时，常见新要求会出现：

- 查询延迟要更低，最好秒级甚至更低
- 并发要更高，很多用户或服务同时查
- 查询模式更偏服务化，而不是人工探索式查询
- 数据更新要更频繁，结果要更快可见

这时问题已经不只是“能不能查到”，而是：

- 查询是否足够稳定
- 高并发下性能是否足够可控
- 数据导入、组织和查询是否可以被统一优化

Doris / Holo 这类一体化 OLAP 引擎的价值，通常就在于把存储组织、索引 / 排序、分布、复制和查询执行做得更一体化。它们相对文件湖查询引擎更偏：

- 服务化分析
- 低延迟聚合
- 更高并发
- 更强的查询体验稳定性

它们为什么会更适合这类场景：

1. 存储与查询协同设计
   - 数据如何分布、如何排序、如何压缩、如何被扫描，通常是一起设计的。
2. 更偏分析服务模型
   - 不只是“把文件查出来”，而是围绕分析查询本身做更深优化。
3. 数据导入和查询路径更短
   - 相比“先落文件、再注册表、再查询”，一体化引擎通常更强调更短的数据到查询路径。

这里还有一个很容易被误判成“其实和 Impala 差不多”的点，需要单独说清：

- 从“都在切分数据、都在并行、都在做副本”这个层面看，Impala 和 Doris 确实有相似性。
- 但真正的差异，不在于它们是否都做了分而治之，而在于这些结构到底属于谁、由谁持续维护。

对 Impala 来说，它更多是在利用底层已经存在的数据组织：

- 分区通常是 Hive 表层的逻辑组织，Impala 负责利用它做分区裁剪。
- `row group` 是 Parquet 文件内部的分块单位，Impala 更多是在读取时利用统计信息跳块。
- 多副本通常首先是 HDFS block 级别的容错和可用性机制，不是 Impala 自己定义的一套分析服务副本语义。

所以 Impala 更像：

- 在文件湖上尽量利用分区、文件格式、统计信息和 MPP 执行，把“现有文件”查快。

对 Doris 来说，这些结构更多是它自己在数据库层主动管理的：

- 表会继续拆成 partition、bucket、tablet。
- tablet 会有自己的 replica，并分布到多个节点。
- tablet 内部还会继续组织成 rowset、segment，并通过 compaction 维持更稳定的读路径。
- 数据写入时就会被放进更适合分析查询的物理组织，而不是等查询时再尽量适配。

因此有几组概念虽然表面相似，但不在同一层：

- Parquet `row group`
  - 是文件内部结构，更像“单个文件里的分块”。
- Doris `tablet`
  - 是表级、集群级的数据分片单元，承担分布式存储、并行调度、副本管理和恢复语义。

- HDFS replica
  - 更偏 block 级容错和数据可用性。
- Doris replica
  - 更偏分析型数据库里的分片副本、版本一致性和服务稳定性。

所以更稳的理解不是：

- Impala 和 Doris 本质一样，只是名字不同。

而是：

- Impala 更像“高性能文件查询引擎”。
- Doris 更像“面向分析服务的一体化 OLAP 数据库”。
- 前者更依赖外部文件布局质量，后者更强调由系统持续维护查询友好的数据形态。

它们的代价也要看到：

- 存储和计算成本可能更高。
- 不是所有冷数据都适合长期放在这类引擎里。
- 如果数据规模极大且大量是低频冷数据，单纯依赖一体化 OLAP 可能不如“湖仓 + 热层”的分层方案经济。

所以 Doris / Holo 往往不是替掉 HDFS / Hive，而是在链路里承担“热数据分析服务层”。

### 3.6 一条典型的数据存储与查询链路里，各层到底应该怎么分工

这一节在回答什么问题：

- 这些组件组合起来时，完整链路该怎么理解。
- 为什么不能只盯某一个组件做百科式解释。

这个问题是怎么解决的：

一条典型数据链路，通常会按“接入处理 -> 冷热分层存储 -> 元数据组织 -> 查询服务”来分层。

一种常见分工可以这样看：

1. 接入与处理层：`Kafka / Flink`
   - 负责事件接入、实时处理、清洗、聚合、路由。
2. 冷数据留存层：`HDFS`
   - 负责海量原始数据、明细数据、中间结果的低成本落地。
3. 元数据与数仓层：`Hive`
   - 负责把文件注册成表，组织分区、格式和仓库语义。
4. 交互式分析层：`Impala / Presto`
   - 负责对 Hive 表做交互式查询、探索式分析、临时报表分析。
5. 热数据分析服务层：`Doris / Holo`
   - 负责低延迟、高并发、服务化分析查询。

这种分层为什么常见，是因为每层解决的问题不同：

1. 存储层关心
   - 容量、成本、容错、留存周期
2. 元数据层关心
   - 表结构、分区、格式、统一管理
3. 查询层关心
   - 执行效率、并行扫描、交互体验
4. 服务层关心
   - 延迟、并发、稳定性、可服务化输出

如果把所有需求都压在单一系统上，边界很快会暴露：

- 全压 MySQL，扫描型分析和资源隔离会出问题。
- 全压 HDFS / Hive，交互延迟和服务能力会不足。
- 全压 Doris / Holo，超大规模冷数据留存和存储成本可能不划算。

所以更稳的思路通常不是“找一个最强系统”，而是先分清冷数据、热数据、批处理、交互分析和服务化查询各自应该落在哪一层。

### 3.7 查询引擎和存储系统到底该怎么选，才能不把选型题答成产品清单

这一节在回答什么问题：

- MySQL、Hive、Impala、Presto、Doris 这些系统到底该怎么选。
- 为什么选型题的关键不是产品介绍，而是约束判断。

这个问题是怎么解决的：

最稳定的选型维度通常不是系统名字，而是下面五件事：

1. 数据规模多大
   - 是百万级、亿级，还是更长历史留存。
2. 查询模式是什么
   - 点查、索引查、范围查，还是大扫描、聚合、多维分析。
3. 时延目标是什么
   - 毫秒级、秒级，还是分钟级。
4. 更新模式是什么
   - 高频更新、实时追加，还是批量导入。
5. 成本约束是什么
   - 能否接受更高存储成本换更低查询延迟，还是更重视冷数据低成本留存。

顺着这些维度去看，边界通常会清晰很多：

#### 适合继续留在 MySQL 的场景

- 事务状态管理
- 索引型在线查询
- 记录级更新频繁
- 数据量和扫描量仍在 OLTP 可控边界内

#### 适合走 HDFS + Hive 的场景

- 海量历史明细留存
- 离线处理和数仓沉淀
- 查询更偏批处理或离线分析
- 需要统一元数据和分区治理

#### 适合走 Impala / Presto 的场景

- 数据已经在 Hive 体系里
- 需要交互式分析查询
- 允许秒级或接近秒级响应
- 更偏人工分析、临时报表、跨源查询

#### 适合走 Doris / Holo 的场景

- 低延迟分析服务
- 较高查询并发
- 多维聚合频繁
- 结果需要更稳定地作为接口或产品能力输出

如果把这些场景放回真实问题里，判断会更稳：

- 做历史数仓明细沉淀，通常优先考虑 HDFS + Hive。
- 在 Hive 数仓上做运营分析、临时探索，通常更偏 Impala / Presto。
- 给业务系统提供高并发的低延迟分析接口，通常更偏 Doris / Holo。

所以选型题真正考的是：你能不能先说清约束，再解释为什么这些约束把系统推向某一类引擎，而不是把产品名字依次背一遍。

## 4. 边界、误区、常见混淆

- HDFS 是分布式存储底座，不是数据库，也不是查询引擎。
- Hive 的价值不只是“提供 SQL”，更重要的是元数据、表语义、分区和文件格式组织。
- `Schema on read` 的重点不是一句“读时解释”，而是要能说明它为什么让存储和表语义解耦，以及为什么这也会把一部分兼容成本推到读侧。
- Hive 和 Impala / Presto 不是简单替代关系，而是“表组织层”和“查询执行层”的分工关系。
- Hive 能执行 SQL，不代表它天然就是低延迟交互式查询引擎；它传统上更偏把 SQL 转成批式或半批式执行任务。
- Impala / Presto 查得快，不代表天然适合高并发、低延迟、服务化分析。
- “Impala 复用 Hive Metastore”不等于必须部署完整 Hive 查询服务；很多生产环境只暴露 Impala，但底层元数据仍然来自 HMS 兼容体系。
- `catalogd` 有 catalog 管理能力，不等于它完全取代了 Hive Metastore 作为元数据权威来源；更常见的是它承担查询侧的加载、缓存和分发。
- `statestored` 不是元数据存储服务，也不是 SQL 执行节点；它更偏集群状态协调和状态广播。
- Doris / Holo 更适合热数据分析服务，但不代表所有冷数据都应该长期放进去。
- 分区和列式格式不是万能优化；分区过碎会放大元数据和小文件成本，列式格式也更偏分析读而不是频繁更新。
- 选型题不要从产品介绍开始，而要先说数据规模、查询模式、时延目标、更新方式和成本约束。

## 5. 自测清单

- 你能不能说明为什么 OLTP 系统不适合承接所有分析压力，真正冲突的是哪些成本模型。
- 你能不能解释 HDFS 的 `NameNode`、`DataNode`、block、多副本分别在控制什么问题。
- 你能不能说清 HDFS 小文件问题本质上是在压哪一层成本。
- 你能不能解释 Hive 的 metastore、`schema on read`、partition、列式格式分别在这条链路里起什么作用。
- 你能不能说明 ORC / Parquet 这类列式格式为什么更适合分析场景，以及列裁剪、压缩、向量化读取分别在省哪一层成本。
- 你能不能解释 Hive 执行 SQL 时为什么更像批式或半批式任务，而 Impala 为什么更适合低延迟交互式查询。
- 你能不能说明 Hive、Impala、Presto 为什么会经常一起出现，而不是互相完全替代。
- 你能不能说清 `catalogd`、`impalad` 和 Hive Metastore 在元数据链路里分别承担什么角色。
- 你能不能说清 `statestored` 为什么不是表元数据服务，而是 Impala 集群里的状态协调和广播组件。
- 你能不能解释为什么有些分析场景会继续走到 Doris / Holo，而不只停在 HDFS + Hive + Impala。
- 你能不能把一条 `Kafka / Flink -> HDFS / Hive -> Impala / Doris` 的典型链路讲顺。
- 你能不能结合一个真实场景说明为什么这里更适合 Hive / Impala，或者更适合 Doris，而不是继续压在 MySQL 上。
