# 排障问答提纲

这份文档用于补“排障 / Linux / 网络 / 线上问题定位”这条主线。

目标不是背很多命令，而是形成资深岗常见排障题的稳定答法。

## 1. 通用答题骨架

排障题建议统一按这条骨架回答：

1. 先确认现象
   - 错误率、RT、吞吐、积压、资源指标，哪个先异常
2. 再判断范围
   - 全局问题、单机问题、单租户问题、单接口问题，还是单下游问题
3. 再沿链路拆
   - 入口、应用、缓存、数据库、MQ、网络、下游，逐层缩小
4. 再判断主瓶颈
   - CPU、内存、锁竞争、GC、慢 SQL、积压、网络超时、磁盘、热点
5. 最后做止损和修复
   - 降级、限流、隔离、扩容、回滚、补偿、监控补齐

## 2. 高频排障题

### 2.1 RT 突然升高

先答：

- 先看是不是全局 RT 抬升，还是某一类接口异常
- 再看资源指标：
  - CPU
  - 内存
  - GC
  - 线程池
  - 连接池
- 再看依赖层：
  - MySQL
  - Redis
  - Kafka
  - HTTP / RPC 下游

可继续展开：

- 如果 CPU 高：
  - 看热点方法、死循环、锁竞争、序列化开销
- 如果 GC 高：
  - 看对象创建过多、缓存失控、大结果集、批量处理不当
- 如果依赖变慢：
  - 看慢 SQL、热 key、下游超时、网络抖动

### 2.2 Kafka 积压

先答：

- 先确认是生产过快还是消费变慢
- 再看积压是否集中在某个 topic、partition、consumer group
- 再看消费端：
  - 线程池
  - 单条处理耗时
  - 下游依赖
  - rebalance

关键追问点：

- 如果只有少数 partition 积压，往往说明热点分区或 key 分布不均
- 如果整体消费都慢，重点看消费逻辑、下游依赖和资源瓶颈

### 2.3 慢 SQL

先答：

- 先确认是执行慢还是等待慢
- 看 `EXPLAIN`
- 看索引是否命中
- 看扫描行数、回表、排序、临时表、锁等待

继续展开：

- 如果 SQL 本身写法有问题，先改语句
- 如果是索引设计问题，补联合索引或改查询模型
- 如果是热点写竞争或大事务，考虑拆分、削峰或异步化

### 2.4 Redis 热 key / 大 key

先答：

- 热 key：
  - 单点访问过高，会把单机打热点
- 大 key：
  - 单次读写、删除、迁移都可能引发阻塞

继续展开：

- 热 key 的处理：
  - 本地缓存
  - 多副本分散
  - 读写削峰
  - 热点隔离
- 大 key 的处理：
  - 拆分结构
  - 渐进删除
  - 避免一次性大批量操作

### 2.5 Full GC 频繁

先答：

- 先看是不是业务流量上涨导致对象创建暴增
- 再看是不是内存配置、缓存策略、结果集大小、批处理方式有问题
- 再结合 GC 日志和堆分析看对象存活情况

重点不要答成：

- 只会背收集器名字

资深岗更看重的是：

- 你有没有清晰的排查链路
- 你能不能把 GC 问题和业务行为对应起来

## 3. Linux / 网络要会讲到什么程度

### Linux

至少要做到：

- 知道看什么，不是命令越多越好
- 能把命令和排查目标对应起来

最小集合：

- `top` / `htop`
- `ps`
- `vmstat`
- `iostat`
- `sar`
- `netstat` / `ss`
- `lsof`
- `tail` / `grep`

### 网络

最少要能讲清：

- TCP 连接建立和关闭的大致过程
- 长连接、短连接、连接池为什么影响性能
- 超时、重试、幂等为什么要一起考虑
- 网络抖动会怎样放大服务问题

## 4. 当前最值得优先准备的排障题

1. 慢 SQL 怎么排
2. Kafka 积压怎么排
3. Redis 热 key / 大 key 怎么排
4. Full GC 和 RT 抖动怎么排
5. 下游 HTTP / RPC 超时增多怎么排
6. 线程池打满怎么排

## 5. 使用建议

- 每周至少抽 2 个排障题口头答
- 答题时必须同时回答：
  - 先看什么
  - 怎么缩小范围
  - 怎么止损
  - 后续怎么避免再发生
