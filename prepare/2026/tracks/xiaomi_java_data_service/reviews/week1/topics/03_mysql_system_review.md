## 3. 核心重点展开

### 3.1 索引和 B+ 树

#### 1）为什么常用 B+ 树

要从“磁盘 IO 和范围查询”去理解：

- 树高低，减少磁盘 IO
- 非叶子节点只存键，单节点能容纳更多索引项
- 叶子节点有序，适合范围查询和排序

这题不要只背“因为快”，而要说出：

- 为什么不是普通二叉树
- 为什么不是只适合等值查询的哈希结构

#### 2）聚簇索引和二级索引要说清

- 聚簇索引：
  - 数据和主键索引组织在一起
- 二级索引：
  - 先按索引列找到主键
  - 再回主键索引取整行

所以回表的本质是：

- 先走二级索引
- 再走主键索引补齐数据

覆盖索引的本质是：

- 需要的列都在索引里
- 不必回表

#### 3）联合索引不要只背“最左前缀”

你要知道最左前缀为什么重要：

- 联合索引本质上按列顺序组织
- 查询条件是否能用上后续列，取决于前导列是否确定

还要知道后续哪些因素会影响利用效果：

- 范围查询
- 排序字段顺序
- 是否发生隐式类型转换
- 是否对列做函数操作
### 3.2 事务、隔离级别、MVCC

#### 1）隔离级别最少要建立什么感觉

你至少要能把这些关系串起来：

- 为什么需要事务隔离
- 脏读、不可重复读、幻读分别是什么
- MySQL InnoDB 默认隔离级别是什么

#### 2）MVCC 的本质

MVCC 的目标不是“完全不加锁”，而是：

- 在很多读场景下，减少读写冲突
- 让一致性读可以看到某个时间点上的数据版本

InnoDB 下常见的理解主线：

- 行记录隐藏字段
- undo log
- Read View

它解决的是：

- 普通读不必和写强烈互斥

#### 3）MVCC 和锁的关系不要说错

要避免把 MVCC 讲成“完全不需要锁”。

更准确的说法是：

- 快照读更多依赖 MVCC
- 当前读仍然会和锁有关
- MySQL 的并发控制不是只有 MVCC 一种手段
### 3.3 `explain` 和慢 SQL 排查

这块要形成稳定步骤，而不是只会说“加索引”。

推荐排查顺序：

1. 先定位具体 SQL：
   - 哪条慢
   - 多慢
   - 频率如何
   - 是否与某类参数或时间段相关
2. 再看执行计划：
   - `type`
   - `key`
   - `rows`
   - `extra`
3. 再判断具体原因：
   - 没命中索引
   - 联合索引没用上
   - 回表代价高
   - 排序 / 分页代价高
   - 临时表 / 文件排序
4. 再决定动作：
   - 调整索引
   - 改 SQL 写法
   - 限制返回量
   - 做缓存
   - 拆查询

#### 1）深分页为什么常慢

这题经常可以顺手补：

- 深分页本质上要跳过大量数据
- 即使只返回少量行，也可能扫描很多行

#### 2）为什么不能滥建索引

因为索引不是纯收益：

- 占空间
- 写入变慢
- 维护成本上升

所以索引题真正考的是“读写平衡”和“执行代价意识”。
## 4. 推荐表达顺序

如果面试官问 MySQL，推荐按下面顺序讲：

1. 先说“数据怎么高效查到”：
   - 索引、回表、覆盖索引、联合索引
2. 再说“并发事务怎么保证正确性和性能”：
   - 隔离级别、MVCC、锁
3. 最后说“线上为什么会慢”：
   - `explain`、扫描行数、索引命中、排序 / 分页代价

## 5. 自测清单

- 为什么 MySQL 常用 B+ 树而不是哈希索引做通用索引结构
- 回表和覆盖索引分别是什么，为什么性能差异明显
- 联合索引为什么不能只背“最左前缀”四个字
- MVCC 解决了什么问题，为什么不能说成“完全不加锁”
- `explain` 最值得优先看的字段有哪些
- 慢 SQL 排查为什么必须先定位 SQL，再看执行计划，再决定优化动作
