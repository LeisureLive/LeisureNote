## 2. Day2 必须掌握的知识框架

### 2.1 JMM

#### 1）JMM 是什么

- JMM 是 Java Memory Model，也就是 Java 内存模型
- 它不是 JVM 的堆、栈、方法区这些内存区域划分
- 它是 Java 在多线程环境下，对共享变量读写行为定义的一套规范

#### 2）为什么需要 JMM

- 线程执行时会有自己的工作内存
- 共享变量最终又存放在主内存
- 再加上编译器优化、CPU 缓存和指令重排序，多线程下就会出现“看不见”和“顺序不一致”的问题

#### 3）JMM 主要解决什么问题

- 可见性：一个线程改了值，另一个线程什么时候能看到
- 有序性：代码写在前面的操作，执行时会不会被重排序
- 原子性语义边界：哪些并发语义需要靠锁、CAS 或原子类来补足

#### 4）它和 `volatile` / `synchronized` 的关系

- `volatile`：主要提供可见性和一定的有序性约束
- `synchronized`：除了可见性和有序性，还通过互斥保证临界区原子性
- 所以 `volatile`、`synchronized`、`final`、原子类，本质上都和 JMM 规则有关

一句话总结：

- JMM 不是“内存长什么样”，而是“多线程下共享变量应该怎么被看见、怎么保持顺序、哪些语义需要额外同步手段保证”

### 2.2 `volatile`

#### 1）`volatile` 能保证什么

- 保证可见性
- 保证一定的有序性，禁止和 `volatile` 读写相关的关键重排序

#### 2）`volatile` 不能保证什么

- 不能保证复合操作的原子性
- 典型例子就是 `i++`

#### 3）为什么 `i++` 不能靠 `volatile`

`i++` 不是一个原子操作，而是三步：

1. 读当前值
2. 加一
3. 写回新值

问题在于：

- `volatile` 只能保证读到的是较新值，写出去的值别人能较快看到
- 但不能把“读、改、写”三个步骤变成一个不可分割的整体

经典冲突例子：

- 线程 A 读到 `i = 1`
- 线程 B 也读到 `i = 1`
- 两边都加一后再写回 `2`
- 最终结果变成 `2`，而不是预期的 `3`

这里最容易说错的一点：

- 不要把 `volatile` 的有序性说成“CPU 一定按源码顺序逐条执行”
- 更准确的说法是：`volatile` 在内存语义上禁止某些会破坏并发语义的重排序

#### 4）什么场景适合用 `volatile`

- 状态标记位，比如停止线程、开关控制
- 一个线程写、多个线程读，且写操作本身不依赖旧值
- 双重检查单例里的实例引用

#### 5）什么场景不适合用 `volatile`

- `i++`、计数器递增这类读改写场景
- 多个变量要一起维护一致性
- 多步业务逻辑必须整体互斥执行的场景

更合适的替代：

- 单变量原子更新：`AtomicInteger`、`AtomicLong`
- 高并发统计：`LongAdder`
- 临界区互斥：`synchronized`、`Lock`

#### 6）双重检查单例为什么必须加 `volatile`

这题今天是明显不熟的点，要重点补。

对象创建可以粗略理解成三步：

1. 分配内存
2. 初始化对象
3. 把引用赋值给变量

如果没有 `volatile`，可能发生关键重排序：

- `1 -> 3 -> 2`

这样会导致：

- 线程 A 先把引用赋值给 `instance`
- 线程 B 看到 `instance != null`，直接返回对象
- 但这个对象其实还没初始化完成

所以这里加 `volatile` 的核心不是“加锁”，而是：

- 禁止实例发布过程中的关键重排序
- 保证一个线程看到实例非空时，这个对象已经完成初始化

一句话总结：

- DCL 单例里的 `volatile`，是为了防止“引用先发布、对象后初始化”

### 2.3 CAS

#### 1）CAS 是什么

- CAS 是 Compare-And-Set，也就是比较并交换
- 核心思路是：更新时先看当前值是不是预期值，如果是才更新，不是就失败或重试

#### 2）为什么 CAS 不加锁也能做并发更新

- 底层依赖的是原子性的“比较 + 更新”操作
- 它不是普通的先 `get` 再 `set`
- 所以不需要把整个临界区都锁住

#### 3）为什么说它是乐观并发控制

- 它默认冲突不会太高，先直接尝试更新
- 如果失败，再重试
- 和悲观锁“先上锁、再操作”的思路不同

#### 4）CAS 的典型问题

- 高竞争下会频繁自旋，浪费 CPU
- 更适合单共享变量或单更新点，不适合复杂临界区一致性控制
- 会有 ABA 问题

#### 5）什么是 ABA

- 当前值一开始是 `A`
- 期间被其他线程改成了 `B`
- 后来又改回 `A`
- 当前线程做 CAS 时只看到“值还是 A”，误以为中间没人改过

#### 6）ABA 怎么处理

- 给值再配一个版本号
- 每次更新时版本一起变
- Java 里常见做法是 `AtomicStampedReference`

一句话总结：

- CAS 的本质是“先比较，再更新；不锁整个过程”，但它只能看当前值，默认看不到中间历史

### 2.4 原子类：`AtomicLong` 和 `LongAdder`

这部分是今天深问过、也比较不熟的点，要记得比普通知识点更细。

#### 1）`AtomicLong` 适合什么

- 维护单点共享值
- 支持 `incrementAndGet`、`addAndGet`、`compareAndSet`
- 适合业务控制值、版本号、序列号、需要 CAS 语义的场景

#### 2）`LongAdder` 为什么更适合高并发统计

不要把它理解成“还是只有一个值，只是快一点”。

它内部更接近：

- 一个公共值 `base`
- 再加一个分散热点用的 `Cell[]`

更新时的思路：

- 低竞争时，先尝试 CAS 更新 `base`
- 竞争变高时，把不同线程分散到不同 `Cell` 上做局部累加
- 最后读值时再做汇总

汇总方式：

- `sum = base + 所有 Cell 的值`

它快的原因不是“算法更神奇”，而是：

- 把原来所有线程竞争一个热点值
- 变成多个线程尽量分散到不同槽位更新
- 从而降低 CAS 冲突

#### 3）`LongAdder` 常用方法怎么理解

- `increment()`：`+1`
- `add(x)`：加上一个值
- `sum()`：读取当前总和
- `longValue()`：对 `LongAdder` 来说，本质上也是把当前累计值按 `long` 取出来，日常业务里几乎可看成等价于 `sum()`
- `sumThenReset()`：读取后清零，适合做周期性统计

实战建议：

- 自己写业务代码时，优先用 `sum()`
- 做监控窗口统计时，优先用 `sumThenReset()`

#### 4）`AtomicLong` 和 `LongAdder` 的边界怎么选

`AtomicLong`：

- 适合业务控制值
- 适合需要 `compareAndSet`
- 适合需要更直接原子语义的场景

`LongAdder`：

- 适合高并发统计值
- 比如 QPS、请求数、命中次数、监控指标
- 不适合账户余额、库存扣减这类业务控制值

一句话总结：

- 业务控制值选 `AtomicLong`
- 高并发统计值选 `LongAdder`

### 2.5 AQS

#### 1）AQS 是什么

- AQS 是 AbstractQueuedSynchronizer
- 可以理解为 Java 并发包里构建锁和同步器的基础框架
- 它本身不是具体的锁，而是把同步控制中的通用逻辑抽象出来了

#### 2）AQS 的核心结构

- 一个 `state` 状态值
- 一个等待队列
- 一套获取和释放同步状态的通用流程

核心过程：

- 线程尝试获取同步状态
- 如果失败，就进入等待队列
- 前驱线程释放后，再唤醒后继节点继续竞争

#### 3）为什么 `ReentrantLock` 和 `CountDownLatch` 都能基于 AQS

这题今天答过，但“共享模式”和“锁竞争”还容易混。

它们的共性不是“都是锁”，而是：

- 都可以抽象成“基于一个 `state` 值判断线程能不能继续执行”

在不同工具里，`state` 的含义不同：

- `ReentrantLock`：表示锁占用情况和重入次数
- `CountDownLatch`：表示剩余计数

所以 AQS 的通用性来自于：

- 状态管理
- 失败排队
- 阻塞唤醒
- 独占 / 共享两套获取模式

#### 4）独占模式和共享模式怎么区分

这部分是今天答得不够稳的点，要重点补。

独占模式：

- 同一时刻通常只允许一个线程成功获取同步状态
- 其他线程失败后进入等待队列
- 典型例子：`ReentrantLock`

共享模式：

- 条件满足时，可以允许多个线程通过
- 或者一个线程释放后，可能有多个等待线程继续执行
- 典型例子：`CountDownLatch`、`Semaphore`

这里最容易说错的是：

- 不要把共享模式说成“多个线程共同更新 `state`”
- 更准确的说法是：多个线程共享“可以继续执行”的条件，或者共享获取同步状态的机会

一句话总结：

- 独占模式更像“一个线程拿到，其他线程等”
- 共享模式更像“条件满足后，多线程都可以继续通过”

### 2.6 `ConcurrentHashMap`

#### 1）为什么不能直接用 `HashMap`

- `HashMap` 线程不安全
- 并发读写、并发扩容时可能出现数据覆盖或不一致结果
- 早期 JDK 下还可能出现结构异常问题

#### 2）为什么不优先用 `Hashtable`

- 它虽然线程安全
- 但同步粒度太粗
- 高并发场景下吞吐会比较差

#### 3）`ConcurrentHashMap` 解决了什么

- 在保证并发安全的前提下，尽量减少锁竞争
- 比 `Hashtable` 更适合高并发读写场景

版本差异知道到这个程度即可：

- JDK 7：分段锁思路
- JDK 8：主要是 CAS + `synchronized` 配合桶级控制

#### 4）它的使用边界是什么

这是今天需要主动补的一句：

- `ConcurrentHashMap` 主要保证单次并发读写操作安全
- 不代表所有复合业务操作都天然线程安全

典型例子：

- 先 `get` 判断不存在
- 再 `put` 新值

这两个步骤之间仍然可能被别的线程插进来。

更稳的做法：

- 用 `putIfAbsent`
- 用 `computeIfAbsent`
- 或者按场景额外加同步控制

## 3. 高频面试题标准答法

### 3.1 JMM 是什么，它解决什么问题

标准答法：

```text
JMM 是 Java Memory Model，也就是 Java 内存模型。它不是 JVM 的堆、栈、方法区这些运行时内存区域划分，而是 Java 在多线程环境下，对共享变量读写行为定义的一套规范。

之所以需要 JMM，是因为线程执行时会有自己的工作内存，再加上 CPU 缓存、编译器优化和指令重排序，多线程下就可能出现一个线程改了值，另一个线程看不到，或者执行顺序和代码表面顺序不一致。

所以 JMM 主要关注三类问题：可见性、有序性，以及并发场景下的原子性语义边界。

像 volatile、synchronized 这些并发手段，本质上都是在 JMM 规则下提供不同程度的保证。
```

### 3.2 `volatile` 能保证什么，为什么不能保证 `i++`

标准答法：

```text
volatile 主要保证两件事。

第一是可见性。一个线程修改了被 volatile 修饰的变量，其他线程能够较快看到最新值。

第二是一定的有序性约束。它会禁止和 volatile 读写相关的关键重排序，所以在对象发布、状态切换这类场景里很重要。

但 volatile 不能保证复合操作的原子性。比如 i++ 其实是读、加一、写回三个步骤组成的复合操作，两个线程可能同时读到同一个旧值，再各自写回，最终发生更新丢失。

所以我的理解是，volatile 更适合状态标记、配置开关和一写多读场景，不适合替代锁去保护复杂临界区。
```

### 3.2.1 双重检查单例为什么必须加 `volatile`

标准答法：

```text
双重检查单例里给实例对象加 volatile，核心原因是对象创建不是绝对单步操作，它大致可以拆成分配内存、初始化对象、把引用赋值给变量三个步骤。

如果没有 volatile，这几个步骤可能发生关键重排序，比如先把引用赋值出去，再执行初始化。这样另一个线程看到实例已经非空，就会直接返回这个对象，但这个对象实际上可能还没初始化完成。

所以这里加 volatile，不是为了加锁，而是为了禁止这种关键重排序，保证一个线程看到实例非空时，这个对象已经完成初始化。
```

### 3.3 CAS 是什么，有什么问题

标准答法：

```text
CAS 是 Compare-And-Set，也就是比较并交换。它的核心是更新数据时先比较当前内存中的值是不是预期值，如果是才更新，不是就失败或重试。

它之所以能在不加锁的情况下实现并发更新，本质上是因为底层依赖的是原子性的比较加更新操作，而不是普通的先读后写。

我一般把它理解成一种乐观并发控制方式，也就是默认冲突不会太高，先尝试直接更新，失败了再重试。

它的典型问题有三个：高竞争下会频繁自旋、只适合单更新点、还有 ABA 问题。如果业务里需要识别中间是否被改过，可以引入版本号，比如用 AtomicStampedReference。
```

### 3.3.1 `AtomicLong` 和 `LongAdder` 怎么选

标准答法：

```text
AtomicLong 和 LongAdder 都能做并发计数，但使用边界不一样。

如果这个值本身是业务控制值，或者我需要 compareAndSet、incrementAndGet 这类明确的原子语义，我会优先选 AtomicLong。因为它维护的是一个单点值，语义更直接。

如果这个值只是统计类指标，比如请求数、QPS、命中次数，而且并发更新非常高，我会优先选 LongAdder。因为它会把更新压力分散到 base 和多个 Cell 上，最后读取时再汇总，总体上比 AtomicLong 更抗竞争。

所以我的原则是：业务控制值选 AtomicLong，高并发统计值选 LongAdder。
```

### 3.4 AQS 是什么

标准答法：

```text
AQS 是 AbstractQueuedSynchronizer，可以理解为 Java 并发包里构建锁和同步器的基础框架。它本身不是具体的锁，而是把同步控制里的通用能力抽象出来了。

它的核心包括一个 state 状态值，以及一个等待队列。线程获取同步状态失败后，会进入队列等待；持有资源的线程释放后，再唤醒后续线程继续竞争。

像 ReentrantLock、CountDownLatch、Semaphore、ReentrantReadWriteLock 这类并发工具，都可以基于 AQS 去实现各自的同步语义。
```

### 3.4.1 为什么 `ReentrantLock` 和 `CountDownLatch` 都能基于 AQS

标准答法：

```text
虽然 ReentrantLock 和 CountDownLatch 用途不一样，但它们都可以抽象成“基于一个状态值来决定线程能不能继续执行”。

在 ReentrantLock 里，state 表示锁的占用和重入次数，线程获取不到锁就进入等待队列，这是独占模式。

在 CountDownLatch 里，state 表示剩余计数，调用 await 的线程会等待这个计数减到 0，再一起继续执行，这是共享模式下的条件等待。

所以 AQS 之所以通用，是因为它把状态管理、失败排队、阻塞唤醒，以及独占和共享两种获取模式统一抽象出来了。
```

### 3.5 为什么并发场景下通常优先用 `ConcurrentHashMap`

标准答法：

```text
HashMap 本身线程不安全，在并发读写尤其是并发扩容时，可能出现数据覆盖、读到不一致结果，早期 JDK 下甚至可能出现结构异常问题。

Hashtable 虽然线程安全，但同步粒度比较粗，高并发下吞吐会比较差。

ConcurrentHashMap 的目标是在保证并发安全的前提下，尽量减少锁竞争，所以它在性能和线程安全之间做了更好的平衡。

不过它主要保证单次并发读写操作安全，不代表所有复合业务操作都天然线程安全。像先判断再更新这类场景，仍然要用 putIfAbsent、computeIfAbsent 或额外同步控制。
```

## 6. 常见失分点

- 把 JMM 说成 JVM 内存区域模型
- 只会说 `volatile` 保证可见性，不会补“一定的有序性约束”
- 说 `volatile` 能保证 `i++` 线程安全
- 把 `volatile` 的有序性说成“CPU 会严格按源码顺序执行”
- 只知道 DCL 要加 `volatile`，但解释不出为什么
- 只会背 CAS 很快，不会说高竞争自旋和 ABA
- 知道 `LongAdder` 更适合高并发，却解释不出为什么
- 把 AQS 的所有实现都当成互斥锁
- 说 `CountDownLatch` 的 `state` 是“锁数量”
- 说 `ConcurrentHashMap` 线程安全，所以所有 `get` + `put` 组合也天然安全
