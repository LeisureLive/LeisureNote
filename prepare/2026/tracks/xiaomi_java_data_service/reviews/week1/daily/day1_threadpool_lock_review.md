## 4. 重点与失分点拆解

### 4.1 线程池参数配置还停留在口号层

- 现象：
  - 能说“IO 密集型线程数可以大一点”
  - 但给真实场景后，还不能顺着指标把初始配置讲完整
- 原因：
  - 参数和容量判断没有串成固定顺序
  - 还没有把“线程池不是万能扩容手段”变成第一反应
- 后续复盘重点：
  - 固定答题顺序：
    - 先算平均并发
    - 再看 CPU 是否扛得住
    - 再看下游稳定承载能力
    - 然后给保护型初始配置
    - 最后补监控和调参路径
- 下次建议直接这样答：
  - 先判断任务类型：
    - CPU 密集
    - IO 密集
  - 再给估算抓手：
    - 平均并发约等于 `TPS x RT`
    - CPU 核需求约等于 `TPS x 单请求 CPU 时间`
    - 线程数可参考 `CPU 核数 x (1 + 等待时间 / 计算时间)`
  - 再补工程约束：
    - 下游扛不扛得住
    - 队列不能无界
    - 拒绝策略要和业务失败策略匹配
  - 最后给一个保护性结论：
    - 线程池配置不是拍脑袋给数字
    - 也不能把调大线程数当成万能解法
  - 如果要压成 1 分钟版本，至少说到这 5 个点：
    - 任务类型
    - 并发估算
    - CPU 是否够
    - 下游承载
    - 监控调参

### 4.2 `Executors` 风险还不够具体

- 现象：
  - 知道生产不推荐直接用 `Executors`
  - 但容易把所有风险混成一句“可能 OOM”
- 原因：
  - 没把不同工厂方法的风险拆开
- 后续复盘重点：
  - 明确拆成两类：
    - `newFixedThreadPool` / `newSingleThreadExecutor`：
      - 主要问题是无界队列
    - `newCachedThreadPool`：
      - 主要问题是线程数快速膨胀
- 下次建议直接这样答：
  - 不建议直接用 `Executors`，核心原因不是“源码不优雅”，而是资源边界不可控
  - `newFixedThreadPool` / `newSingleThreadExecutor`：
    - 风险是无界队列
    - 任务堆积后会带来内存上涨、延迟失控，严重时 OOM
  - `newCachedThreadPool`：
    - 风险是线程数可能快速膨胀
    - 会带来 CPU 抖动、上下文切换增多、系统不稳定
  - 更稳的做法是显式创建 `ThreadPoolExecutor`
  - 至少要把：
    - 线程数
    - 队列
    - 拒绝策略
    控制在可预期范围内

### 4.3 锁选型容易变成特性背诵

- 现象：
  - 能列出 `ReentrantLock` 支持可中断、超时、公平锁、多个 `Condition`
  - 但项目里如何选还不够自然
- 原因：
  - 缺少“简单互斥 vs 复杂并发控制”的稳定切分
- 后续复盘重点：
  - 固定一句话原则：
    - 简单互斥优先 `synchronized`
    - 复杂控制优先 `ReentrantLock`
- 下次建议直接这样答：
  - 如果只是保护一小段简单临界区，我优先用 `synchronized`
  - 原因是：
    - 语法简单
    - 异常时自动释放锁
    - 可维护性更好
  - 如果需要更复杂的锁控制，我会选 `ReentrantLock`
  - 典型场景包括：
    - 可中断获取锁
    - 超时获取锁
    - 公平锁
    - 多个 `Condition`
  - 收口时固定说一句：
    - 简单互斥看稳不看花
    - 复杂并发控制再上 `ReentrantLock`
