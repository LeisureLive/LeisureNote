## 3. 核心重点展开

### 3.1 GC 基础和常见收集器

#### 1）GC 这块真正要建立的主线

GC 不要学成“收集器名字大全”，而要按一条因果链去理解：

1. 程序不断创建对象，为什么必须回收
2. JVM 怎么判断对象还能不能继续活着
3. 为什么不同生命周期的对象要分区域管理
4. JVM 回收对象时常见的基本算法是什么
5. 不同收集器是在什么目标下做不同取舍
6. 当回收跟不上时，会出现什么线上症状

如果这条主线顺了，GC 这一块才算进入“能系统复习”的状态。

#### 2）为什么需要 GC

因为 Java 程序运行时会不断创建对象，而对象生命周期差异极大：

- 有的只活一个方法调用
- 有的活一个请求周期
- 有的会被缓存、队列、上下文引用很久

如果不回收那些已经不再被使用的对象：

- 堆空间会被不断吃掉
- 内存占用持续抬升
- 分配新对象越来越困难
- 最终可能触发频繁 GC、长停顿甚至 OOM

所以 GC 的本质是：

- 自动回收“程序以后不再需要”的对象空间

但它不是免费能力，因为回收本身也会消耗 CPU、带来停顿、影响吞吐。

#### 3）JVM 怎么判断对象是否还活着

现代 JVM 更核心的判断方式不是引用计数，而是可达性分析。

##### 为什么不主要使用引用计数

引用计数的思路很直观：

- 一个对象被引用一次，计数加一
- 引用失效一次，计数减一
- 计数为零就可回收

但它有一个经典问题：

- 循环引用对象即使业务上已经没用了，计数也可能不为零

例如两个对象互相引用，但外部已经没人能访问它们：

- 用引用计数很难直接判定它们可回收
- 用可达性分析则可以从根出发，发现它们整体已经不可达

所以 JVM 更核心的是：

- 从一组 GC Roots 出发做可达性分析

##### 可达性分析到底在做什么

基本思路是：

- 从一组“根对象”开始沿引用链往下找
- 能找到的对象，说明当前仍然可达
- 找不到的对象，说明已经不可达，后续可被回收

第一轮复习时，对 GC Roots 至少有这个感觉：

- 线程栈里引用到的对象
- 类静态字段引用到的对象
- 常量引用到的对象
- JNI 等本地方法持有的对象

这部分最重要的结论是：

- Java 里的“内存泄漏”很多时候不是对象业务上没用了
- 而是对象仍然能从 GC Roots 沿引用链找到，所以 GC 不会回收

#### 4）GC 不是一发现不可达就立刻物理删除

这块第一轮不用钻到所有细节，但要知道：

- GC 判断“对象是否可回收”
- 和“对象空间什么时候真正被回收”
- 是两个相关但不完全等同的步骤

对面试来说，先抓住主线就够：

- 可达性分析负责判断对象存活性
- 收集算法和收集器负责真正回收空间

#### 5）为什么堆里要分代或分区域管理

因为对象生命周期非常不均匀。

经验上通常是：

- 大量对象生得快、死得也快
- 少量对象会活得比较久

如果所有对象都放在一块堆里，每次回收都统一处理：

- 扫描成本会很高
- 回收效率会很差

所以 JVM 会倾向按对象生命周期做区域化管理。

第一轮至少建立下面这个结构：

- 年轻代：
  - 主要放新创建、短生命周期对象
- 老年代：
  - 主要放存活较久、晋升上来的对象

它的目的不是“分区好看”，而是：

- 让不同生命周期对象用不同回收策略
- 让大部分短命对象快速被清掉
- 减少每次 GC 的整体成本

#### 6）年轻代为什么通常回收得更频繁

因为新对象通常先进入年轻代，而年轻代里短命对象非常多。

所以年轻代回收的典型特点是：

- 发生更频繁
- 单次处理对象更多但大多数都能很快被回收

这也是为什么很多业务服务里看到的 GC，大量都是先发生在年轻代。

#### 7）对象为什么会晋升到老年代

对象晋升是连接“正常对象生命周期”和“老年代压力”的关键桥梁。

主线先记这几个点：

- 新对象一般先进入年轻代
- 经过多次回收仍然存活的对象，可能会晋升到老年代
- 某些大对象也可能更早进入老年代
- Survivor 空间承载不下时，也可能触发更早晋升

为什么这题重要：

- 老年代增长过快，往往比年轻代回收频繁更危险
- 很多 Full GC 问题本质上都和对象晋升、长生命周期对象、缓存膨胀有关

#### 8）GC 常见基本算法为什么要知道

这一层是很多人复习 GC 时缺掉的中间层。

收集器不是凭空工作的，底层一定依赖某种回收算法思路。第一轮至少知道下面三类：

##### 标记-清除

思路：

- 先标记存活对象
- 再清掉未标记对象

优点：

- 思路直接

缺点：

- 可能留下内存碎片

##### 复制算法

思路：

- 把存活对象复制到另一块连续空间
- 然后整体清空原区域

优点：

- 回收后空间规整
- 适合“死得快、活得少”的场景

缺点：

- 需要额外空间做复制

这也是为什么它很适合年轻代：

- 年轻代里大量对象很快就死掉，真正需要复制的存活对象通常不多

##### 标记-整理

思路：

- 先标记存活对象
- 再把存活对象往一端移动，整理出连续空间

优点：

- 既能回收无用对象，又能减少碎片

代价：

- 整理过程更重

所以你可以把三种算法先粗略对应起来：

- 短命对象多的区域，适合偏复制思路
- 长生命周期对象多的区域，更常考虑碎片和整理问题

#### 9）Stop-The-World 为什么会被频繁提到

GC 题里经常会听到 STW，也就是 `Stop-The-World`。

它的直观含义是：

- 在某些 GC 阶段，应用线程需要暂停
- JVM 先完成某些关键回收动作，再恢复业务线程

为什么要知道这个概念：

- 因为 GC 对业务最直接的影响，经常不是“有没有回收成功”
- 而是“停顿有多长，RT 抖动有多明显”

所以面试里说 GC，不要只说内存回收，还要有“停顿成本”的意识。

#### 10）Minor GC、Major GC、Full GC 应该怎么理解

不要把这几个词死背成唯一标准定义，因为不同资料和实现里表述会有差异。

第一轮更稳的理解方式是：

- Minor GC：
  - 主要针对年轻代
  - 往往更频繁、单次相对轻
- Major GC：
  - 常用来指老年代回收
  - 但术语口径可能不完全一致
- Full GC：
  - 往往是更重、更全局的回收动作
  - 停顿和影响通常更值得警惕

面试里更推荐这样回答：

- 不强调名词争议
- 强调这次 GC 影响了哪些区域、停顿多大、问题重不重

#### 11）常见收集器全景图要怎么先建立起来

虽然 Week1 重点放在 CMS 和 G1，但“常见收集器”这题如果完全没有全景图，回答会显得视野偏窄。

第一轮更稳的做法是先建立一个粗颗粒地图：

- 年轻代常见语境：
  - Serial
  - ParNew
  - Parallel Scavenge
- 老年代常见语境：
  - Serial Old
  - CMS
  - Parallel Old
- 更现代、整体视角更强的收集器：
  - G1
- 更新一代的低停顿收集器：
  - ZGC
  - Shenandoah

但 Week1 不需要把这些全部展开成参数表。更合理的理解是：

- 先知道“谁大致在哪个时代和目标下出现”
- 再把 CMS 和 G1 作为在线 Java 服务里最值得先讲顺的重点

为什么 Week1 重点放 CMS 和 G1：

- 资深后端面试里，高频问题更常落在“低停顿”“大堆”“线上 RT 影响”这些维度
- CMS 和 G1 最适合拿来建立“收集器在做什么取舍”的主线

这里要顺手补一个时效边界：

- 不同 JDK 版本默认收集器和对 CMS 的支持情况会变化
- 所以回答“线上默认用什么”这类问题时，要结合当前运行环境验证，不能把旧版本经验讲成绝对事实

#### 12）CMS 和 G1 第一轮应该怎么理解

CMS 和 G1 这部分如果只记“一个低停顿，一个可预测”，还是太粗。更稳的方式是按同一套框架去理解和回答：

1. 它主要想解决什么问题
2. 它的基本思路是什么
3. 它的代价和短板是什么
4. 它更适合什么场景

##### CMS 应该怎么理解

如果面试官问 CMS，更稳的回答思路是：

- CMS 是一个更偏“降低停顿时间”的老年代收集器
- 它的核心思路是：
  - 尽量把一部分标记和清理工作放到和用户线程并发执行的阶段
- 它想解决的问题是：
  - 传统回收停顿太长，影响在线业务 RT

第一轮不用死背所有阶段名，但至少要知道它大致经历：

- 初始标记
- 并发标记
- 重新标记
- 并发清除

你可以先把它理解成：

- “先快速停一下，标记根上直接可达对象”
- “再并发地往下追踪”
- “最后再补一次关键修正”
- “然后并发清理无用对象”

这样回答的好处是：

- 你不是在背术语
- 而是在说明 CMS 为什么能降低停顿

CMS 的典型优点：

- 更适合对停顿敏感的在线业务
- 相比完全重停顿式回收，更容易降低单次暂停时间

CMS 的典型短板：

- 因为主要是标记-清除思路，所以容易产生内存碎片
- 并发阶段会和业务线程一起竞争 CPU
- 还可能遇到浮动垃圾问题

所以如果要把 CMS 讲完整，最稳的一句话是：

- CMS 的核心价值是尽量降低老年代回收停顿，但代价是碎片、并发开销和更复杂的回收压力管理

##### G1 应该怎么理解

如果面试官问 G1，更稳的回答思路是：

- G1 是一个更强调“面向大堆、可预测停顿”的收集器
- 它不是简单按连续的大块年轻代 / 老年代去管堆
- 而是把堆拆成多个 Region，再按回收收益和停顿目标选择回收哪些 Region

它想解决的问题是：

- 堆越来越大后，传统分代回收的停顿和管理成本更难控制
- 在线业务更希望 GC 停顿可预测，而不是偶尔突然很长

第一轮先建立这几个核心认知：

- 堆被拆成很多 Region
- 年轻代、老年代不再是物理上绝对固定的大连续块
- G1 会优先回收“垃圾收益更高”的 Region
- 它会围绕停顿目标去做回收选择

所以 G1 的第一层理解不是“名字更新的收集器”，而是：

- 用更细粒度的堆管理方式，去换更好的停顿可控性

G1 的典型优点：

- 更适合较大堆场景
- 更强调停顿可预测
- 通过 Region 管理和整理能力，通常比 CMS 更不容易被碎片问题困住

G1 的代价和边界：

- 机制更复杂
- 回收决策和维护开销也更高
- 不代表所有场景都一定比 CMS 更优

所以更稳的一句话是：

- G1 的核心价值是用 Region 和收益优先回收的思路，在较大堆场景下追求更可预测的停顿表现

##### CMS 和 G1 怎么比较更稳

不要把比较做成“谁先进谁落后”的价值判断，更适合按维度比较：

- 目标侧重点：
  - CMS 更容易从“降低老年代回收停顿”去理解
  - G1 更容易从“较大堆 + 可预测停顿”去理解
- 堆管理方式：
  - CMS 仍然更接近传统分代回收语境
  - G1 用 Region 做更细粒度管理
- 碎片问题：
  - CMS 更容易遇到碎片压力
  - G1 通常更擅长处理这类问题
- 适用场景：
  - CMS 更适合先从“低停顿老年代回收器”这个角色理解
  - G1 更适合先从“大堆、停顿可控”这个角色理解

如果面试官让你一句话比较，可以这样说：

- CMS 和 G1 都在解决 GC 对在线业务的影响，但 CMS 更像是围绕老年代低停顿做并发回收，G1 更像是围绕大堆和可预测停顿做细粒度堆管理

##### 这一块怎么答才不像背书

如果现场被问“CMS 和 G1 有什么区别”，推荐按这个顺序讲：

1. 先说两者共同点：
   - 都是在尽量降低 GC 对业务的影响
2. 再说核心差异：
   - CMS 偏老年代低停顿并发回收
   - G1 偏 Region 管理和可预测停顿
3. 再说代价：
   - CMS 有碎片和并发开销问题
   - G1 机制更复杂，也有自己的管理成本
4. 最后说场景：
   - 一个更容易从低停顿理解
   - 一个更容易从大堆和停顿可控理解

#### 14）这一块最该形成的复习框架

GC 复习时，最好固定成下面这条“可直接回答”的框架：

1. 先讲对象生命周期：
   - 为什么需要回收
   - 怎么判断对象存活
2. 再讲堆为什么分区域：
   - 为什么年轻代 / 老年代分开
   - 对象为什么会晋升
3. 再讲回收算法：
   - 标记-清除为什么会带来碎片
   - 复制算法为什么适合短命对象
   - 标记-整理为什么更重但更规整
4. 再讲业务影响：
   - STW 为什么会影响 RT
   - Full GC 为什么比 Minor GC 更值得警惕
5. 最后讲收集器取舍：
   - CMS 在优化什么，代价是什么
   - G1 在优化什么，代价是什么

如果按这个框架复习，你后面回答 GC 题时就不容易散成：

- 一堆名词
- 一堆阶段名
- 一堆参数

而会更像一条完整的技术理解链路。
### 3.2 频繁 Full GC / OOM 的排查路径

这一块如果只背命令和工具，很容易答得像“会用几个命令”，不像真正理解 JVM 问题。

更稳的复习方式是按下面这条链路：

- 先分症状：
  - 你看到的是 OOM、Full GC 频繁、停顿长、CPU 高，还是 RT 抖动
- 再做判断：
  - 这些现象更像是对象分配过快、对象回不掉、堆外内存问题，还是根本不是 GC 主因
- 再找证据：
  - 监控、GC 日志、线程栈、heap dump 分别在验证什么
- 最后才落动作：
  - 改代码、改对象生命周期、扩容、限流，还是调参数

#### 1）先分清线上到底是哪一类 JVM 问题

JVM 线上问题最常见的几类表象是：

- OOM
- Full GC 频繁
- GC 停顿长
- CPU 高
- RT 抖动

这些现象会同时出现，但不能混成同一个问题去答。

第一轮先建立下面这个区分：

- OOM：
  - 更像“某块内存资源已经耗尽”
- Full GC 频繁：
  - 更像“回收跟不上对象增长，或者老年代压力持续偏大”
- GC 停顿长：
  - 更像“某次回收代价过高，对业务线程暂停太久”
- CPU 高：
  - 可能是 GC，也可能是死循环、锁竞争、线程池空转、热点代码
- RT 抖动：
  - 可能是 GC，也可能是线程池堆积、下游超时、磁盘 / 网络抖动

所以第一步不是“立刻分析堆”，而是：

- 先判断当前主症状到底是什么

#### 2）这一块真正考什么

这类题真正考的不是你记不记得 `jmap`、`jstat`，而是你能不能沿着下面这条因果链分析：

- 现象是什么
- 这个现象更像哪类内存 / 生命周期问题
- 你准备用什么证据验证
- 验证后你会采取什么动作

如果这条链讲不出来，排障题就会变成工具名堆砌。

#### 3）排查 JVM 问题的稳定总框架

建议固定成下面这条：

1. 先看监控和现象面：
   - Full GC 频率
   - GC 停顿时间
   - 堆使用率曲线
   - 老年代增长速度
   - CPU、RT、错误率、吞吐是否同步恶化
2. 再做第一层判断：
   - 是对象分配过快
   - 还是对象回不掉
   - 是堆问题、元空间问题、直接内存问题
   - 还是根因根本不在 GC
3. 再看 GC 日志和线程状态：
   - 哪个区域压力最大
   - 回收后内存能不能明显下降
   - 是否存在明显晋升压力
   - CPU 高时线程主要在忙什么
4. 再回到应用和业务变化：
   - 流量是否突增
   - 缓存是否膨胀
   - 批量任务是否引入大对象
   - 线程池堆积是否让对象滞留更久
   - 下游超时是否让请求上下文长期不释放
5. 必要时做堆分析和对象定位：
   - `jstat`
   - `jmap`
   - heap dump
   - MAT
6. 最后再决定动作：
   - 改代码
   - 缩短对象生命周期
   - 限流 / 降级
   - 调整缓存策略
   - 控制批量任务
   - 扩容
   - 最后再考虑 JVM 参数

这套框架最重要的地方在于：

- 先判断问题形态
- 再找对象生命周期证据
- 最后才判断是代码、容量、流量，还是参数问题

#### 4）频繁 Full GC 应该怎么系统理解

频繁 Full GC 最常见的本质是：

- 老年代压力持续大
- 回收动作变重
- 但回收后空间释放又不明显

回答这题时，最好沿下面这条线说：

##### 第一步：先看“是不是持续性问题”

要先判断：

- 是偶发 Full GC 尖峰
- 还是长时间持续高频

因为偶发尖峰更可能和：

- 批量任务
- 临时流量
- 一次性大对象

有关；而持续高频更像：

- 对象生命周期设计有问题
- 老年代一直被占住

##### 第二步：再看“回收后到底降了多少”

这是区分问题性质的关键。

- 如果 Full GC 后堆明显回落：
  - 更像对象分配过快，或者短时间流量 / 任务把内存打高了
- 如果 Full GC 后堆几乎不回落：
  - 更像长生命周期对象、大对象、缓存、集合膨胀，甚至内存泄漏

所以你要形成一个稳定判断：

- “回收后降不下来”比“发生了 Full GC”更值得警惕

##### 第三步：再问老年代为什么会持续上涨

常见原因包括：

- 缓存过大或热点数据长期不淘汰
- 批量任务持有大量数据
- 大对象或大集合长期存活
- 请求上下文、异步任务、线程池队列导致对象滞留
- 下游超时或重试导致对象生命周期被拉长

##### 第四步：最后才决定动作

动作不是固定只有“调参数”。

更常见的正确方向是：

- 缩短对象生命周期
- 拆小批处理
- 限制缓存大小
- 避免一次性加载大结果集
- 限流 / 扩容
- 参数优化只是最后一层补充

#### 5）OOM 应该怎么系统理解

OOM 不是一句“内存不够”就能交代的，它首先在问：

- 到底是哪一类内存耗尽了

第一轮至少要形成下面这个映射：

- `Java heap space`：
  - 更常指向堆问题
  - 常见于对象太多、缓存膨胀、大集合、大批量数据
- `Metaspace`：
  - 更常指向类元信息区域压力
  - 常见于动态生成类过多、类加载器泄漏
- `Direct buffer memory`：
  - 更常指向直接内存问题
  - 常见于 NIO / Netty / 堆外缓冲区使用不当

所以 OOM 第一反应应该是：

- 先定位“是哪块内存”
- 再判断“为什么这块内存会持续增长”

而不是笼统地说：

- 机器内存不够

#### 6）CPU 高和 RT 抖动为什么不能直接归因给 GC

这两个现象最容易被误判。

##### CPU 高

CPU 高有很多可能来源：

- GC 线程占用过高
- 业务线程死循环
- 热点代码计算过重
- 锁竞争和自旋
- 线程池空转

所以 CPU 高时除了看 GC，还要看：

- 线程栈
- 热点线程
- CPU 消耗是否集中在 GC 线程

##### RT 抖动

RT 抖动也不一定是 GC。

它可能来自：

- GC 停顿
- 线程池队列积压
- 下游 RPC / 数据库抖动
- 锁竞争
- 网络波动

所以 RT 抖动这题更稳的回答方式是：

- 先判断抖动是否和 GC 停顿时间同步
- 如果不同步，再看线程池、下游依赖和锁竞争

#### 7）为什么很多 JVM 问题最后都要回到业务对象生命周期

因为 JVM 不是脱离业务独立运转的。

很多表面上的 GC / OOM 问题，最终都能追到业务层：

- 缓存 key 太多，堆被撑大
- 批量任务一次拉太多数据，大对象太多
- 下游超时导致请求对象滞留
- 异步队列堆积导致大量任务上下文无法释放
- 集合不断增长但没有及时清理

所以资深岗回答 JVM 问题时，不能只说：

- 哪个区满了
- 用了哪个工具

还要能回答：

- 为什么业务会制造出这种对象生命周期

#### 8）工具分别在验证什么

这块不要背成命令表，而要知道它们分别回答什么问题。

- `jstat`：
  - 更适合快速看 GC 趋势、各区域使用和回收频率
- GC 日志：
  - 更适合看每次 GC 的时机、停顿、回收前后变化、晋升压力
- `jmap` / heap dump：
  - 更适合把问题落到对象和类级别
- MAT：
  - 更适合看大对象、Retained Size、引用链和可疑泄漏点
- 线程栈：
  - 更适合判断 CPU 高是不是 GC 之外的线程问题

所以排查顺序通常是：

- 先用轻量工具看趋势
- 再用 dump 和分析工具做深挖

#### 9）这一块常见失分方式

- 一上来就说“调 JVM 参数”
- 只会背工具名，不知道每个工具在验证什么
- 只盯 GC，不看 CPU / RT / 错误率 / 流量变化
- 把 OOM、Full GC、CPU 高、RT 抖动混成一个问题
- 完全不结合业务对象生命周期

#### 10）这一块最适合怎么复习

把这类题按“症状 -> 判断 -> 证据 -> 动作”四步去练：

- 症状：
  - 看到什么现象
- 判断：
  - 你怀疑是哪类问题，先画出根因树
- 证据：
  - 你准备用什么监控、日志、线程栈、dump 去验证
- 动作：
  - 你最终会怎么改，为什么不是先调参数

如果能按这个结构去讲，排障题就会从“工具题”变成“诊断题”。
## 4. 边界、误区、常见追问

### 4.1 容易说错的边界

- GC 的核心目标不只是“把内存清掉”，还包括控制停顿和吞吐代价。
- 可达性分析和“对象什么时候真正被回收”不是完全同一个步骤。
- Minor GC、Major GC、Full GC 这些词在不同资料和实现里口径可能有差异，不要讲成绝对唯一标准。
- STW 不只会出现在 Full GC 语境里，讨论 GC 题时要把“停顿成本”单独拎出来。
- OOM 不等于内存泄漏，很多 OOM 是流量、缓存、批量任务或对象生命周期设计导致的资源耗尽。
- CPU 高和 RT 抖动不一定就是 GC 问题，不能看到 JVM 就只盯 GC。
- G1 不等于所有场景都优于 CMS，收集器选择本质上是目标和代价的取舍。
- CMS 相关结论带有明显版本背景，具体默认和支持情况要结合当前 JDK 环境判断。

### 4.2 高频追问方向

- 为什么 JVM 更核心地用可达性分析，而不是引用计数。
- 年轻代 / 老年代划分到底在优化什么。
- 对象晋升为什么会直接影响 Full GC 风险。
- 标记-清除、复制、标记-整理各自代价是什么。
- STW 为什么会直接体现在业务 RT 上。
- 常见收集器全景图应该怎么建立，为什么 Week1 重点看 CMS 和 G1。
- CMS 和 G1 分别在优化什么，代价是什么。
- 频繁 Full GC 为什么要先看回收后内存是否明显回落。
- OOM 为什么要先区分 heap、metaspace、direct memory。
- CPU 高和 RT 抖动为什么不能直接归因给 GC。

### 4.3 这块最值得刻意训练的表达习惯

- 回答 GC 时，不要直接背收集器名字，要先从对象生命周期和停顿成本讲起。
- 回答排障题时，不要先背工具命令，要先讲“症状 -> 判断 -> 证据 -> 动作”。
- 回答收集器题时，不要做价值判断，要按目标、机制、代价、场景四层讲。
- 这块有明显版本敏感信息，尤其是默认收集器和 CMS 状态，记得结合当前环境验证。

## 5. 推荐表达顺序

如果面试官问 GC 或线上排障，推荐按下面顺序讲：

1. 先说对象生命周期：
   - 为什么需要 GC，怎么判断对象存活
2. 再说回收代价：
   - 分代、晋升、碎片、STW 各自意味着什么
3. 再说收集器取舍：
   - CMS、G1 分别在优化什么，代价是什么
4. 最后说排障：
   - 症状、判断、证据、动作怎么连起来
