# 算法专题 02：哈希表

适用目标：

- 系统理解哈希表题的适用边界
- 分清 `HashSet` 和 `HashMap` 的典型用法
- 建立“存在性 / 映射 / 计数 / 分组 / 前缀和 + 哈希”的题型框架

## 1. 哈希表的核心特点

- 用空间换时间
- 平均情况下支持 `O(1)` 查询、插入、删除
- 最适合解决“是否存在、出现次数、位置映射、分组归类”问题
- Java 里最常用的是：
  - `HashSet`
  - `HashMap`

## 2. 什么题应该优先想到哈希表

如果题目有下面这些信号，通常优先想到哈希表：

- “判断是否存在”
- “找某个值对应的另一个值”
- “找重复元素”
- “统计出现次数”
- “按特征分组”
- “希望把暴力 `O(n^2)` 降到 `O(n)`”

## 3. 哈希表题常考模式

看这一节时，建议你每个模式都固定想三件事：

- 哈希结构里到底存什么
- 为什么这样存能把复杂度降下来
- 这个模式的标准主流程是什么

### 3.1 存在性判断

核心思路：

- 用 `HashSet` 快速判断元素是否出现过

典型题：

- `217. 存在重复元素`
- `128. 最长连续序列`

通用套路：

1. 先判断题目是不是只关心“某个值有没有出现过”。
2. 如果只关心存在性，优先用 `HashSet`，不要一上来就上 `HashMap`。
3. 常见主流程是：
   - 遍历元素
   - 先查是否存在
   - 再决定是否加入集合
4. 如果题目还需要更多信息，比如索引或次数，`Set` 就不够了，要升级成 `Map`。

你真正要会的是：

- 分清“只要存在性”还是“还要额外信息”
- 避免明明一个 `Set` 能做，却写成啰嗦的 `Map`

### 3.2 补数 / 映射关系

核心思路：

- 遍历当前元素时，快速查“目标值减去当前值”是否已经出现

典型题：

- `1. 两数之和`

关键点：

- 先查再放，还是先放再查，要根据题意决定
- 容易在重复元素上出错

通用套路：

1. 先想清楚当前元素需要和谁配对。
2. 再决定哈希表里该存什么映射，最常见的是：
   - `值 -> 下标`
   - `值 -> 次数`
3. 常见主流程是：
   - 遍历当前元素 `x`
   - 计算它需要的“搭档”
   - 先查搭档是否已出现
   - 再决定要不要把当前元素放进哈希表
4. 如果题目不允许元素和自己配对，就必须非常注意更新顺序。

你真正要会的是：

- 不是只会写 `target - nums[i]`
- 而是能解释为什么这里必须先查再放，或者必须先放再查

### 3.3 计数统计

核心思路：

- `HashMap<元素, 次数>`

典型题：

- `242. 有效的字母异位词`
- `347. 前 K 个高频元素`
- `169. 多数元素`

通用套路：

1. 先判断你统计的对象范围大不大：
   - 范围固定且很小，可以用数组
   - 范围不固定或值域大，用 `HashMap`
2. 再决定统计方式：
   - 单独统计一次
   - 两边同时加减抵消
   - 先统计再做 TopK / 排序 / 堆处理
3. 常见主流程是：
   - 遍历元素并更新频次
   - 再根据频次做比较、校验或筛选
4. 这类题的核心不是“把元素放进 map”，而是明确 map 的 value 表示什么。

你真正要会的是：

- 看到“出现次数”先想到计数
- 再判断该用数组计数还是哈希计数

### 3.4 分组归类

核心思路：

- 为每个元素生成“分类 key”
- 相同 key 的归到同一组

典型题：

- `49. 字母异位词分组`

常见 key：

- 排序后的字符串
- 字母频次数组转字符串

通用套路：

1. 先不要两两比较元素，而是先想“同一组元素有什么共同特征可以做 key”。
2. 只要能为每个元素构造出稳定且唯一的分类 key，就可以直接做：
   - `HashMap<key, List<元素>>`
3. 常见主流程是：
   - 遍历每个元素
   - 计算它的分类 key
   - 把它追加到对应分组
4. `49. 字母异位词分组` 常见 key 有两种：
   - 排序后的字符串：写起来直观
   - 频次数组转签名：更贴近“字符组成相同”本质
5. 这类题最常见的误区是：
   - 虽然用了哈希思想
   - 但仍然在做两两比较
   - 复杂度没有真正降下来

你真正要会的是：

- 先找稳定 key
- 再直接分桶
- 而不是先建一堆统计结果，再回头两两配对

### 3.5 前缀和 + 哈希表

核心思路：

- 用哈希表记录“某个前缀和出现过几次 / 最早出现在哪”
- 把区间问题转成前缀差值问题

典型题：

- `560. 和为 K 的子数组`
- `525. 连续数组`

这是非常高频、非常值得专项训练的一类题。

通用套路：

1. 先定义滚动前缀和 `prefixSum`。
2. 再判断这题到底要从哈希表里拿什么历史信息：
   - 前缀和出现次数
   - 前缀和最早出现位置
3. 把区间问题改写成“当前前缀和”和“历史前缀和”的关系。
4. 常见主流程是：
   - 更新当前前缀和
   - 查询历史上是否有目标前缀和
   - 统计答案或更新最优值
   - 最后更新当前前缀和信息
5. 这类题经常要做初始化：
   - `0 -> 1`
   - 或 `0 -> -1`
   初始化含义必须能讲清。

你真正要会的是：

- 先把区间问题翻译成前缀差
- 再明确 map 存的是“次数”还是“位置”

## 4. `HashSet` 和 `HashMap` 怎么选

优先判断你到底需要保存什么：

- 只关心“存不存在”：优先 `HashSet`
- 需要记录“元素 -> 索引”：用 `HashMap`
- 需要记录“元素 -> 次数”：用 `HashMap`
- 需要记录“分类 key -> 一组元素”：用 `HashMap`

一个常见失误：

- 其实只需要 `Set`，却硬用 `Map`
- 这样代码更啰嗦，也更容易让思路发散

## 5. 哈希表题的高频坑点

- 只会用 `Map`，但不知道什么时候 `Set` 更合适
- 忽略重复元素
- 更新顺序写反，导致把当前元素自己匹配上自己
- 复杂度说成绝对 `O(1)`，更严谨的说法应是“平均 `O(1)`”
- 设计分组 key 时不稳定、不唯一
- 只用了哈希表，但仍然写出了重复扫描，复杂度并没有真正降下来

## 6. 哈希表题的面试表达模板

```text
这题核心是把原本需要反复查找的信息提前存进哈希结构里，用空间换时间。
如果我需要判断某个值是否存在，我会优先想到 HashSet；
如果我需要记录元素和索引、元素和次数的映射，就用 HashMap。
这样通常能把原本的 O(n^2) 降到 O(n)。
```

## 7. 本专题关键训练样例

### 7.1 `1. 两数之和`

本质：

- 哈希表最基础的补数查找题
- 重点不是“会用 map”，而是“遍历过程中如何在线匹配”

你需要掌握：

- 为什么要先查再放
- 为什么能把 `O(n^2)` 降到 `O(n)`
- 如何处理重复元素

### 7.2 `49. 字母异位词分组`

本质：

- 哈希表分组归类题
- 重点是如何构造稳定的分类 key

你需要掌握：

- 为什么排序后字符串可以作为 key
- 为什么频次数组也可以作为 key
- 两种 key 的取舍分别是什么

### 7.3 `560. 和为 K 的子数组`

本质：

- 前缀和 + 哈希表
- 重点是把区间问题转成“前缀差”

你需要掌握：

- 为什么这题不能直接套普通滑动窗口
- 为什么 map 要存“出现次数”而不是“最近位置”
- 为什么初始化要放 `0 -> 1`

## 9. 本专题练习规划

### 第一阶段：模板入门

1. `1. 两数之和`
   - 训练重点：
     - 补数查找
     - 先查再放
2. `242. 有效的字母异位词`
   - 训练重点：
     - 计数统计
     - 数组计数和 `HashMap` 计数的取舍

### 第二阶段：高频主战

1. `49. 字母异位词分组`
   - 训练重点：
     - 分组 key 设计
     - 分类问题为什么适合 `HashMap`
2. `560. 和为 K 的子数组`
   - 训练重点：
     - 前缀和 + 哈希
     - 区分“计数型”和“最长长度型”前缀和题
3. `128. 最长连续序列`
   - 训练重点：
     - `HashSet` 做存在性判断
     - 为什么只从起点开始枚举

### 第三阶段：变体 / 验收

1. `347. 前 K 个高频元素`
   - 训练重点：
     - 哈希统计和堆 / 桶排序的配合
     - 训练“哈希不一定单独使用”的意识
2. `525. 连续数组`
   - 训练重点：
     - 和 `560` 对照：
       - 一个存次数
       - 一个存最早位置

## 10. 专题验收标准

- 能先判断这题该用 `HashSet` 还是 `HashMap`
- 能区分哈希题里的：
  - 存在性判断
  - 索引映射
  - 次数统计
  - 分组归类
  - 前缀和 + 哈希
- 能讲清 `1`、`49`、`560` 三题分别代表什么模式
- 能解释“为什么用了哈希以后复杂度真的降下来了”
