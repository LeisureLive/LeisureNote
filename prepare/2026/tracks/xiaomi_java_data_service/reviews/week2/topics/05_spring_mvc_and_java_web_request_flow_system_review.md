## 1. 这一块在解决什么问题

Spring MVC / Java Web 这块不是在考你背多少接口名，而是在考你能不能把下面四件事讲清楚：

1. 一个 HTTP 请求进入服务端后，为什么要先经过 Servlet 容器。
2. Spring MVC 为什么能在 Java Web 之上把请求分发、参数绑定、返回值处理统一起来。
3. 参数绑定、拦截器、异常处理、返回 JSON 为什么都能挂在同一条请求链路上。
4. 过滤器、拦截器、AOP 到底各拦在哪一层，为什么面试里总被放在一起问。

把这些点串起来，可以这样理解：

- Web 服务先要有一个能接收 HTTP 请求、管理 Servlet 生命周期的基础运行环境。
- Servlet 容器先把请求收进来，Spring MVC 再在这个基础上做更细的控制器分发和方法级处理。
- 请求一旦进入统一分发器，参数绑定、返回值处理、异常处理就都可以复用同一套扩展链路。
- 不同横切能力拦截的位置不同，才会引出过滤器、拦截器、AOP 的分工边界。

所以这块真正要建立的是三种能力：

- 能解释 Web 请求为什么要先经过 Servlet 容器，再进入 Spring MVC。
- 能解释 Spring MVC 请求处理主流程为什么能统一接住参数、返回值和异常。
- 能解释过滤器、拦截器、AOP 的层级差异和适用场景。

## 3. 核心重点展开

### 3.1 为什么 Web 请求要先进入 Servlet 容器，而不是直接交给 Spring MVC

这一节在整条主线里解决的是第一个问题：

- Java Web 基础运行环境在解决什么。
- 为什么 Spring MVC 不是脱离 Servlet 体系单独存在的。

#### 先直接回答这个问题

Web 请求要先进入 Servlet 容器，是因为服务端必须先有一个符合 Servlet 规范的运行环境来接收 HTTP 请求、管理 Servlet 生命周期、创建 `HttpServletRequest` / `HttpServletResponse` 这些基础对象。Spring MVC 并不是绕过这层基础设施，而是在这层之上继续做请求分发和方法调用增强。

#### 再用场景把问题钉住

浏览器发起一个 HTTP 请求后，最先面对的不是 Controller，而是像 Tomcat 这样的容器。容器先把网络请求解析成 Servlet 规范下的请求对象，再把它交给入口 Servlet。Spring MVC 的 `DispatcherServlet` 就是被挂在这里的。

#### 是什么

- Servlet 容器：提供 Web 请求接入和 Servlet 运行环境。
- Spring MVC：构建在 Servlet 体系之上的请求分发框架。

#### 为什么有它

因为 Spring MVC 要解决的是“怎么优雅处理请求”，而不是“怎么从网络层收包并管理 Servlet 运行时”。

#### 怎么理解它的机制

第一层，Servlet 容器负责监听端口、接收 HTTP 请求、创建请求响应对象。

第二层，请求按照映射关系进入某个 Servlet。

第三层，Spring MVC 把 `DispatcherServlet` 作为统一入口挂上去。

第四层，后面的控制器分发、参数绑定、异常处理，都是从这个入口继续往下发生的。

#### 面试时怎么讲才不空

“Spring MVC 不是直接接网络请求，它是建立在 Servlet 容器之上的。容器先负责接住 HTTP 请求和提供 Servlet 运行环境，Spring MVC 再通过 `DispatcherServlet` 统一做控制器分发、参数绑定和返回值处理。所以回答 MVC 主流程时，我会先把 Servlet 容器这层交代清楚。”

### 3.2 Spring MVC 请求处理主流程为什么能把请求统一分发到控制器方法

这一节在整条主线里解决的是第二个问题：

- `DispatcherServlet` 到底在做什么。
- 请求为什么能从 URL 一路稳定走到某个 Controller 方法。

#### 先直接回答这个问题

Spring MVC 请求处理主流程的核心，是 `DispatcherServlet` 把“请求怎么找到处理器、怎么调用目标方法、怎么处理返回结果”统一收口了。它不是简单转发一下，而是把请求分发、方法适配、参数绑定、结果处理全都接成了一条标准化链路。

#### 再用场景把问题钉住

比如请求 `GET /orders/123` 进来之后，Spring MVC 不会让你手工解析 URL 再自己 `if-else` 分发，而是会：

- 先根据请求路径找到匹配的处理器。
- 再找到能调用这个处理器的适配器。
- 完成方法参数准备。
- 执行目标 Controller 方法。
- 最后统一处理返回值。

#### 是什么

Spring MVC 主流程可以先压成下面几步：

1. 请求进入 `DispatcherServlet`。
2. `HandlerMapping` 找到目标处理器。
3. `HandlerAdapter` 负责以统一方式调用处理器方法。
4. Controller 执行业务逻辑。
5. 返回值进入统一处理链路。

#### 为什么有它

因为 Web 请求处理不能靠每个 Controller 各写一套分发和调用逻辑，必须统一收口，才能复用参数绑定、异常处理、返回值处理等能力。

#### 怎么理解它的机制

第一层，`DispatcherServlet` 先做统一入口。

第二层，`HandlerMapping` 负责把请求和具体处理器关联起来。

第三层，`HandlerAdapter` 负责屏蔽不同处理器调用方式的差异。

第四层，处理完成后再进入统一的结果处理链路。

#### 面试时怎么讲才不空

“Spring MVC 的核心不是记组件名，而是理解它为什么能统一分发请求。我一般会按‘请求进 `DispatcherServlet` -> 找处理器 -> 找适配器 -> 调 Controller -> 统一处理返回值’这条链来讲，这样后面参数绑定、异常处理、拦截器都能顺着接上。”

### 3.3 参数绑定、返回值处理、异常处理为什么都能挂在同一条请求链路上

这一节在整条主线里解决的是第三个问题：

- 为什么 Spring MVC 不只是“路由到方法”这么简单。
- 为什么参数、返回值、异常这些通用能力可以统一治理。

#### 先直接回答这个问题

参数绑定、返回值处理、异常处理之所以都能挂在同一条链路上，是因为 Spring MVC 在调用 Controller 之前和之后都保留了统一扩展点。它不是把请求直接扔给某个方法就结束，而是把“入参准备、方法调用、结果转换、异常兜底”全部纳入同一个执行框架。

#### 再用场景把问题钉住

比如一个 Controller 方法声明：

- 路径变量要绑定到方法参数。
- 请求体要反序列化成对象。
- 返回对象要自动转成 JSON。
- 抛出的业务异常要统一转成错误响应。

这些能力如果靠每个方法自己处理，代码会非常散。Spring MVC 统一调用链的价值，就是把这些工作抽出来复用。

#### 是什么

- 参数绑定：把请求里的路径、查询参数、请求体转换成方法参数。
- 返回值处理：把方法返回结果转换成视图或 JSON 响应。
- 异常处理：把控制器抛出的异常纳入统一错误处理链。

#### 为什么有它

因为 Web 层的大量工作都是共性流程，适合统一框架化，而不是让每个接口自己重复实现。

#### 怎么理解它的机制

第一层，调用 Controller 前，Spring MVC 会根据参数解析规则准备方法入参。

第二层，方法执行后，返回值会交给对应处理器决定是走视图解析还是消息转换。

第三层，如果执行过程中抛异常，异常解析器会尝试把它转换成统一响应结果。

第四层，这使得 Web 层可以把业务逻辑和通用协议处理拆开。

#### 面试时怎么讲才不空

“Spring MVC 不只是把请求路由到 Controller，它还统一接住了参数绑定、返回值转换和异常处理。因为这些都挂在同一条执行链上，所以接口层可以比较薄，很多协议和错误处理逻辑都由框架复用。”

### 3.4 过滤器、拦截器、AOP 为什么总被一起问，它们到底怎么分工

这一节在整条主线里解决的是第四个问题：

- 三种常见横切手段分别拦在哪一层。
- 为什么它们不能互相完全替代。

#### 先直接回答这个问题

过滤器、拦截器、AOP 总被一起问，是因为它们都能做横切增强，但作用层级不同。Filter 更靠近 Servlet 容器层，Interceptor 更靠近 Spring MVC 请求处理层，AOP 更靠近 Spring Bean 方法调用层。它们不是谁更高级，而是分别适合拦不同位置的问题。

#### 再用场景把问题钉住

比如：

- 跨域、编码、统一请求包装，通常更适合放在 Filter。
- 登录校验、接口级审计、Controller 调用前后处理，通常更适合放在 Interceptor。
- 事务、Service 日志、权限切面这类 Bean 方法增强，更适合放在 AOP。

#### 是什么

- Filter：Servlet 规范层的过滤器。
- Interceptor：Spring MVC 请求执行链上的拦截器。
- AOP：Spring Bean 方法调用链上的横切增强。

#### 为什么有它

因为请求处理和业务方法调用不是同一层，横切逻辑要贴着最合适的拦截点落位。

#### 怎么理解它的机制和边界

第一层，Filter 在请求进入 Servlet 之后、进入 Spring MVC 之前就可以拦截。

第二层，Interceptor 更贴近 Controller 执行前后，但它已经站在 Spring MVC 链路里。

第三层，AOP 针对的是 Spring 容器里的 Bean 方法调用，不天然感知 HTTP 请求语义。

第四层，所以问题不是“哪个都能做”，而是“当前需求更适合拦在哪一层”。

#### 面试时怎么讲才不空

“这三者都能做横切，但层级不同。Filter 偏 Servlet 层，Interceptor 偏 MVC 请求链，AOP 偏 Spring Bean 方法增强。面试里我会先按拦截层级分，再举一个适用场景，不会把它们答成三个平行定义。”

## 4. 边界、误区、常见追问

- 不要一上来背 `DispatcherServlet`、`HandlerMapping`、`HandlerAdapter`，先把请求主流程讲顺。
- Spring MVC 不是脱离 Servlet 体系单独存在的。
- 过滤器、拦截器、AOP 都能做横切，但拦截位置不同，不能混成一个概念。
- JSON 返回、参数绑定、异常处理这些能力，最好放在同一条请求链路里解释。

5. 最后讲过滤器、拦截器、AOP 的分工边界。
