# 算法专题 07：堆 / 优先队列

适用目标：

- 建立堆题的统一认知：这类题的关键不是“排序后取结果”，而是“动态维护当前最值得保留的一批元素”
- 分清什么时候该用最小堆，什么时候该用最大堆
- 建立 TopK、多路归并、数据流维护、优先级调度这几类高频堆题模板
- 能把堆题从“会调用 PriorityQueue”提升到“能解释为什么这里必须用堆而不是排序”

## 1. 堆题的核心特点

- 堆不是为了得到完整有序结果，而是为了快速拿到当前最值：
  - 最小堆可以快速拿最小值
  - 最大堆可以快速拿最大值
- 堆最适合的场景是：
  - 数据持续进入
  - 只关心前 `k` 个最值
  - 每次都要快速取出当前优先级最高的元素
- Java 面试里最常用的实现是 `PriorityQueue`
- 很多堆题的本质不是“把所有元素都排好序”，而是：
  - 在扫描过程中，只保留当前最有价值的一小部分候选
  - 或者在多个有序来源里，始终挑下一个最优元素

## 2. 什么题应该优先想到堆 / 优先队列

如果题目有下面这些信号，通常优先考虑堆：

- “第 `k` 大 / 第 `k` 小”
- “TopK 高频 / TopK 最值”
- “数据流里持续维护中位数 / 第 `k` 大”
- “每次都要取当前最小 / 最大 / 最早结束 / 最高优先级”
- “合并多个有序链表 / 有序数组”
- “任务调度、会议室分配、最小代价连接”

如果题目只是一次性求整个数组排序结果，通常排序就够，不一定需要堆。

## 3. 堆题常考模式

看这一节时，建议你每个模式都固定追问三件事：

- 堆里到底要存什么：值、下标、频次对，还是节点对象
- 为什么这里要保留一部分候选，而不是等全部数据处理完再排序
- 当前题是“固定容量维护”，还是“持续弹出当前最优”

### 3.1 TopK：固定容量维护最值集合

适用场景：

- 找第 `k` 大 / 第 `k` 小
- 找前 `k` 个高频元素
- 只关心结果中的前 `k` 个，而不关心完整排序

代表题：

- `215. 数组中的第 K 个最大元素`
- `347. 前 K 个高频元素`
- `973. 最接近原点的 K 个点`

核心理解：

- TopK 题最常见做法不是“全量排序后取前 `k` 个”，而是：
  - 用一个容量为 `k` 的堆，持续维护当前最优的 `k` 个候选
- 如果要求“前 `k` 大”，常见做法是：
  - 用最小堆保存当前最大的 `k` 个元素
  - 这样堆顶就是这 `k` 个里最小的那个，也就是第 `k` 大
- 如果要求“前 `k` 小”，常见做法反过来。

通用套路：

1. 先判断题目是不是只关心前 `k` 个结果，而不是完整有序序列。
2. 如果是，就考虑“固定容量堆”而不是全量排序。
3. 常见主流程是：
   - 遍历元素
   - 先把候选放入堆
   - 如果堆大小超过 `k`，弹出最不该保留的元素
4. 关键是选对堆类型：
   - 保留最大的 `k` 个，用最小堆
   - 保留最小的 `k` 个，用最大堆
5. 这样整体复杂度通常是 `O(n log k)`，当 `k << n` 时很有价值。

你真正要会的是：

- 能解释为什么“找第 `k` 大”常用最小堆，而不是最大堆
- 能说明堆顶元素到底代表什么淘汰边界

### 3.2 哈希 + 堆：频次排序与 TopK 高频

适用场景：

- 先统计频次，再从高频对象里筛前 `k` 个
- 元素本身不是直接比较值大小，而是比较统计结果

代表题：

- `347. 前 K 个高频元素`
- `692. 前 K 个高频单词`

核心理解：

- 这类题往往不是纯堆题，而是“两阶段”：
  - 先用哈希表统计
  - 再用堆维护前 `k` 个
- 真正比较的对象通常不是元素本身，而是：
  - 频次
  - 频次 + 字典序
  - 自定义优先级

通用套路：

1. 先用 `HashMap` 统计每个元素的频次。
2. 再遍历 `map.entrySet()`，把条目放入堆。
3. 如果题目只要 TopK，通常仍然用固定容量堆，把堆控制在 `k`。
4. 如果有多重排序条件，要把比较器想清楚：
   - 第一关键字比什么
   - 相等时第二关键字比什么
5. 这类题的关键不是 API，而是“比较器表达的业务优先级”要准确。

你真正要会的是：

- 能先把问题拆成“统计”和“筛选”两步
- 能说清堆比较器为什么这样写，而不是只会试错调整

### 3.3 多路归并：多个有序来源里持续取最优

适用场景：

- 合并 `k` 个有序链表
- 合并多个有序数组
- 从多个有序来源里，每次取当前最小或最大

代表题：

- `23. 合并 K 个升序链表`
- `373. 查找和最小的 K 对数字`
- `378. 有序矩阵中第 K 小的元素`

核心理解：

- 这类题不是 TopK，而是“多个来源各自局部有序，如何快速找到全局下一个最优元素”
- 堆里保存的通常不是普通数值，而是“每个来源当前的候选头部”
- 每次弹出最优候选后，再把它所在来源的下一个元素补进堆

通用套路：

1. 先判断题目是不是有多个“各自局部有序”的来源。
2. 如果是，就不要把所有元素摊平后再排序，而是考虑多路归并。
3. 常见主流程是：
   - 把每个来源的第一个候选放入堆
   - 弹出当前全局最优元素
   - 把它所在来源的下一个元素继续入堆
4. 堆里通常需要保存：
   - 当前值
   - 当前元素属于哪个来源
   - 在来源中的位置
5. 这类题的重点是把“局部有序”转成“全局逐步有序”。

你真正要会的是：

- 能识别“多路归并”题感，而不是看到多个链表就手写两两合并
- 能说清为什么每次只要维护各来源当前头部就够了

### 3.4 数据流维护：动态维护第 `k` 大、中位数等结果

适用场景：

- 数据持续流入，要求随时返回结果
- 每插入一个数，都要快速得到当前第 `k` 大或中位数

代表题：

- `703. 数据流中的第 K 大元素`
- `295. 数据流的中位数`

核心理解：

- 这类题最大的特征是：
  - 数据不是一次性给完，而是持续到来
  - 所以全量排序每次重算代价太高
- `703` 的核心仍然是固定容量堆。
- `295` 的核心则是双堆：
  - 一个最大堆维护较小的一半
  - 一个最小堆维护较大的一半

通用套路：

1. 先判断题目是不是“在线处理”而不是“离线一次计算”。
2. 如果每次都要快速取极值或中位数，就考虑堆的动态维护。
3. 常见主流程有两类：
   - 固定容量堆维护第 `k` 大
   - 双堆维护数据流平衡
4. 双堆题最关键的是维护两个不变量：
   - 两边元素数量平衡
   - 一边所有元素都不大于另一边所有元素

你真正要会的是：

- 能区分“单堆就够”还是必须“双堆平衡”
- 能讲清双堆各自维护的是哪一半数据

### 3.5 优先级调度：按当前最早结束或最高优先级决策

适用场景：

- 会议室分配
- CPU 任务调度
- 合并代价最小
- 每次选择当前最早结束、最小代价或最高优先级的任务

代表题：

- `253. 会议室 II`
- `621. 任务调度器`
- `1167. 连接棒材的最低费用`

核心理解：

- 这类题里，堆不再只是“保留前 `k` 个”，而是“每一步都要根据当前最优选择推进状态”
- 堆顶往往代表：
  - 最早释放的资源
  - 当前最小代价
  - 当前最该执行的任务
- 这类题经常和贪心思想绑定：
  - 每一步先做当前最优决策
  - 再更新系统状态

通用套路：

1. 先判断题目是不是在重复执行“取当前最优，再更新剩余状态”的过程。
2. 如果是，就考虑优先队列是否能把“当前最优”快速取出来。
3. 常见主流程是：
   - 把候选资源或任务按优先级放入堆
   - 每次弹出堆顶做处理
   - 根据处理结果把新状态重新放回堆
4. 这类题关键不只是会用堆，还要先想清楚“贪心选择依据是什么”。

你真正要会的是：

- 能识别“当前最优决策反复发生”的题感
- 能说清堆顶元素代表系统里的哪类关键资源或候选

## 4. 最小堆和最大堆怎么选

先问自己：你想快速拿到哪一类元素？

- 想快速拿最小值：用最小堆
- 想快速拿最大值：用最大堆
- 想保留最大的 `k` 个元素：常用最小堆
- 想保留最小的 `k` 个元素：常用最大堆

一个高频误区：

- 题目要“第 `k` 大”，就机械上最大堆
- 其实如果你的目标是维护“当前最大的 `k` 个候选”，最自然的是最小堆，因为它方便淘汰这 `k` 个里最弱的那个

## 5. 堆题的高频坑点

- 没分清“需要完整排序”还是“只要动态最值”
- 堆里该存值、索引还是对象没想清楚
- TopK 题把堆类型选反
- 比较器逻辑写乱，尤其是多关键字排序
- 只会全量入堆，没意识到固定容量堆能把复杂度降到 `O(n log k)`
- 多路归并题没有保存来源信息，弹出后不知道下一个该补谁
- 双堆题只会入堆，不会维护两边数量和平衡关系

## 6. 堆题的面试表达模板

```text
这题我会先判断它是不是在反复维护“当前最值得保留的一批元素”或“当前最优候选”。
如果只是 TopK，我通常不会先全量排序，而是用一个容量为 k 的堆动态维护候选，
这样复杂度可以从 O(n log n) 降到 O(n log k)。

如果题目是多个有序来源里持续取最优，我会考虑多路归并，
堆里保存每个来源当前的头部候选；
如果题目是数据流实时维护结果，我会再判断是单堆还是双堆。
```

   - 训练重点：
