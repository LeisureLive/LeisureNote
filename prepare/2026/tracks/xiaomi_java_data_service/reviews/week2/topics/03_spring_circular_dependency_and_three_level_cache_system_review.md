## 1. 这一块在解决什么问题

循环依赖这块不是在考你会不会背 `singletonFactories`、`earlySingletonObjects`、`singletonObjects`，而是在考你能不能把下面四件事讲清楚：

1. 什么叫循环依赖，为什么 Spring 容器创建 Bean 时会遇到这个问题。
2. 为什么 Spring 不是“所有循环依赖都能解决”，而是只能解决一部分。
3. 三级缓存到底在解决什么问题，为什么不是二级缓存就够了。
4. 构造器注入、原型 Bean、AOP 代理介入时，为什么循环依赖会有不同结果。

把这四个问题串起来，主线其实也很稳定：

- Spring 容器要创建 Bean，同时还要处理依赖注入。
- 如果 A 依赖 B，B 又依赖 A，容器在创建过程中就会出现“对象还没完全创建好，但另一个 Bean 已经要用它”的问题。
- 直接等对象完全创建完再暴露已经来不及，所以 Spring 需要“提前暴露一个可引用对象”。
- 但提前暴露的不一定是原始对象，还可能是后续需要增强的代理对象，这就引出了三级缓存。
- 即便如此，Spring 也不是万能的，因为有些依赖阶段太早，有些作用域不适合缓存复用。

所以这块真正要建立的是三种能力：

- 能解释循环依赖为什么会发生在 Bean 创建期。
- 能解释三级缓存是在解决“提前暴露”和“可能需要代理”这两个问题。
- 能说明 Spring 能解决什么场景，不能解决什么场景。

## 2. 先把主线串起来，再看知识树

### 2.1 先用一条因果链把全文串起来

如果你觉得循环依赖这块容易散，通常不是三级缓存名字没背住，而是没有先抓住下面这条因果链：

1. 先看循环依赖为什么会出现：Bean 创建和依赖注入是分阶段的。
2. 再看为什么必须“提前暴露”：否则另一个 Bean 在注入时拿不到依赖对象。
3. 再看为什么提前暴露不能只暴露原始对象：因为这个 Bean 后面可能还要被 AOP 等机制增强成代理对象。
4. 再看三级缓存为什么存在：它本质上是在管理“成品对象、提前暴露对象、可生成提前对象的工厂”三种状态。
5. 最后看哪些场景仍然解决不了：构造器注入、原型 Bean、某些代理和依赖链组合。

所以这篇文档不是在讲几个缓存名词，而是在回答一条连续问题链：

- 为什么会出现循环依赖。
- 为什么要提前暴露对象。
- 为什么提前暴露还要考虑代理对象。
- 三级缓存到底在管什么。
- 哪些场景 Spring 仍然救不了。

## 3. 核心重点展开

### 3.1 循环依赖为什么会出现在 Spring Bean 创建过程中

这一节在回答什么问题：

- 循环依赖为什么是容器创建期问题。
- 为什么它不是一个“普通引用关系”这么简单。

这个问题是怎么解决的：

循环依赖会出现在 Spring Bean 创建过程中，是因为 Bean 的“实例化”和“依赖注入完成”不是同一时刻。如果 A 在创建时需要 B，而 B 在创建时又反过来需要 A，就会出现一个典型问题：A 还没完全初始化完成，但 B 已经需要拿到 A 的引用。所以循环依赖的本质，不是“两个类互相认识”，而是“两个 Bean 都在创建过程中，同时依赖对方已经可用”。

放回最小场景里看，比如 `ServiceA` 注入 `ServiceB`，`ServiceB` 又注入 `ServiceA`。如果是 Setter 注入，Spring 可能先把 `ServiceA` 实例化出来，再在属性填充阶段注入 `ServiceB`。当它去创建 `ServiceB` 时，又需要 `ServiceA`，这时如果没有提前暴露机制，就会发现 `ServiceA` 还没走完整个生命周期。

从机制上可以拆成三层：

1. Spring 创建 Bean 不是一步到位，而是先实例化、再填充属性、再初始化。
2. 只要在属性填充阶段出现互相依赖，就可能触发循环依赖问题。
3. 问题的关键不在于“有没有对象”，而在于“对方要的那个对象能不能在当前阶段被安全引用”。

所以循环依赖题真正要回答的是：为什么对象创建分阶段这件事，会让“半成品对象引用”变成一个必须被处理的问题。

### 3.2 为什么 Spring 只能解决一部分循环依赖，而不是所有情况都能解决

这一节在回答什么问题：

- Spring 的能力边界到底在哪。
- 为什么面试里不能把这题答成“Spring 可以解决循环依赖”。

这个问题是怎么解决的：

Spring 只能解决一部分循环依赖，因为它依赖的前提是：对象能够先被实例化，再在后续阶段把“早期引用”暴露出去。这通常只适用于单例 Bean、且依赖发生在属性填充阶段的场景。对于构造器注入、原型 Bean 等情况，这个前提不成立，所以 Spring 解决不了。

把这个边界放回场景里会更直观。如果 A 和 B 都通过构造器注入依赖彼此，那么 A 连第一步实例化都过不去，因为创建 A 前就必须先拿到完整 B，而创建 B 前又必须先拿到完整 A。这个时候连“提前暴露一个半成品对象”的机会都没有，所以循环依赖不再是缓存策略问题，而是创建顺序本身无法展开的问题。

这也是为什么这题不能只背结论，而要讲清前提：

1. 必须先能拿到实例化后的对象。
2. 依赖最好发生在属性填充阶段，而不是构造阶段。
3. 作用域必须允许容器缓存和复用这个早期引用。

所以更准确的表述不是“Spring 能解决循环依赖”，而是“Spring 能解决一部分单例 Bean、属性注入场景下的循环依赖”。超出这个前提，容器就没有足够的时机和结构把问题化解掉。

### 3.3 三级缓存到底在解决什么问题，为什么不是二级缓存就够了

这一节在回答什么问题：

- 三级缓存的本质作用是什么。
- 为什么它不是单纯多放一层 Map。

这个问题是怎么解决的：

三级缓存的核心，不是为了把缓存层数搞复杂，而是为了管理循环依赖处理过程中同一个 Bean 的三种不同状态：

1. 这个 Bean 已经完全创建好了，可以直接给别人用。
2. 这个 Bean 还没创建完，但已经生成过一个“早期引用”，别人可以先拿去解环。
3. 这个 Bean 还没创建完，也还没真正生成早期引用，但容器需要保留一种“如果后面有人来要，我再按需生成”的能力。

所以三级缓存真正要解决的，不是“缓存命中率”，而是下面三个问题：

- Bean 还没初始化完成时，能不能先暴露一个引用给别的 Bean。
- 这个提前暴露的引用，到底应该是原始对象还是代理对象。
- 这个早期引用一旦生成，能不能保证后续所有地方拿到的是同一个结果，而不是一会儿原始对象、一会儿代理对象。

换句话说，三级缓存真正解决的是“提前暴露 + 延迟决策 + 引用一致性”，而不是简单分层。

#### 3.3.1 先看最简单的问题：为什么需要提前暴露

先看一个没有 AOP 的最小例子：

- `AService` 依赖 `BService`
- `BService` 又依赖 `AService`
- 两边都是单例，且通过字段或 setter 注入

创建过程大致是：

1. Spring 开始创建 A。
2. A 已经实例化出来，但属性还没填完，也还没初始化完成。
3. Spring 发现 A 需要 B，于是转去创建 B。
4. B 在注入时又发现自己需要 A。

这时问题就来了：A 还没完全创建好，但 B 已经急着要 A。如果 Spring 什么都不做，就会卡死在“等 A 完成”和“先给 B 一个 A”之间。

所以 Spring 必须支持一件事：

- A 还没完全创建好时，先把一个“可引用的早期 A”交给 B。

这就是提前暴露的意义。循环依赖真正难的，不是“两个类互相引用”，而是“Bean 还处在创建中，就要先把引用借出去”。

#### 3.3.2 为什么二级缓存看起来能跑，但一般场景不够

如果只看上面的简单例子，你很容易觉得：

- 一级缓存放成品对象
- 二级缓存放提前暴露的对象

好像就够了。

这个直觉在“没有 AOP、没有事务代理”的简单场景里不完全错，因为这时提前暴露原始对象，看起来也能把环解开。

但 Spring 设计的不是只支持最简单场景，而是要支持通用场景，尤其是 AOP / 事务代理介入后的情况。问题就在这里变复杂了。

看一个更关键的例子：

- `AService` 依赖 `BService`
- `BService` 依赖 `AService`
- `AService` 上有 `@Transactional`，最终会被代理

这时如果只有二级缓存，Spring 在 A 刚实例化完时就必须做决定：到底往二级缓存里放什么？

第一种做法：提前放原始 A

- 好处是简单，A 一实例化完就能放进去。
- 但问题是：B 注入到的是原始 A，而容器最后对外暴露的可能是代理 A。

结果就会出现：

- B 手里拿的是原始 A
- 容器最终放出去的是代理 A

这会导致同一个 Bean 出现两种语义不同的引用。B 后面调用 A 时，可能绕过事务或其他 AOP 增强。

第二种做法：提前放代理 A

- 这能解决引用不一致问题。
- 但问题是：A 还没真正被人依赖时，你就已经把“到底要不要生成代理、生成什么代理”这个决定做死了。
- 更重要的是，很多 Bean 最终根本不会发生循环依赖，也不需要这么早生成早期代理。

所以二级缓存不够，不是因为“它少一层”，而是因为：

- 它只能存“已经决定好的结果”
- 却无法优雅表达“先别急着决定，等真正有人来拿时再生成”

而 Spring 恰恰需要这种“延迟决定”的能力。

#### 3.3.3 第三级到底多做了什么

第三级缓存 `singletonFactories` 放的不是对象本身，而是一个对象工厂，可以理解成：

- “如果后面真的有人需要这个 Bean 的早期引用，你再来调我，我到时候决定返回什么。”

这就把“是否生成早期引用、生成原始对象还是代理对象”的决策，从“实例化刚完成时”推迟到了“真正发生依赖查找时”。

所以三层缓存更准确的分工是：

1. 一级缓存 `singletonObjects`
   - 放完全初始化完成的单例成品 Bean。
2. 二级缓存 `earlySingletonObjects`
   - 放已经生成出来的早期引用。
3. 三级缓存 `singletonFactories`
   - 放还没有真正生成早期引用前，用来按需生成它的工厂。

这里最关键的一点是：

- 三级缓存存的是“生成能力”
- 二级缓存存的是“生成结果”

这两个角色不能混。

#### 3.3.4 用一个带事务代理的例子把三级缓存走一遍

继续用上面的例子：

- `AService` 依赖 `BService`
- `BService` 依赖 `AService`
- `AService` 上有 `@Transactional`

创建链路可以这样理解：

1. Spring 开始创建 A，先实例化出原始 A。
2. A 还没初始化完成，Spring 不会立刻把某个具体对象塞进二级缓存，而是先往三级缓存里放一个工厂。
3. 这里的工厂不是用来“重新创建一个 A”，而是和当前这个已经实例化出来的原始 A 绑定在一起。更准确地说，它是在保留一种能力：如果后面有人来拿 A 的早期引用，我就基于当前这个原始 A，通过 `getEarlyBeanReference` 判断应该返回原始 A 还是早期代理 A。
4. 接着 Spring 发现 A 需要 B，于是去创建 B。
5. B 在属性注入时又发现自己需要 A。
6. 这时 Spring 先查一级缓存，没有；再查二级缓存，也没有；最后查到三级缓存里有 A 的工厂。
7. Spring 调用这个工厂，生成 A 的早期引用：
   - 如果 A 不需要代理，就返回原始 A。
   - 如果 A 需要代理，就返回早期代理 A。
8. 生成好的早期引用会被放进二级缓存，同时三级缓存里的工厂被移除。
9. B 拿到这个早期引用后完成创建。
10. 最后 A 再继续完成自己的属性填充、初始化和后续流程，并作为最终单例进入一级缓存。也就是说，第 1 步实例化出来的原始 A 并没有被丢掉，它仍然是后续依赖注入、初始化、代理生成所依赖的目标对象；三级缓存里的工厂只是把“基于这个原始 A 生成什么样的早期引用”这个决定延迟了。

这里第三级最关键的价值就是：

- 在“真正有人要 A 之前”，Spring 不提前做死决策。
- 一旦真的有人要了，再生成一次早期引用，并把结果固定到二级缓存里。
- 这样 B 拿到的 A，和容器最终对外暴露的 A，才更有机会保持一致。

#### 3.3.5 为什么还需要二级缓存，不能只有一级和三级

这也是个容易被追问的点。

如果只有一级和三级，没有二级，那么每次有人来拿早期 A 时，都得重新调用工厂。这样会带来两个问题：

- 早期引用可能被重复生成，尤其在代理场景下，会增加引用不一致风险。
- “到底暴露原始对象还是代理对象”这个决策会被反复执行，结果不够稳定。

所以二级缓存的作用不是多余，而是把三级缓存已经生成出来的“早期引用结果”固定下来，保证后面别人再来拿时，拿到的是同一个对象。

可以把这三者压成一句话：

- 三级缓存负责延迟生成。
- 二级缓存负责固定早期结果。
- 一级缓存负责保存最终成品。

#### 3.3.6 这一节真正要记住什么

- 三级缓存不是“二级缓存再多一层”，而是把“按需生成早期引用”和“缓存已经生成好的早期引用”拆开了。
- 没有 AOP 的简单场景下，二级缓存看起来像是也能工作；但 Spring 要解决的是通用场景，尤其是代理介入时的引用一致性问题。
- 真正难的不是“先把对象放出去”，而是“什么时候放、放什么、放出去后是不是所有地方都拿到同一个引用”。
- 所以三级缓存的本质，不是层数，而是三件事：
  - 提前暴露
  - 延迟决策
  - 代理一致性

### 3.4 哪些场景仍然解决不了，为什么代理介入后结果会不一样

这一节在回答什么问题：

- 为什么 Spring 不是万能修复器。
- 为什么构造器注入、原型 Bean、AOP 代理介入时，循环依赖会有不同结果。

这个问题是怎么解决的：

Spring 处理循环依赖的能力，建立在“单例 + 属性注入 + 可提前暴露引用”这套前提上。只要这套前提被破坏，结果就会变。构造器注入的问题在于对象连实例化都还没完成，就已经要求拿到完整依赖；原型 Bean 的问题在于它本来就不走单例缓存复用主线，容器也很难像单例一样稳定管理它的早期引用；AOP 代理介入时，问题则从“能不能暴露对象”进一步升级成“暴露出去的到底该是原始对象还是代理对象”。

所以这些场景的差异，不是零散边界，而是在检验循环依赖处理的三个核心前提有没有被满足：

1. 有没有机会先实例化出一个对象。
2. 容器能不能在后续阶段继续管理并复用这个对象引用。
3. 这个早期引用和最终容器里的对象语义能不能保持一致。

这也是为什么面试官会追问构造器注入、原型 Bean 和代理问题。它们不是额外刁难，而是在确认你有没有真正理解循环依赖处理依赖的前提条件。一旦把这三个前提讲清楚，很多边界就不会再显得像背诵题。

## 4. 边界、误区、常见追问

- 不要把循环依赖讲成“两个类互相引用”，重点是 Bean 创建期和依赖注入期的冲突。
- 不要把三级缓存答成单纯记忆题，关键是“提前暴露 + 代理一致性”。
- 构造器注入场景通常解决不了，因为连实例化都过不去。
- Prototype Bean 不适合依赖这套单例缓存思路。
- 面试官经常继续追问：如果有 AOP 代理，提前暴露出来的到底应该是谁。

## 5. 自测清单

- 你能不能先不背缓存名，直接讲清循环依赖为什么会发生。
- 你能不能解释为什么 Spring 只能解决一部分循环依赖。
- 你能不能说明三级缓存的关键不是层数，而是“提前暴露 + 代理一致性”。
- 你能不能讲清构造器注入为什么通常解决不了。
- 你能不能说明代理介入后，为什么循环依赖问题会进一步变复杂。
- 你能不能用 1-2 分钟把“为什么需要三级缓存”讲顺。
