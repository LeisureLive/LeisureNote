## 1. 这一块在解决什么问题

Redis 面试不要只答“它快”，而是要理解：

1. 为什么它适合做缓存和高频访问存储
2. 使用缓存后会引入哪些稳定性问题
3. Redis 自己的可靠性和一致性边界在哪里

## 3. 核心重点展开

### 3.1 Redis 为什么快

推荐从四个角度讲：

- 基于内存，避免磁盘随机 IO
- 数据结构针对典型场景做了优化
- 单线程命令执行路径减少锁竞争和线程切换
- 网络模型和协议相对高效

要顺手补一条边界：

- 快不代表没有性能边界
- 大 key、热 key、阻塞命令、持久化压力都会让性能明显下降
### 3.2 数据结构和使用场景

不要只背 API，要记“为什么用这个结构”：

- `String`：
  - 缓存对象、计数、分布式锁基础
- `Hash`：
  - 存对象字段，适合局部读写
- `List`：
  - 消息队列、简单有序集合
- `Set`：
  - 去重、集合运算
- `ZSet`：
  - 排行榜、延迟队列、带权重排序
### 3.3 穿透、击穿、雪崩

这三题不要只背定义，要记“问题来源 + 对应手段”。

- 穿透：
  - 数据本来就不存在
  - 处理手段：
    - 参数校验
    - 布隆过滤器
    - 空值缓存
- 击穿：
  - 热点 key 失效瞬间，大量请求回源
  - 处理手段：
    - 热点不过期
    - 互斥重建
    - 逻辑过期
    - 提前续期
- 雪崩：
  - 大量 key 同时失效，或者 Redis 整体不可用
  - 处理手段：
    - 过期时间打散
    - 多级缓存
    - 限流降级
    - 高可用部署

这部分继续深挖时，优先补：

- 热点 key 重建失败怎么办
- 雪崩时为什么不能无脑回源
### 3.4 RDB、AOF、主从、哨兵、集群

#### 1）持久化先把取舍讲清

- RDB：
  - 恢复快
  - 文件紧凑
  - 但快照间隔期间有数据丢失风险
- AOF：
  - 更有利于降低数据丢失窗口
  - 但文件更大，重写和恢复成本要考虑

#### 2）高可用先按层次理解

- 主从：
  - 解决读扩展和基础冗余
- 哨兵：
  - 解决故障发现和主从切换
- 集群：
  - 解决更大规模的数据分片和高可用
### 3.5 缓存一致性和工程边界

资深岗回答 Redis，最好不要停在“性能”层。

至少要能说到：

- Redis 和数据库双写会有顺序问题
- 写 DB、删缓存是常见思路，但也有失败窗口
- 需要结合：
  - 重试
  - 补偿
  - 延迟双删
  - 消息通知

还要知道 Redis 的边界：

- 它适合做高性能缓存
- 但不适合替代强一致主存储
## 4. 推荐表达顺序

如果面试官问 Redis，推荐按下面顺序讲：

1. 先说性能来源：
   - 内存、数据结构、线程模型、IO 路径
2. 再说工程问题：
   - 穿透、击穿、雪崩、大 key、热 key
3. 再说可靠性和一致性：
   - RDB / AOF、高可用、缓存和数据库双写边界

## 5. 自测清单

- Redis 为什么快，能否避免只说“纯内存”
- 不同数据结构为什么适合不同业务场景
- 穿透、击穿、雪崩为什么容易被混淆，怎么区分
- RDB 和 AOF 的核心取舍是什么
- 主从、哨兵、集群各自解决什么问题
- 缓存一致性为什么是 Redis 面试里真正决定深度的部分
