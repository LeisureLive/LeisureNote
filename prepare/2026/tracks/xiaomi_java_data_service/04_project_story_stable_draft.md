这份文档用于把核心项目整理成“可直接上场”的稳定表达。

目标：

- 先解决“项目主线不稳”的问题
- 再解决“可以讲，但一追问就散”的问题
- 默认每个项目至少准备：
  - 1 分钟版
  - 3 分钟版

## 1. 项目一：实时数据订阅 + 实时查询

### 1.0 代码仓库与模块映射

- 仓库：
  - `horizon-subscription`
  - `sensors-hubble`
- 关键模块：
  - `horizon-subscription/subscription-server`
    - 承担订阅配置、元数据、RPC 与内部服务编排
  - `horizon-subscription/subscription-sender`
    - Flink 发送作业，消费 Kafka 流，做消息转换，再通过 `SenderSink` 发往下游
  - `horizon-subscription/subscription-receiver`
    - gRPC 接收服务，接收上游发送结果并负责落到订阅缓存 / 队列
  - `horizon-subscription/subscription-scheduler`
    - 定时任务与补偿类逻辑
  - `sensors-hubble/sensors-hubble-subscribe-server`
    - 实时标签 / 画像订阅服务，提供 gRPC 接口与任务调度
  - `sensors-hubble/sensors-hubble-subscribe-stream-job`
    - Flink 流作业，消费 profile/item/tag 相关流并写入下游存储

### 1.0.1 已确认的关键信息

- 数据来源：
  - 用户属性数据来自数仓服务，写入 Kudu 后，再通过 Kudu duplication 订阅出来，经处理后发送到订阅服务
  - 其他类型数据来自导入流，在导入流处理并入库时分流发送到订阅服务
- 查询侧存储：
  - 使用 Pegasus Client
  - 主要采用 `hashKey + sortKey + value` 的存储模型
- 容错与监控：
  - 重试主要依赖 Flink，整体保证 `at-least once`
  - 各模块处理过程中补了去重策略，尽量控制重复消费影响
  - 监控告警主要依赖 Prometheus
- 量级与扩容：
  - 数据量级取决于导入链路整体量级
  - 3 节点下峰值 QPS 接近 `2w/s`
  - 主要通过提高 Flink job 并行度扩容
  - 当前默认并行度为 3，单 task 的 `slotNum` 也配置为 3，因此默认只启动 1 个 taskManager
- 个人负责边界：
  - 发送链路主要负责 sender 这一块，整体基本都有参与
  - 重点主导 sender 缓存结构设计
  - 重点优化 sender 发送攒批逻辑
  - 补充了发送目标为自定义 Kafka 的能力，并支持认证

### 1.1 1 分钟版

我这边有一个比较能体现数据服务场景的项目，是内部的实时数据订阅和实时查询系统。上游一部分是数仓服务，比如用户属性会先写 Kudu，再通过 Kudu duplication 订阅出来；另一部分数据来自导入流，在处理并入库时会分流到订阅服务。这个系统本质上是给内部业务方提供统一的数据消费能力，不是简单把消息转发出去，而是要把进入平台的事件流按订阅配置做规则匹配、字段加工，再投递到不同目标端。  
我主要参与的是 sender 这一块，整体基本都有参与，重点主导了 sender 缓存结构设计和发送攒批逻辑优化，还补了自定义 Kafka 发送目标并支持认证。这个项目里一个很实际的难点，是处理过程依赖元数据，但元数据更新没有强通知机制，只能靠缓存刷新，所以会有短暂不一致窗口。围绕这个问题，我参与做了链路优化和扩展性改造，在 3 节点下峰值 QPS 可以接近 2w/s，也可以通过提高 Flink 并行度继续扩。

### 1.2 3 分钟版

这个项目的背景是，内部很多业务方都需要实时拿数据做消费或者查询，如果每个业务自己去对接原始链路，接入成本会很高，而且规则、字段和目标端协议也很难统一。所以平台侧就做了一套统一的数据服务能力，核心包括实时订阅和实时查询。上游来源主要有两类：一类是数仓服务，比如用户属性会先写到 Kudu，再通过 Kudu duplication 拉出来；另一类是导入流，导入流在处理并入库时会顺手把数据分流到订阅服务。

这个项目真正难的地方，不是消费消息以后再发一下，而是中间这层运行时处理很复杂。因为一条事件可能会命中多个订阅关系，扇出之后发送量会放大；发送前还要结合元数据做字段加工、映射或者过滤；而且不同下游目标端能力差异很大，慢下游、失败重试、隔离治理这些问题都会直接影响主链稳定性。

我主要参与的是数据处理和发送这条链路，重点在 sender。这里面一个很典型的难点，是处理过程依赖元数据，但元数据更新没有强通知机制，所以系统只能依赖缓存刷新，这就会带来短暂不一致窗口。这个问题决定了你不能只盯吞吐，还要同时考虑一致性窗口、失败补偿和下游隔离。

我在这个项目里比较有代表性的工作，主要有两块：

- 一块是 sender 链路的性能和扩展性优化，重点包括缓存结构设计和发送攒批逻辑优化。
- 一块是能力扩展，补了发送目标为自定义 Kafka 的场景，并支持开启认证。
- 再往上一层，是把这套能力逐步往平台化方向推进，也就是不只是给单一场景做功能，而是考虑不同订阅规则、不同目标端和运行时治理怎么统一承接。

从仓库结构上也能看到它是分层的，不是一个单模块服务：

- `subscription-server` 和 `hubble-subscribe-server` 更偏控制面，负责订阅配置、元数据、任务和 gRPC 服务。
- `subscription-sender` 和 `subscribe-stream-job` 更偏执行面，负责消费 Kafka 流、做消息转换和实际发送。
- `subscription-receiver` 更像接收面，负责通过 gRPC 接收并继续落到缓存或者队列。

查询这边的存储没有直接走传统关系库，而是用了 Pegasus Client，底层更接近 `hashKey + sortKey + value` 这样的模型，这也比较适合用来回答“为什么查询不是直接查事务库”这类问题。

容错这边，重试主要依赖 Flink，本身保证的是 `at-least once`，然后我们在具体模块处理时补了去重策略，尽量控制重复处理带来的影响。监控告警主要依赖 Prometheus。

所以这个项目我更愿意把它讲成一条数据服务平台链路。它比较能体现我的地方，不只是写了某个模块，而是我在 sender 这一层做过比较完整的设计和优化，同时对上游链路、查询存储、容错和扩展方式也有比较系统的理解。

## 2. 项目二：id-mapping

### 2.0 代码仓库与模块映射

- 仓库：
  - `horizon-inflow`
  - `horizon-identity`
- 关键模块：
  - `horizon-inflow/application/inflow-handlers`
    - 实时入流处理链，包含 `IdMappingHandler`、`SubscribeDataHandler` 等核心 handler
  - `horizon-inflow/application/inflow-chain`
    - 实时 pipeline、source、router、worker 组织
  - `horizon-inflow/application/inflow-importer`
    - 批量导入 Flink 作业与批处理链路
  - `horizon-identity/horizon-identity-engine-v2`
    - 批量 ID mapping 处理、策略和状态管理
  - `horizon-identity/horizon-identity-jobscheduler`
    - 离线 remapping、修复和调度任务
  - `horizon-identity/horizon-identity-skv-proxy`
    - 高性能 SKV 代理层，负责 HTTP 服务、队列、锁和本地缓存
  - `horizon-identity/horizon-identity-sink`
    - remapping 结果向 Infinity / SDW 侧的落地

### 2.1 1 分钟版

`id-mapping` 这块我做的是一套身份映射相关的数据处理服务。这个系统历史包袱比较重，所以难点不只是把功能做出来，而是要在兼容旧逻辑的前提下持续演进。我参与得比较多的一块，是围绕性能瓶颈推进高性能模式改造，通过保留高频主路径、裁剪部分低频能力，同时减少对单点模块 `skv-proxy` 的依赖，把单机处理性能从 3k-4k/s 提升到了 10k+/s，并且支持横向扩展。这个项目比较能体现我的地方，是它不是单纯做优化，而是要在历史兼容、功能边界和处理性能之间做取舍。

### 2.2 3 分钟版

`id-mapping` 这类系统的难点，主要在于它不是一个新系统，而是历史债务比较重，新功能还要兼容旧逻辑，所以你没有办法像做新服务那样直接推翻重来。很多时候真正难的是怎么在可兼容、可迁移、可回退的前提下把主路径做快。

我参与比较多的是高性能模式改造。原来的处理链路里，一方面历史兼容逻辑比较多，另一方面也依赖一些单点模块，所以单机处理性能大概只有 3k-4k/s，扩展性也比较受限。后来我们围绕主路径做了比较明确的取舍：保留高频核心能力，裁掉一部分低频路径，同时尽量减少对单点模块 `skv-proxy` 的依赖，最后把单机性能提升到 10k+/s，并且支持横向扩展。

这个项目最值得讲的，不只是“我把性能做上去了”，而是为什么这种优化一定伴随边界重划。因为高性能模式不是白来的，本质上是你要决定哪些能力必须保留，哪些低频能力可以不再走同一条主路径，哪些历史包袱必须继续兼容，哪些可以拆开治理。

从实际代码链路上看，这个系统也不是一个单模块服务，而是几层协同：

- 在入流侧，`IdMappingHandler` 会把事件按批次送进 `IdMappingProcessor`。
- 在引擎侧，`IdMappingProcessor` 会初始化 `IdentityEngineContext`，通过 `SKVProxyClient` 调 identity engine 做批量处理，`IDBatchHandler` 和 `MergeIfPossibleStrategy` 负责核心的映射和合并逻辑。
- 在代理层，`SKVProxyService` 提供高性能 HTTP 服务，内部有队列、线程池、本地缓存和多 key 锁，承接高频读写。
- 在调度侧，jobscheduler 负责离线 remapping、修复和任务同步。

所以这个项目我比较愿意讲成一条协同链路，而不是只讲某个 engine 策略或者某个接口优化。因为真正把性能做上去，通常不是只改一个点，而是要把入流处理、存储代理和离线修复能力一起考虑。

## 3. 可以直接对外讲的链路总结

### 3.1 订阅 + 实时查询

可以压成一句话：

- 这是一个典型的数据服务平台链路，控制面负责订阅元数据和任务编排，执行面用 Flink / Kafka 处理实时流并进行消息转换和下发，接收面通过 gRPC 和缓存 / 队列承接下游消费。

### 3.2 导入流 + id-mapping

可以压成一句话：

- 这是一个典型的“实时入流 + identity engine + 高性能存储代理 + 离线 remapping 调度”协同系统，既要处理实时映射，也要处理批量导入、历史修复和兼容性演进。
