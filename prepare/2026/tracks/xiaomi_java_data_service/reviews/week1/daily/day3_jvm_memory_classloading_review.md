## 4. 重点与失分点拆解

### 4.1 内存结构回答还不够像“运行时模型”

- 现象：
  - 能列出程序计数器、栈、堆、方法区、直接内存
  - 但第一反应还不是“职责 + 典型问题”
- 原因：
  - 目前更像记忆名词，还没有完全变成运行时视角
- 后续复盘重点：
  - 固定回答顺序：
    - 先按线程私有 / 线程共享分类
    - 再补各区域职责
    - 最后补典型异常和边界
- 下次建议直接这样答：
  - JVM 内存结构我一般先按线程私有和线程共享来讲
  - 线程私有：
    - 程序计数器
    - 虚拟机栈
    - 本地方法栈
  - 线程共享：
    - 堆
    - 方法区
  - 另外还经常一起问直接内存
  - 然后补各自职责：
    - 栈管方法执行现场
    - 堆管对象实例
    - 方法区管类级别信息
    - 直接内存管部分堆外 IO 场景
  - 最后再补边界：
    - JVM 内存结构和 JMM 不是一回事
    - 方法区和元空间不是同一层概念
    - 直接内存不是堆的一部分

### 4.2 对象创建细节还需要更稳的表达

- 现象：
  - 五步主线能背出来
  - 但零值初始化、对象头、直接内存和对象分配位置容易混
- 原因：
  - 对“对象创建为什么是运行时问题”理解刚建立，还没有完全收敛成稳定答法
- 后续复盘重点：
  - 固定只补三件事：
    - 零值初始化解决什么问题
    - 对象头大致有什么信息
    - 为什么“对象主要在堆上”比“对象一定在堆上”更严谨
  - 后续不要再把 NIO 说成“对象直接分配到堆外”
- 下次建议直接这样答：
  - 对象创建不是执行一个 `new` 就结束了
  - JVM 内部一般至少会经历五步：
    - 类加载检查
    - 内存分配
    - 零值初始化
    - 设置对象头
    - 构造初始化
  - 零值初始化的作用是：
    - 保证默认值语义成立
    - 避免旧内存脏数据残留
    - 让对象先处于合法、可预测的基础状态
  - 对象头可以先粗略讲成：
    - `Mark Word`
    - 类型指针
  - 最后主动补一句：
    - 绝大多数对象分配在堆上
    - 但在逃逸分析等优化下不能绝对化成“一定在堆上”

### 4.3 类加载追问链路还不够稳

- 现象：
  - `Loading / Linking / Initialization` 能说出来
  - 但 `Resolution` 和 `loadClass / findClass / defineClass` 一开始解释不顺
- 原因：
  - 只记了阶段名，没有把“类名 -> 字节流 -> 定义类”的链路完全讲清
- 后续复盘重点：
  - 固定两条线：
    - `Resolution`：
      - 符号引用 -> 运行时可直接定位的引用
    - `loadClass / findClass / defineClass`：
      - 入口调度
      - 查找字节流
      - 定义成 `Class`
  - 后续口答时不要一上来就讲源码方法细节
- 下次建议直接这样答：
  - 类加载主流程我一般按：
    - `Loading`
    - `Linking`
    - `Initialization`
    来讲
  - `Loading`：
    - 根据类名获取字节流
    - 读入 JVM
    - 构造运行时类结构
  - `Linking`：
    - `Verification`
    - `Preparation`
    - `Resolution`
  - `Resolution` 可以直接解释成：
    - 把 class 文件里按名字写的符号引用
    - 解析成运行时可以直接定位目标的引用
  - 如果被追问类加载器怎么找类：
    - `loadClass` 负责整体调度
    - `findClass` 负责查找字节流
    - `defineClass` 负责把字节流定义成 `Class`

### 4.4 双亲委派和类身份的表述还需要标准化

- 现象：
  - 知道双亲委派的价值
  - 但流程表达容易不够标准
  - 类身份的结论能说，但跨类加载器影响还不够自然
- 原因：
  - 主线已经理解，但缺少“稳定答案模板”
- 后续复盘重点：
  - 双亲委派固定说法：
    - 先检查是否已加载
    - 再委派父加载器尝试加载
    - 父加载器加载不了，当前加载器再自己加载
  - 类身份固定说法：
    - 类加载器 + 全限定类名
    - 不同加载器定义的同名类，在 JVM 看来可能是不同类型
- 下次建议直接这样答：
  - 双亲委派是类加载时的默认机制
  - 当前类加载器收到请求后：
    - 先检查是否已经加载过
    - 没加载过就委派父加载器先尝试加载
    - 父加载器加载不了，当前加载器再自己加载
  - 它主要解决两个问题：
    - 类唯一性
    - 安全性
  - 类身份则由：
    - 类加载器
    - 全限定类名
    共同决定
  - 所以即使全限定类名一样，只要定义加载器不同，JVM 里也可能是两个不同类型

  - 代码手感在恢复，但题型识别后的口头模板还没完全固定
- 后续复盘重点：
  - 对滑动窗口题固定先说 4 件事：
    - 为什么是滑动窗口
    - 窗口里维护什么状态
    - 什么时候扩张、什么时候收缩
    - 为什么这样不会漏解
  - `3` 这题要主动补：
    - 窗口内维护“无重复字符”
  - `209` 这题要主动补：
    - 数组元素全是正整数，所以窗口和单调
- 下次建议直接这样答：
  - `3` 这题：
    - 连续子串 + 最长 + 窗口内不能有重复字符

    - 连续子数组 + 最短 + 区间和约束
    - 一旦窗口和 `>= target`，就要先更新答案再收缩窗口
    - 正整数保证右扩和只会变大、左缩和只会变小
