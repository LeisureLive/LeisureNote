## 1. 这一块在解决什么问题

Spring 事务这块不是在考你背 `REQUIRED`、`REQUIRES_NEW` 有几个，而是在考你能不能把下面五件事讲清楚：

1. 为什么业务代码需要 Spring 来统一托管事务边界。
2. 声明式事务为什么能成立，它和 AOP / 代理是什么关系。
3. 隔离级别和底层数据库事务能力分别在解决什么问题，为什么它们不能被 Spring 注解单独替代。
4. 事务传播行为到底在解决什么问题，什么时候该拆新事务，什么时候应该共用事务。
5. `@Transactional` 为什么经常失效，失效的根因到底是代理边界、异常处理还是方法定义方式。

把这些点串起来，主线其实也很稳定：

- 业务代码要保证多步数据库操作的一致性。
- 如果每个方法都手工开启、提交、回滚事务，维护成本高，也容易出错。
- Spring 希望通过声明式方式把事务边界托管起来。
- 这类托管很多时候依赖代理增强，所以事务题天然会和 AOP 连在一起。
- 但事务不只是“同成同败”，还要看并发下的数据可见性和隔离成本，这就会引出隔离级别和底层数据库边界。
- 一旦调用路径没经过代理，或者异常、方法边界不符合规则，事务就可能失效。

所以这块真正要建立的是三种能力：

- 能解释 Spring 事务在解决什么工程问题。
- 能解释声明式事务为什么能生效，以及它对代理边界的依赖。
- 能解释隔离级别、传播行为和失效场景，而不是只背注解参数。

## 2. 先把主线串起来，再看知识树

### 2.1 先用一条因果链把全文串起来

如果你觉得 Spring 事务容易散，通常不是传播行为没记住，而是没有先抓住下面这条因果链：

1. 先看为什么需要 Spring 事务：多步数据库操作需要一致性边界。
2. 再看声明式事务为什么成立：它不是魔法，本质上还是在调用前后接管事务开启、提交和回滚。
3. 再看它为什么经常和 AOP 一起问：因为很多声明式事务就是基于代理增强实现的。
4. 再看隔离级别为什么不能跳过：因为事务不只要“成败一致”，还要处理并发事务之间看见什么、锁住什么。
5. 再看传播行为为什么存在：因为一个方法可能在另一个事务方法里被调用，需要定义“加入还是新开”。
6. 最后看 `@Transactional` 为什么失效：调用没经过代理、异常没有触发回滚规则、方法边界不符合代理增强条件。

所以这篇文档不是在讲几个注解参数，而是在回答一条连续问题链：

- 为什么要托管事务。
- Spring 为什么能托管事务。
- 隔离级别和数据库边界在解决什么问题。
- 传播行为在解决什么问题。
- `@Transactional` 为什么会失效。
- 遇到失效时应该怎么定位根因。

## 3. 核心重点展开

### 3.1 Spring 事务在解决什么问题，为什么很多系统更偏向声明式事务

这一节在回答什么问题：

- 为什么业务系统需要 Spring 来统一管理事务。
- 为什么声明式事务通常比手工事务更常见。

这个问题是怎么解决的：

Spring 事务在解决的是“多步数据库操作如何稳定地纳入同一个一致性边界”这个问题。声明式事务更常见，是因为它把开启、提交、回滚这些样板逻辑从业务代码里抽离出来，让事务规则集中配置，业务代码更聚焦。它的核心价值不是少写几行代码，而是让事务边界和回滚规则更统一、更可维护。

把这件事放回一个最小场景里更容易理解。比如一个下单操作需要扣减库存、写订单表、写流水表，而这三步必须同成同败。你当然可以在业务代码里手工写 `begin / commit / rollback`，但一旦这种模式在大量方法里反复出现，业务代码很快就会被样板逻辑污染。Spring 的声明式事务，就是把这层公共事务控制抽出去，让事务语义变成统一托管的基础设施。

从机制上看，这件事真正解决的是三类问题：

1. 让多步数据库操作拥有清晰的一致性边界。
2. 让事务控制从业务代码里抽走，减少重复模板和人为疏漏。
3. 让事务规则可以集中配置，而不是散落在每个方法里各自维护。

收益是事务边界更统一、业务代码更干净；代价是你必须理解事务不再是“手写在哪里就生效在哪里”，而是更多依赖 Spring 的托管语义和调用链。

### 3.2 声明式事务为什么能生效，它和 AOP / 代理是什么关系

这一节在回答什么问题：

- Spring 为什么能在你不手工写事务模板代码的情况下管理事务。
- 为什么事务题经常会直接追到 AOP 和代理。

这个问题是怎么解决的：

声明式事务之所以能生效，很多时候是因为 Spring 用代理接管了方法调用。调用进入代理后，事务拦截器先判断这个方法是否需要事务，再决定开启事务、执行目标方法，并在调用结束后提交或回滚。所以事务本质上不是注解本身生效，而是“带事务语义的调用路径”被代理拦住了。

放回场景里看，一个被 `@Transactional` 标注的 Service 方法，外部调用时真正拿到的往往是代理对象：先进入代理，代理发现这个方法需要事务，开启事务，执行目标方法，再根据执行结果提交或回滚。如果调用没有经过代理，这条链就不成立，事务也就可能失效。

这也是为什么事务和 AOP 总是绑在一起讲：

1. `@Transactional` 只是声明“这里需要事务语义”。
2. 真正让它生效的是事务拦截器和代理调用链。
3. 所以事务题最后一定会回到“调用有没有经过代理”这条主线上。

收益是事务控制可以统一接入，不需要把模板逻辑散落进业务方法；边界是只要代理调用链被绕开，声明式事务就不再自动成立。

### 3.3 隔离级别和数据库事务边界在这里解决什么问题，为什么不能脱离底层数据库单独理解

这一节在回答什么问题：

- Spring 事务配置为什么不能脱离数据库事务能力单独理解。
- 隔离级别为什么回答到最后一定会回到并发异常、锁和数据库实现边界。

这个问题是怎么解决的：

隔离级别在这里解决的是“并发事务之间允许看到什么、要避免什么异常”这个问题。Spring 可以帮你声明事务边界，也可以配置隔离级别，但它并不会凭空创造数据库没有的事务能力，真正落地这些隔离语义的还是底层数据库引擎。所以事务题如果只讲 `@Transactional` 和传播行为，不讲隔离级别与数据库边界，就会缺一半。

更准确地说：

- Spring 负责声明和托管事务边界。
- 数据库负责真正实现隔离、锁、可见性和并发控制。

把这件事放回场景里更容易看清。比如一个下单场景里，你希望读取库存时不要读到未提交数据，也希望同一事务里多次读取结果尽量稳定，还要考虑并发更新时是否会有脏读、不可重复读、幻读或锁冲突。这些问题不是 Spring 注解本身能单独解决的，而是要靠底层数据库在当前隔离级别下给出实际语义。Spring 在这里做的，是把“这段代码需要怎样的事务语义”声明清楚。

所以这一节真正要讲清的是：事务不只是“同成同败”，还包括并发执行时的数据正确性；隔离级别越强，通常一致性越稳，但并发代价、锁冲突和吞吐压力也可能更高。回答事务题时，如果能把“Spring 声明层”和“数据库实现层”区分开，整条主线就不会停留在注解参数层面。

### 3.4 事务传播行为在解决什么问题，`@Transactional` 为什么经常失效

这一节在回答什么问题：

- 为什么事务传播行为会存在。
- 失效场景到底该怎么分类理解。

这个问题是怎么解决的：

事务传播行为解决的是“一个事务方法调用另一个事务方法时，到底怎么衔接当前事务边界”的问题。更直白一点说，它在定义下面这些规则：

- 当前线程里已经有事务时，内层方法是加入、挂起还是拒绝这个事务。
- 当前线程里没有事务时，内层方法是自己开一个、无事务执行，还是直接报错。

所以传播行为讨论的不是“事务会不会生效”，而是“如果事务已经存在或不存在，当前方法应该怎么处理”。真正面试里最容易混的是：很多人只会背 `REQUIRED`、`REQUIRES_NEW`，但不知道它们分别在“已有事务”和“没有事务”两种情况下怎么表现。

#### 3.4.1 先用一个统一视角看传播行为

可以把传播行为压成一个二维问题：

1. 外层调用进来时，当前线程里有没有事务。
2. 内层方法希望怎么对待这个事务上下文。

按这个思路，常见传播行为可以这样理解：

1. `REQUIRED`
   - 有事务：加入当前事务。
   - 没事务：自己新开一个事务。
   - 它是默认值，适合绝大多数“同成同败”的业务主流程。

2. `REQUIRES_NEW`
   - 有事务：挂起外层事务，自己新开一个独立事务。
   - 没事务：自己新开一个事务。
   - 适合“我需要独立提交或独立回滚”的动作，比如审计日志、操作流水、补偿记录。

3. `SUPPORTS`
   - 有事务：加入当前事务。
   - 没事务：就按无事务方式执行。
   - 适合“有事务最好，没有也能跑”的查询或弱一致性逻辑。

4. `NOT_SUPPORTED`
   - 有事务：挂起当前事务，自己按无事务方式执行。
   - 没事务：本来就无事务执行。
   - 适合明确不希望被事务包裹的代码，比如某些长耗时读操作、对事务不友好的外部调用。

5. `MANDATORY`
   - 有事务：加入当前事务。
   - 没事务：直接报错。
   - 适合强制要求“调用我时必须已经有事务”的内部基础能力。

6. `NEVER`
   - 有事务：直接报错。
   - 没事务：无事务执行。
   - 适合明确要求“绝不能在事务里调用”的逻辑，但业务代码里实际用得不多。

#### 3.4.2 这几个最常用的传播行为到底怎么选

真正高频的，通常还是下面四个：

- `REQUIRED`
  - 默认优先考虑它。
  - 适合订单创建、库存扣减、账户扣款这类必须同成同败的主链路。

- `REQUIRES_NEW`
  - 当你明确希望“外层回滚不影响我提交”或“我失败也别把主流程一起打回去”时再选。
  - 典型例子是审计日志、发送补偿记录、失败留痕。
  - 代价是事务边界被切开，理解和排障成本也会上去。

- `SUPPORTS`
  - 更像“随缘加入事务”，一般不拿来承载核心写流程。
  - 更适合读方法或可事务可无事务的辅助逻辑。

- `NOT_SUPPORTED`
  - 适合明确不希望被事务包住的逻辑。
  - 比如某些长耗时读操作，或者不想长期占着事务资源的外部调用。

可以用一个最小场景来串起来：

- `createOrder()` 是主流程，扣库存、写订单、写支付单必须一起成功：
  - 通常用 `REQUIRED`
- `recordAuditLog()` 只是记录审计日志，失败不能拖垮主流程：
  - 可以考虑 `REQUIRES_NEW`
- `queryPromotionRules()` 只是查配置，有事务就顺带加入，没有也不强求：
  - 可以考虑 `SUPPORTS`
- `exportLargeReport()` 是一个长耗时导出，不希望长时间占着事务资源：
  - 可以考虑 `NOT_SUPPORTED`

#### 3.4.3 为什么 `@Transactional` 经常失效

传播行为定义的是“事务边界如何衔接”，但它有一个大前提：事务增强本身得先真正生效。也就是说，传播行为不是写上就一定能执行到，前面还有事务代理链这道门。

常见失效场景可以稳定归成三类：

1. 代理边界问题
   - 同类内部调用：`this.methodB()` 或直接 `methodB()` 会绕过代理。
   - 自己 `new` 对象，不经过 Spring 容器。
   - 初始化阶段调用事务方法，比如在 `@PostConstruct` 里依赖事务。

2. 方法边界问题
   - `private`、`final`、代理不可达的方法边界，会让事务增强无法正常拦截。
   - 接口代理和类代理的可增强边界也不完全一样。

3. 回滚规则问题
   - 异常被自己吞掉了，事务拦截器看起来就是“正常返回”。
   - 抛的是 checked exception，但没有配 `rollbackFor`。
   - 传播行为本身配得没问题，但事务语义和你的预期不一致。

还可以再补一个非常高频、但容易误判成“事务失效”的场景：

- 新线程 / `@Async`
  - 这更多是事务上下文和线程边界问题，不等于外层事务代理没生效。

#### 3.4.4 这一节真正要记住什么

所以这组问题真正要统一起来看的是四件事：

1. 传播行为讨论的是“事务怎么衔接”，不是“事务一定会生效”。
2. 面试里最常用的是 `REQUIRED`、`REQUIRES_NEW`、`SUPPORTS`、`NOT_SUPPORTED`，重点是讲清“已有事务 / 没有事务”两种情况下分别怎么表现。
3. 生效前提仍然是调用必须走代理、方法边界必须可增强、异常和回滚规则必须符合预期。

这样回答的好处是，传播行为和失效问题不会被拆成两套孤立知识点，而是都能回到“事务调用链怎么衔接、怎么生效、为什么失效”这一条主线上。

## 4. 边界、误区、常见追问

- 不要把事务题答成只会背传播行为枚举。
- `@Transactional` 生效的关键不是注解写上了，而是调用链是否经过代理。
- 同类内部调用是最经典的失效场景。
- 默认回滚规则通常和异常类型有关，异常被吞也会影响结果。
- Spring 事务配置不能脱离底层数据库事务能力单独理解。

## 5. 自测清单

- 你能不能先不背传播行为，直接讲 Spring 事务在解决什么问题。
- 你能不能说明为什么 `@Transactional` 的关键是“有没有走到代理”。
- 你能不能说明隔离级别为什么不能脱离底层数据库能力单独理解。
- 你能不能用场景解释 `REQUIRED` 和 `REQUIRES_NEW` 怎么选。
- 你能不能把事务失效归类成代理边界、方法边界、异常回滚规则三类。
- 你能不能用 1-2 分钟把“为什么事务会失效”讲顺。
