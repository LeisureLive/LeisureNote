## 1. 这一块在解决什么问题

Spring AOP 这块不是在考你记住了几个注解，而是在考你能不能把下面四件事讲清楚：

1. 为什么业务系统需要一种“不改业务代码也能加公共逻辑”的机制。
2. Spring 为什么没有直接改字节码主逻辑，而是大多通过代理做增强。
3. JDK 动态代理和 CGLIB 各自依赖什么前提，分别适合什么场景。
4. AOP 能解决什么，不能解决什么，为什么它经常和事务、日志、鉴权一起出现。

如果把这些点串成一条主线，可以这样理解：

- 业务代码里总会出现日志、事务、权限、监控这类横切逻辑。
- 如果把这些逻辑硬写进每个方法，代码会重复、耦合、难维护。
- 所以 Spring 需要一种在目标方法前后插入增强逻辑的机制。
- 这类机制在 Spring 里主要靠代理对象实现，而不是直接修改原对象行为。
- 代理方案不同，就会引出 JDK Proxy 和 CGLIB 的差异。
- 代理边界不同，又会继续引出自调用失效、方法可见性、final 限制这些常见追问。

所以 AOP 这块真正要建立的是三种能力：

- 能解释 AOP 在解决什么工程问题。
- 能解释 Spring AOP 为什么通常是“代理增强”而不是“直接织入全部字节码”。
- 能解释 AOP 的边界，以及它为什么经常和事务失效问题连在一起问。

## 3. 核心重点展开

### 3.1 为什么系统需要 AOP，而不是把公共逻辑直接写进业务代码

这一节在整条主线里解决的是第一个问题：

- AOP 到底在解决什么问题。
- 为什么它不是一个“语法花活”，而是工程化分层手段。

#### 先直接回答这个问题

系统需要 AOP，本质上是因为很多逻辑不是业务主流程的一部分，但又会反复出现在大量方法里，比如日志、事务、权限、监控、审计。如果把这些逻辑都硬塞进业务代码，结果通常是：

- 代码重复。
- 主流程被横切逻辑淹没。
- 改一个公共能力时要改很多地方。

所以 AOP 的价值不是“让代码更炫”，而是把横切关注点从业务主逻辑里抽出来，让业务代码更聚焦，也让公共能力更容易统一治理。

#### 再用一个最小场景把问题钉住

例如订单服务里的多个方法都要做：

- 打入口日志。
- 鉴权。
- 开事务。
- 统计耗时。

如果这些逻辑都写在每个方法里，业务代码很快就会变成“重复模板 + 少量真实逻辑”。AOP 的作用就是把这类重复模板抽出去，让调用进入方法前后自动执行公共逻辑。

#### 是什么

AOP 可以先理解成一种“把横切逻辑挂到目标方法调用链上”的机制。

它不是替代面向对象，而是在 OOP 之上补一层“跨多个对象都要生效的统一增强能力”。

#### 为什么有它

它要解决的是：

- 公共逻辑重复。
- 业务代码和治理逻辑耦合。
- 横切能力难以统一维护。

#### 怎么理解它的机制

第一层，AOP 不直接改你的业务方法代码，而是在调用进入目标方法之前或之后插入逻辑。

第二层，这种插入逻辑在 Spring 里通常不是靠修改原对象本身，而是靠生成一个代理对象来包住原对象。

第三层，调用方拿到的往往不是原始目标对象，而是代理对象，所以调用会先经过代理，再决定是否执行增强逻辑，最后再进入目标方法。

#### 面试时怎么讲才不空

推荐这样讲：

“AOP 本质上是在解决横切关注点问题，比如日志、事务、权限、监控这类逻辑会散落在很多业务方法里。Spring 通常不会让你手工把这些逻辑写进每个方法，而是通过代理对象在方法调用前后做统一增强。这样做的好处是业务代码更聚焦，公共逻辑也更容易集中治理。”

### 3.2 Spring 为什么大多通过代理实现 AOP，而不是直接改原对象

这一节在整条主线里解决的是第二个问题：

- Spring 为什么能在不侵入业务代码的情况下做增强。
- 为什么“代理”会成为 Spring AOP 的核心实现思路。

#### 先直接回答这个问题

Spring 大多通过代理实现 AOP，是因为代理方案足够通用、和容器结合自然，而且能在运行时对 Bean 做增强。它的核心不是改掉原对象，而是让外部调用先经过代理对象，再由代理决定是否追加前置、后置或环绕逻辑。

这意味着 Spring AOP 的关键不是“目标对象被改写了”，而是“调用路径被接管了”。

#### 再用场景把问题钉住

比如一个 Service Bean 被 Spring 容器托管后，如果它匹配某个切面规则，Spring 可以在创建 Bean 时额外生成一个代理对象：

- 调用方注入的是代理对象。
- 代理对象内部再持有目标对象。
- 每次调用方法时，先经过代理判断需不需要增强。

这就是为什么你写的业务代码没有显式调用日志或事务逻辑，但运行时还是会生效。

#### 是什么

代理可以先理解成“目标对象外面包一层壳”。

调用先到壳，再由壳决定：

- 直接放行。
- 先做增强再放行。
- 包一层环绕逻辑后再调用目标方法。

#### 为什么有它

因为 Spring 需要：

- 不侵入业务代码。
- 在容器创建 Bean 时统一增强。
- 让事务、日志、权限这些公共能力都走统一接入方式。

#### 怎么理解它的机制

第一层，Spring 在 Bean 创建和后处理阶段判断这个 Bean 是否需要增强。

第二层，如果需要增强，就为它生成代理对象。

第三层，外部依赖注入到的通常是代理对象，而不是裸目标对象。

第四层，代理在方法调用时执行通知链，最后再调用目标方法。

#### 面试时怎么讲才不空

“Spring AOP 的核心不是直接修改业务类，而是让容器在合适时机生成代理对象。调用方实际调用的是代理，代理里会串起通知链，再进入目标方法。所以它本质上接管的是调用路径，而不是手工把公共逻辑塞进业务代码。”

### 3.3 JDK 动态代理和 CGLIB 的区别，为什么这题经常和 AOP 一起问

这一节在整条主线里解决的是第三个问题：

- 两种常见代理方案分别依赖什么前提。
- 为什么代理边界会直接影响 AOP 是否生效。

#### 先直接回答这个问题

JDK 动态代理是基于接口生成代理，前提是目标对象实现了接口；CGLIB 是基于继承目标类生成子类代理，不依赖接口，但会受到 final 类和 final 方法限制。它们之所以总和 AOP 一起问，是因为 Spring AOP 落地时必须选一种代理方式，而代理方式决定了增强边界。

#### 再用场景把问题钉住

如果你的 Service 通过接口暴露能力，JDK 动态代理通常就够用；如果目标类没有接口，Spring 就更可能退回到 CGLIB。面试官追问这题，不是为了听你背定义，而是想看你是否知道：

- 为什么有时增强能生效。
- 为什么有时因为方法边界或类定义方式，增强会失效。

#### 是什么

- JDK 动态代理：基于接口。
- CGLIB：基于继承和字节码生成子类。

#### 为什么会有两种方案

因为 Java 世界里并不是所有目标对象都以接口方式暴露能力。Spring 需要同时覆盖“有接口”和“无接口”两类场景。

#### 怎么理解它们的机制和边界

第一层，JDK 动态代理生成的是“实现同一组接口的代理对象”，所以调用必须通过接口方法进入。

第二层，CGLIB 生成的是目标类的子类，通过重写可代理的方法来做增强。

第三层，这就带来典型边界：

- final 类不能被继承。
- final 方法不能被重写。
- private 方法本来就不在可重写边界上。

#### 面试时怎么讲才不空

“JDK 动态代理和 CGLIB 本质上都在解决‘怎么生成代理对象’这个问题。前者基于接口，后者基于继承。Spring 为什么要同时支持两种方案，是因为业务类不一定都实现接口。真正重要的不是背结论，而是知道代理边界会影响增强是否生效，比如 final 方法、private 方法、自调用这些问题，后面都会继续追到事务失效。”

### 3.4 为什么 AOP 这题总会继续追到事务、自调用失效这些问题

这一节在整条主线里解决的是第四、第五和第六个问题：

- 为什么 AOP 题不会只停留在“能不能做增强”。
- 为什么事务题和自调用失效题，本质上经常和 AOP 走的是同一条主线。

#### 先直接回答这个问题

AOP 这题总会继续追到事务和自调用失效，是因为 Spring 里很多横切能力本来就是靠代理挂到调用链上的，事务只是其中最常见的一种。如果你只会说“Spring AOP 用代理实现”，但说不清调用到底什么时候会绕过代理、哪些方法边界会让增强失效，那你的 AOP 理解就还停留在概念层。

所以这类追问真正想确认的是：

- 你知不知道代理接管的是“外部调用路径”。
- 你知不知道只要调用没走代理，AOP 和事务这类增强就都可能一起失效。

#### 再用场景把问题钉住

比如一个 Service 的外部方法被事务代理增强了：

- 外部调用进入代理，事务和日志都能生效。
- 但这个方法内部如果直接 `this.otherMethod()`，调用就可能直接落到目标对象本身。

这时不是只有事务会出问题，而是所有依赖这条代理链的增强能力，都可能被一起绕过去。

#### 是什么

可以先这样理解：

- AOP 负责把横切逻辑挂到方法调用链上。
- 事务增强是 AOP / 代理主线上的典型应用。
- 自调用失效本质上是调用路径没有再经过代理。

#### 为什么会有这些追问

因为面试官不是只想听你懂代理名词，而是想看你能不能把“代理实现机制”和“增强生效边界”连起来。

#### 怎么理解它的机制和边界

第一层，Spring AOP 的很多能力都依赖代理对象接管外部方法调用。

第二层，只要调用从外部进入代理，通知链、事务拦截器这类增强逻辑才有机会执行。

第三层，如果是同类内部调用、不可代理的方法边界、或某些 final / private 限制，调用可能直接落到目标对象，从而绕过增强。

第四层，所以 AOP、事务失效、自调用失效这些问题，本质上都在问同一件事：调用路径有没有经过代理，以及代理能不能覆盖当前方法边界。

#### 面试时怎么讲才不空

“AOP 题之所以总会追到事务和自调用失效，是因为它们很多时候共用同一条代理主线。代理真正接管的是外部调用路径，只要调用没走代理，日志、事务这类增强都可能一起失效。所以我回答 AOP 不会只停在‘JDK Proxy 和 CGLIB 有什么区别’，还会补一句代理边界决定增强边界。”

## 4. 边界、误区、常见追问

- AOP 不适合替代明确业务逻辑，它更适合横切关注点。
- Spring AOP 重点是运行时代理，不要和更强织入能力的 AspectJ 混成一个概念。
- 自调用可能绕过代理，导致增强不生效。
- final 类、final 方法、private 方法会影响代理增强边界。
- 面试官经常从 AOP 直接追到事务，因为两者很多时候共用代理主线。
