## 1. 这一块在回答什么问题

系统设计这块真正要回答的，不是“我能不能列出很多组件”，而是下面这条连续问题：

1. 面对一个真实系统需求，为什么不能一上来就画架构图。
2. 什么信息会直接决定你后面到底该不用缓存、MQ、分库分表、OLAP 引擎还是异步链路。
3. 一条系统链路到底应该怎样拆，才能知道瓶颈会落在写入、查询、状态、存储还是下游依赖。
4. RPC、缓存、消息队列、负载均衡、分布式事务这些常见组件，到底分别在系统里控制什么问题。
5. 为什么高可用、扩展性、监控、容灾不是最后补一句“我们也会做”，而是设计时就要一起进入主线。

所以这篇文档的主线不是“系统设计题模板”，而是先建立一条稳定的判断路径：

- 先明确需求和约束。
- 再估算规模和流量特征。
- 再拆核心链路和状态边界。
- 再把常见分布式组件放到链路里，而不是孤立列举。
- 最后补可用性、扩展性、监控和容灾，形成完整设计闭环。

只有这条主线稳定了，后面做实时数据订阅、实时查询、数据平台设计题时，才不会一题一个答法。

## 2. 主线

1. 先看问题边界：没有需求、时延、规模、一致性目标，后面的架构选择都没有判断依据。
2. 再看核心链路：请求从哪里进、数据在哪里变、结果落到哪里、哪些步骤必须同步、哪些步骤适合异步。
3. 再看状态和瓶颈：系统真正吃紧的是 CPU、内存、磁盘、网络、锁竞争、下游依赖，还是数据分布。
4. 再看分布式组件：RPC、缓存、MQ、负载均衡、事务不是答案本身，而是控制具体问题的手段。
5. 最后看治理闭环：限流、降级、隔离、监控、扩容、容灾，决定系统在故障和增长下还能不能继续工作。

## 3. 核心问题展开

### 3.1 为什么系统设计题不能一上来就画图或堆组件

这一节在回答什么问题：

- 系统设计为什么必须先问需求和约束。
- 为什么不先问清约束，后面的方案很容易看起来完整但其实站不住。

这个问题是怎么解决的：

系统设计不能一上来就画图，是因为系统不是为了使用组件而存在，而是为了满足一组业务目标和工程约束而存在。只要约束没有钉住，所谓架构图通常只是默认假设堆出来的样子，既无法判断是否过度设计，也无法判断是否会在关键位置失手。

真正决定系统设计方向的，通常是下面几类约束：

1. 读多还是写多。
2. 数据是实时处理还是离线处理。
3. 延迟要求是毫秒、秒级还是分钟级。
4. 强一致要求有多强，是否允许最终一致。
5. 峰值流量、数据规模、增长速度大概在什么量级。
6. 是否多租户、是否跨团队使用、是否需要平台化能力。

为什么这些约束这么重要？因为它们会直接把后续设计往不同方向推：

- 如果是高频读、低延迟查询，缓存和索引设计会成为前排问题。
- 如果是高吞吐写入，异步化、批处理、削峰填谷会更重要。
- 如果是分析型查询，MySQL 继续硬扛通常就不是优选。
- 如果是平台化系统，元数据、权限、配额、监控不会是附加功能，而是主能力。

这也是为什么系统设计里先问清约束，本质上是在控制两类风险：

- 方案明显不匹配实际问题。
- 组件虽然都列了，但没有一个真正落在最关键的矛盾上。

代价是前期会花一点时间澄清问题，看起来不像“马上开始设计”；但收益是后面每一步都有判断依据，而不是靠经验硬猜。

### 3.2 一条系统链路到底该怎么拆，才能知道瓶颈和职责落点

这一节在回答什么问题：

- 系统链路应该按什么方式拆。
- 为什么链路拆不清，后面的缓存、MQ、事务、扩容都会讲散。

这个问题是怎么解决的：

系统链路拆分的核心，不是按模块名堆层，而是先把数据和请求的流动方向拆出来。对大多数后端和数据服务系统来说，至少要先分清四条线：

1. 入口线
   - 请求、事件或数据从哪里进入系统。
2. 处理线
   - 进入之后要经过哪些同步或异步处理。
3. 状态线
   - 哪些地方会落状态、缓存、中间结果或元数据。
4. 输出线
   - 结果最终要返回给谁、写到哪里、如何被下游消费。

只有把这四条线拆出来，很多问题才有位置：

- 同步还是异步，取决于输出线的时延目标和失败处理方式。
- 缓存该放哪，取决于状态线里热点读写落在哪里。
- MQ 该不该引入，取决于处理线是不是需要削峰、解耦和重试缓冲。
- 事务边界怎么划，取决于状态线和输出线是不是要跨多个系统同时成功。

举个抽象一点的例子，实时数据订阅系统的链路就不应只说“接入层、服务层、存储层”，而更应该拆成：

- 数据从哪里来。
- 如何匹配订阅规则。
- 元数据在哪里查、怎么缓存。
- 投递是同步发还是异步发。
- 重试、去重、顺序、失败隔离落在哪。

这类拆法的收益是：

- 你会更容易看到真正的关键路径。
- 你会更容易分清哪些问题发生在请求主链，哪些问题发生在异步旁路。
- 后面的监控、限流、降级、容灾都能找到准确落点。

边界在于：链路拆得过粗，细节会丢；拆得过细，又会失去主线。所以系统设计里的正确拆法，应该是“先拆出关键路径，再把关键路径上的状态、依赖和瓶颈钉住”。

### 3.3 RPC、缓存、MQ、负载均衡、分布式事务为什么必须放回同一条主线里讲

这一节在回答什么问题：

- 分布式基础组件到底分别控制什么问题。
- 为什么它们不能作为平铺知识点去答。

这个问题是怎么解决的：

这些组件会总被放在一起问，不是因为它们属于同一个技术栈，而是因为它们经常同时出现在一条分布式链路里，各自控制不同的风险和成本。

可以把它们放回一条典型服务链路里理解：

1. `RPC`
   - 解决的是服务拆分后，调用方如何稳定地访问下游服务。
   - 它把本来散落在业务代码里的节点发现、超时、重试、序列化、治理能力收口。
2. `缓存`
   - 解决的是热点读、重复计算和主存储承压问题。
   - 它本质上是在用额外空间和一致性复杂度，换更低延迟和更高吞吐。
3. `MQ`
   - 解决的是同步链路过长、峰值突发、系统解耦和失败缓冲问题。
   - 它把“必须当场做完”改成“可以异步消化”。
4. `负载均衡`
   - 解决的是请求如何在多个实例间分配，避免部分节点过载。
   - 它本质上是在控制流量分布和节点选择。
5. `分布式事务`
   - 解决的是跨多个状态节点时，系统到底如何控制一致性边界。
   - 它本质上不是“让系统更高级”，而是在控制跨系统更新的风险。

为什么这几个东西不能分开零散地讲？因为它们经常是在同一个系统里一起配合：

- 上游先通过 RPC 调下游。
- 热点结果被缓存吸收。
- 非关键同步动作改走 MQ。
- 多实例靠负载均衡分流。
- 少数必须跨状态一致的场景，才谨慎处理分布式事务边界。

也就是说，它们不是五个独立答案，而是一组围绕吞吐、时延、耦合、一致性和稳定性做取舍的手段。

真正的难点不在于知道这些名词，而在于知道什么时候该上、什么时候不该上：

- 缓存不是越多越好，它会引入一致性和失效治理问题。
- MQ 不是天然更稳，它会把同步复杂度换成异步复杂度。
- 重试和事务边界不当，反而会把重复执行和数据不一致放大。
- 负载均衡也不等于自动均衡，热 key、会话粘性、长连接模型都会让分配变形。

所以更稳的讲法永远是：先说链路，再说这些组件分别在链路里控制哪类问题。

### 3.4 系统设计里为什么必须把高可用、扩展性和监控一起纳入主线

这一节在回答什么问题：

- 为什么高可用、扩展性、监控不是设计完成后顺手补的一层。
- 为什么很多方案看起来能跑，但一遇到故障和增长就撑不住。

这个问题是怎么解决的：

一个系统只要上线运行，就一定会面对两类压力：

1. 故障压力
   - 节点挂掉、下游抖动、流量突增、慢 SQL、缓存失效、消息积压。
2. 增长压力
   - QPS 提升、数据量增长、租户增多、查询维度增加、任务数增加。

所以系统设计如果只设计“正常路径”，没有设计“异常时怎么活、增长时怎么扩”，这个设计就还没闭环。高可用、扩展性和监控必须一起进入主线，是因为它们分别回答：

- 出问题时怎么止损。
- 量上来时怎么扩。
- 什么时候知道已经出问题了。

这三者通常会落到下面几层设计里：

1. 高可用
   - 限流、降级、隔离、熔断、超时、重试、故障切换。
2. 扩展性
   - 无状态扩容、分片、异步化、冷热分层、读写分离、引擎拆分。
3. 监控
   - QPS、RT、错误率、积压、命中率、线程池、GC、下游依赖、任务延迟。

它们为什么必须一起看？因为很多治理动作本身就是相互制约的：

- 想扩容，得先知道瓶颈是 CPU、IO、存储还是热点分布。
- 想降级，得知道哪些功能是核心路径，哪些是可延后路径。
- 想熔断和限流，得先有足够监控去判断是不是到了危险阈值。

所以一个设计能不能像资深岗答案，关键往往不在于你列没列 Redis、Kafka，而在于你有没有主动把“故障时如何继续工作、增长时如何继续扩、异常时如何被及时发现”补进来。

### 3.5 面对“量级涨 10 倍”或“高并发 / 高可用怎么做”这类追问，到底该怎么继续往下走

这一节在回答什么问题：

- 设计题被继续追问时，应该沿什么维度往下展开。
- 为什么很多人一被追问“量级涨 10 倍怎么办”就开始泛泛而谈。

这个问题是怎么解决的：

“量级涨 10 倍怎么办”这类追问，本质上是在看你是否知道系统的瓶颈会在哪里先暴露，以及你会按什么顺序处理。稳定的展开方式，不是列一个通用组件清单，而是按瓶颈来源拆：

1. 入口压力
   - 网关、接入层、限流、鉴权、配额。
2. 计算压力
   - 同步链路是否过长、能否异步化、能否拆分热路径。
3. 状态压力
   - 数据库、缓存、索引、状态膨胀、热 key、分库分表。
4. 存储和查询压力
   - MySQL 是否继续适合、是否该引入 OLAP 引擎、是否需要冷热分层。
5. 运维和治理压力
   - 监控、报警、压测、容灾、故障演练、发布策略。

为什么要这么拆？因为“量级涨 10 倍”从来不是一个点的问题，而是整条链路的成本重新分布。你要做的不是背一个标准答案，而是判断：

- 谁会先成为瓶颈。
- 是不是可以先通过缓存、异步化、索引优化、分层存储来缓解。
- 什么时候只是调优已经不够，必须改架构边界。

这类追问的稳定答法，核心不在于“我知道很多中间件”，而在于“我知道该优先处理哪一层，为什么先动这一层，以及这样做会引入什么新边界”。

所以系统设计题真正的上限，不取决于图画得多复杂，而取决于你能不能把增长、故障、瓶颈和治理继续按同一条主线展开。

## 4. 边界、误区、常见追问

- 系统设计不是组件选型比赛，没有需求和约束就谈架构，结论通常站不住。
- 链路拆分不要只按模块名，要按数据流、状态流、同步 / 异步边界来拆。
- RPC、缓存、MQ、负载均衡、分布式事务不是平铺组件列表，而是围绕不同成本和风险的控制手段。
- 高可用不是“多部署几台机器”就结束，真正关键的是限流、降级、隔离、超时、重试和止损路径。
- 扩展性也不是默认分库分表，先判断瓶颈在入口、计算、状态、查询还是下游依赖。
- 分布式事务不要无脑上。很多场景更合理的是控制边界、降低同步要求、接受可控的最终一致。

## 5. 自测清单

- 你能不能解释为什么系统设计题必须先问需求、时延、规模和一致性约束。
- 你能不能把一个系统拆成入口线、处理线、状态线、输出线，而不是只说服务分层。
- 你能不能说明 RPC、缓存、MQ、负载均衡、分布式事务分别在链路里控制什么问题。
- 你能不能解释为什么缓存和 MQ 虽然常见，但都会引入新的边界和治理复杂度。
- 你能不能说明高可用、扩展性和监控为什么必须在设计阶段一起进入主线。
- 你能不能面对“量级涨 10 倍怎么办”时，先按瓶颈位置拆，而不是直接堆中间件。
- 你能不能把一题系统设计继续追问到故障止损、扩容和监控，而不是只停在正常流程。
