# 算法专题 09：回溯

适用目标：

- 建立回溯题的统一认知：回溯的本质不是“递归枚举”，而是“在决策树上做选择、撤销选择、继续搜索”
- 分清排列、组合、子集、切割、约束构造这几类高频回溯题
- 能把回溯题从“会写 `dfs`”提升到“能解释路径、选择列表、结束条件、去重和剪枝”
- 减少回溯题最常见的重复答案、状态污染、剪枝失效问题

## 1. 回溯题的核心特点

- 回溯本质上是在一棵“决策树”上做深度优先搜索
- 每一层递归通常都在回答同一个问题：
  - 当前还可以做哪些选择
  - 做出某个选择后，接下来怎么继续搜索
- 回溯题最重要的不是“会递归”，而是下面四件事是否清楚：
  - 当前路径 `path` 表示什么
  - 当前还能选什么 `choices`
  - 什么时候到达结束条件
  - 返回上一层时，哪些状态必须撤销
- 很多题表面不同，但骨架都类似：
  - 选择一个候选
  - 更新路径和状态
  - 递归进入下一层
  - 回来后撤销选择

## 2. 什么题应该优先想到回溯

如果题目有下面这些信号，通常优先考虑回溯：

- “返回所有可能结果”
- “列出所有组合 / 排列 / 子集”
- “需要把每种可行方案都找出来”
- “要求生成满足约束的所有字符串 / 序列 / 分组”
- “题目本身天然在做尝试、撤销、再尝试”

如果题目只求最优值、不要求列出所有方案，未必一定要用回溯，很多时候 DP、贪心或图搜索更合适。

## 3. 回溯题常考模式

看这一节时，建议你每个模式都固定追问三件事：

- 这一层递归到底在决定什么
- 下一层是否还能重复使用当前元素
- 重复答案来自哪里，要靠什么方式去重

### 3.1 子集型：每个元素选或不选

适用场景：

- 返回所有子集
- 每个元素都可以选择加入或不加入
- 不强调顺序，只关心一个集合的组成

代表题：

- `78. 子集`
- `90. 子集 II`

核心理解：

- 子集题的核心不是“排列顺序”，而是“从当前位置开始，还能选哪些元素”
- 为了避免重复，一般用 `startIndex` 控制后续只往后选，不回头选前面的元素
- 这类题通常每到一个节点，当前路径本身就已经是一组合法答案

通用套路：

1. 把当前路径先加入结果集，因为任意阶段的路径都可能代表一个子集。
2. 从 `startIndex` 开始遍历候选元素。
3. 选中一个元素后加入路径。
4. 递归到下一层，并把起点推进到 `i + 1`。
5. 返回上一层时撤销刚才加入的元素。

你真正要会的是：

- 能说清为什么子集题通常一进递归就收集答案
- 能解释为什么用 `startIndex` 就能避免顺序不同但元素相同的重复结果

### 3.2 组合型：从候选里选若干个满足条件

适用场景：

- 选出若干个数组成一个目标值
- 固定选 `k` 个元素
- 组合只看元素集合，不看顺序

代表题：

- `77. 组合`
- `39. 组合总和`
- `216. 组合总和 III`

核心理解：

- 组合题和子集题很像，区别通常在于：
  - 有明确目标条件，如选满 `k` 个或凑成目标和
  - 到达终点前，不是所有路径都要加入答案
- 这类题最关键的是判断：
  - 当前元素能否重复使用
  - 下一层起点是 `i` 还是 `i + 1`

通用套路：

1. 先定义终止条件：
   - 选满了 `k` 个
   - 当前和等于 target
   - 当前和已经超过 target
2. 从 `startIndex` 开始枚举候选。
3. 选中候选后更新路径和状态。
4. 如果元素可重复使用，下一层仍从 `i` 开始；
   如果不可重复使用，下一层从 `i + 1` 开始。
5. 回来后撤销当前选择。

你真正要会的是：

- 能说清 `39` 这类题为什么递归下一层还是从 `i` 开始
- 能区分“组合”和“排列”在顺序处理上的根本差异

### 3.3 排列型：顺序本身就是答案的一部分

适用场景：

- 返回所有排列
- 不只是选哪些元素，还要区分选的先后顺序
- 每个元素通常只能用一次，除非题目明确允许重复使用

代表题：

- `46. 全排列`
- `47. 全排列 II`

核心理解：

- 排列题和组合题最大的区别是：
  - 组合里 `[1,2]` 和 `[2,1]` 算同一种
  - 排列里它们是两种不同答案
- 所以排列题通常不能只靠 `startIndex` 控制，而要显式记录哪些元素已经被使用过

通用套路：

1. 用 `used[]` 或等价结构记录每个元素是否已经在当前路径里使用。
2. 每层都从头遍历所有候选元素。
3. 如果某个元素已经被使用，就跳过。
4. 选中未使用元素，加入路径并标记 `used[i] = true`。
5. 回来后撤销路径并恢复 `used[i] = false`。
6. 当路径长度等于原数组长度时，收集答案。

你真正要会的是：

- 能解释为什么排列题不能像组合题那样只靠 `startIndex`
- 能说清 `used[]` 记录的是“当前路径占用状态”，不是全局永久删除

### 3.4 切割型：在字符串或数组上切分合法片段

适用场景：

- 把字符串切成若干段，每段都满足约束
- 题目本质上是在枚举切割位置

代表题：

- `131. 分割回文串`
- `93. 复原 IP 地址`

核心理解：

- 这类题每一层递归不是“选一个元素”，而是“决定下一刀切在哪里”
- 当前路径通常保存已经切出的若干段结果
- 当前起点 `startIndex` 表示下一段从哪里开始选

通用套路：

1. 用 `startIndex` 表示当前还没切分的起点。
2. 枚举终点 `end`，形成一段候选子串。
3. 判断这段是否满足条件：
   - 是否是回文
   - 是否是合法 IP 段
   - 是否满足其他约束
4. 如果合法，就加入路径并递归处理剩余部分。
5. 当 `startIndex` 到达末尾时，说明整段已经切完，可以收集答案。

你真正要会的是：

- 能识别“切割位置”就是这类题的决策变量
- 能说清为什么合法性判断通常发生在递归前，而不是最后统一校验

### 3.5 约束构造型：在生成过程中持续满足条件

适用场景：

- 生成括号、生成合法序列、棋盘摆放
- 每走一步都要满足某些约束，否则后续无需继续搜索

代表题：

- `22. 括号生成`
- `51. N 皇后`
- `17. 电话号码的字母组合`

核心理解：

- 这类题不是简单枚举，而是在构造过程中持续约束状态
- 关键是把非法分支尽早剪掉，而不是等生成完整结果后再统一过滤
- 这类题的回溯价值很高，因为剪枝效果通常非常明显

通用套路：

1. 明确当前构造状态：
   - 已放了多少个左括号和右括号
   - 当前棋盘占用情况
   - 当前拼出的字符串
2. 每一层先判断有哪些合法选择。
3. 对每个合法选择：
   - 更新状态
   - 递归进入下一层
   - 回来后撤销状态
4. 如果某个选择已经违反约束，就不要进入下一层。

你真正要会的是：

- 能说清“剪枝前置”为什么对这类题尤其重要
- 能把“合法选择集合”表达清楚，而不是盲目枚举后再过滤

## 4. 去重和剪枝怎么理解

这是回溯题最容易讲乱、也最容易写错的部分。

### 4.1 去重：防止同一层生成重复答案

常见场景：

- 输入本身有重复元素
- 不同选择路径可能生成相同答案

典型题：

- `90. 子集 II`
- `47. 全排列 II`
- `40. 组合总和 II`

核心理解：

- 去重通常不是“全局不能再用这个值”，而是“同一层里，相同值的分支不要重复展开”
- 所以常见做法是：
  - 先排序
  - 再在循环里跳过同层重复值

常见判断方式：

- 如果 `nums[i] == nums[i - 1]`，并且前一个相同元素在当前层已经被跳过，那么当前也跳过
- 组合 / 子集题里更常见的写法是：
  - `if (i > startIndex && nums[i] == nums[i - 1]) continue;`

你真正要会的是：

- 能区分“同层去重”和“同枝可重复使用”不是一回事
- 能讲清为什么先排序是为了让重复元素相邻，从而方便剪掉重复分支

### 4.2 剪枝：提前终止不可能成为答案的分支

常见场景：

- 当前和已经超过目标值
- 剩余元素数量不够凑满答案
- 当前状态已经违反题目约束

核心理解：

- 剪枝的本质不是“优化代码”，而是“减少无意义搜索”
- 回溯题如果不剪枝，虽然也可能做出来，但效率通常很差
- 真正有效的剪枝依赖两点：
  - 你知道什么样的状态已经不可能成功
  - 你能在递归前就识别出来

典型例子：

- `39. 组合总和` 里当前和已经大于 target，就没必要继续向下搜
- `77. 组合` 里剩余元素数量不够凑满 `k` 个，也可以提前结束
- `22. 括号生成` 里右括号数量超过左括号，立刻剪掉

你真正要会的是：

- 能说明“这个分支为什么已经不可能成功”
- 剪枝尽量放在递归前，而不是递归到底再统一过滤

## 5. 回溯题的统一判断框架

做回溯题时，可以先按这 5 步判断：

1. 当前路径 `path` 表示什么
2. 当前可选集合 `choices` 是什么
3. 结束条件是什么
4. 下一层如何缩小搜索空间：
   - `startIndex`
   - `used[]`
   - 约束状态
5. 哪些分支需要去重或剪枝

如果这 5 件事没想清楚，回溯代码一般会边写边乱。

## 6. 回溯题的高频坑点

- 不清楚当前层到底在做什么决策
- 组合题和排列题的控制方式混淆
- 路径加入答案后没有深拷贝，导致结果被后续修改污染
- 回溯回来后忘记撤销状态
- 去重条件写错，把合法分支也剪掉了
- 只会暴力枚举，不会做任何剪枝
- 结束条件放错位置，导致漏答案或多答案

## 7. 回溯题的面试表达模板

```text
这题我会先把它抽象成一棵决策树：
每一层要决定什么，当前路径 path 表示什么，
以及从当前状态出发还能做哪些选择。

如果题目只关心元素集合、不关心顺序，我会更偏组合 / 子集型写法，
通常用 startIndex 控制后续只往后选；
如果顺序本身也是答案的一部分，我会改用排列型写法，并用 used[] 记录当前路径里哪些元素已被占用。

接下来我会明确结束条件，再看是否需要去重和剪枝，
最后按“做选择 -> 递归 -> 撤销选择”的回溯骨架展开。
```

## 9. 你复习回溯题时最低要做到什么程度

- 能区分：
  - 子集型
  - 组合型
  - 排列型
  - 切割型
  - 约束构造型
- 能独立写出：
  - `78` 的子集模板
  - `46` 的排列模板
  - `39` 的组合型回溯模板
- 能讲清：
  - `startIndex` 在组合 / 子集题里控制什么
  - `used[]` 在排列题里控制什么
  - 去重和剪枝分别在解决什么问题
- 口头表达时，先说：
  - 当前路径是什么
  - 当前层选择列表是什么
  - 结束条件和剪枝条件是什么

- 能区分：
  - 选或不选
  - 选若干个满足条件

- 能讲清：
  - `78` 为什么一进递归就可以收集答案

- 能做到回溯题不只是“会写 dfs”，而是能先说清：
  - 当前路径代表什么
  - 当前层有哪些选择
  - 结束条件、剪枝条件、撤销动作分别是什么
