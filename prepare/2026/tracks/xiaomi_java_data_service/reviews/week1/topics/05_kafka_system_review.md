## 3. 核心重点展开

### 3.1 Kafka 为什么吞吐高

第一轮复习先建立下面这几个原因：

- 顺序追加写对磁盘更友好
- 分区机制带来并行能力
- 批量发送、批量拉取提升吞吐
- 零拷贝等机制降低数据传输开销
### 3.2 分区、副本、消费组各解决什么问题

- Partition：
  - 解决并行度和扩展性
- Replica：
  - 解决可靠性和故障恢复
- Consumer Group：
  - 解决消费扩展和负载均衡

这题一定要避免混讲：

- 分区不是为了高可用
- 副本不是为了扩并行
### 3.3 如何理解“不丢消息”

Kafka 的可靠性不能只看某一个环节，要按链路讲：

#### 1）生产端

- 合理设置 `acks`
- 配合重试
- 关键场景开启幂等生产者

#### 2）Broker 端

- 依赖副本机制和 ISR
- 避免单副本故障直接丢数据

#### 3）消费端

- 业务处理成功后再提交 offset
- 避免“先提交、后失败”

最后补一句工程现实：

- Kafka 更多提供“至少一次”语义下的工程保证
- 业务层还要配合幂等和补偿
### 3.4 顺序性和重复消费

#### 1）顺序性

Kafka 主要保证：

- 分区内顺序

它不能天然保证：

- 全局顺序

如果业务要顺序消费，常见做法是：

- 让同一业务 key 路由到同一分区
- 消费端不要把同一分区消息并发打乱

#### 2）重复消费

常见来源：

- 业务处理成功，但 offset 没提交成功
- Rebalance
- 消费端重试

所以重复消费的正确解决方向通常是：

- 业务幂等
- 去重表
- 唯一键
- 状态机校验
### 3.5 积压和 Rebalance

这块很适合资深岗往下追问。

消费积压排查优先看：

- 分区数是否足够
- 消费端并行度是否受限
- 下游 RT 是否抬高
- 是否有消息体过大或批量参数不合理

Rebalance 要知道的最小集合：

- 它会导致消费暂停和分区重新分配
- 对 RT 和稳定性有影响
- 频繁重平衡往往意味着消费组不稳定
## 4. 推荐表达顺序

如果面试官问 Kafka，推荐按下面顺序讲：

1. 先说整体模型：
   - Topic、Partition、Replica、Consumer Group
2. 再说工程目标：
   - 吞吐、扩展性、可靠性
3. 再说高频难点：
   - 不丢消息、顺序性、重复消费、积压
4. 最后说取舍：
   - 吞吐、顺序、一致性、幂等之间怎么平衡

## 5. 自测清单

- Kafka 为什么高吞吐，能否不只说“异步”
- Partition、Replica、Consumer Group 分别解决什么问题
- “保证不丢消息”为什么一定要按生产端、Broker、消费端三段讲
- `acks=all` 为什么不是免费的
- Kafka 为什么只能天然保证分区内顺序
- 重复消费为什么不能只靠 MQ 自己解决
- 积压排查为什么必须看分区、消费能力和下游耗时
