## 1. 这一块在解决什么问题

这一块不是只在考你背多少类名，而是在考你能不能把下面六件事讲清楚：

1. 为什么数据访问层通常不直接在业务代码里手写 JDBC，而 MyBatis 和 JPA 分别站在“SQL 显式可控”和“对象状态托管”两条不同路线。
2. Mapper 接口为什么能像普通 Java 接口一样被调用，但最后却能执行到数据库。
3. JPA 为什么经常看起来“不用显式写 update 也能改库”，它背后的持久化上下文和脏检查到底在做什么。
4. 一条 SQL 从 Mapper 方法调用，到参数绑定、JDBC 执行、结果映射，中间到底经历了哪些关键环节。
5. 动态 SQL、一级缓存、二级缓存为什么都不是独立功能，而是挂在同一条执行链上的不同阶段；JPA 的延迟加载和 N+1 又为什么总和 ORM 一起被问。
6. 分页和性能问题为什么很多时候不能简单归因于框架本身，而要继续回到 SQL、数据库和接入方式本身判断。

把这些点串起来，主线其实很稳定：

- 业务系统需要数据访问层，但不希望每个方法都重复写连接获取、参数绑定、结果集映射和资源释放。
- MyBatis 选择的路线不是彻底隐藏 SQL，而是让开发者继续显式写 SQL，同时把 JDBC 样板和执行链统一托管起来。
- JPA 选择的路线则更进一步：不仅托管执行链，还托管事务里的实体对象，让开发者更多围绕对象模型而不是 SQL 本身开发。
- 一旦 Mapper 方法调用被代理接住，或者实体对象进入持久化上下文，后面就可以统一挂接 SQL 定位、参数处理、缓存、结果映射、对象状态同步等能力。
- 动态 SQL 决定“最终发出去的 SQL 长什么样”，缓存决定“这次到底要不要真的查数据库”；延迟加载决定“关联对象现在查还是以后查”，而 N+1 恰好就是 ORM 在这一步最容易暴露的代价。
- 所以这块面试题真正想确认的，不是你会不会背接口，而是你能不能把“方法调用 / 对象访问 -> SQL 执行 -> 结果返回 -> 对象状态 / 缓存 / 分页边界”讲成一条完整主线。

所以这块真正要建立的是四种能力：

- 能把 MyBatis 执行一条 SQL 的主流程讲顺，而不是只说“Mapper 底层是代理”。
- 能把 JPA 的持久化上下文、脏检查、延迟加载和 N+1 串成一条 ORM 语义主线，而不是把它们背成四个孤立术语。
- 能把动态 SQL、缓存、分页，以及 ORM 的常见坑都放回数据访问链路一起理解，而不是拆成几套孤立知识点。
- 能把 MyBatis 和 JPA 的价值、边界和适用场景讲清，不把数据库层问题简单甩给框架。

## 3. 核心重点展开

### 3.1 为什么需要 MyBatis，而不是在业务代码里直接手写 JDBC

这一节在回答什么问题：

- MyBatis 的工程价值到底是什么。
- 为什么它不是“多包一层没必要的框架”，又为什么它和重量级 ORM 不完全一样。

这个问题是怎么解决的：

需要 MyBatis，不是因为 JDBC 不能用，而是因为手工 JDBC 在工程上太容易把业务代码拖进样板泥潭。你真正重复的并不是 SQL 本身，而是下面这些动作：

- 获取连接
- 创建 `PreparedStatement`
- 绑定参数
- 处理 `ResultSet`
- 手工把列映射成对象字段
- 处理异常和关闭资源

如果这些动作散落在每个 DAO / Repository 方法里，代码很快就会出现两个问题：

1. SQL 逻辑和 JDBC 样板混在一起，真正业务意图不清晰。
2. 参数绑定、结果映射、资源释放这些技术细节很容易重复、出错、风格不统一。

MyBatis 的路线很明确：

- 不替你隐藏 SQL
- 但把 JDBC 执行样板和对象映射过程统一托管起来

这也是它和 JPA / Hibernate 这类 ORM 经常被一起问时最大的区别之一：

- MyBatis 更强调 SQL 显式可控
- ORM 更强调对象模型驱动和持久化上下文能力

所以 MyBatis 更适合的场景通常是：

- SQL 复杂、查询形态多变
- 你希望精确控制 SQL
- 更关心执行链和数据库行为，而不是让框架尽量替你抽象 SQL

但它的边界也很明确：

- 它帮你组织 SQL 执行流程
- 不会替你自动优化糟糕的 SQL
- 也不会像 ORM 那样提供强持久化上下文、脏检查、对象状态管理那整套语义

如果把它压成一句话，就是：

- MyBatis 解决的是“SQL 继续由你控制，但 JDBC 样板和映射链由框架接管”这个问题。

### 3.2 JPA / Hibernate 为什么会自动管理对象状态，持久化上下文、脏检查、延迟加载和 N+1 到底是怎么串起来的

这一节在回答什么问题：

- JPA 为什么经常看起来比 MyBatis 更“省代码”。
- 为什么你改了对象字段却没显式调用 update，数据库还是被更新了。
- 为什么同样是查列表，JPA 有时会在循环里打出很多 SQL。

这个问题是怎么解决的：

JPA / Hibernate 比 MyBatis 多做的一层事，不是“把 SQL 写法换了一种”，而是把事务里的实体对象也纳入框架管理。它要解决的是：

- 开发者能不能更多围绕对象模型开发，而不是每次都显式思考 SQL 调用。
- 在一个事务里，框架能不能帮你维护对象状态、对象关系，以及最终和数据库的一致性同步。

先看持久化上下文。

持久化上下文可以理解成：

- 当前事务内由 JPA 管理的一批实体对象

这些对象一旦被 `find`、`getReference`、JPQL 查询等方式加载进来，就不再只是普通 POJO，而是：

- 受管实体对象

这件事最直接带来的能力就是：

- 同一个事务里，同一个主键的实体通常只维护一份对象实例
- 框架能追踪这个对象后续有没有被改动

这就引出了脏检查。

脏检查本质上是在事务提交或 `flush` 时做一件事：

- 比较受管对象现在的状态和刚进入持久化上下文时的状态

如果发现字段发生了变化，JPA 就会自动生成对应的 `update`。所以 JPA 常见的写法是：

- 查出实体
- 修改字段
- 不显式写 update SQL
- 事务提交时自动同步数据库

这也是它和 MyBatis 的一个本质区别：

- MyBatis 里你改对象，只是改了内存里的 Java 字段；要改数据库，必须显式调用更新 SQL
- JPA 里你改受管实体，框架可能在事务提交时自动把变更刷回数据库

这套能力让简单 CRUD 很顺，但它也带来了新的复杂度：

- SQL 什么时候发、发多少、是否提前 `flush`，不再像 MyBatis 一样完全显式
- 事务里如果挂了很多实体对象，框架就要多维护很多状态，排查成本也会上升

再看延迟加载。

延迟加载的核心问题是：

- 关联对象到底要不要在主对象加载时一起查出来

JPA 默认会尝试把这件事做成“按需加载”：

- 当前先把主对象查出来
- 关联对象只放一个代理或占位关系
- 真正访问关联属性时再补发 SQL

这种能力的好处是：

- 避免一开始把完整对象图全部查出来
- 对只需要主对象字段的场景更节省

但它的风险也非常典型。

第一个坑是：

- `LazyInitializationException`

也就是：

- 你在事务里查出了主对象
- 事务结束后才去访问延迟加载属性
- 这时持久化上下文已经关闭，框架已经没有条件再去补查数据库

第二个坑就是：

- N+1 查询

N+1 的典型链路是：

1. 先查 1 次主对象列表
2. 再在循环里访问每个对象的延迟加载关联
3. 于是每个对象又各打 1 次 SQL

最后 SQL 数量就变成：

- 1 次主查询
- N 次关联查询

这也是为什么 JPA 的坑经常不是“不会用 API”，而是：

- 你以为只是在操作对象
- 实际上框架在背后做了很多你没显式写出来的数据库动作

把 JPA 这条主线压缩成一句话，就是：

- JPA / Hibernate 的核心价值不是帮你少写几行 DAO，而是通过持久化上下文把“对象状态”和“数据库状态”挂在一起；脏检查让简单更新更顺，延迟加载让对象图加载更灵活，但 N+1 和事务外访问延迟属性也正是这套能力最常见的代价。

### 3.3 MyBatis 执行一条 SQL 的主流程到底是怎么串起来的，Mapper 接口为什么能执行到数据库

这一节在回答什么问题：

- Mapper 方法为什么不是一个普通空接口调用。
- 从 `userMapper.selectById(id)` 到真正发 SQL，中间到底发生了什么。

这个问题是怎么解决的：

MyBatis 执行一条 SQL 的主流程，本质上是：

- Mapper 接口方法调用
- 被代理对象接住
- 定位到对应的 `MappedStatement`
- 通过 `SqlSession` 和 `Executor` 组织执行
- 再把结果映射回 Java 对象

先把最容易被问的第一层讲清：Mapper 为什么能执行 SQL。

MyBatis 底层的核心思路是：

- Mapper 接口不是自己执行 SQL
- 真正接住调用的是代理对象

也就是说，当你调用：

```java
userMapper.selectById(id)
```

真正被执行的不是接口本身，而是代理对象的 `invoke(...)`。它会根据：

- 当前接口类型
- 当前方法
- 当前参数

去决定应该调用哪个 MyBatis 语句。

再往下走，执行链可以按下面顺序理解：

1. 先通过 Mapper 方法定位 `MappedStatement`
   - 你可以把 `MappedStatement` 理解成“这条语句的完整元数据定义”
   - 里面包含 SQL 来源、参数映射、结果映射、缓存配置、语句类型等信息

2. 再进入 `SqlSession`
   - 它是 MyBatis 工作的核心接口
   - 承接执行 mapped statements、获取 mapper、管理事务等动作

3. `SqlSession` 再把执行请求交给 `Executor`
   - `Executor` 是真正组织“查不查缓存、要不要查库、怎么调 StatementHandler”的执行核心
   - 常见执行器类型有简单执行、复用执行、批量执行等

4. `Executor` 往下再交给 `StatementHandler`
   - 负责准备 JDBC `Statement`
   - 包括创建、参数化、执行 SQL 等动作

5. 参数绑定由 `ParameterHandler` 处理
   - 把 Java 参数对象，绑定成 JDBC 预编译参数

6. 查询结果由 `ResultSetHandler` 处理
   - 把 JDBC `ResultSet` 转成目标对象、集合、`resultMap` 对应结构

所以如果把整条链压缩成一句话，就是：

- Mapper 负责提供面向接口的调用入口，`SqlSession` 负责承接执行，`Executor` 负责组织执行策略，`StatementHandler` / `ParameterHandler` / `ResultSetHandler` 负责把一次数据库访问真正落到 JDBC 层并把结果映射回来。

再补一个在 Spring 项目里非常容易遇到的现实差异。

在裸 MyBatis 里：

- 你通常是 `SqlSessionFactory -> SqlSession -> getMapper()`

在 MyBatis-Spring 里：

- 真正被注入的往往不是原生 `DefaultSqlSession`
- 而是线程安全、能参与 Spring 事务的 `SqlSessionTemplate`

这意味着在 Spring 项目里，Mapper 调用链虽然概念上还是：

- Mapper -> `SqlSession` -> `Executor`

但实际承接 `SqlSession` 语义的通常是：

- `SqlSessionTemplate`

它负责：

- 选择当前事务关联的 `SqlSession`
- 管理 session 生命周期
- 提交 / 回滚 / 关闭
- 把 MyBatis 异常转成 Spring `DataAccessException`

这一节真正要讲清的，不是死记类图，而是方法调用怎么一路被翻译成数据库动作。只要主线讲成：

- 方法调用 -> 代理 -> `MappedStatement` -> `SqlSession` -> `Executor` -> JDBC 执行 -> 结果映射

后面缓存、动态 SQL、插件、分页这些点都更容易挂回去。

### 3.4 动态 SQL、一级缓存、二级缓存为什么总和执行主流程一起问，它们到底分别作用在链路的哪里

这一节在回答什么问题：

- 为什么动态 SQL 和缓存不能脱离执行链单独理解。
- 缓存为什么既能提速，又会带来一致性边界问题。

这个问题是怎么解决的：

动态 SQL 和缓存总和主流程一起问，是因为它们影响的是同一次 SQL 执行链的不同阶段：

- 动态 SQL 决定“最终真正执行的 SQL 长什么样”
- 缓存决定“这次到底要不要真的打数据库”

先看动态 SQL 到底挂在哪一段。

如果把一次查询调用压成更细的链路，可以先记住这一版：

1. Mapper 方法调用进入代理
2. 根据方法定位 `MappedStatement`
3. 根据参数生成最终 SQL，也就是生成 `BoundSql`
4. 根据 `MappedStatement + RowBounds + 最终 SQL + 参数值` 生成缓存键
5. 先查一级 / 二级缓存
6. 缓存没命中，才创建 `StatementHandler`
7. 再做 JDBC 参数绑定、执行 SQL、结果集映射
8. 把结果写回一级缓存，并在合适时机写入二级缓存

这里最关键的一点是：

- 动态 SQL 发生在缓存判断之前

因为缓存命不命中，不是看“你调用的是不是同一个 Mapper 方法”，而是看：

- 这次最终 SQL 是不是一样
- 参与 SQL 的参数值是不是一样
- 分页参数是不是一样

如果最终 SQL 还没生成出来，缓存键就根本没法稳定计算。

#### 3.4.1 动态 SQL 在调用链里具体怎么实现

动态 SQL 的核心不是 XML 标签本身，而是：

- `MappedStatement` 里持有的 `SqlSource`

你可以把 `SqlSource` 理解成：

- “如何根据入参得到最终 SQL” 的对象

它至少分两种典型情况：

- 静态 SQL
  - SQL 文本基本固定，只需要做 `#{}` 占位符解析
- 动态 SQL
  - SQL 文本本身会因为参数不同而变化，比如 `if`、`where`、`set`、`foreach`

对动态 SQL 来说，MyBatis 在启动阶段不会直接把 SQL 固化成最终字符串，而是会先把 XML 解析成一棵 SQL 节点树。你可以把常见标签大致理解成这些节点：

- `if`
- `choose`
- `trim` / `where` / `set`
- `foreach`
- 普通文本节点

真正执行时，流程更像这样：

1. 先拿到参数对象
2. 用参数对象驱动这棵 SQL 节点树
3. 每个节点决定自己要不要把一段 SQL 追加到上下文里
4. 最终拼出一条完整 SQL 文本
5. 再把 `#{}` 解析成 JDBC `?`
6. 同时生成参数映射列表
7. 最后得到 `BoundSql`

所以 `BoundSql` 里真正重要的是两件事：

- 最终 SQL 字符串
- 这条 SQL 对应的参数映射信息

这也是为什么说动态 SQL 不只是“模板语法”，而是执行链前半段的真正入口控制点。因为它直接决定了：

- 本次查询走哪条 where 条件
- 批量 `in (...)` 里有几个参数
- update 到底更新哪些列
- 最后缓存键长什么样

举一个最典型的例子：

```xml
<select id="selectByCondition" resultType="User">
  select * from user
  <where>
    <if test="status != null">
      and status = #{status}
    </if>
    <if test="name != null and name != ''">
      and name like concat('%', #{name}, '%')
    </if>
  </where>
</select>
```

同一个 Mapper 方法，在不同参数下，最终可能生成三种不同 SQL：

- `select * from user`
- `select * from user where status = ?`
- `select * from user where status = ? and name like concat('%', ?, '%')`

从缓存和执行器视角看，这已经不是“一条语句”，而是三条不同的最终查询形态。

#### 3.4.2 一级缓存具体作用在哪一层，怎么实现

一级缓存作用的位置，不是在 JDBC 最底层，而是在：

- `Executor` 层

更准确地说，MyBatis 的本地缓存是 `BaseExecutor` 维护的。你可以把它理解成：

- 在同一个 `SqlSession` 生命周期内，`Executor` 持有一个本地 map

当一次 query 进来时，执行器不会立刻创建 `Statement` 去打数据库，而是先做这些事：

1. 拿到 `MappedStatement`
2. 拿到 `BoundSql`
3. 基于以下信息生成 `CacheKey`
   - `MappedStatement` 的 id
   - `RowBounds` 的 offset / limit
   - 最终 SQL 字符串
   - SQL 参数值
   - 环境标识等附加信息
4. 用这个 `CacheKey` 先查本地缓存

所以一级缓存的命中条件，本质上不是“同一个方法又调了一次”，而是：

- 同一个 `SqlSession`
- 同样的 statement
- 同样的最终 SQL
- 同样的参数
- 同样的分页信息

只有这些组合起来一致，才可能直接命中。

一级缓存为什么默认比较安全？

因为它的边界很小：

- 只在当前 `SqlSession` 内可见
- 一次请求、一段事务里比较容易保证语义一致

但它也不是永远都在。下面这些情况通常会清掉或影响一级缓存：

- 执行了 `insert` / `update` / `delete`
- 手动 `commit` / `rollback`
- 手动 `clearCache`
- `LocalCacheScope` 配成了 `STATEMENT`

这里面最值得面试强调的是：

- 一级缓存不是给业务做热点缓存的
- 它更像“当前会话里的重复查询去重”

所以你在 Spring 项目里如果每个 Mapper 调用背后都是不同的会话边界，一级缓存的体感往往并不强。

#### 3.4.3 二级缓存具体作用在哪一层，怎么实现

二级缓存不是 `BaseExecutor` 自己那张本地 map 的简单放大版，它的位置更靠外一层，通常可以理解成：

- `Executor` 外围的缓存装饰层

MyBatis 的典型做法是用一个带缓存能力的执行器去包装底层执行器。于是查询调用会变成：

- 先经过缓存执行器
- 再决定是否下钻到真正的数据库执行器

二级缓存的几个关键点要分开讲。

第一，它的作用域更大。

- 一级缓存是 `SqlSession` 级别
- 二级缓存通常是 mapper namespace 级别

也就是说，它不是“当前会话私有”，而是：

- 多个会话只要访问的是同一个 namespace 下、同一组缓存键，就可能共享结果

第二，它的写入时机更谨慎。

二级缓存通常不会在查询结束那一瞬间立刻把结果扔进共享缓存，而是会先放在事务性的暂存区里，等事务真正提交后再刷进共享缓存。这样做是为了避免：

- 查询结果还没和事务最终状态对齐，就被别的会话读走

第三，它的失效是按 statement 配置和 namespace 语义联动的。

常见写法里：

- 查询语句可以使用缓存
- 更新语句通常会触发缓存清理

原因很直接：

- 二级缓存共享范围更大
- 一旦更新后不及时清理，就更容易读到旧数据

所以二级缓存真正难的从来不是“会不会开”，而是：

- 共享范围扩大后，失效策略还能不能跟上业务一致性要求

这也是为什么很多项目宁愿用 Redis 这种显式业务缓存，也不愿把 MyBatis 二级缓存开得很重。因为二级缓存的问题往往不是性能本身，而是：

- 排障时你不容易第一时间想到“这是不是 namespace 缓存残留”

#### 3.4.4 动态 SQL 和缓存为什么必须放在一起讲

因为缓存键的生成依赖最终 SQL，而最终 SQL 又取决于动态 SQL。

这会带来三个非常实际的结果。

第一，同一个 Mapper 方法，不同条件分支，缓存键完全不同。

也就是说：

- 不是“同一个方法天然共享一个缓存”
- 而是“同一个方法在不同最终 SQL 形态下，对应不同缓存键”

第二，`foreach` 场景特别容易让缓存键变得离散。

比如：

- `id in (1,2,3)`
- `id in (1,2,3,4)`

哪怕只多一个参数，最终 SQL 文本都不同，缓存键也不同。这也是为什么批量动态查询场景下，缓存命中率未必高。

第三，动态 update / select 的条件差异会直接影响缓存复用价值。

如果一个查询方法支持很多可选条件，那么：

- 从接口层看它像一个方法
- 从缓存层看它其实像一组不同查询

所以缓存收益高不高，不只取决于“有没有缓存”，也取决于：

- 最终 SQL 形态是不是足够稳定

#### 3.4.5 用一条完整调用链把动态 SQL 和缓存挂起来

如果面试官追问“到底是怎么串起来的”，你可以按下面这条更细的链来答：

1. 调用 Mapper 方法
2. 代理根据接口方法定位 `MappedStatement`
3. `MappedStatement` 持有的 `SqlSource` 根据入参生成 `BoundSql`
4. 执行器基于 `MappedStatement + RowBounds + BoundSql + 参数值` 生成 `CacheKey`
5. 先查一级缓存
6. 如果启用了二级缓存，再按 namespace 语义查共享缓存
7. 缓存都没命中，才创建 `StatementHandler`
8. `ParameterHandler` 把参数绑定到 JDBC `PreparedStatement`
9. 执行 SQL
10. `ResultSetHandler` 做结果映射
11. 结果先进入一级缓存
12. 事务提交后，再按规则写入或刷新二级缓存

这条链里最关键的认知顺序是：

- 先有最终 SQL
- 再有缓存键
- 再决定是否查库

顺序不能反。

这一节真正要讲清的是：

- 动态 SQL 不是独立模板语法题，而是 `SqlSource -> BoundSql` 这一段的执行逻辑
- 一级缓存不是 JDBC 驱动特性，而是 `Executor` 层的会话级缓存
- 二级缓存不是一级缓存的简单放大版，而是带事务边界和 namespace 语义的共享缓存
- 动态 SQL 会直接影响缓存键，所以它和缓存本质上属于同一条调用链上的前后两个环节

### 3.5 MyBatis 和 JPA / Hibernate 的核心差异到底在哪里，项目里应该怎么选

这一节在回答什么问题：

- 两套框架为什么经常被一起问，却又不该简单回答成“一个好一个不好”。
- 项目里什么时候更适合 MyBatis，什么时候更适合 JPA。

这个问题是怎么解决的：

MyBatis 和 JPA 的核心差异，不在于“谁更高级”，而在于它们的抽象中心不同。

MyBatis 的抽象中心是：

- SQL

你写什么 SQL、怎么分页、怎么走索引、怎么做统计，基本都由你显式决定；框架帮你解决的是：

- Mapper 代理
- 参数绑定
- 结果映射
- 动态 SQL
- 缓存和插件挂点

JPA 的抽象中心是：

- 实体对象及其关系

你更多围绕实体、仓库接口、对象关系和事务语义去开发；框架帮你解决的是：

- 持久化上下文
- 脏检查
- 实体生命周期管理
- 延迟加载
- 对象关系映射

所以它们最大的工程体验差异是：

- MyBatis 更显式，SQL 更透明，排查链路更直接
- JPA 更自动，简单 CRUD 开发更快，但 SQL 行为和对象加载行为不总是显式可见

如果继续往下压缩，可以按下面四组差异记忆。

第一组：控制力和开发效率的取舍。

- MyBatis 更适合：
  - 查询条件复杂
  - 报表和统计多
  - SQL 需要精细调优
  - DBA / 后端需要直接围绕 SQL 对齐
- JPA 更适合：
  - 标准 CRUD 较多
  - 实体关系比较稳定
  - 希望快速交付常规后台功能

第二组：对象状态语义。

- MyBatis 没有完整的实体状态托管语义，改对象不会自动落库
- JPA 有持久化上下文和脏检查，改受管对象可能自动落库

第三组：性能问题暴露方式。

- MyBatis 的性能问题更常见于：
  - SQL 写法
  - 索引设计
  - 映射过重
  - 插件误用
- JPA 的性能问题除了 SQL 本身，还经常包括：
  - 延迟加载使用不当
  - N+1 查询
  - 对象图过重
  - `flush` 时机和 ORM 语义理解不清

第四组：复杂查询表达成本。

- MyBatis 在复杂条件查询、聚合、报表、数据服务接口上通常更直接
- JPA 在复杂查询里经常会走到：
  - JPQL
  - `Specification`
  - `Criteria API`
  - 原生 SQL

所以真实项目里常见的经验结论是：

- 如果系统偏数据服务、查询复杂、SQL 调优频繁，MyBatis 往往更稳
- 如果系统偏标准业务后台、实体关系稳定、希望快速完成 CRUD，JPA 往往更省代码
- 如果项目既有复杂查询又有标准后台，也完全可能：
  - 主体 CRUD 用 JPA
  - 复杂报表或高性能查询单独落到 MyBatis / 原生 SQL

把这节压成最短回答，就是：

- MyBatis 是 SQL-first，适合需要显式控制 SQL 和执行代价的场景；JPA 是 Entity-first，适合标准 CRUD 和对象关系更稳定的系统。前者更透明，后者更自动；前者更容易围绕 SQL 调优，后者更容易踩延迟加载和 N+1 这类 ORM 语义坑。

### 3.6 分页和性能问题为什么不能简单归因于框架，本质上应该怎么分层判断

这一节在回答什么问题：

- 为什么分页题最后总会追到 SQL 和数据库本身。
- MyBatis 和 JPA 在分页和性能问题里到底该背多大的锅。

这个问题是怎么解决的：

分页和性能问题不能简单归因于框架，是因为 MyBatis 和 JPA 都主要负责的是：

- 组织 SQL 执行链
- 处理参数绑定和结果映射
- 在合适的位置挂缓存、插件、动态 SQL 或对象状态管理等能力

但真正决定查询代价的，很多时候仍然是：

- SQL 本身怎么写
- 索引怎么建
- 执行计划怎么样
- 是否存在深分页
- 是否有大结果集回传
- ORM 的访问模式是否把查询次数放大了

先说分页。

MyBatis 里经常会提到两种分页思路：

1. `RowBounds`
2. 物理分页插件 / SQL 改写

这里最容易被误解的是 `RowBounds`。

`RowBounds` 本质上只是一个：

- `offset`
- `limit`

包装对象。它更多是一种框架层分页抽象，不等于数据库已经高效完成物理分页。所以如果你把分页答成“RowBounds 就是高性能分页”，这就容易被追问穿。

#### 3.6.1 `RowBounds` 具体是怎么做分页的

`RowBounds` 的关键点不是“改 SQL”，而是：

- 把分页参数沿着 MyBatis 调用链往下传

调用链里大致会发生下面几步：

1. 业务层调用查询方法，同时把 `RowBounds(offset, limit)` 传进来
2. `Executor.query(...)` 在执行过程中把 `RowBounds` 一路传给 `StatementHandler` 和结果集处理逻辑
3. 真正发给数据库的 SQL 通常并没有被改写
4. 查询结果返回后，MyBatis 再根据 `offset` / `limit` 对结果做截取

这意味着：

- `RowBounds` 默认更接近逻辑分页，而不是数据库原生物理分页

如果继续往细一点看，它通常不是在 SQL 层直接补一个 `limit ?, ?`，而是：

- SQL 照常执行
- JDBC 拿到结果集
- MyBatis 在结果集处理阶段跳过前 `offset` 条，再取接下来的 `limit` 条

所以它更像：

- “结果集层面的分页裁剪”

这也是为什么它的问题很明显：

- 如果 SQL 本身返回结果很多，数据库和网络传输的成本已经发生了
- MyBatis 即使最后只留下 20 条，前面被跳过的数据也已经查出来了

再说得更直白一点：

- `RowBounds(100000, 20)` 不代表数据库只扫 20 行
- 很可能数据库和驱动已经把大量结果参与了处理，只是 MyBatis 最后只给你 20 条

所以它适合的场景通常比较有限：

- 数据量不大
- 只是想用统一接口抽象分页
- 或者某些驱动 / 中间层能对 `RowBounds` 做额外优化

但在常规 MySQL / PostgreSQL 这类场景下，面试里更稳的说法是：

- `RowBounds` 是框架层分页参数抽象，不应默认等价于高性能物理分页

#### 3.6.2 物理分页插件 / SQL 改写具体是怎么做的

物理分页插件的思路和 `RowBounds` 正好相反。它不是“查完再裁”，而是：

- 在 SQL 发给数据库之前，先把 SQL 改成数据库真正认识的分页 SQL

典型链路通常是：

1. 拦截 MyBatis 的查询执行过程
2. 识别当前查询是不是分页查询
3. 读取分页参数，比如页码、页大小、offset、limit
4. 按数据库方言把原 SQL 改写成分页 SQL
5. 再把改写后的 SQL 交给数据库执行

比如原 SQL 是：

```sql
select id, name, created_at
from orders
where status = ?
order by id desc
```

插件改写后，发给 MySQL 的可能变成：

```sql
select id, name, created_at
from orders
where status = ?
order by id desc
limit ?, ?
```

如果是 Oracle / SQL Server / PostgreSQL，方言写法又会不同。所以分页插件通常都要做一件事：

- 根据数据库类型选择不同的 SQL 改写规则

从 MyBatis 扩展点角度看，这类插件最常见的做法是拦截：

- `Executor.query(...)`
- 或 `StatementHandler.prepare(...)`

你不用死记哪个插件拦哪一层，但要知道它的本质都是：

- 在真正执行 JDBC 之前改写 SQL 或调整执行参数

很多分页插件还会顺手做第二件事：

- 自动生成一条 `count` SQL 去查总条数

这时一次“分页查询”在数据库侧往往会变成两次调用：

1. 一次 `count(*)`
2. 一次带 `limit/offset` 的分页数据查询

所以分页插件带来的能力不只是“少写一点代码”，而是：

- 帮你把分页逻辑挂到执行链前半段
- 让数据库只返回当前页需要的数据
- 但也可能额外引入总数查询成本

#### 3.6.3 为什么物理分页通常比 `RowBounds` 更合理

因为真正昂贵的成本主要发生在：

- 数据库扫描
- 排序
- 网络传输
- JDBC 结果集处理

物理分页的好处是：

- 让数据库尽量只返回当前页需要的数据

而 `RowBounds` 的问题是：

- 数据库和驱动层的大部分成本往往已经发生，MyBatis 只是最后做了一次截断

所以在真实项目里，如果数据量稍微大一点，更常见的结论就是：

- 优先走数据库物理分页，而不是依赖 `RowBounds`

#### 3.6.4 为什么物理分页也不等于性能一定没问题

这也是分页题最容易被答浅的地方。

很多人一听到“物理分页”，就以为问题已经解决了。实际上，它只是把分页能力下推到了数据库，并不等于：

- SQL 一定高效
- 深分页一定没问题

比如下面这条 SQL：

```sql
select * from orders order by id limit 100000, 20
```

它虽然是物理分页，但数据库仍然可能需要：

- 先扫描大量数据
- 做排序或定位
- 再跳过前 100000 条

所以性能瓶颈依然可能很大。

这时真正该看的就不是“是不是插件分页”，而是：

- 排序列上有没有合适索引
- 能不能改成基于游标 / 上次最大 id 的翻页
- 能不能避免深分页
- `count(*)` 是不是本身就很贵

#### 3.6.5 用调用链把两种分页方式挂起来

如果面试官追问“到底怎么做”，你可以这样对比着答：

`RowBounds`：

1. Mapper 调用时传入 `RowBounds`
2. SQL 本身通常不改
3. 数据库先执行原查询
4. MyBatis 在结果集处理阶段按 `offset` / `limit` 做截取

物理分页插件：

1. Mapper 调用时传入分页参数
2. 插件在执行前拦截查询
3. 按数据库方言改写 SQL，补上 `limit/offset` 或等价分页语法
4. 数据库只返回当前页数据
5. 如有需要，再额外执行 `count` SQL

把这两者压成一句话，就是：

- `RowBounds` 更像框架层逻辑分页参数；物理分页插件则是在执行前改写 SQL，把分页真正下推到数据库。

在真实项目里，更常见的是：

- 通过插件改写 SQL，走数据库物理分页

这时 MyBatis 的角色是：

- 提供插件挂点和执行链扩展能力

但深分页问题本身并不会因此消失。比如：

```sql
select * from orders order by id limit 100000, 20
```

即使框架层分页接得再优雅，数据库还是可能要先跳过大量记录，代价仍然很高。这个问题本质不在 MyBatis，而在：

- SQL 分页策略本身
- 索引与排序条件
- 数据量级

再看性能问题为什么不能一股脑怪框架。

排查查询慢时，至少要先分三层：

1. SQL / 数据库层
   - SQL 本身是否慢
   - 索引是否合理
   - 是否存在回表、排序、临时表、深分页
   - 执行计划是否异常

2. 映射与 ORM 层
   - `resultMap` 是否过重
   - 嵌套映射 / 集合映射是否导致多次查询或结果集膨胀
   - 返回对象是否过大
   - JPA 是否触发了延迟加载、N+1、过重对象图
   - 是否为了方便返回实体，实际多查了很多业务不需要的数据

3. 框架接入层
   - 插件是否过多
   - 缓存是否误用
   - 会话和事务接入是否合理
   - 是否存在批量场景却没用 `ExecutorType.BATCH`
   - JPA 是否发生了意外 `flush`
   - 仓库接口和查询方式是否让 SQL 行为变得不透明

如果你只会说“这是 MyBatis 的分页问题”“这是 JPA 太慢”或者“这是框架慢”，通常是不够的。更稳的回答应该是：

- MyBatis 可以放大一个差的 SQL，也可以把分页接入组织得更规范
- JPA 可以让简单 CRUD 更顺，也可能因为对象关系和延迟加载把查询次数放大
- 但它们都不决定底层查询本身一定高效，最终还是要回到 SQL、索引、执行计划和实际访问模式判断

这一节真正要讲清的是：

- 框架执行链问题、ORM 语义问题和数据库执行代价问题必须分层判断
- MyBatis 和 JPA 都负责组织访问语义，但都不负责替你自动优化数据库

## 4. 边界、误区、常见追问

- 不要把 MyBatis 讲成全自动 ORM，它更强调 SQL 显式可控和 JDBC 样板收口。
- 不要把 JPA 讲成“只是少写 SQL 的 DAO 框架”，它真正多出来的是持久化上下文、脏检查和对象关系管理语义。
- Mapper 接口本身不会执行 SQL，真正接住调用的是代理对象；在 Spring 项目里，还要记得补一句 `SqlSessionTemplate` 经常承担实际 session 管理责任。
- 不要把“改了实体就能自动更新数据库”理解成魔法，前提是这个实体还在当前持久化上下文里，且事务提交或 `flush` 发生了。
- 不要把执行流程只答成“Mapper -> SqlSession -> Executor”，继续讲清 `MappedStatement`、参数处理、结果映射分别在干什么。
- 动态 SQL 不是为了方便拼字符串，而是为了让条件化 SQL 在执行链前半段仍然保持结构化。
- 一级缓存和二级缓存不能只讲“能提速”，必须继续讲作用范围、失效时机和一致性边界。
- 二级缓存要谨慎，不是因为它一定不好，而是因为共享范围一大，一致性和排障复杂度会明显上升。
- 不要把延迟加载只讲成“性能优化”，它的前提是持久化上下文仍然可用；一旦在事务外访问延迟属性，就可能抛出延迟加载异常。
- 不要把 N+1 讲成数据库问题，它更常见的是 ORM 访问模式问题：主查询没错，但循环访问关联属性让查询次数爆炸。
- 不要把分页性能问题都甩给 MyBatis 或 JPA，先分清是框架执行链问题、ORM 语义问题，还是 SQL / 索引 / 执行计划问题。
- `RowBounds` 可以讲，但不要把它讲成“物理分页等于高性能分页”的保证。
- JPA 的默认自动化能力不等于适合所有场景；复杂查询、报表、数据服务接口不一定适合强行围绕实体对象建模。

## 5. 自测清单

- 你能不能先不背类名，直接讲清 MyBatis 为什么能在 JDBC 和 ORM 之间形成一个独立定位，以及 JPA 为什么比它多一层对象状态管理语义。
- 你能不能把“Mapper 方法调用 -> 代理 -> `MappedStatement` -> `SqlSession` -> `Executor` -> JDBC 执行 -> 结果映射”这条链讲顺。
- 你能不能说明持久化上下文、脏检查、延迟加载、N+1 分别在解决什么问题，又各自带来什么代价。
- 你能不能说明为什么 MyBatis-Spring 里很多时候真正参与事务的是 `SqlSessionTemplate`，而 JPA 里事务又会决定持久化上下文的边界。
- 你能不能解释动态 SQL 为什么会影响最终 SQL 形态，并进一步影响缓存判断和查询行为。
- 你能不能讲清一级缓存和二级缓存的作用范围、收益和一致性边界，以及为什么 JPA 的问题更常落在延迟加载和 N+1 上。
- 你能不能按“SQL / 数据库层、映射与 ORM 层、框架接入层”三层去分析查询慢的问题。
- 你能不能用 2-3 分钟把“MyBatis 和 JPA 的能力差异、常见坑和适用场景”讲顺。
