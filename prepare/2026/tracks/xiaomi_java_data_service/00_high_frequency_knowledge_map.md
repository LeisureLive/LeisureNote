# 小米 Java 研发工程师（数据服务）面试高频知识地图

## 1. 这份文档解决什么问题

这份文档不是替代 `01_1month_plan.md`，而是给那份计划补一层“知识导航”。

当前 1 个月计划的问题不在于日拆分不合理，而在于：

- 每天直接进入专题问答，缺少稳定的知识总览
- 进入问答前没有先明确“今天这个专题到底要答到什么深度”
- 输出动作偏早，导致你经常一边查漏、一边组织表达，耗时偏高

这份知识地图的作用是：

- 帮你先建立模块级框架，再进入当天计划
- 帮你区分哪些知识点只要会讲主线，哪些必须能承接深挖
- 让你每天复习时先有“总图”，再做“局部细节”

## 2. 推荐使用方式

### 每天开始前先做 10-15 分钟模块预热

步骤固定如下：

1. 先定位今天对应的知识模块
2. 只看该模块的“核心框架”和“高频问题”
3. 明确今天要拿稳的 1-2 个主问题
4. 再进入当天专项复习和问答
5. 最后只输出提纲或口头回答，不要求每题都写成长文

### 答题深度统一分 3 层

- `L1`：1 分钟版本
  - 说清定义、作用、适用场景
- `L2`：3 分钟版本
  - 说清核心机制、关键流程、常见对比
- `L3`：深挖版本
  - 能承接边界、原理、线上案例、项目结合

建议默认目标：

- P0 模块至少达到 `L2`
- 你的强项模块尽量冲到 `L3`
- 当前短板模块先保 `L1 -> L2`，不要一上来追求细枝末节

### 掌握度标记

- `A`：主线稳定，追问也能展开
- `B`：主线能答，但深挖容易乱
- `C`：知道概念，但还讲不顺

每天复习后，给当天模块打一个 `A/B/C` 即可。

## 4. 高频知识模块梳理

### 4.1 Java 并发

#### 核心框架

- 线程与线程池
- 锁机制：`synchronized`、`ReentrantLock`
- JMM、`volatile`
- CAS、原子类、ABA
- AQS 及常见同步器
- 并发容器：`ConcurrentHashMap`

#### 必须拿稳的高频题

- 线程池参数如何配置
- 线程池执行流程是什么
- 为什么不建议直接使用 `Executors`
- `synchronized` 和 `ReentrantLock` 的区别
- `volatile` 能保证什么，不能保证什么
- 为什么 `i++` 不能靠 `volatile`
- CAS 是什么，ABA 怎么解决
- AQS 是什么，解决了什么问题
- `ConcurrentHashMap` 为什么线程安全

#### 常见追问

- 拒绝策略怎么选
- 双重检查单例为什么必须加 `volatile`
- `AtomicLong` 和 `LongAdder` 怎么选
- 公平锁和非公平锁差异
- `ThreadLocal` 的使用边界和泄漏风险
- 线程池监控要看哪些指标

#### 你当前更该补的点

- 线程池参数配置要带容量视角，不要只背公式
- `volatile` / JMM / DCL 这组问题要能连起来讲
- AQS 不必抠源码，但要会说“队列 + 状态 + 唤醒”的主线

#### 推荐输出物

- “线程池参数如何配置”3 分钟回答
- “`volatile` 能保证什么”2 分钟回答

### 4.2 JVM

#### 核心框架

- 运行时内存结构
- 对象创建过程
- 类加载流程与双亲委派
- GC 基础与常见收集器
- 常见线上问题：OOM、频繁 GC、CPU 飙高

#### 必须拿稳的高频题

- JVM 运行时内存结构有哪些
- 栈、堆、方法区、直接内存有什么区别
- 对象创建流程是什么
- 类加载有哪些阶段
- 双亲委派是什么，解决了什么问题
- 常见垃圾回收器的区别
- Full GC 频繁怎么排查

#### 常见追问

- JMM 和 JVM 内存结构有什么区别
- 对象一定分配在堆上吗
- `prepare` 和 `initialize` 的区别
- CMS 和 G1 的差异
- OOM 排查链路是什么
- CPU 高、GC 多、RT 抖动时先看什么指标

#### 你当前更该补的点

- JVM 内存结构和 JMM 一定要彻底区分开
- GC 不必追全量参数，重点是“场景 + 排查路径”
- 线上排查题要尽量结合真实案例回答

#### 推荐输出物

- “JVM 内存结构”3 分钟回答
- “频繁 Full GC 排查路径”提纲

### 4.3 MySQL / SQL

#### 核心框架

- B+ 树索引
- 联合索引、最左前缀、回表、覆盖索引
- 事务、隔离级别、MVCC

#### 必须拿稳的高频题

- 为什么 MySQL 索引常用 B+ 树

- 什么是回表、覆盖索引

- 慢 SQL 怎么排查
- `explain` 主要看哪些字段

#### 常见追问

- 为什么不建议滥建索引
- `count(*)` 为什么有时慢
- 索引失效常见场景
- 深分页怎么优化
- 幻读是怎么回事，MySQL 如何处理

#### 你当前更该补的点

- 索引题不要只停留在定义，要会解释执行代价
- 慢 SQL 排查必须有稳定步骤
- OLTP 和 OLAP 的差异要能联系你的数据平台背景去说

#### 推荐输出物

- “慢 SQL 怎么排查”3 分钟回答

## 5. 和 1 个月计划如何配合

### 第 1 周

- 只盯 5 个模块：并发、JVM、MySQL、Redis、Kafka
- 每天先看知识地图对应模块，再进入当日复习文档
- 目标不是每题写完，而是先把主问题答稳

### 第 2 周

- 只盯 4 个模块：Spring、Spring Boot/MVC、MyBatis、Dubbo
- 输出以“流程图 + 回答提纲”为主
- 如果某天问答特别卡，先退回看本知识地图，不要硬扛

### 第 3 周

- 重点切到系统设计与数据服务场景
- 复用前两周的中间件和存储知识，不单独割裂复习

### 第 4 周

- 以混合复习和模拟为主
- 每天按模块抽查自己最弱的 2 个问题
- 只做查漏补缺，不再扩展低频知识

## 6. 当前最推荐的执行顺序

如果你已经在执行 `01_1month_plan.md`，不建议推翻重来，建议这样调整：

1. 先用 2-3 天把这份知识地图过一遍，建立总图
2. 然后继续按原日计划推进
3. 每天进入问答前，先看今天对应模块的框架和高频题
4. 每周只挑 2-3 个最核心问题写成完整答案
5. 其余问题先做到“能口头讲 + 有提纲”

结论：

- 你确实需要单独补一块“面试高频知识点的系统梳理”
- 但不需要额外起一套平行计划
- 最有效的方式是：保留月计划，新增知识地图作为前置导航层
