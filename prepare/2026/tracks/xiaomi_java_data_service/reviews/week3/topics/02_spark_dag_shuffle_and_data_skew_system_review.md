## 1. 这一块在回答什么问题

Spark 这块真正要回答的是下面这条主线：

1. 一段大数据处理逻辑提交上去之后，Spark 到底是怎么把它拆成可以分布式执行的任务的。
2. 为什么同样是“分布式计算”，有的任务很顺，有的任务会在中间某一步突然变慢。
3. `shuffle` 为什么总被认为是 Spark 成本最高、最容易出问题的一层。
4. 为什么资源看起来很多，任务却还是会被少数节点或少数 key 拖住。
5. 缓存、广播、分区这些优化手段到底各自在解决什么成本，而不是为什么它们“听起来像优化”。

所以这块的主线不是背 `RDD`、`Stage`、`Task` 这些名词，而是：

- Spark 先把逻辑计算组织成 DAG。
- DAG 再按依赖关系切成多个 stage。
- stage 内部由多个 task 并行处理分区数据。
- 宽依赖会触发 shuffle，而 shuffle 往往是网络、磁盘、序列化和内存成本最集中的地方。
- 数据倾斜、本地性不足、缓存误用，都会把成本集中到少数节点上。

只有把这条链讲顺，Spark 才不会变成术语列表。

## 2. 主线

1. 先看执行入口：Spark 并不是把整段代码广播出去同时跑，而是先在 Driver 侧形成可调度的执行计划。
2. 再看任务拆分：真正决定执行边界的不是代码行数，而是依赖关系，尤其是宽依赖。
3. 再看 shuffle：只要数据需要跨分区重新分发，成本就会从 CPU 计算变成磁盘、网络、排序和内存管理。
4. 再看数据倾斜：即使总数据量不大，分布不均也会把少数 task 拖成整条链的尾部。
5. 最后看优化手段：缓存、广播、分区不是固定套路，而是分别针对不同成本项下药。

## 3. 核心问题展开

### 3.1 Spark 的一个作业从提交到执行，到底是怎么跑起来的

这一节在回答什么问题：

- Spark 任务为什么不是“每台机器执行同一段代码”这么简单。
- Driver、Executor、调度器分别在做什么。

这个问题是怎么解决的：

Spark 的执行起点在 Driver。无论是 RDD、DataFrame 还是 Spark SQL，提交逻辑之后，Driver 都会先把计算关系整理成可执行计划，再由调度器拆分任务并下发给 Executor 执行。也就是说，Spark 先在控制面回答“应该怎么拆”，再在执行面回答“谁来跑这些分片任务”。

这套运行时里，几个角色的职责边界很清楚：

1. `Driver`
   - 负责持有应用主逻辑、生成 DAG、做任务调度协调。
2. `Executor`
   - 负责真正执行 task、读写数据、参与 shuffle、缓存分区数据。
3. `Cluster Manager`
   - 负责资源分配，给 Driver 和 Executor 提供运行资源。

这样拆的意义在于，Spark 不把“业务逻辑表达”和“分布式执行管理”混在一起。你写的是数据处理逻辑，Spark 负责把它翻译成一组任务，并安排这些任务如何分布到集群里跑。

这个设计带来的收益主要有三点：

1. 计算逻辑先在 Driver 端统一组织，依赖关系更容易被全局优化。
2. Executor 只专注执行 task，不需要每个节点都自己重新推导一遍全局计划。
3. 调度、执行、资源管理分层后，失败恢复和性能分析都有更明确的落点。

代价也很直接：

- Driver 会成为作业控制面的关键节点，计划太大、元数据太重、调度压力太高时会成为瓶颈。
- Executor 不是无限扩容就能解决一切，因为后续还有 shuffle、数据分布、外部存储等约束。

所以 Spark 作业不是“把代码丢到集群上跑”，而是“先形成统一执行计划，再把计划拆成分布式任务执行”。

### 3.2 为什么写了一串转换逻辑后，Spark 往往不会立刻执行

这一节在回答什么问题：

- 为什么 Spark 经常说自己是懒执行。
- `transformation`、`action`、`job` 之间到底是什么关系。

这个问题是怎么解决的：

Spark 不会在你每写一行转换逻辑时就立刻执行，是因为它要先把整段数据处理关系收集起来，再决定怎样统一拆分和优化。像 `map`、`filter`、`select`、`join` 这类 transformation，更多是在不断补充“我要对数据做什么”；真正触发执行的，通常是 `count`、`collect`、`save`、`foreach` 这类 action，因为只有到了 action，系统才真正知道“这次需要把结果算到哪里、算到什么程度”。

这背后的关键不是“为了偷懒不执行”，而是为了先保留依赖关系，再统一决定：

1. 哪些转换可以串在一起。
2. 哪些地方必须切 stage。
3. 哪些代价可以通过整体计划优化掉。

这也是为什么 Spark 常说自己不是“立即执行引擎”，而是先形成 lineage 或逻辑计划，再在 action 到来时生成真正的物理执行任务。

这里还要继续区分三个层次：

1. `transformation`
   - 描述数据怎么被转换。
   - 它先积累依赖关系，不立即产出最终结果。
2. `action`
   - 告诉 Spark 这次真的要把结果算出来。
   - 它是执行触发点。
3. `job`
   - 是 action 触发后形成的一次完整执行提交。
   - 一个 Spark application 里可以因为多个 action 形成多个 job。

如果把这三个层次再压成一句话来看：

- `job` 主要受 action 影响。触发一次真正执行，通常就会形成一次 job。
- `stage` 主要受 shuffle 或宽依赖影响。只要数据不能继续在本地分区内流水化处理，通常就会形成新的 stage 边界。
- `task` 主要受分区数影响。一个 stage 里有多少个分区，通常就会对应多少个并行 task。

这层区分很重要，因为很多性能问题都和“重复 action 导致重复 job”有关。表面上你可能只是多写了一次结果输出，但系统实际上可能把前面的整段 lineage 重新执行一遍。缓存是否值得加、某段中间结果是否要复用，都要回到这个边界来判断。

一个很典型的例子是：一段 `filter -> join -> groupBy` 的结果，既要写下游表，又要做一次 `count` 校验，还要 `show` 几条样本看结果。代码里看起来只是复用了同一个中间变量，但对 Spark 来说，这其实对应多个 action。只要中间结果没有被缓存，后面的每个 action 都可能再次触发一个 job，把前面那段读源数据、join、shuffle、聚合的 lineage 再跑一遍。

这也是 `persist` 或 `cache` 发挥作用的边界。Spark 默认依赖 lineage 做容错和重算，所以 transformation 本身不会自动把每一步中间结果都物化下来。只有下面这类情况，缓存才通常值得考虑：

- 某一段中间结果后面会被反复复用。
- 这段结果的生成过程本身已经包含较高的扫描、join、shuffle 或聚合成本。

相反，如果上游只是很轻的 `filter`、`select`，或者结果只会被消费一次，那么缓存本身带来的内存占用和管理成本，往往不值得支付。

这里还要把 `persist` 的存储位置说清楚。它不是把结果默认写回 HDFS，也不是放在 Driver 里统一保存，而是按分区把结果缓存在 Executor 侧。更准确地说，第一次 action 真正触发计算后，Executor 会把已经算出的分区结果放进自己的 `BlockManager`。这些缓存块具体放在内存、本地磁盘，还是两者结合，取决于 `StorageLevel`。后续再复用这段结果时，Spark 会优先直接读取这些已经物化的分区，而不是重新回到 lineage 从头推导。

`MEMORY_ONLY` 和 `MEMORY_AND_DISK` 的区别，关键不只是访问速度，而是缓存放不下时系统会怎样退化：

- `MEMORY_ONLY` 会尽量把分区结果保留在内存里，放不下的分区不会自动落盘。后续如果又有 action 需要这些未缓存成功的分区，Spark 只能沿着原来的 lineage 重新计算。
- `MEMORY_AND_DISK` 会把内存放不下的那部分分区写到 Executor 本地磁盘中。读取速度会比纯内存慢一些，但能避免因为部分缓存失败而反复重算上游链路。

对于包含大表扫描、宽依赖、聚合或去重的结果集来说，这种差别往往直接对应作业的稳定性差异。

所以懒执行在 Spark 里不是一个口号，它真正控制的是：

- 是否给优化器留下整体计划空间。
- 是否避免过早物化中间结果。
- 是否能把多个局部变换合并到更少的执行边界里。

代价也很清楚：

- 不真正触发 action 时，很多问题不会暴露出来。
- 一个 application 里多个 action 可能让你误以为“逻辑只写了一次”，但底层已经执行了多轮 job。
- 如果对 lineage、cache 和 action 边界没有感觉，作业会很容易重复计算。

### 3.3 DAG 为什么会被切成 Stage 和 Task，这个拆分到底在控制什么成本

这一节在回答什么问题：

- Spark 为什么总在讲 DAG、stage、task。
- 它们分别在控制什么边界。

这个问题是怎么解决的：

Spark 会把一个作业表示成 DAG，是因为一个分布式作业不是线性步骤，而是多个数据转换节点及其依赖关系。真正决定执行边界的关键，不是你代码里写了多少个算子，而是相邻算子之间的数据依赖是不是可以局部完成。

这里最关键的区分是：

1. `narrow dependency`
   - 上游一个分区的数据只需要被少量、通常是一个下游分区消费。
   - 这类依赖通常可以在本地连续执行，不需要大规模跨节点搬数据。
2. `wide dependency`
   - 上游多个分区的数据需要重新分发给下游分区。
   - 这会触发 shuffle，也意味着 stage 边界出现。

一个具体例子更容易把这件事看清：

```scala
val lines = sc.textFile("/logs", 4)

val result = lines
  .flatMap(_.split(" "))
  .map(word => (word, 1))
  .reduceByKey(_ + _, numPartitions = 2)
  .filter { case (_, cnt) => cnt > 10 }

result.saveAsTextFile("/out")
```

这段逻辑里：

- `saveAsTextFile` 是 action，所以这次提交会形成 1 个 job。
- `textFile -> flatMap -> map` 之间都是窄依赖，通常会被放在同一个 stage 里连续执行。
- `reduceByKey` 需要把相同 key 的数据从多个上游分区汇聚到一起，所以会触发 shuffle，也就形成了 stage 边界。
- `filter` 和后面的写出，通常会继续放在 shuffle 之后的 stage 中执行。
    - job 的数量主要看 action 的次数。 cache/persist 不会减少 job 数量，只是第二个 job 可能复用缓存结果。
    - stage 的数量主要看 shuffle / 宽依赖
    - task 的数量主要看分区数

如果输入有 4 个分区，而 `reduceByKey` 指定了 2 个目标分区，那么这个 job 往往会表现成：

1. `Stage 0`
   - 读取 4 个输入分区，执行 `flatMap + map`。
   - 在 `reduceByKey` 的 map-side 按目标分区分桶，并把 shuffle 数据写出来。
   - 这一阶段通常会有 4 个 task，因为它面对的是 4 个输入分区。
2. `Stage 1`
   - 读取上一个 stage 写出的 shuffle 数据。
   - 在 `reduceByKey` 的 reduce-side 拉取、合并并完成聚合。
   - 再继续执行后面的 `filter` 和结果写出。
   - 这一阶段通常会有 2 个 task，因为下游只有 2 个目标分区。

这里最容易混淆的一点是：触发 shuffle 的算子并不是“完整地属于前一个 stage”或“完整地属于后一个 stage”。更准确地说，它在物理执行上会横跨两个 stage：

- 前一个 stage 负责 shuffle write，也就是按目标分区组织并写出中间结果。
- 后一个 stage 负责 shuffle read，也就是把属于自己的那部分数据拉回来，再继续聚合或 join。

为什么 stage 要沿着 shuffle 来切？因为 shuffle 前后，成本结构已经变了：

- shuffle 之前，很多操作可以沿着分区本地连续做下去。
- shuffle 之后，数据必须重新分桶、排序、落盘、传输，再在下游重新聚合。

也就是说，stage 的边界不是语法边界，而是“还能不能在局部连续执行”的边界。Spark 按这个边界切阶段，能带来两个收益：

1. 让可以流水化的局部计算尽量在同一阶段内做完，减少中间落盘和调度切换。
2. 让失败恢复更聚焦，只回滚到必要阶段，而不是整条链从头开始。

`task` 则是 stage 内真正被调度执行的最小单元，一般和分区强相关。一个 stage 里有多少 task，本质上是在回答“这个阶段的数据被切成了多少并行分片”。所以 task 数量、分区数量、并行度，最后都会影响实际执行效率和资源利用。

代价在于：

- 分区切得过细，调度和小任务开销会上来。
- 分区切得过粗，单个 task 时间过长，局部热点更明显。
- 宽依赖太多，stage 被频繁打断，shuffle 成本会把整体吞吐拉下来。

所以 DAG、stage、task 不是一组需要背的层级，而是在控制“哪些计算可以连续局部完成，哪些地方必须重新分发数据”。

### 3.4 Shuffle 为什么总是 Spark 里最贵的一层

这一节在回答什么问题：

- shuffle 为什么容易成为 Spark 的主要成本中心。
- 它到底贵在什么地方。

这个问题是怎么解决的：

shuffle 之所以贵，是因为它把原本主要发生在本地分区里的计算，变成了一次跨分区、跨节点的数据重分发。只要发生了重分发，系统就不再只是算一遍函数，而是要额外承担分桶、排序、序列化、磁盘写入、网络传输、下游拉取、合并和反序列化这些成本。

一条典型 shuffle 成本链可以压成下面几步：

1. map 侧先根据目标分区重新划分数据。
2. 数据往往要排序、缓冲、必要时落盘。
3. 下游 task 再从多个上游节点拉取属于自己的那部分数据。
4. 拉回来的数据还要再合并、排序或聚合，才能继续后续计算。

所以 shuffle 贵，并不是因为“多了一个网络步骤”，而是它把多种高成本动作堆在一起了：

- 网络传输成本
- 磁盘 spill 和 merge 成本
- 排序和序列化成本
- 内存占用与 GC 成本
- 下游等待最慢 map 任务完成的尾部成本

这也是为什么很多 Spark 作业在非 shuffle 阶段看起来很快，一到 `groupBy`、`reduceByKey`、`join`、`distinct` 这类操作就突然慢下来。不是这些操作“逻辑复杂”，而是它们背后要求系统重新分发数据。

shuffle 也不是纯坏事。它解决的是“相同 key 的数据必须汇聚到一起”或“某个 join 需要把相关数据对齐”的问题。没有 shuffle，很多聚合和关联根本做不成。所以问题不在于“要不要 shuffle”，而在于：

- 是否真的需要这次重分发。
- 能不能提前局部聚合，缩小传输体积。
- 分区策略是否合理。
- 是否可以用广播、预聚合、分桶等手段降低代价。

所以 shuffle 的本质不是一个神秘黑盒，而是“跨节点重新组织数据”这件事天然昂贵。

这里还要继续往前压一步：并不是所有聚合算子都会把这份成本推到一样高。像 `groupByKey` 和 `reduceByKey` 看起来都在按 key 聚合，也都会触发 shuffle，但它们进入 shuffle 的数据形态并不一样。

一个很短的例子就能看出来。假设有两个分区：

```text
Partition 0: (a,1) (a,2) (b,1)
Partition 1: (a,3) (b,4) (b,5)
```

如果使用 `groupByKey`，系统需要把同一个 key 的所有原始 value 都拉到一起，所以更接近直接把 `(a,1) (a,2) (a,3)`、`(b,1) (b,4) (b,5)` 这些原始记录推进 shuffle。  
如果使用 `reduceByKey(_ + _)`，则可以先在每个分区内做局部聚合，例如先得到 `(a,3) (b,1)` 和 `(a,3) (b,9)`，再把这些已经压缩过的中间结果推进 shuffle。

所以两者真正的区别不是“谁会不会 shuffle”，而是：

- `groupByKey` 更接近把原始数据完整推入 shuffle。
- `reduceByKey` 可以先做 map-side combine，再把压缩后的结果推进 shuffle。

这也是为什么 Spark 里的很多优化不是“执行时再想办法”，而是算子选择本身就在决定 shuffle 的成本上限。系统不只是问“能不能聚合”，还在问“聚合之前，有多少数据能先在本地被压缩掉”。

### 3.5 数据倾斜为什么会把整条作业拖慢，它和资源不足不是一回事

这一节在回答什么问题：

- 数据倾斜到底是什么。
- 为什么看起来总资源不少，作业却还是被少数 task 拖住。

这个问题是怎么解决的：

数据倾斜本质上是数据分布不均。Spark 的并行执行默认假设“不同分区的负载大致可控”，但如果某些 key 极端集中、某些分区远大于其他分区、某些 join 一侧存在热点值，那么少数 task 会处理远超平均值的数据量，形成明显长尾。

它和单纯“资源不够”不是一回事。资源不够通常表现为整体都慢；数据倾斜更常见的表现是：

- 大多数 task 已经完成，只有少数 task 长时间不结束。
- 个别 Executor 的 CPU、内存、磁盘或网络压力异常高。
- 某个 stage 的最后阶段明显被个别分区拖住。

为什么它会这么难受？因为分布式任务通常要等最慢那个 task 收尾，后续阶段才能继续。也就是说，哪怕 95% 的数据都处理完了，只要剩下那个热点分区还没结束，整条作业就还在等。

常见成因一般有四类：

1. 热 key
   - 业务上天然集中，比如某个默认值、热门用户、异常统一值。
2. join 两侧数据分布极不均匀
   - 某些 key 在大表和小表里都特别集中。
3. 上游分区策略不合理
   - 分区数过少或分区规则无法均摊数据。
4. 小文件或数据组织问题
   - 某些分区虽然逻辑记录数不大，但底层文件布局让读取异常集中。

治理思路也必须对准原因，而不是统一回答“加机器”：

- 热 key 需要考虑打散、两阶段聚合、业务侧拆分。
- 例如按 `province` 做聚合时，`unknown` 占了大部分数据，可以先把 `unknown` 打散成 `unknown_0 ~ unknown_9` 做一轮局部聚合，再汇总回真正的 `unknown`。
- 大小表 join 可能更适合广播小表。
- 例如行为日志去 join 只有几千行的渠道维表时，可以直接广播维表，避免维表也跟着大表一起 shuffle。
- 分区不合理需要重分区或调整 key 设计。
- 例如按 `country` 分区时 90% 数据都落到 `CN`，可以改成按 `country + dt` 分区，或直接提高并行分区数来摊平负载。
- 小文件或数据布局问题更适合先做 compaction 或重写分区，而不是只调 Spark 参数。
- 例如某天分区下堆了几万个小 Parquet 文件，少数 task 会被文件打开、关闭和随机读拖慢，这时先合并小文件往往比盲目加机器更有效。
- Spark SQL 场景可以结合自适应执行能力去减轻部分倾斜，但前提仍然是数据模式允许。
- 例如 join 跑起来后才发现少数 shuffle partition 明显大很多，这时 AQE 可以尝试拆分 skewed partition 或改写 join 策略来减轻长尾。

所以数据倾斜真正暴露的是“数据分布结构和执行模型不匹配”，而不是单纯机器不够。

这件事之所以容易被误判，是因为它常常表现成“少数 task 特别慢”，表面看像资源不够，实际上更像负载没摊平。真实排查时，通常要继续观察下面这些信号：

- 同一个 stage 里，大多数 task 很快结束，只有少数 task 长时间拖尾。
- 少数 task 的 `shuffle read`、输入记录数、spill 体积明显高于其他 task。
- 个别 Executor 的 CPU、内存或磁盘 IO 明显高于整体平均。

为什么这些治理手段会有效，也要继续解释到机制层：

- 打散 key
  - 是把原来集中到一个分区的大 key 拆到多个分区先并行处理。
  - 本质是用后续多一次聚合，换前面不再由一个 task 独扛全部压力。
- 广播小表
  - 是避免 join 时让小表也参与大规模 shuffle。
  - 本质是用更多 Executor 本地内存，换更少的数据重分发。
- 两阶段聚合
  - 是先在局部把数据压缩，再做全局汇总。
  - 本质是削减进入最终热点分区的数据量。

只有把“现象 -> 机制 -> 治理手段为什么有效”接起来，数据倾斜这块才不只是停留在“知道有这个词”。

### 3.6 缓存、广播、分区这些优化手段，到底各自在解决什么问题

这一节在回答什么问题：

- Spark 常见优化手段为什么不能当成固定口诀。
- 它们分别在降低哪一类成本。

这个问题是怎么解决的：

Spark 里的常见优化手段看起来很多，但本质上都在针对前面提到的成本项下手：

1. `cache / persist`
   - 解决的是“同一份中间结果被重复计算”的成本。
   - 适合多次复用、上游代价高、后续分支多的场景。
2. `broadcast`
   - 解决的是“为了 join，把小表也一起参与 shuffle”的成本。
   - 当一侧足够小，广播给各 Executor 通常比全量重分发更便宜。
3. `partitioning`
   - 解决的是“数据怎样分到不同 task 手里更均衡、更少重分发”的问题。
   - 本质是把后续代价前置到数据组织阶段。

这些手段之所以不能背成口诀，是因为它们都会引入自己的代价：

- 缓存会吃内存，缓存不命中或缓存过多会反过来带来淘汰、GC 和 spill。
- 广播并不适合中等偏大的表，广播对象过大同样会压 Executor 内存。
- 重分区本身可能就是一次昂贵的 shuffle，如果收益不明确，可能越调越慢。

所以判断标准不是“这个优化名词是否常见”，而是：

- 我当前在重复付哪类成本。
- 这个手段能不能精准削掉这类成本。
- 它引入的新成本是否比节省下来的成本更小。

这才是 Spark 优化里真正稳定的主线。

还要再补一个容易被忽略的判断：这些优化手段里，有些是在执行前就要做出的静态决策，有些则更适合交给运行时根据真实数据分布再调整。也就是说，优化不只发生在“写代码阶段”，还可能发生在“执行阶段”。

### 3.7 AQE、Skew Join 和运行时计划调整为什么会越来越重要

这一节在回答什么问题：

- 为什么同一条 SQL 或同一段 Spark 作业，执行前很难完全知道最优计划。
- 自适应执行到底在根据什么信息调整计划。

这个问题是怎么解决的：

Spark 在真正运行前，能看到的是统计信息、逻辑依赖和预估计划，但很多关键成本只有作业跑起来后才会暴露出来，例如：

- 某次 shuffle 之后每个分区到底有多大。
- 某个 join 一侧实际上是不是已经小到可以广播。
- 某些分区是否明显比其他分区更大，已经形成 skew。

AQE，也就是自适应执行，解决的就是这个问题：先让作业跑到关键边界，再根据运行时拿到的真实统计信息，对后续物理计划做调整。它的价值不在于“神奇提速”，而在于减少因为前期估计不准导致的明显错误计划。

这类调整通常集中在几件事上：

1. 动态合并或调整 shuffle 分区
   - 避免分区过细导致小 task 过多，或分区过粗导致局部 task 过重。
2. 动态改写 join 策略
   - 某一侧数据比预估更小时，可能转成 broadcast join，减少重分发。
3. 对明显倾斜的分区做特殊处理
   - 把超大分区拆开，避免一个 task 独自拖尾。

这背后的本质不是“Spark 更聪明了”，而是：

- 计划优化不能只依赖静态推测。
- 运行时真实数据分布，往往比执行前估计更重要。

但 AQE 也不是万能解：

- 如果上游数据模式本身就极端不均，AQE 只能减轻，不一定能根治。
- 如果逻辑层选择了代价很高的算子，运行时优化也只能在有限空间里修补。
- 如果统计信息不足或链路过于复杂，AQE 也不一定能改到最优。

所以 AQE 的意义，不是替代你理解 shuffle、join 和倾斜，而是在你已经理解这些成本结构之后，再给运行时一次“按真实情况修正计划”的机会。

## 4. 边界与误区

- DAG 不是执行日志，它是依赖关系图；真正切 stage 的关键是宽依赖和 shuffle。
- 懒执行不等于什么都不做；它是在 action 触发前先保留 lineage 和计划优化空间。
- 一个 application 可以因为多个 action 触发多个 job，如果中间结果没缓存，前序计算可能被重复执行。
- `MEMORY_ONLY` 不等于“放不下就自动降级到磁盘”；放不下的分区后续很可能直接按 lineage 重算，这也是缓存策略最容易被误判的地方。
- stage 变多不一定坏，关键看是不是因为不必要的重分发把流水化执行打断了。
- shuffle 不等于失败，它是很多聚合和 join 的必要代价，问题是要控制次数和规模。
- `groupByKey` 之类的高成本操作，问题不在“名字不好”，而在它更容易把原始数据完整推入 shuffle。
- 数据倾斜不等于资源少，很多时候是数据分布和分区策略本身出了问题。
- `cache`、`broadcast`、`repartition` 都不是默认应该加的优化，它们都必须回答“我到底在消什么成本”。
- AQE 不是万能优化器，它更像基于运行时统计信息做计划修正，而不是替代对执行模型的理解。
- 单纯加 Executor 或加 core，解决不了热 key、慢 shuffle、错误分区这些结构性问题。

## 5. 自测清单

- 是否能说明 Driver、Executor、调度器分别在 Spark 里承担什么职责。
- 是否能解释为什么 transformation 往往不会立刻执行，而 action 会触发 job。
- 是否能解释为什么宽依赖往往会成为 stage 边界。
- 是否能把 shuffle 的成本讲成“分桶 -> 排序 / spill -> 网络传输 -> 拉取合并”的连续过程。
- 是否能说明为什么 `groupBy`、`join`、`distinct` 这类操作经常会突然变慢。
- 是否能解释为什么 `reduceByKey` 一类操作通常比直接把原始值全部推入 shuffle 更省成本。
- 是否能区分“整体资源不足”和“数据倾斜导致长尾”这两类问题。
- 是否能结合一个真实业务场景说明热 key 是怎么把少数 task 拖成瓶颈的。
- 是否能解释缓存、广播、重分区分别在削减哪一类成本。
- 是否能解释 AQE、broadcast join、skew join 分别在什么边界下会有收益。
