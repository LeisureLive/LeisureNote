## 1. 这一块在回答什么问题

这一块真正要回答的，不是“背几个 Linux 命令”或“会说几个监控指标”，而是下面这条连续问题：

1. 线上出问题时，为什么最怕的不是不会命令，而是没有稳定的判断顺序。
2. 为什么 CPU 飙高、RT 抖动、慢 SQL、热 key、MQ 积压、频繁 Full GC 这些现象虽然表面不同，本质上都在回答“瓶颈落在哪一层”。
3. 为什么排障不能只盯应用代码，而必须同时看请求链路、线程、依赖、缓存、数据库、MQ 和系统资源。
4. 为什么资深岗排障题更重视“先看什么、如何缩小范围、如何止损”，而不是命令背得多不多。
5. 为什么性能优化也不是泛讲“加缓存、加机器”，而是先识别成本结构，再决定该优化哪一层。

所以这篇文档的主线应该是：

- 先从现象入手，不先猜结论。
- 再用监控和链路把问题缩小到某一层。
- 再结合线程、资源、依赖和数据路径定位根因。
- 再先做止损，再做长期修复。
- 最后把经验沉淀成可复用的排障路径，而不是一次性操作记录。

## 2. 主线

1. 先看现象：是 CPU、内存、GC、RT、错误率、积压还是吞吐异常。
2. 再看范围：是单机问题、单实例问题、单租户问题、单依赖问题，还是系统级问题。
3. 再看链路：入口、线程池、缓存、数据库、MQ、下游依赖分别有没有异常。
4. 再看根因：资源打满、锁竞争、慢依赖、数据热点、错误索引、背压或配置问题哪个更像源头。
5. 最后看动作：先止损、再修复、再预防。

## 3. 核心问题展开

### 3.1 为什么线上排障不能一上来就登机器看日志

这一节在回答什么问题：

- 排障为什么必须先看现象和范围。
- 为什么一上来就看日志或猜代码，往往会把时间浪费在错误方向上。

这个问题是怎么解决的：

线上排障最稳的起点，不是先 SSH 上去，也不是先搜日志关键字，而是先确认“系统到底出现了什么异常表现”。因为很多现象虽然最后都落到服务异常，但根因完全不同：

1. RT 抖动可能来自下游超时、线程池打满、GC 停顿或慢 SQL。
2. 错误率上升可能来自配置变更、依赖故障、连接池耗尽或流量峰值。
3. 吞吐下降可能来自限流、积压、热点或资源耗尽。

所以排障的第一步必须先钉住现象和范围：

- 指标是哪一个先异常。
- 异常开始于什么时间点。
- 影响的是所有请求还是某类请求。
- 影响的是所有实例还是少数实例。
- 是否和发布、流量峰值、依赖变更时间吻合。

这一步的价值在于快速把问题缩到更小范围：

- 如果只有少数实例异常，更像单机资源、局部热 key 或实例级配置问题。
- 如果所有实例一起抖，更像共享依赖、数据库、缓存、MQ 或网关层问题。
- 如果只影响某个租户或某类接口，更像数据分布或特定路径问题。

所以稳定的排障顺序应该是“先现象、后范围”，而不是“先看代码、先猜根因”。

### 3.2 监控指标为什么必须先于命令行操作，它们分别在告诉你什么

这一节在回答什么问题：

- 监控指标为什么是排障主入口。
- CPU、内存、线程、GC、RT、QPS、错误率分别在提示什么方向。

这个问题是怎么解决的：

监控之所以必须先看，是因为它能最快回答“问题最像发生在哪一层”。不同指标虽然不会直接给出根因，但会明显缩小搜索空间：

1. `RT`
   - 提示请求完成时间是否明显变慢。
   - 更偏用户感知层和调用链整体表现。
2. `错误率`
   - 提示是不是已经出现大量失败，以及失败是否集中在特定接口或特定依赖。
3. `QPS`
   - 提示问题是吞吐突然增高、下降，还是请求结构变化。
4. `CPU`
   - 提示是否有高计算、死循环、频繁序列化、锁竞争、自旋或热点逻辑。
5. `内存 / GC`
   - 提示对象膨胀、缓存失控、大结果集、泄漏、频繁 Full GC 等方向。
6. `线程池 / 连接池`
   - 提示是否已经出现等待堆积和资源耗尽。

为什么这些指标必须先一起看？因为单看某一个很容易误判：

- CPU 高不一定是业务逻辑重，也可能是频繁重试、日志打爆、序列化风暴。
- RT 高不一定是本服务慢，也可能是下游调用卡住。
- 错误率不高也不等于没问题，积压和延迟可能已经在持续扩大。

所以监控的价值不在于“看到很多图”，而在于先判断：

- 问题更像算力问题、等待问题、内存问题，还是依赖问题。
- 异常是在入口开始，还是在内部链路后半段开始。

只有这一步做对了，后面的命令行、线程栈、SQL、日志分析才有正确方向。

### 3.3 CPU 飙高、RT 抖动、Full GC、慢 SQL 为什么要按不同路径排查

这一节在回答什么问题：

- 常见高频故障现象为什么不能用一套模板硬套。
- 这些问题分别应该沿哪条路径往下缩小。

这个问题是怎么解决的：

这些现象表面上都叫“线上性能问题”，但它们切入点不同，所以排查顺序也必须不同。

1. `CPU 飙高`
   - 先看是单机还是多机。
   - 再看线程栈、热点方法、死循环、频繁序列化、日志风暴、锁竞争。
   - 它更偏“谁在持续占用算力”。
2. `RT 抖动`
   - 先分本服务执行慢还是等待慢。
   - 再看下游依赖、线程池、数据库、缓存命中、网络超时。
   - 它更偏“请求在哪一段被拖长了”。
3. `频繁 Full GC`
   - 先看堆是否持续增长、回收后是否回不去、对象增长来自哪类请求。
   - 再看缓存、大对象、大结果集、内存泄漏、数据堆积。
   - 它更偏“内存和对象生命周期出了什么问题”。
4. `慢 SQL`
   - 先分稳定慢还是抖动慢。
   - 再分执行慢还是等待慢。
   - 再看执行计划、索引、锁等待、热点行、深分页、连接池。
   - 它更偏“数据库内部执行成本还是外部竞争成本”。

这些路径为什么要分开？因为它们最终关注的瓶颈维度不同：

- CPU 主要盯计算和线程行为。
- RT 主要盯调用链和等待。
- GC 主要盯对象和内存行为。
- SQL 主要盯执行计划和数据库竞争。

所以排障时不能只背一个大而全模板，而是要先判断问题的第一性状，再走对应路径。

### 3.4 MQ 积压、热 key、线程池打满为什么本质上都在暴露流量与资源的不匹配

这一节在回答什么问题：

- 为什么这些看起来不同的问题会经常一起出现。
- 它们本质上在暴露什么系统矛盾。

这个问题是怎么解决的：

MQ 积压、热 key、线程池打满表面上分别发生在消息系统、缓存层和执行层，但它们共同暴露的是同一个矛盾：流量分布和资源承载能力不匹配。

1. `MQ 积压`
   - 说明生产速度长期高于消费速度，或者某些分区 / 某类消息消费明显偏慢。
2. `热 key`
   - 说明访问流量集中在少数 key 上，缓存、数据库或下游资源被局部热点压垮。
3. `线程池打满`
   - 说明单位时间进入的工作量已经超过线程池和下游资源可承接范围，任务开始排队。

为什么它们经常会联动？

- 某类热点请求先打穿缓存，回源数据库后让执行时间拉长。
- 执行时间拉长后线程池堆积，异步消费速度下降。
- 消费下降后 MQ 开始积压，最终连锁成全链路抖动。

所以这类问题的处理重点，不只是“扩容”：

- 要先看是不是数据分布失衡。
- 要看是慢依赖拖慢了消费，还是消费模型本身太弱。
- 要看线程池参数是不是和实际业务负载匹配。
- 要看是否需要对热点做单独隔离、单独缓存、单独限流。

也就是说，这类问题真正暴露的不是“某个组件不行”，而是整条链路的流量形态和资源配置没有对齐。

### 3.5 Linux、网络和压测为什么不是附属知识，而是排障和性能优化的基础层

这一节在回答什么问题：

- Linux 命令、网络基础和压测为什么会成为排障题的底层支撑。
- 为什么它们不能被当成独立冷知识背诵。

这个问题是怎么解决的：

很多应用层故障最终都要落回操作系统和网络层才能解释清楚：

1. 线程很多但 CPU 不高，可能是等待 IO 或等待锁，而不是纯业务逻辑慢。
2. 接口超时不一定是代码执行慢，也可能是网络抖动、连接池耗尽、连接建立成本异常。
3. 压测里吞吐上不去，不一定是服务端处理慢，也可能是客户端连接模型、网络带宽、文件句柄或系统参数限制。

所以 Linux 和网络的价值，不是让你背命令，而是帮助你回答：

- 当前瓶颈更像应用层、JVM 层、系统层还是网络层。
- 请求是在排队、在等待、在超时，还是在反复重试。
- 连接是建立慢、传输慢，还是服务端处理慢。

压测的价值也一样。它不是“把机器打满看看”，而是为了在更可控环境里回答：

- 系统真实瓶颈落在哪。
- 当并发上来时，是 CPU 先满、线程池先打满，还是下游先掉速。
- 限流、缓存、异步化、隔离这些手段到底有没有真实收益。

所以这三块不是附属冷知识，而是线上排障和性能优化的基础判断层。

### 3.6 排障和性能优化为什么必须先止损，再修复，再预防

这一节在回答什么问题：

- 线上问题处理为什么不能只盯根因。
- 为什么“先止损、再修复、再预防”是资深岗更稳定的处理顺序。

这个问题是怎么解决的：

线上问题一旦已经影响用户，系统最优先要做的不是立刻写出最优解，而是先把损失控制住。因为在真实生产环境里，继续拖着等完整分析，很可能会把局部问题放大成更大范围故障。

这就是为什么更稳的处理顺序通常是：

1. `止损`
   - 限流、降级、熔断、切流、回滚、扩容、关闭次要功能。
2. `修复`
   - 找到真正瓶颈并做针对性调整，例如改索引、调线程池、修缓存策略、调整消费模型。
3. `预防`
   - 补监控、补告警、补压测、补隔离、补容量评估、补自动化演练。

为什么这个顺序重要？因为它把“业务影响控制”和“技术优化完善”拆开了：

- 止损优先保住服务。
- 修复优先解决这次真正的根因。
- 预防优先减少同类问题再次发生。

如果没有这三层，很多所谓的排障经验最后只会变成：

- 这次问题处理掉了。
- 但下次相似问题还会再来。

所以资深岗排障题真正看重的，不是你会不会几条命令，而是你有没有完整的问题处理闭环。

## 4. 边界、误区、常见追问

- 排障不要一上来先猜代码问题，先看现象和范围。
- 监控不是看一张图，要用一组指标判断问题更像落在哪一层。
- CPU 高、RT 高、GC 多、慢 SQL 分别走不同路径，不要用一套模板硬套。
- MQ 积压、热 key、线程池打满很多时候不是独立问题，而是一条资源失衡链。
- Linux 和网络不是附属冷知识，它们是帮助你判断瓶颈位置的基础层。
- 线上问题先止损，再修复，再预防；不要为了追求一次性定位最优解而延误止损。
