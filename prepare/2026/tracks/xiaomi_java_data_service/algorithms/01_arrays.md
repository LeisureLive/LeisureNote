# 算法专题 01：数组

适用目标：

- 系统理解数组题的核心特点
- 能识别数组题常见模式
- 为后续刷题和面试表达提供模板

## 1. 数组的核心特点

- 内存连续
- 支持按下标 `O(1)` 访问
- 尾部追加通常较高效
- 中间插入 / 删除通常需要搬移元素，代价较高
- 很多题的本质是“如何在遍历过程中维护一段有意义的区间”

## 2. 什么题应该优先想到数组

如果题目有下面这些特征，通常优先考虑数组思维：

- 输入本身就是数组，要求原地修改
- 需要找连续区间、子数组、窗口
- 需要按位置关系处理：左边、右边、前缀、后缀
- 需要通过下标做状态表达
- 需要频繁访问相邻元素

## 3. 数组题常考模式

建议你看这一节时，不要只记“题型名字”，而要固定追问三件事：

- 这题为什么会落到这个模式
- 这个模式的核心状态是什么
- 代码里先做什么、后做什么

### 3.1 遍历与模拟

适用场景：

- 规则明确，按顺序处理每个元素
- 更考查边界控制和实现细节

典型题：

- `88. 合并两个有序数组`
- `54. 螺旋矩阵`
- `59. 螺旋矩阵 II`

通用套路：

1. 先把题目拆成“访问顺序 + 规则更新 + 停止条件”三部分。
2. 明确当前到底在维护什么：
   - 当前下标
   - 当前方向
   - 当前边界
   - 当前结果
3. 先写主循环框架，再填每一步的更新逻辑，不要一上来把所有分支揉在一起。
4. 每处理完一轮，就立刻更新边界或状态，避免重复访问或漏访问。

你真正要会的是：

- 能先把题目翻译成几个“机械动作”
- 能说清为什么循环会结束
- 能控制边界，而不是写到一半靠调试补洞

### 3.2 双指针

适用场景：

- 左右收缩
- 快慢指针
- 原地去重 / 原地删除
- 有序数组上的搜索和配对

常见信号：

- “有序数组”
- “原地删除 / 原地去重”
- “找两数 / 多数之和”

典型题：

- `26. 删除有序数组中的重复项`
- `27. 移除元素`
- `167. 两数之和 II - 输入有序数组`
- `11. 盛最多水的容器`

通用套路：

1. 先判断是哪一类双指针：
   - 左右夹逼
   - 快慢指针
   - 同向扫描
2. 先定义两个指针各自代表什么，再写移动规则。
3. 必须回答“为什么只移动这一边不会漏解”，这个理由通常来自：
   - 有序性
   - 单调性
   - 原地覆盖时的读写分工
4. 如果是原地修改题，常见写法是：
   - 快指针负责扫描
   - 慢指针负责写入有效结果
5. 如果是有序数组夹逼题，常见写法是：
   - 左指针太小就右移
   - 右指针太大就左移

你真正要会的是：

- 不是只会“写两个指针”
- 而是能解释每次移动的依据和不变量

### 3.3 滑动窗口

适用场景：

- 连续子数组 / 连续子串问题
- 窗口内需要动态维护某种状态
- 目标是求最长、最短，或统计满足条件的连续区间

#### 1. 使用信号

如果题目出现下面这些特征，优先考虑滑动窗口：

- 题目对象是连续子数组 / 连续子串
- 题目要求窗口内满足某种约束，例如：
  - 无重复
  - 至多 `k` 个不合法元素
  - 覆盖某些字符
  - 和达到某个阈值
- 目标通常是：
  - 最长合法区间
  - 最短满足条件区间
  - 统计满足条件的区间
  - 找固定长度窗口中的最优结果
- 暴力写法会自然变成：
  - 枚举左端点，再不断向右扩
  - 或枚举所有子串 / 子数组

一句话判断：

- 只要题目是“连续区间 + 区间状态可维护 + 左边界右移后旧状态可删除”，就优先考虑滑动窗口。

#### 2. 算法定义

滑动窗口本质上是在维护一个连续区间，以及这个区间对应的状态。

核心只有 3 个部分：

- 窗口边界：
  - `left`
  - `right`
- 窗口状态：
  - 频次数组 / 哈希表
  - 当前和
  - 当前种类数
  - 坏元素个数
  - 已满足条件的计数
- 窗口不变量：
  - 当前窗口是否合法
  - 当前窗口是否已经满足目标条件

滑动窗口真正要做的事情，不是机械移动双指针，而是：

- 把右侧新元素纳入窗口状态
- 在需要时移动左边界并删除旧状态
- 在“合法 / 不合法 / 满足 / 不满足”之间切换
- 在合适时机更新答案

#### 3. 区间写法怎么选

滑动窗口常见有两种区间写法。

##### 3.1 左闭右闭 `[left, right]`

含义：

- `right` 指向当前已经在窗口里的最后一个元素

常见特点：

- 窗口长度通常写成 `right - left + 1`
- 常见写法是 `for (int right = 0; right < n; right++)`

##### 3.2 左闭右开 `[left, right)`

含义：

- `right` 指向下一个准备进入窗口的位置

常见特点：

- 窗口长度通常写成 `right - left`
- 外层循环条件常见写法是 `while (right < n)`
- 进入循环后，先处理 `nums[right]` 入窗，再让 `right++`

##### 3.3 怎么选

结论：

- 这几类滑动窗口题型都不强绑定某一种区间写法
- 关键不是题型，而是整段代码的语义必须统一

默认建议：

- 专题和模板优先统一成 `[left, right)`
- 因为：
  - 长度公式统一是 `right - left`
  - `right` 更自然地表示“下一个待进入位置”
  - 更适合写成“先入窗，再修复，再更新答案”

什么时候用 `[left, right]` 也可以：

- 你更习惯把 `right` 理解成“当前窗口最后一个元素”
- 你更习惯在窗口长度里写 `right - left + 1`

一句话要求：

- 选一种写法后，窗口语义、长度公式、推进顺序、答案更新时机必须保持一致。

#### 4. 常见题型与对应套路

##### 4.1 固定长度窗口

特征：

- 窗口长度固定不变
- 每次只需要维护“进一个、出一个”

代表题：

- `438. 找到字符串中所有字母异位词`

通用套路：

1. 把右侧新元素纳入窗口状态
2. 如果窗口长度超过固定值，就移动左边界并删除旧状态
3. 当窗口长度恰好等于目标长度时，判断是否满足题意

核心点：

- 重点不是“合法 / 不合法”
- 而是固定长度下的状态维护

##### 4.2 变长窗口：求最长合法区间

特征：

- 窗口内有合法性约束
- 目标是求最长

代表题：

- `3. 无重复字符的最长子串`
- `485. 最大连续 1 的个数`
- `1004. Max Consecutive Ones III`

通用套路：

1. 把右侧新元素纳入窗口状态
2. 如果窗口不合法，就不断移动左边界修复窗口
3. 窗口重新合法后，更新最长答案

核心点：

- 不合法就缩
- 合法时更新答案

##### 4.3 变长窗口：求最短满足区间

特征：

- 窗口需要先满足条件
- 目标是求最短

代表题：

- `209. 长度最小的子数组`
- `76. 最小覆盖子串`

通用套路：

1. 把右侧新元素纳入窗口状态
2. 一旦窗口满足条件，就不断移动左边界压缩窗口
3. 在压缩过程中更新最短答案
4. 直到窗口重新不满足，再继续处理后续元素

核心点：

- 满足了就尽量缩
- 和“最长合法窗口”正好相反

##### 4.4 容错型窗口

特征：

- 窗口内允许最多 `k` 个不满足条件的元素
- 常见表述是：
  - 最多替换 `k` 次
  - 最多包含 `k` 个 `0`
  - 最多删除 / 修改 `k` 个字符

代表题：

- `1004. Max Consecutive Ones III`

通用套路：

1. 维护一个“坏元素计数”
2. 把右侧新元素纳入窗口状态时更新坏元素数
3. 如果坏元素数超过 `k`，就移动左边界修复
4. 合法时更新答案

核心点：

- 本质上还是“最长合法窗口”
- 只是合法条件变成了“坏元素数不超过 `k`”

#### 5. 通用骨架

下面默认按 `[left, right)` 写。

##### 5.1 最长合法窗口

```java
while (right < n) {
    // 1. 把 nums[right] 纳入窗口状态
    add(nums[right]);
    right++;

    // 2. 如果窗口不合法，就移动 left 修复
    while (窗口不合法) {
        remove(nums[left]);
        left++;
    }

    // 3. 窗口合法时更新答案
    update(right - left);
}
```

##### 5.2 最短满足窗口

```java
while (right < n) {
    // 1. 把 nums[right] 纳入窗口状态
    add(nums[right]);
    right++;

    // 2. 只要窗口满足条件，就尽量收缩
    while (窗口满足条件) {
        update(right - left);
        remove(nums[left]);
        left++;
    }
}
```

##### 5.3 固定长度窗口

```java
while (right < n) {
    // 1. 把 nums[right] 纳入窗口状态
    add(nums[right]);
    right++;

    // 2. 如果窗口长度超过固定值，就收缩
    if (right - left > fixedLen) {
        remove(nums[left]);
        left++;
    }

    // 3. 当窗口长度刚好等于固定值时，判断答案
    if (right - left == fixedLen) {
        update();
    }
}
```

如果采用 `[left, right]`，则更常见写法是：

```java
for (int right = 0; right < n; right++) {
    add(nums[right]);

    while (窗口不合法) {
        remove(nums[left]);
        left++;
    }

    update(right - left + 1);
}
```

### 3.4 前缀和

适用场景：

- 频繁查询某个区间和
- 子数组求和类问题
- 需要把“区间信息”转成“两个前缀的差”

典型题：

- `303. 区域和检索 - 数组不可变`
- `724. 寻找数组的中心下标`
- `560. 和为 K 的子数组`（与哈希表结合）

通用套路：

1. 先把“区间信息”改写成“两个前缀的差”。
2. 明确定义：
   - `prefix[i]` 到底表示前 `i` 个元素之和
   - 还是表示到下标 `i` 为止的和
3. 如果是离线多次查询区间和，通常先建前缀和数组。
4. 如果是边遍历边统计答案，通常用“滚动前缀和 + 其他结构”。
5. 写代码时一定把下标和定义对齐，不然很容易差一位。

你真正要会的是：

- 看到区间和先想到“能不能转前缀差”
- 能把数学等式稳定翻译成代码，而不是凭感觉写

### 3.5 前后缀分解

适用场景：

- 每个位置的答案依赖左边信息和右边信息
- 不适合直接暴力枚举左右两边

典型题：

- `238. 除自身以外数组的乘积`
- `42. 接雨水`

通用套路：

1. 先识别每个位置的答案是否同时依赖左边信息和右边信息。
2. 如果左右两边的信息可以独立计算，就考虑拆成：
   - 左侧信息
   - 右侧信息
   - 最后合并
3. 常见写法有两种：
   - 两个辅助数组分别保存左信息和右信息
   - 先存一边，再用滚动变量补另一边
4. 这类题的价值在于避免“每个位置都回头扫左边和右边”的暴力做法。

你真正要会的是：

- 能先看出“答案依赖左右两侧”
- 再自然想到把左右信息拆开计算

### 3.6 原地交换 / 原地标记

适用场景：

- 要求 `O(1)` 额外空间
- 元素值和下标之间存在天然映射关系

典型题：

- `41. 缺失的第一个正数`
- `448. 找到所有数组中消失的数字`

通用套路：

1. 先判断元素值和下标之间是否存在天然映射关系。
2. 如果题目要求 `O(1)` 额外空间，经常要考虑：
   - 把值放回它“应该去的位置”
   - 或直接在原数组上做标记
3. 原地交换题常见流程是：
   - 当前值合法且不在目标位置，就持续交换
   - 直到当前位置无法继续交换
4. 原地标记题常见流程是：
   - 用正负号或特殊值记录“这个位置代表的数字见过没有”
5. 这类题最怕死循环和覆盖错误，写之前一定先讲清交换条件和停止条件。

你真正要会的是：

- 先看出“值和位置能不能绑定”
- 再决定是交换归位还是原地标记

### 3.7 二分查找

适用场景：

- 数组有序
- 答案空间具有单调性

典型题：

- `704. 二分查找`
- `34. 在排序数组中查找元素的第一个和最后一个位置`
- `875. 爱吃香蕉的珂珂`

通用套路：

1. 先分清这是哪一类二分：
   - 有序数组找目标
   - 找左边界 / 右边界
   - 对答案空间做二分
2. 先定义搜索区间，再统一边界风格：
   - 左闭右闭
   - 或左闭右开
3. 每次取 `mid` 后，要明确当前是在判断：
   - 值相等关系
   - 还是某个单调条件是否成立
4. 如果是答案空间二分，核心不是数组有序，而是“可行性随答案单调变化”。
5. 二分写不稳，通常不是代码能力问题，而是：
   - 区间定义没定死
   - 更新规则和区间定义不一致

你真正要会的是：

- 先讲清单调性或边界目标
- 再套统一模板，而不是每题临场重写一套二分

## 4. 数组题的高频坑点

- 下标越界
- 左闭右闭、左闭右开不统一
- 原地修改时覆盖掉后面还要用的数据
- 忘记处理空数组、长度为 `1` 的数组
- 写出了正确思路，但“定义”和“代码实现”不一致
- 复杂度判断不严谨，以为用了数组就默认 `O(n)`

## 5. 数组题的面试表达模板

面试里可以优先用这个结构表达：

1. 先说输入特征
2. 再说为什么想到某种数组技巧
3. 再说核心状态定义
4. 再说遍历顺序
5. 最后给复杂度和边界

例子：

```text
这题输入是数组，而且每个位置的答案依赖左边和右边的信息，所以我会考虑前后缀分解。
我定义 prefix[i] 表示 i 左边元素的乘积，suffix[i] 表示 i 右边元素的乘积，
那么答案就是 prefix[i] * suffix[i]。
这样只需要线性遍历两到三次，时间复杂度 O(n)。
```

## 6. 本专题关键训练样例

### 6.1 `238. 除自身以外数组的乘积`

本质：

- 数组题
- 重点是前后缀分解
- 不需要哈希表

你需要掌握：

- 如何定义前缀积和后缀积
- 如何保证定义与代码一致
- 如何优化到 `O(1)` 额外空间

### 6.2 `167. 两数之和 II - 输入有序数组`

本质：

- 数组题
- 重点是双指针
- 关键信号是“有序数组 + 找两数之和”

题目约束下要先想到：

- 不要先写暴力 `O(n^2)`
- 目标是把时间复杂度降到 `O(n)`
- 额外空间应控制在 `O(1)`

为什么能用双指针：

- 数组已经有序
- 左指针指向较小值，右指针指向较大值
- 当前和如果偏大，就只能让右指针左移
- 当前和如果偏小，就只能让左指针右移

这题真正要掌握的不是“会写”，而是：

- 为什么看到“有序数组”会想到双指针
- 为什么 `sum > target` 时可以放心移动右指针
- 为什么 `sum < target` 时可以放心移动左指针
- 为什么每次移动都不会漏解

面试里建议按这个顺序回答：

1. 先说输入特征：有序数组，找两个数之和
2. 再说为什么不用暴力：`O(n^2)` 不必要
3. 再说双指针定义：一个从左，一个从右
4. 再说移动规则：
   - `sum > target`：右指针左移
   - `sum < target`：左指针右移
   - `sum == target`：直接返回
5. 最后补复杂度和边界

你需要掌握：

- 双指针为什么能把复杂度降到 `O(n)`
- 内部更推荐用 `0-based` 指针，返回结果时再转成 `1-based`
- 如果题目不保证一定有解，循环条件不要直接写 `while (true)`
- 口头表达时，要把“移动方向不会漏解”的理由说清楚

2. `209. 长度最小的子数组`
   - 训练重点：
     - 滑动窗口何时扩张、何时收缩
     - 为什么“数组元素全为正数”很关键

### 第二阶段：高频主战

1. `167. 两数之和 II - 输入有序数组`
   - 训练重点：
     - 有序数组上的双指针判断
     - 为什么移动方向不会漏解
2. `238. 除自身以外数组的乘积`
   - 训练重点：
     - 前后缀分解
     - 如何从两个辅助数组优化到更低额外空间
3. `11. 盛最多水的容器`
   - 训练重点：
     - 双指针不是只会“夹逼”，而是要能解释为什么移动短板

### 第三阶段：变体 / 验收

1. `42. 接雨水`
   - 训练重点：
     - 一题多解的比较：
       - 双指针
       - 前后缀最大值
       - 单调栈
     - 训练从专题交叉视角思考问题
2. `41. 缺失的第一个正数`
   - 训练重点：
     - 原地交换 / 下标映射
     - 训练边界控制能力

## 9. 专题验收标准

- 能先判断这题更像：
  - 双指针
  - 滑动窗口
  - 前缀和
  - 前后缀分解
  - 原地标记
- 能讲清 `167` 和 `11` 中双指针为什么不会漏解
- 能讲清 `238` 的状态定义，并写出不容易错的代码
- 遇到数组题时，不再只说“这是数组题”，而是能继续说出具体模式
