## 1. 这一块在解决什么问题

RPC / 服务治理这块不是在考你背组件名，而是在考你能不能把下面四件事讲清楚：

1. 为什么服务拆分后，系统需要一套比 HTTP 手写调用更稳定的远程调用机制。
2. 一次 RPC 调用从调用方发起到服务方执行返回，中间到底经过了哪些环节。
3. 注册发现、负载均衡、超时、重试、容错、序列化这些治理能力为什么总会一起出现。
4. Dubbo、Spring Cloud、gRPC 分别更适合解决什么问题，它们的边界和取舍是什么。

把这些点串起来，可以这样理解：

- 服务一旦拆开，调用就从本地方法调用变成了远程调用。
- 远程调用天然会引入网络、节点、协议、超时、失败重试这些新问题。
- 所以系统不能只解决“调通”，还要解决“怎么稳定地调、怎么治理、怎么演进”。
- 不同框架和体系，对调用模型、治理粒度和生态整合的侧重点不同。

所以这块真正要建立的是三种能力：

- 能解释一次 RPC 调用链到底怎么跑。
- 能解释治理能力为什么必须成体系，而不是单点罗列。
- 能解释 Dubbo、Spring Cloud、gRPC 的定位差异和适用边界。

2. 再看一次调用怎么完成：
   - 服务发现。

   - 序列化。
   - 网络传输。

   - 结果返回。
3. 再看为什么治理能力一定会跟上：

## 3. 核心重点展开

### 3.1 为什么系统需要 RPC，而不是服务之间都手工调 HTTP 就够了

这一节在整条主线里解决的是第一个问题：

- RPC 在分布式系统里到底在解决什么。
- 为什么它不是“换个协议名词”这么简单。

#### 先直接回答这个问题

系统需要 RPC，是因为服务拆分后，原本进程内的方法调用变成了跨网络的远程调用。手工调 HTTP 当然也能通信，但当服务数量、接口数量、治理需求上来后，只靠手写请求会把服务发现、负载均衡、超时、重试、监控这些公共能力散落在业务代码里。RPC 的价值，就是把远程调用抽象成统一调用模型，并把这些治理能力收口。

#### 再用场景把问题钉住

比如订单服务要调库存服务、用户服务、营销服务，如果每个地方都手工拼 URL、自己做节点选择、自己处理超时重试，业务代码会非常散。RPC 框架做的不是替你写业务，而是把这些公共通信和治理逻辑框起来。

#### 是什么

RPC 可以先理解成：

- 一套把远程服务调用包装成类似本地方法调用体验的通信和治理机制。

#### 为什么有它

因为远程调用除了“能调通”，还要解决：

- 服务地址怎么找。
- 失败怎么处理。
- 多节点怎么选。
- 调用质量怎么观测。

#### 怎么理解它的机制

第一层，调用方拿到的通常不是远程服务本体，而是一个代理引用。

第二层，代理负责把方法调用转成网络请求。

第三层，服务治理能力围绕这条调用链统一生效。

#### 面试时怎么讲才不空

“RPC 的价值不是把 HTTP 换个名字，而是把远程调用抽象成统一模型，并把服务发现、负载均衡、超时重试、监控这些公共能力收口。否则服务一多，业务代码里会到处散着通信和治理逻辑。”

### 3.2 一次 RPC 调用到底是怎么从调用方走到服务方再返回的

这一节在整条主线里解决的是第二个问题：

- RPC 调用链到底怎么跑。
- 为什么代理、注册中心、负载均衡、序列化会被放在同一条主线上。

#### 先直接回答这个问题

一次 RPC 调用的主流程，本质上是“调用方通过代理发起方法调用 -> 根据注册发现拿到服务节点 -> 经过负载均衡选择目标实例 -> 把请求序列化并通过网络发出去 -> 服务端反序列化、执行方法 -> 把结果再编码返回”。重点不是背组件名，而是知道每一层在远程调用里承担什么职责。

#### 再用场景把问题钉住

比如 `orderService` 要调 `inventoryService.deduct()`：

- 本地代码像是在调接口方法。
- 但底层其实要先定位可用服务节点。
- 再选择某个实例发起请求。
- 服务端收到请求后再还原成实际方法调用执行。

#### 是什么

可以先把主流程压成下面几步：

1. 调用方通过代理发起调用。
2. 通过注册中心或配置拿到服务地址。
3. 通过负载均衡选择目标节点。
4. 请求序列化并通过网络发送。
5. 服务端执行并返回结果。

#### 为什么有它

因为远程调用不能像本地方法一样直接进栈，它必须显式处理网络和节点问题。

#### 怎么理解它的机制

第一层，代理让业务代码仍然可以按接口方式调用。

第二层，服务发现负责告诉调用方现在有哪些可用节点。

第三层，负载均衡负责在多个节点之间做选择。

第四层，序列化和网络传输负责把方法调用跨机器送达。

第五层，服务端再完成反序列化、执行业务、返回结果。

#### 面试时怎么讲才不空

“我回答 RPC 主流程时会按‘代理调用 -> 服务发现 -> 负载均衡 -> 序列化传输 -> 服务端执行 -> 返回结果’这条链讲。这样后面为什么要有超时、重试、熔断这些治理能力，也能自然接上。”

### 3.3 为什么超时、重试、负载均衡、容错这些治理能力不能分开零散地讲

这一节在整条主线里解决的是第三个问题：

- 为什么远程调用题最终一定会落到治理。
- 治理能力之间为什么本来就是互相配合的。

#### 先直接回答这个问题

这些治理能力不能分开零散地讲，是因为它们都在回答同一个问题：远程调用天然不稳定，系统该如何把失败控制在可接受范围内。超时在定义等待边界，重试在定义失败后的补救策略，负载均衡在分散流量压力，容错和熔断在防止故障扩散。它们不是并列清单，而是一组配套手段。

#### 再用场景把问题钉住

如果某个服务实例抖动：

- 没有超时，调用线程可能一直卡住。
- 没有负载均衡，流量可能继续压到坏节点。
- 重试策略不当，又可能把下游打得更重。
- 没有熔断降级，故障可能沿调用链继续扩散。

#### 是什么

- 超时：限定调用最大等待时间。
- 重试：调用失败后的再次尝试策略。
- 负载均衡：在多个节点间分配请求。
- 容错 / 熔断：在故障场景下控制影响面。

#### 为什么有它

因为远程调用比本地调用多了网络不确定性、节点不确定性和下游依赖风险。

#### 怎么理解它们的机制和边界

第一层，超时先定义“多久算失败”。

第二层，重试定义“失败后是否值得再试，以及怎么试”。

第三层，负载均衡决定“流量优先打到谁”。

第四层，容错和熔断负责在持续失败时把问题截断。

第五层，治理重点不是每个点都开，而是组合后是否符合业务幂等性、延迟目标和稳定性要求。

#### 面试时怎么讲才不空

“远程调用题我不会把治理点答成术语列表，而是会先说它们都在解决‘远程调用不稳定’这个总问题。超时定边界，重试做补救，负载均衡分流量，熔断防扩散，这样回答才像一套治理体系。”

### 3.4 Dubbo、Spring Cloud、gRPC 为什么总被放在一起比较，它们到底该怎么选

这一节在整条主线里解决的是第四个问题：

- 三种常见体系的定位差异是什么。
- 为什么它们不是简单的谁替代谁。

#### 先直接回答这个问题

Dubbo、Spring Cloud、gRPC 总被放在一起比较，是因为它们都在解决分布式服务调用问题，但关注重点不同。Dubbo 更偏 Java 内部服务高性能 RPC 和细粒度治理，Spring Cloud 更偏整套微服务生态和治理组件协同，gRPC 更偏基于 protobuf / HTTP2 的高性能、跨语言接口通信。它们不是简单替代关系，而是取决于系统语言栈、治理诉求和生态偏好。

#### 再用场景把问题钉住

比如：

- 如果主要是 Java 内部服务，强调高性能和成熟 RPC 治理，Dubbo 往往更自然。
- 如果团队已经深度使用 Spring 体系，重视配置中心、网关、熔断、链路治理整体协同，Spring Cloud 更容易融进生态。
- 如果存在强跨语言诉求、接口契约明确、追求高性能二进制协议，gRPC 会更有优势。

#### 是什么

- Dubbo：偏 RPC 框架与服务治理。
- Spring Cloud：偏微服务治理生态组合。
- gRPC：偏高性能、跨语言的 RPC 通信框架。

#### 为什么会有这些差异

因为分布式系统的核心诉求不只一种，有的更看重内部调用效率，有的更看重生态整合，有的更看重跨语言契约。

#### 怎么理解它们的机制和边界

第一层，Dubbo 更强调接口调用体验、注册发现和 RPC 治理能力。

第二层，Spring Cloud 更强调围绕 Spring 生态做服务治理整合。

第三层，gRPC 更强调协议、IDL、跨语言和高性能传输。

第四层，选型时不能只看单点性能，还要看团队生态、运维复杂度、治理能力和语言栈。

#### 面试时怎么讲才不空

“我回答 Dubbo、Spring Cloud、gRPC 的差异，不会简单说哪个更好，而是会按‘内部 Java 服务治理、微服务生态整合、跨语言高性能通信’三个方向来分。这样选型逻辑会更稳，也更像工程判断。”

## 4. 边界、误区、常见追问

- 不要把 RPC 讲成“就是远程调接口”，重点要补网络不确定性和治理问题。
- 远程调用主流程一定要能讲成一条链，不要拆成孤立术语。
- 治理能力不是越多越好，要结合幂等性、稳定性和延迟目标判断。
- Dubbo、Spring Cloud、gRPC 不要答成组件列表，重点是定位和选型边界。

## 5. 推荐表达顺序

建议这样讲：

1. 先讲为什么服务拆分后需要 RPC。
2. 再讲一次远程调用的主流程。
3. 再讲为什么治理能力必须成体系。
4. 最后讲 Dubbo、Spring Cloud、gRPC 的定位差异和选型边界。
