# LeisureNote Agent Guide

## 仓库定位

这是一个以个人技术学习、知识沉淀、面试准备和代码训练为目标的仓库。

Agent 在本仓库中工作时，应优先服务以下目标：

- 帮助维护和整理后端技术知识体系
- 帮助补充与知识点配套的代码练习
- 帮助沉淀问题排查经验和面试复习材料
- 在保证可读性和可维护性的前提下，做最小可行改动

## 仓库结构

- `code/algorithm`
  - Java Maven 模块
  - 主要用于算法练习和示例代码
- `code/bigdata`
  - Java Maven 模块
  - 主要用于大数据相关示例代码
- `note/technical_points`
  - 按主题整理的知识点笔记
- `note/problem_solving`
  - 问题排查、开发问题与经验归纳
- `prepare`
  - 面试准备材料、阶段性整理内容

## 工作原则

- 优先最小改动，避免无关重构
- 优先可回滚、可验证的修改
- 不确定时明确说明假设和信息边界，不要编造
- 修改前先理解目标目录和上下文，避免跨主题误改
- 对代码、笔记、目录整理三类任务，分别采用不同策略

## 代码修改规则

- 仅修改与任务直接相关的模块或文件
- 保持现有 Maven 模块边界，除非任务明确要求调整
- 除非明确要求，否则不要升级依赖版本
- 除非确有必要，否则不要修改 public API
- 优先补充最小必要测试，而不是顺手做大规模重构
- 不做大范围格式化，不修改无关文件
- 对示例代码，优先正确性、可读性和学习价值

## 文档与笔记修改规则

- 保留原有中文语义和作者表达意图
- 优先整理结构、归类、命名和索引，不做无必要重写
- 除非明确要求，否则不要删除现有内容
- 对版本、依赖、命令、配置敏感的信息，提示结合当前环境验证
- 如果是面试准备类内容，优先输出结构化、可复习的表达

## 目录整理规则

- 优先保持当前主结构稳定：
  - `note/technical_points`
  - `note/problem_solving`
  - `prepare`
- 新增内容时，优先放到最相关的主题目录
- 移动或重命名文件时，要说明原因和影响范围
- 避免一次性做大规模目录迁移，除非任务明确要求

## 常见任务处理方式

### 1. 补充代码练习

- 先阅读对应主题笔记和模块结构
- 优先补最小可运行示例
- 如果适合，加最小测试或验证说明
- 输出改动说明、影响范围和验证方式

### 2. 整理技术笔记

- 先判断这是知识点笔记、问题排查，还是面试准备材料
- 优先统一命名、目录归类和标题层级
- 如果涉及内容补充，保持简洁、准确、面向开发实践

### 3. 代码审查

重点关注：

- 明显 bug 风险
- 边界条件遗漏
- 空指针、异常处理、资源释放
- 命名是否清晰
- 是否存在重复逻辑
- 是否缺少最小验证或测试

输出 review 时，优先按风险等级给出结论和建议。

### 4. 提交改动

在执行 `git commit` 前，应先：

- 检查是否包含无关改动
- 说明本次改动内容
- 说明验证方式
- 给出简洁清晰的 commit message 建议

如果用户明确要求提交，再执行 commit。

## 验证命令

对 Java 模块，优先使用 Maven 验证：

```bash
cd code/algorithm && mvn test
cd code/bigdata && mvn test
```

如果命令因环境、依赖或外部条件失败，需要明确说明失败原因和未验证项。

## 输出要求

默认使用中文回答，必要时保留英文技术术语。

输出应尽量包含：

- 改了什么
- 为什么改
- 影响范围
- 验证方式
- 风险点或待确认项

## 不建议做的事情

- 不要擅自删除笔记内容
- 不要擅自提交 `.idea/` 等纯本地环境配置
- 不要随意扩大 `target/`、`.class` 等构建产物改动范围
- 不要在没有明确目标的情况下进行大规模重命名或重组
