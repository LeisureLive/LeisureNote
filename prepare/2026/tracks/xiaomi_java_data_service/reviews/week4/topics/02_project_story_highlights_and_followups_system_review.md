## 1. 这一块在回答什么问题

这一块真正要弄懂的，不是“项目介绍怎么背”，而是项目为什么需要被复盘成一条稳定叙事，以及这条叙事应该承接哪些判断：

1. 为什么很多项目明明做过不少事，面试里却只能讲成“我负责了某模块、做了几个优化点”。
2. 为什么面试官听项目，真正想判断的不是你是否参与过，而是你是否理解业务背景、核心矛盾、关键取舍和结果影响。
3. 为什么项目故事不能只讲职责和技术栈，而必须讲清“为什么做、怎么做、为什么这么做、结果怎样、代价是什么”。
4. 为什么亮点和难点不是分开背的两个列表，而是同一条演进主线里的两个侧面。
5. 为什么项目追问之所以容易把人问散，不是因为问题太多，而是因为项目主线本身没有搭稳。

所以这篇文档的主线不是给你一段可背稿，而是帮你把项目重新理解成一条稳定的故事链：

- 先讲项目为什么存在，它在业务里解决什么矛盾。
- 再讲系统为什么会难，核心难点落在哪些链路和边界上。
- 再讲你具体推动了什么变化，为什么这样做，而不是别的做法。
- 再讲这些变化带来了什么收益，同时引入了什么新边界。
- 最后把项目整理成可展开、可追问、可切换深度的稳定叙事。

## 2. 主线

1. 先看项目背景：业务在缺什么能力，为什么现有方案不够。
2. 再看核心矛盾：性能、扩展性、一致性、治理、协作里真正最痛的是什么。
3. 再看你的动作：你推动了哪些关键设计、优化或重构。
4. 再看结果：性能、稳定性、扩展性、接入能力、维护成本分别发生了什么变化。
5. 再看代价和边界：这次改动引入了什么新复杂度，哪些问题还没被彻底解决。
6. 最后看展开方式：如何切成 1 分钟版、3 分钟版和追问展开版。

## 3. 核心问题展开

### 3.1 为什么很多项目介绍会失败，不是因为做得少，而是因为主线搭错了

这一节在回答什么问题：

- 项目介绍为什么常常讲不出价值。
- 什么样的讲法会让项目显得像“参与过”，而不是“真正理解过”。

这个问题是怎么解决的：

项目介绍最常见的失败，不在于项目本身不够好，而在于叙事顺序一上来就错了。很多人一开口就是：

- 项目名称
- 技术栈
- 我负责某个模块
- 做了几个优化

这种讲法的问题是：它没有先告诉面试官“这个项目为什么值得存在”，也没有告诉面试官“你解决的核心矛盾是什么”。于是后面的技术点再多，也很容易变成离散事实。

更稳的项目主线，至少要先回答下面四个判断：

1. 这个项目在业务里解决什么问题
2. 为什么这件事当时难，不是普通 CRUD
3. 你负责的那部分，卡住的核心矛盾是什么
4. 你做的改动为什么真的改变了系统边界或工程状态

也就是说，项目故事不是“工作流水账”，而是一个因果链：

- 业务需要什么能力
- 原有系统为什么扛不住
- 你为什么要这样设计
- 这样设计换来了什么
- 又带来了什么新边界

只有先把这条主线搭好，后面的亮点、难点、追问才会自然落位。

### 3.2 面试官真正想从项目里判断什么，为什么职责描述远远不够

这一节在回答什么问题：

- 面试官听项目时到底在判断什么。
- 为什么只讲“我做了什么”远远不够。

这个问题是怎么解决的：

资深岗面试官听项目，通常不是在收集经历，而是在判断你有没有下面几类能力：

1. 业务理解力
   - 你是否知道这个系统为什么存在，而不是只知道自己写了哪些代码。
2. 技术判断力
   - 你是否知道为什么选这个方案，而不是其他方案。
3. 复杂问题拆解能力
   - 你是否能把瓶颈、故障、数据规模、依赖边界讲清楚。
4. 结果意识
   - 你是否知道这次改动对吞吐、稳定性、接入成本、运维成本的影响。
5. 反思能力
   - 你是否知道这次改动没有解决什么、又引入了什么新边界。

这就是为什么项目介绍不能停在：

- 我负责某模块
- 我优化了性能
- 我支持了某能力

因为这些信息最多说明你参与过，不能说明你是否具备架构判断和工程取舍能力。更稳的讲法应该是：

- 我负责哪一段链路
- 这段链路当时最痛的矛盾是什么
- 我做了什么设计或调整来解决这个矛盾
- 为什么这套调整比别的路径更适合当时约束
- 最终结果是什么，又留下了什么后续问题

所以职责描述只是起点，不是项目故事的主体。

### 3.3 一个稳定的项目故事到底应该按什么结构组织

这一节在回答什么问题：

- 项目应该按什么顺序讲，才能让后面的亮点和追问都有位置。
- 为什么项目表达要像一条链，而不是几个模块介绍。

这个问题是怎么解决的：

一个稳定的项目故事，建议至少按下面 6 层组织：

1. `业务背景`
   - 这个项目服务谁，解决什么问题。
2. `系统目标`
   - 这个系统最重要的能力目标是什么，是实时性、扩展性、可靠性，还是接入效率。
3. `核心矛盾`
   - 当时最难的地方是什么，是吞吐、规则复杂度、元数据依赖、慢下游、历史兼容，还是查询成本。
4. `我的动作`
   - 你具体做了哪些关键设计、优化、抽象或重构。
5. `结果变化`
   - 指标、能力边界、运维效率、接入能力发生了什么变化。
6. `新边界`
   - 这次改动没有彻底解决什么，又引入了什么治理复杂度。

为什么这个结构稳定？因为它天然支持后续追问：

- 面试官问“业务背景”，你有第一层。
- 面试官问“为什么难”，你有第三层。
- 面试官问“你到底做了什么”，你有第四层。
- 面试官问“有什么结果”，你有第五层。
- 面试官问“有没有代价”，你有第六层。

所以项目表达不应该先讲技术栈，而应该先让面试官知道：

- 这个项目不是普通功能
- 你理解它的业务和技术矛盾
- 你能对自己的改动负责

### 3.4 亮点和难点为什么不能分开背，而必须沿着同一条主线展开

这一节在回答什么问题：

- 项目亮点和项目难点为什么常常讲散。
- 为什么它们应该是同一条故事线里的两个侧面。

这个问题是怎么解决的：

很多人会把“亮点”和“难点”分别列成两个清单：

- 亮点：性能提升、支持扩展、架构优化
- 难点：元数据一致性、慢下游、历史兼容

这种拆法虽然方便记，但面试里很容易出问题，因为面试官通常不是按“亮点题”和“难点题”分开问，而是顺着同一条演进链往下追：

- 你为什么要做这个优化
- 优化前卡在哪里
- 优化后解决了什么
- 又带来了什么新问题

也就是说，亮点和难点本来就是一条链：

- 难点定义了为什么要改
- 亮点定义了改完带来的结果
- 新边界定义了这次改动的代价

所以更稳的讲法是：

1. 先讲一个明确难点
   - 比如元数据依赖导致链路吞吐和一致性之间有冲突。
2. 再讲你怎么处理
   - 比如引入缓存、分层投递、隔离目标端。
3. 再讲结果
   - 吞吐提升、接入能力增强、横向扩展成立。
4. 再讲代价
   - 带来短暂不一致窗口，需要版本校验和补偿机制。

这样亮点和难点才会互相支撑，而不是像两个孤立列表。

### 3.5 项目追问为什么会把人问散，真正散掉的是哪一层

这一节在回答什么问题：

- 面试里的项目追问为什么特别容易把人问散。
- 追问真正暴露的薄弱点是什么。

这个问题是怎么解决的：

项目追问之所以容易把人问散，通常不是因为问题太难，而是因为项目主线只搭到了“职责层”，没有搭到“因果层”。一旦面试官继续追问：

- 为什么这么设计
- 为什么不用另一种方案
- 代价是什么
- 如果量级再涨 10 倍怎么办

你就会发现原先的“我做了什么”已经不够用了。

项目追问真正常见的 5 类方向是：

1. `为什么`
   - 为什么这么设计，不这么设计会怎样。
2. `取舍`
   - 你在性能、一致性、扩展性、复杂度之间做了什么交换。
3. `边界`
   - 这个方案什么时候不适用。
4. `故障`
   - 出问题时怎么止损、怎么恢复。
5. `演化`
   - 如果量级、场景或租户继续增长，下一步怎么演进。

也就是说，项目追问其实是在逼你从“做事的人”切换成“理解系统的人”。所以项目主线如果没有先搭到：

- 核心矛盾
- 方案取舍
- 新边界

那么一被追问就会散。

### 3.6 结合你当前两个核心项目，哪些素材最值得优先打磨

这一节在回答什么问题：

- 你当前简历素材里，哪些点最值得优先作为项目主线。
- 哪些点应该先补信息，否则很难支撑深挖。

这个问题是怎么解决的：

基于 `prepare/2026/profile/00_interview_master.md` 当前已知素材，你最值得优先打磨的两个项目分别是：

#### 项目 1：实时数据订阅 + 实时查询

当前已知最强素材有三类：

1. 场景语境强
   - 天然贴近数据服务 / 数据平台岗位。
2. 指标结果明确
   - 单机无带宽限制下发送性能可达 `6w/s`，并支持横向扩展。
3. 难点真实
   - 数据处理依赖元数据，但元数据更新缺少通知机制，只能依赖缓存刷新，存在短暂不一致窗口。

这意味着它最适合被打磨成下面这条主线：

- 为什么内部业务方需要统一的实时消费与查询能力
- 为什么订阅系统和查询系统会一起面对吞吐、规则、元数据和多目标治理问题
- 你具体在哪条链路上推动了性能、扩展性和平台能力的升级
- 为什么元数据依赖会成为真实难点

但这条项目线当前还缺几块信息，必须后补，否则深挖容易断：

- 数据来源和事件规模
- 查询系统的数据存储模型
- 失败重试、监控告警、目标端隔离策略
- 更完整的架构链路

#### 项目 2：id-mapping

当前已知最强素材有两类：

1. 结果变化明显
   - 单机处理性能从 `3k-4k/s` 提升到 `10k+/s`，并支持横向扩展。
2. 取舍感强
   - 引入高性能模式，抛弃部分低频功能，摆脱对单点模块 `skv-proxy` 的依赖。

这意味着它最适合被打磨成：

- 历史系统技术债务如何影响性能和迭代
- 为什么需要引入新的模式，而不是在旧逻辑上继续堆补丁
- 你是如何在功能完整性和处理性能之间做取舍

但它当前缺的信息更多，因此更适合先作为第二项目打磨：

- 你具体负责的核心模块
- 更准确的技术栈
- 高性能模式的关键设计
- 哪些能力被保留，哪些被牺牲

所以从面试性价比看：

1. 先把 `实时数据订阅 + 实时查询` 打造成主项目
2. 再把 `id-mapping` 打造成“历史系统治理 + 性能取舍”型副项目

### 3.7 一个项目应该怎样切成 1 分钟版、3 分钟版和追问展开版

这一节在回答什么问题：

- 为什么同一个项目必须能切换不同深度。
- 不同版本分别要保住什么信息。

这个问题是怎么解决的：

项目表达不能只有一种长度，因为不同轮次、不同面试官给你的时间完全不同。稳定的做法是至少准备三种层级：

#### 1 分钟版：保住项目价值和你的角色

目标是快速让面试官知道：

- 这是什么项目
- 解决什么问题
- 你负责哪一段
- 最有代表性的结果是什么

这一版不要讲太多细节，重点是建立项目价值感。

#### 3 分钟版：保住主线和关键取舍

目标是让面试官听明白：

- 业务背景
- 核心矛盾
- 你的关键动作
- 结果变化
- 一个真实难点

这一版是最常用版本，应该是你最稳定的主版本。

#### 追问展开版：保住链路、边界和代价

目标是当面试官继续问时，你能顺着同一条主线往下展开：

- 为什么这么设计
- 为什么不用另一种方案
- 性能瓶颈在哪里
- 一致性问题怎么接受和修正
- 后面继续演进会怎么做

这时最重要的不是“多讲”，而是仍然不丢主线。

## 4. 边界、误区、常见混淆

- 项目故事不是工作流水账，不能只讲职责、模块和技术栈。
- 不要把亮点和难点拆成两个孤立清单，它们本来就是同一条演进链。
- 平台化不是“支持更多场景”这么简单，关键看能力是不是可复用、可配置、可治理。
- 性能数字只是结果，不是项目价值本身；必须继续解释为什么会有这些结果。
- 讲项目时不要只讲“做了什么”，还要讲“为什么这么做、代价是什么、还有什么没完全解决”。
- 追问把人问散，通常不是问题太难，而是项目主线没有搭到取舍和边界这一层。
- 如果某个项目当前素材不足以支撑深挖，就要明确补信息，而不是用想象把空白补满。

## 5. 自测清单

- 你能不能解释为什么很多项目介绍会失败，不是因为做得少，而是因为主线搭错了。
- 你能不能用“业务背景 -> 核心矛盾 -> 我的动作 -> 结果变化 -> 新边界”这条线讲一个项目。
- 你能不能把亮点和难点放回同一条演进链，而不是分开背。
- 你能不能说明项目追问为什么会把人问散，真正散掉的是哪一层。
- 你能不能指出你当前两个核心项目里，哪个更适合作为主项目，为什么。
- 你能不能把一个项目切成 1 分钟版、3 分钟版和追问展开版。
- 你能不能明确说出你当前项目素材里还缺哪些关键信息，需要后续补齐。
