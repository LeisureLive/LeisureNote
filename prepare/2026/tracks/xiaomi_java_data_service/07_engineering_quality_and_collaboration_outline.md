# 工程质量与协作表达提纲

这份文档用于补“工程质量、测试意识、发布回滚、跨团队协作”这条线。

这部分很少像八股题那样单独问，但在资深岗项目面、系统设计面、行为面里经常被间接判断。

## 1. 工程质量要回答什么

最少要能稳定回答这几类问题：

1. 你怎么保证改动可维护
2. 你怎么控制上线风险
3. 你怎么看测试和回归
4. 复杂需求怎么拆，怎么和其他角色协作
5. 出线上问题后怎么复盘和改进

## 2. 常用表达骨架

### 2.1 代码质量

可直接答：

- 我比较重视最小可行改动，尽量避免无关重构，因为线上系统里越大范围改动，验证和回滚成本越高。
- 对复杂逻辑，我会优先把关键边界、失败路径和约束讲清，而不是只追求把功能做出来。
- 如果是历史系统，我会优先做可回滚、可灰度的调整，而不是一次性推翻。

### 2.2 测试意识

可直接答：

- 我不会把测试理解成“补几条 case 就结束”，更重要的是先识别主风险点：
  - 主流程是否正确
  - 边界条件是否覆盖
  - 失败路径是否可控
  - 回归影响面有多大
- 对核心链路改动，我通常会优先确认最小验证集合，必要时补自动化测试，同时配合灰度和监控验证真实效果。

### 2.3 上线与回滚

可直接答：

- 对高风险改动，我更关注可灰度、可回滚、可观测，而不是一次性全量上线。
- 如果变更涉及核心链路，我会尽量做到：
  - 开关控制
  - 灰度发布
  - 监控指标提前补齐
  - 异常时能快速回退到旧路径

### 2.4 协作表达

可直接答：

- 跨团队协作时，我习惯先把目标、约束、接口边界和风险讲清楚，避免后面反复对齐。
- 如果依赖产品、数据、前端、测试或其他服务团队，我会先确认：
  - 谁负责什么
  - 哪些节点会影响排期
  - 哪些风险需要提前暴露

## 3. 资深岗更容易被追问的点

### 3.1 你怎么保证不是“只会写功能”

建议答法：

- 我会主动关注三个层面：
  - 当前需求能不能稳定上线
  - 改动是否影响后续维护成本
  - 系统是否因为这次变更留下新的治理风险

### 3.2 你怎么做技术支持和质量保障

建议答法：

- 如果是重点项目，我会把质量保障拆成几步：
  - 先识别核心风险
  - 再确定验证路径
  - 再补监控、告警和回滚方案
  - 上线后再观察关键指标，必要时做补偿或修正

### 3.3 你怎么处理协作中的分歧

建议答法：

- 我一般不会先争结论，而是先把目标、约束和代价摊开。
- 如果方案有分歧，我更倾向于把问题变成可比较的维度：
  - 开发成本
  - 风险
  - 上线时间
  - 可维护性
  - 后续扩展性

## 4. 当前最值得优先准备的表达题

1. 你如何保证线上改动的质量
2. 你如何设计测试和回归范围
3. 你如何控制高风险需求的上线风险
4. 你如何推动跨团队项目落地
5. 你如何做事后复盘和持续改进

## 5. 使用建议

- 不要把这份文档当行为面稿子硬背
- 更适合在项目面和系统设计面里穿插使用
- 每次模拟面试后，把答得空泛的点继续回填到这里
