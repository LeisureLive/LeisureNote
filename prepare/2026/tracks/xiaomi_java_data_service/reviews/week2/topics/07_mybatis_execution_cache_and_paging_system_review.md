## 1. 这一块在解决什么问题

MyBatis 这块不是在考你背多少类名，而是在考你能不能把下面五件事讲清楚：

1. 为什么数据访问层通常不直接在业务代码里手写 JDBC，而 MyBatis 恰好卡在一个“SQL 可控、样板收口”的中间位置。
2. Mapper 接口为什么能像普通 Java 接口一样被调用，但最后却能执行到数据库。
3. 一条 SQL 从 Mapper 方法调用，到参数绑定、JDBC 执行、结果映射，中间到底经历了哪些关键环节。
4. 动态 SQL、一级缓存、二级缓存为什么都不是独立功能，而是挂在同一条执行链上的不同阶段。
5. 分页和性能问题为什么很多时候不能简单归因于 MyBatis 本身，而要继续回到 SQL、数据库和接入方式本身判断。

把这些点串起来，主线其实很稳定：

- 业务系统需要数据访问层，但不希望每个方法都重复写连接获取、参数绑定、结果集映射和资源释放。
- MyBatis 选择的路线不是彻底隐藏 SQL，而是让开发者继续显式写 SQL，同时把 JDBC 样板和执行链统一托管起来。
- 一旦 Mapper 方法调用被代理接住，后面就可以统一挂接 SQL 定位、参数处理、执行器、缓存、结果映射等能力。
- 动态 SQL 决定“最终发出去的 SQL 长什么样”，缓存决定“这次到底要不要真的查数据库”，分页则会把执行链里的性能问题放大出来。
- 所以 MyBatis 面试题真正想确认的，不是你会不会背接口，而是你能不能把“方法调用 -> SQL 执行 -> 结果返回 -> 缓存/分页边界”讲成一条完整主线。

所以这块真正要建立的是三种能力：

- 能把 MyBatis 执行一条 SQL 的主流程讲顺，而不是只说“Mapper 底层是代理”。
- 能把动态 SQL、缓存、分页放回执行链一起理解，而不是拆成三套孤立知识点。
- 能把 MyBatis 的价值和边界讲清，不把数据库层问题简单甩给框架。

## 3. 核心重点展开

### 3.1 为什么需要 MyBatis，而不是在业务代码里直接手写 JDBC

这一节在回答什么问题：

- MyBatis 的工程价值到底是什么。
- 为什么它不是“多包一层没必要的框架”，又为什么它和重量级 ORM 不完全一样。

这个问题是怎么解决的：

需要 MyBatis，不是因为 JDBC 不能用，而是因为手工 JDBC 在工程上太容易把业务代码拖进样板泥潭。你真正重复的并不是 SQL 本身，而是下面这些动作：

- 获取连接
- 创建 `PreparedStatement`
- 绑定参数
- 处理 `ResultSet`
- 手工把列映射成对象字段
- 处理异常和关闭资源

如果这些动作散落在每个 DAO / Repository 方法里，代码很快就会出现两个问题：

1. SQL 逻辑和 JDBC 样板混在一起，真正业务意图不清晰。
2. 参数绑定、结果映射、资源释放这些技术细节很容易重复、出错、风格不统一。

MyBatis 的路线很明确：

- 不替你隐藏 SQL
- 但把 JDBC 执行样板和对象映射过程统一托管起来

这也是它和 JPA / Hibernate 这类 ORM 经常被一起问时最大的区别之一：

- MyBatis 更强调 SQL 显式可控
- ORM 更强调对象模型驱动和持久化上下文能力

所以 MyBatis 更适合的场景通常是：

- SQL 复杂、查询形态多变
- 你希望精确控制 SQL
- 更关心执行链和数据库行为，而不是让框架尽量替你抽象 SQL

但它的边界也很明确：

- 它帮你组织 SQL 执行流程
- 不会替你自动优化糟糕的 SQL
- 也不会像 ORM 那样提供强持久化上下文、脏检查、对象状态管理那整套语义

如果把它压成一句话，就是：

- MyBatis 解决的是“SQL 继续由你控制，但 JDBC 样板和映射链由框架接管”这个问题。

这也是为什么回答 MyBatis 时，最稳的起手式不是背类名，而是先把它放回“手工 JDBC 和重量级 ORM 之间的工程取舍”去讲。

### 3.2 MyBatis 执行一条 SQL 的主流程到底是怎么串起来的，Mapper 接口为什么能执行到数据库

这一节在回答什么问题：

- Mapper 方法为什么不是一个普通空接口调用。
- 从 `userMapper.selectById(id)` 到真正发 SQL，中间到底发生了什么。

这个问题是怎么解决的：

MyBatis 执行一条 SQL 的主流程，本质上是：

- Mapper 接口方法调用
- 被代理对象接住
- 定位到对应的 `MappedStatement`
- 通过 `SqlSession` 和 `Executor` 组织执行
- 再把结果映射回 Java 对象

先把最容易被问的第一层讲清：Mapper 为什么能执行 SQL。

MyBatis 官方 API 文档明确说明：

- Mapper 接口底层是通过 `MapperProxy` 这个 `InvocationHandler` 代理起来的

也就是说，当你调用：

```java
userMapper.selectById(id)
```

真正被执行的不是接口本身，而是代理对象的 `invoke(...)`。它会根据：

- 当前接口类型
- 当前方法
- 当前参数

去决定应该调用哪个 MyBatis 语句。

再往下走，执行链可以按下面顺序理解：

1. 先通过 Mapper 方法定位 `MappedStatement`
   - 你可以把 `MappedStatement` 理解成“这条语句的完整元数据定义”
   - 里面包含 SQL 来源、参数映射、结果映射、缓存配置、语句类型等信息

2. 再进入 `SqlSession`
   - MyBatis 官方文档明确指出：`SqlSession` 是 MyBatis 工作的核心 Java 接口
   - 它负责执行 mapped statements、获取 mapper、管理事务等

3. `SqlSession` 再把执行请求交给 `Executor`
   - `Executor` 可以理解成真正组织“查不查缓存、要不要查库、怎么调 StatementHandler”的执行核心
   - 常见执行器类型有简单执行、复用执行、批量执行等

4. `Executor` 往下再交给 `StatementHandler`
   - 负责准备 JDBC `Statement`
   - 包括创建、参数化、执行 SQL 等动作

5. 参数绑定由 `ParameterHandler` 处理
   - 把你的 Java 参数对象，绑定成 JDBC 预编译参数

6. 查询结果由 `ResultSetHandler` 处理
   - 把 JDBC `ResultSet` 转成目标对象、集合、`resultMap` 对应结构

所以如果把整条链压缩成一句话，就是：

- Mapper 负责提供面向接口的调用入口，`SqlSession` 负责承接执行，`Executor` 负责组织执行策略，`StatementHandler` / `ParameterHandler` / `ResultSetHandler` 负责把一次数据库访问真正落到 JDBC 层并把结果映射回来。

再补一个你在 Spring 项目里非常容易遇到的现实差异。

在裸 MyBatis 里：

- 你通常是 `SqlSessionFactory -> SqlSession -> getMapper()`

在 MyBatis-Spring 里，官方文档明确说明：

- 真正被注入的往往不是原生 `DefaultSqlSession`
- 而是线程安全、能参与 Spring 事务的 `SqlSessionTemplate`

这意味着在 Spring 项目里，Mapper 调用链虽然概念上还是：

- Mapper -> SqlSession -> Executor

但实际承接 `SqlSession` 语义的通常是：

- `SqlSessionTemplate`

它负责：

- 选择当前事务关联的 `SqlSession`
- 管理 session 生命周期
- 提交 / 回滚 / 关闭
- 把 MyBatis 异常转成 Spring `DataAccessException`

这也是为什么面试里如果明确在 Spring 项目语境下问 MyBatis 执行流程，你最好补一句：

- 在 MyBatis-Spring 里，很多时候底层真实参与 Spring 事务管理的是 `SqlSessionTemplate`

这一节真正要讲清的，不是死记类图，而是方法调用怎么一路被翻译成数据库动作。只要主线讲成：

- 方法调用 -> 代理 -> `MappedStatement` -> `SqlSession` -> `Executor` -> JDBC 执行 -> 结果映射

后面缓存、动态 SQL、插件、分页这些点都更容易挂回去。

### 3.3 动态 SQL、一级缓存、二级缓存为什么总和执行主流程一起问，它们到底分别作用在链路的哪里

这一节在回答什么问题：

- 为什么动态 SQL 和缓存不能脱离执行链单独理解。
- 缓存为什么既能提速，又会带来一致性边界问题。

这个问题是怎么解决的：

动态 SQL 和缓存总和主流程一起问，是因为它们影响的是同一次 SQL 执行链的不同阶段：

- 动态 SQL 决定“最终真正执行的 SQL 长什么样”
- 缓存决定“这次到底要不要真的打数据库”

先看动态 SQL。

MyBatis 官方文档明确说明，它提供了一套 XML 动态 SQL 语言，核心标签包括：

- `if`
- `choose / when / otherwise`
- `trim / where / set`
- `foreach`

这套机制在解决什么问题？

- 不是为了炫技拼字符串
- 而是为了在“查询条件可选、更新字段可选、批量参数可变”这类场景下，让 SQL 仍然保持结构化、可维护

它本质上作用在执行链前半段：

- 先根据参数和动态标签生成最终 SQL
- 再把这个 SQL 变成 `BoundSql`
- 然后后面的参数绑定、缓存判断、JDBC 执行都基于这个最终 SQL 往下走

所以动态 SQL 真正回答的是：

- 同一个 Mapper 语句，最终发出去的 SQL 到底是什么

再看缓存。

MyBatis 官方文档说明：

- 默认只有本地 session 级缓存，也就是一级缓存
- 二级缓存需要在 mapper XML 里显式配置 `<cache/>`

一级缓存是什么？

- 作用域通常是同一个 `SqlSession`
- 本质上是“这一次会话里，相同查询不要重复打数据库”

它为什么存在？

- 因为同一个会话上下文中，重复查询完全相同的数据是一种常见浪费
- 在这个范围内做本地缓存，一致性相对容易控制

但它的边界也要讲清：

- 它不是跨请求、跨会话共享缓存
- 默认作用范围小，和会话强绑定
- 官方 Java API 文档还提到 `LocalCacheScope`，也就是本地缓存范围是可配置的

二级缓存是什么？

- 通常是 namespace 级别
- 作用范围比一级缓存大
- 需要显式开启

MyBatis 官方 Mapper XML 文档说明，配置 `<cache/>` 之后，典型行为包括：

- 当前 mapper namespace 下 select 结果会被缓存
- insert / update / delete 会刷新缓存
- 默认有一定缓存淘汰和读写语义配置

所以二级缓存的真正价值是：

- 提升更大范围内的查询复用能力

但它的真正风险也在这里：

- 共享范围更大
- 一致性边界更复杂
- 失效策略更难判断
- 一旦业务读写频繁、跨 namespace、跨服务边界复杂，就很容易出现“缓存命中了，但语义不稳”的问题

这也是为什么很多项目对一级缓存比较自然，对二级缓存却会比较谨慎。不是因为二级缓存不能用，而是因为：

- 缓存收益扩大了
- 一致性和失效管理难度也同步扩大了

如果把动态 SQL 和缓存统一放回执行链，可以这样记：

1. 动态 SQL 阶段
   - 先决定本次最终 SQL 和参数形态
2. 缓存判断阶段
   - 再决定这次是不是可以直接复用已有结果
3. 真正查库阶段
   - 缓存没命中才继续走 JDBC 执行
4. 结果映射与缓存写回阶段
   - 把查到的结果映射回来，并视情况进入缓存

这一节真正要讲清的是：

- 动态 SQL 不是独立模板语法题
- 缓存也不是独立“提速题”
- 它们都属于同一条 SQL 执行链里的关键环节

只要这条关系讲清，后面再问你“为什么某条 SQL 没命中缓存”“为什么动态 SQL 会影响缓存键”“为什么二级缓存容易有一致性边界”，主线都比较稳。

### 3.4 分页和性能问题为什么不能简单归因于 MyBatis，本质上应该怎么分层判断

这一节在回答什么问题：

- 为什么分页题最后总会追到 SQL 和数据库本身。
- MyBatis 在分页和性能问题里到底该背多大的锅。

这个问题是怎么解决的：

分页和性能问题不能简单归因于 MyBatis，是因为 MyBatis 主要负责的是：

- 组织 SQL 执行链
- 处理参数绑定和结果映射
- 在合适的位置挂缓存、插件、动态 SQL 等能力

但真正决定查询代价的，很多时候仍然是：

- SQL 本身怎么写
- 索引怎么建
- 执行计划怎么样
- 是否存在深分页
- 是否有大结果集回传

先说分页。

MyBatis 里经常会提到两种分页思路：

1. `RowBounds`
2. 物理分页插件 / SQL 改写

这里最容易被误解的是 `RowBounds`。

MyBatis 官方 Java API 文档明确给出了 `RowBounds`，它本质上是一个：

- `offset`
- `limit`

包装对象。但官方文档也明确提醒：不同驱动对它的效率支持不同。也就是说：

- `RowBounds` 不等于数据库原生 `limit offset` 已经高效执行好了
- 它更多是一种框架层分页抽象

所以如果你把分页答成“RowBounds 就是高性能分页”，这就容易被追问穿。

在真实项目里，更常见的是：

- 通过插件改写 SQL，走数据库物理分页

这时 MyBatis 的角色是：

- 提供插件挂点和执行链扩展能力

但深分页问题本身并不会因此消失。

比如：

```sql
select * from orders order by id limit 100000, 20
```

即使框架层分页接得再优雅，数据库还是可能要先跳过大量记录，代价仍然很高。这个问题本质不在 MyBatis，而在：

- SQL 分页策略本身
- 索引与排序条件
- 数据量级

再看性能问题为什么不能一股脑怪框架。

排查 MyBatis 查询慢时，至少要先分三层：

1. SQL / 数据库层
   - SQL 本身是否慢
   - 索引是否合理
   - 是否存在回表、排序、临时表、深分页
   - 执行计划是否异常

2. 映射与结果处理层
   - `resultMap` 是否过重
   - 嵌套映射 / 集合映射是否导致多次查询或结果集膨胀
   - 返回对象是否过大

3. 框架接入层
   - 插件是否过多
   - 缓存是否误用
   - 会话和事务接入是否合理
   - 是否存在批量场景却没用 `ExecutorType.BATCH`

如果你只会说“这是 MyBatis 的分页问题”或者“这是框架慢”，通常是不够的。更稳的回答应该是：

- MyBatis 可以放大一个差的 SQL，也可以把分页接入组织得更规范
- 但它不决定底层查询本身是否高效

这一节真正要讲清的是：

- 框架执行链问题和数据库执行代价问题必须分层判断
- MyBatis 负责组织调用，不负责替你自动优化数据库

只要这层分工讲清，后面面试官继续追问插件分页、深分页、批量写入、缓存误用时，你都更容易稳住主线。

## 4. 边界、误区、常见追问

- 不要把 MyBatis 讲成全自动 ORM，它更强调 SQL 显式可控和 JDBC 样板收口。
- Mapper 接口本身不会执行 SQL，真正接住调用的是代理对象；在 Spring 项目里，还要记得补一句 `SqlSessionTemplate` 经常承担实际 session 管理责任。
- 不要把执行流程只答成“Mapper -> SqlSession -> Executor”，继续讲清 `MappedStatement`、参数处理、结果映射分别在干什么。
- 动态 SQL 不是为了方便拼字符串，而是为了让条件化 SQL 在执行链前半段仍然保持结构化。
- 一级缓存和二级缓存不能只讲“能提速”，必须继续讲作用范围、失效时机和一致性边界。
- 二级缓存要谨慎，不是因为它一定不好，而是因为共享范围一大，一致性和排障复杂度会明显上升。
- 不要把分页性能问题都甩给 MyBatis，先分清是框架执行链问题，还是 SQL / 索引 / 执行计划问题。
- `RowBounds` 可以讲，但不要把它讲成“物理分页等于高性能分页”的保证。
