## 1. 这一块在解决什么问题

Spring 事务这块不是在考你背 `REQUIRED`、`REQUIRES_NEW` 有几个，而是在考你能不能把下面五件事讲清楚：

1. 为什么业务代码需要 Spring 来统一托管事务边界。
2. 声明式事务为什么能成立，它和 AOP / 代理是什么关系。
3. 隔离级别和底层数据库事务能力分别在解决什么问题，为什么它们不能被 Spring 注解单独替代。
4. 事务传播行为到底在解决什么问题，什么时候该拆新事务，什么时候应该共用事务。
5. `@Transactional` 为什么经常失效，失效的根因到底是代理边界、异常处理还是方法定义方式。

把这些点串起来，可以这样理解：

- 业务代码要保证多步数据库操作的一致性。
- 如果每个方法都手工开启、提交、回滚事务，维护成本高，也容易出错。
- Spring 希望通过声明式方式把事务边界托管起来。
- 这类托管很多时候依赖代理增强，所以事务题天然会和 AOP 连在一起。
- 但事务不只是“同成同败”，还要看并发下的数据可见性和隔离成本，这就会引出隔离级别和底层数据库边界。
- 一旦调用路径没经过代理，或者异常、方法边界不符合规则，事务就可能失效。

所以这块真正要建立的是三种能力：

- 能解释 Spring 事务在解决什么工程问题。
- 能解释声明式事务为什么能生效，以及它对代理边界的依赖。
- 能解释隔离级别、传播行为和失效场景，而不是只背注解参数。

## 3. 核心重点展开

### 3.1 Spring 事务在解决什么问题，为什么很多系统更偏向声明式事务

这一节在整条主线里解决的是第一个问题：

- 为什么业务系统需要 Spring 来统一管理事务。
- 为什么声明式事务通常比手工事务更常见。

#### 先直接回答这个问题

Spring 事务在解决的是“多步数据库操作如何稳定地纳入同一个一致性边界”这个问题。声明式事务更常见，是因为它把开启、提交、回滚这些样板逻辑从业务代码里抽离出来，让事务规则集中配置，业务代码更聚焦。

它的核心价值不是少写几行代码，而是让事务边界和回滚规则更统一、更可维护。

#### 再用一个最小场景把问题钉住

例如一个下单操作需要：

- 扣减库存。
- 写订单表。
- 写流水表。

如果这三步必须同成同败，你就需要事务边界。如果每个方法都手工写 `begin / commit / rollback`，业务代码很快就会被样板逻辑污染。Spring 的声明式事务就是把这层公共事务控制抽出去。

#### 是什么

Spring 事务可以先理解成：

- 你声明某个方法需要事务。
- Spring 在方法调用前后帮你接管事务生命周期。

#### 为什么有它

因为业务里事务是高频共性需求，而手工控制事务：

- 重复。
- 容易漏回滚。
- 很难统一规范。

#### 怎么理解它的机制

第一层，业务方法本身不直接处理事务模板代码。

第二层，Spring 通过代理拦截调用，在进入方法前决定是否开启事务。

第三层，方法成功时提交，出现符合规则的异常时回滚。

#### 面试时怎么讲才不空

“Spring 事务的价值不是少写几行代码，而是把事务边界统一托管起来。业务里很多操作需要同成同败，如果每个方法都手工控制事务，代码会重复且容易出错。声明式事务本质上是让 Spring 在方法调用前后接管事务生命周期，让事务规则集中管理。”

### 3.2 声明式事务为什么能生效，它和 AOP / 代理是什么关系

这一节在整条主线里解决的是第二个问题：

- Spring 为什么能在你不手工写事务模板代码的情况下管理事务。
- 为什么事务题经常会直接追到 AOP 和代理。

#### 先直接回答这个问题

声明式事务之所以能生效，很多时候是因为 Spring 用代理接管了方法调用。调用进入代理后，事务拦截器先判断这个方法是否需要事务，再决定开启事务、执行目标方法，并在调用结束后提交或回滚。所以事务本质上不是注解本身生效，而是“带事务语义的调用路径”被代理拦住了。

#### 再用场景把问题钉住

一个被 `@Transactional` 标注的 Service 方法，外部调用时真正拿到的往往是代理对象：

- 先进入代理。
- 代理发现这个方法需要事务。
- 开启事务。
- 调目标方法。
- 根据执行结果提交或回滚。

如果调用没有经过代理，这条链就不成立，事务也就可能失效。

#### 是什么

声明式事务可以看成一种典型的 AOP 增强：

- 横切关注点是事务控制。
- 代理对象负责把事务逻辑织入方法调用链。

#### 为什么有它

因为事务本身就是一种典型的横切能力，适合通过统一增强方式接入。

#### 怎么理解它的机制

第一层，`@Transactional` 只是声明“这里需要事务语义”。

第二层，真正让它生效的是事务拦截器和代理调用链。

第三层，只要调用链没走代理，声明就不会自动变成事务控制。

#### 面试时怎么讲才不空

“`@Transactional` 不是加了注解就自动有事务，本质上还是 Spring 通过代理拦截方法调用，再在调用前后加上事务管理逻辑。所以事务题和 AOP 常常是同一条主线：关键不是注解本身，而是调用有没有走到代理。”

### 3.3 隔离级别和数据库事务边界在这里解决什么问题，为什么不能脱离底层数据库单独理解

这一节在整条主线里解决的是第三个问题：

- Spring 事务配置为什么不能脱离数据库事务能力单独理解。
- 隔离级别为什么回答到最后一定会回到并发异常、锁和数据库实现边界。

#### 先直接回答这个问题

隔离级别在这里解决的是“并发事务之间允许看到什么、要避免什么异常”这个问题。Spring 可以帮你声明事务边界，也可以配置隔离级别，但它并不会凭空创造数据库没有的事务能力，真正落地这些隔离语义的还是底层数据库引擎。所以事务题如果只讲 `@Transactional` 和传播行为，不讲隔离级别与数据库边界，就会缺一半。

更准确地说：

- Spring 负责声明和托管事务边界。
- 数据库负责真正实现隔离、锁、可见性和并发控制。

#### 再用场景把问题钉住

比如一个下单场景里：

- 你希望读取库存时不要读到未提交数据。
- 你希望同一事务里多次读取结果尽量稳定。
- 你还要考虑并发更新时是否会有脏读、不可重复读、幻读或锁冲突。

这些问题不是 Spring 注解本身能单独解决的，而是要靠底层数据库在当前隔离级别下给出实际语义。Spring 在这里做的是把“这段代码需要怎样的事务语义”声明清楚。

#### 是什么

- 隔离级别：定义并发事务之间的数据可见性和异常容忍边界。
- 数据库事务边界：定义底层数据库实际能提供哪些隔离和锁语义。

#### 为什么有它

因为事务不只是“同成同败”，还包括并发执行时的数据正确性。如果不讨论隔离级别，事务题就只剩“开关事务”，没有谈到一致性和并发成本之间的平衡。

#### 怎么理解它的机制和边界

第一层，最常见的隔离级别讨论，本质上是在回答会不会出现脏读、不可重复读、幻读这类并发异常。

第二层，Spring 可以把事务方法声明成某个隔离级别，但真正怎么实现，取决于底层数据库，比如 MySQL InnoDB 会通过 MVCC、锁和当前读等机制落地。

第三层，隔离级别越强，通常一致性越稳，但并发代价、锁冲突和吞吐压力也可能更高。

第四层，所以事务题里不能只说“我配置了某个隔离级别”，还要继续补一句：这个隔离级别在底层数据库里意味着什么并发效果和性能代价。

#### 面试时怎么讲才不空

“Spring 事务不能脱离数据库单独理解。Spring 负责托管事务边界和配置隔离级别，但真正落地这些隔离语义的还是数据库本身。回答隔离级别时，我会先讲它在解决哪些并发异常，再补底层数据库通常靠 MVCC、锁这些机制实现，以及隔离增强会带来什么并发代价。”

### 3.4 事务传播行为在解决什么问题，`@Transactional` 为什么经常失效

这一节在整条主线里解决的是第四和第五个问题：

- 为什么事务传播行为会存在。
- 失效场景到底该怎么分类理解。

#### 先直接回答这个问题

事务传播行为解决的是“一个事务方法调用另一个事务方法时，到底共用当前事务，还是新开事务”的问题。`@Transactional` 经常失效，根因通常不在数据库，而在事务增强链没有正确生效，常见原因包括：

- 同类方法内部调用绕过代理。
- 方法可见性或定义方式不符合代理增强边界。
- 异常被吞掉或异常类型不触发默认回滚规则。

#### 再用场景把问题钉住

例如外层方法要做主流程，内层方法要记审计日志：

- 如果日志失败不能影响主流程，可能考虑 `REQUIRES_NEW`。
- 如果必须同成同败，通常继续留在同一个事务里。

这就是传播行为的意义，它不是让你背 7 个枚举，而是在定义调用链里的事务边界。

再比如一个 Service 里：

- `methodA()` 标了事务。
- 它内部直接 `this.methodB()`。

这时 `methodB()` 即使也标了事务，调用也可能没经过代理，从而导致你以为事务会生效，实际上没有走完整事务增强链。

#### 是什么

- 传播行为：定义已有事务上下文下如何处理当前方法。
- 失效场景：定义“为什么声明了事务，但运行时事务语义没有按预期生效”。

#### 为什么有它

因为真实系统调用链不是单层的，事务边界必须能跨方法协同定义。

#### 怎么理解它的机制和边界

第一层，最常用的是：

- `REQUIRED`：有就加入，没有就新开。
- `REQUIRES_NEW`：挂起当前事务，自己新开。
- `NESTED`：在支持条件下走嵌套保存点语义。

第二层，传播行为讨论的是“事务怎么衔接”，不是“事务一定会生效”。

第三层，生效前提仍然是调用必须走代理、异常必须符合回滚规则、方法边界必须可被增强。

#### 面试时怎么讲才不空

“传播行为本质上是在定义事务调用链怎么衔接，不是让人背枚举。最常用的是 `REQUIRED` 和 `REQUIRES_NEW`，前者偏共享上下文，后者偏隔离子流程。`@Transactional` 失效的根因通常也不复杂，核心就是三类：调用没走代理、方法边界不支持增强、异常没有触发回滚规则。面试里我会先把这三类说清，再结合一个自调用或异常被吞的例子展开。”

## 4. 边界、误区、常见追问

- 不要把事务题答成只会背传播行为枚举。
- `@Transactional` 生效的关键不是注解写上了，而是调用链是否经过代理。
- 同类内部调用是最经典的失效场景。
- 默认回滚规则通常和异常类型有关，异常被吞也会影响结果。
- Spring 事务配置不能脱离底层数据库事务能力单独理解。

## 5. 推荐表达顺序

建议这样讲：

1. 先讲 Spring 事务在解决什么工程问题。
2. 再讲声明式事务为什么能成立，以及它和代理的关系。
3. 再讲隔离级别在解决哪些并发问题，以及它依赖数据库能力落地。
4. 再讲传播行为在定义什么边界。
5. 最后讲 `@Transactional` 失效的三类典型根因。

- 你能不能先不背传播行为，直接讲 Spring 事务在解决什么问题。

- 你能不能说明隔离级别为什么不能脱离底层数据库能力单独理解。
- 你能不能用场景解释 `REQUIRED` 和 `REQUIRES_NEW` 怎么选。
