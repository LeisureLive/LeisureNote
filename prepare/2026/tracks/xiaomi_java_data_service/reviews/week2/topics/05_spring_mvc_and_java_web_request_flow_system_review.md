## 1. 这一块在解决什么问题

Spring MVC / Java Web 这块不是在考你背多少接口名，而是在考你能不能把下面四件事讲清楚：

1. 一个 HTTP 请求进入服务端后，为什么要先经过 Servlet 容器。
2. Spring MVC 为什么能在 Java Web 之上把请求分发、参数绑定、返回值处理统一起来。
3. 参数绑定、拦截器、异常处理、返回 JSON 为什么都能挂在同一条请求链路上。
4. 过滤器、拦截器、AOP 到底各拦在哪一层，为什么面试里总被放在一起问。

把这些点串起来，主线其实很稳定：

- Web 服务先要有一个能接收 HTTP 请求、管理 Servlet 生命周期的基础运行环境。
- Servlet 容器先把请求收进来，Spring MVC 再在这个基础上做更细的控制器分发和方法级处理。
- 请求一旦进入统一分发器，参数绑定、返回值处理、异常处理就都可以复用同一套扩展链路。
- 不同横切能力拦截的位置不同，才会引出过滤器、拦截器、AOP 的分工边界。

所以这块真正要建立的是三种能力：

- 能解释 Web 请求为什么要先经过 Servlet 容器，再进入 Spring MVC。
- 能解释 Spring MVC 请求处理主流程为什么能统一接住参数、返回值和异常。
- 能解释过滤器、拦截器、AOP 的层级差异和适用场景。

## 2. 先把主线串起来，再看知识树

### 2.1 先用一条因果链把全文串起来

如果你觉得 Spring MVC 容易散，通常不是组件名没记住，而是没有先抓住下面这条主线：

1. 先看为什么请求要先进入 Servlet 容器：HTTP 请求要先被网络层和 Servlet 规范接住。
2. 再看 Spring MVC 为什么能接管请求：它把一个 `DispatcherServlet` 放在请求入口，统一做分发。
3. 再看请求进入分发器后发生什么：找处理器、适配调用、完成参数绑定、执行控制器方法。
4. 再看为什么返回值、异常、视图或 JSON 也都能统一处理：因为它们都挂在同一条请求执行链上。
5. 最后看过滤器、拦截器、AOP 为什么总被一起问：它们都能做横切，但拦截位置和能力边界不同。

所以这篇文档不是在讲几个并列组件，而是在回答一条连续问题链：

- 请求先到哪里。
- Spring MVC 怎么接住请求。
- 请求怎么被分发到控制器。
- 参数、返回值、异常怎么统一处理。
- 不同拦截手段怎么分工。

## 3. 核心重点展开

### 3.1 为什么 Web 请求要先进入 Servlet 容器，而不是直接交给 Spring MVC

这一节在回答什么问题：

- Java Web 基础运行环境在解决什么。
- 为什么 Spring MVC 不是脱离 Servlet 体系单独存在的。

这个问题是怎么解决的：

Web 请求要先进入 Servlet 容器，是因为服务端必须先有一个符合 Servlet 规范的运行环境来接收 HTTP 请求、管理 Servlet 生命周期、创建 `HttpServletRequest` / `HttpServletResponse` 这些基础对象。Spring MVC 并不是绕过这层基础设施，而是在这层之上继续做请求分发和方法调用增强。

放回场景里看，浏览器发起一个 HTTP 请求后，最先面对的不是 Controller，而是像 Tomcat 这样的容器。容器先把网络请求解析成 Servlet 规范下的请求对象，再把它交给入口 Servlet。Spring MVC 的 `DispatcherServlet` 就是被挂在这里的。

从机制上看，这件事可以拆成四层：

1. Servlet 容器负责监听端口、接收 HTTP 请求、创建请求响应对象。
2. 请求按照映射关系进入某个 Servlet。
3. Spring MVC 把 `DispatcherServlet` 作为统一入口挂上去。
4. 后面的控制器分发、参数绑定、异常处理，都是从这个入口继续往下发生的。

所以这一节真正要讲清的是：Spring MVC 解决的是“怎么优雅处理请求”，而不是“怎么从网络层收包并管理 Servlet 运行时”。收益是 Web 层能力分层更清晰；边界是回答 MVC 主流程时，不能跳过 Servlet 容器这一层直接开始背组件名。

### 3.2 Spring MVC 请求处理主流程为什么能把请求统一分发到控制器方法

这一节在回答什么问题：

- `DispatcherServlet` 到底在做什么。
- 请求为什么能从 URL 一路稳定走到某个 Controller 方法。

这个问题是怎么解决的：

Spring MVC 请求处理主流程的核心，是 `DispatcherServlet` 把“请求怎么找到处理器、怎么调用目标方法、怎么处理返回结果”统一收口了。它不是简单转发一下，而是把请求分发、方法适配、参数绑定、结果处理全都接成了一条标准化链路。

放回场景里看，比如请求 `GET /orders/123` 进来之后，Spring MVC 不会让你手工解析 URL 再自己 `if-else` 分发，而是会先根据请求路径找到匹配的处理器，再找到能调用这个处理器的适配器，完成方法参数准备，执行目标 Controller 方法，最后统一处理返回值。

可以把主流程压成下面几步：

1. 请求进入 `DispatcherServlet`。
2. `HandlerMapping` 找到目标处理器。
3. `HandlerAdapter` 负责以统一方式调用处理器方法。
4. Controller 执行业务逻辑。
5. 返回值进入统一处理链路。

这套设计的价值在于，请求分发和具体 Controller 的业务实现被拆开了。这样参数绑定、异常处理、返回值转换就不必每个控制器各写一套。代价则是如果只记一串类名、不理解每一层在调用链中的职责，主流程题就很容易答散。

### 3.3 参数绑定、返回值处理、异常处理为什么都能挂在同一条请求链路上

这一节在回答什么问题：

- 为什么 Spring MVC 不只是“路由到方法”这么简单。
- 为什么参数、返回值、异常这些通用能力可以统一治理。

这个问题是怎么解决的：

参数绑定、返回值处理、异常处理之所以都能挂在同一条链路上，是因为 Spring MVC 在调用 Controller 之前和之后都保留了统一扩展点。它不是把请求直接扔给某个方法就结束，而是把“入参准备、方法调用、结果转换、异常兜底”全部纳入同一个执行框架。

放回场景里更容易理解。一个 Controller 方法可能同时要求：路径变量绑定到方法参数、请求体反序列化成对象、返回对象自动转成 JSON、抛出的业务异常统一转成错误响应。这些能力如果靠每个方法自己处理，代码会非常散。Spring MVC 统一调用链的价值，就是把这些工作抽出来复用。

从机制上可以这样看：

1. 调用 Controller 前，Spring MVC 会根据参数解析规则准备方法入参。
2. 方法执行后，返回值会交给对应处理器决定是走视图解析还是消息转换。
3. 如果执行过程中抛异常，异常解析器会尝试把它转换成统一响应结果。
4. 所以 Web 层可以把业务逻辑和协议处理、错误处理拆开。

这也是为什么回答 Spring MVC 时，最好不要只讲“请求找到 Controller 了”。真正关键的是：同一条请求链，为什么能统一接住参数、返回值和异常，统一治理的收益是什么，以及这套统一治理为什么能让 Controller 层保持更薄。

### 3.4 过滤器、拦截器、AOP 为什么总被一起问，它们到底怎么分工

这一节在回答什么问题：

- 三种常见横切手段分别拦在哪一层。
- 为什么它们不能互相完全替代。

这个问题是怎么解决的：

过滤器、拦截器、AOP 总被一起问，是因为它们都能做横切增强，但作用层级不同。Filter 更靠近 Servlet 容器层，Interceptor 更靠近 Spring MVC 请求处理层，AOP 更靠近 Spring Bean 方法调用层。它们不是谁更高级，而是分别适合拦不同位置的问题。

放回场景里看会更直观：跨域、编码、统一请求包装通常更适合放在 Filter；登录校验、接口级审计、Controller 调用前后处理通常更适合放在 Interceptor；事务、Service 日志、权限切面这类 Bean 方法增强，则更适合放在 AOP。

从机制和边界上可以这样区分：

1. Filter 在请求进入 Servlet 之后、进入 Spring MVC 之前就可以拦截。
2. Interceptor 更贴近 Controller 执行前后，但它已经站在 Spring MVC 链路里。
3. AOP 针对的是 Spring 容器里的 Bean 方法调用，不天然感知 HTTP 请求语义。
4. 所以问题不是“哪个都能做”，而是“当前需求更适合拦在哪一层”。

这一节真正要建立的不是三个平行定义，而是一种层级感。只要能先按拦截位置分层，再补适用场景，这题就不容易答混。

## 4. 边界、误区、常见追问

- 不要一上来背 `DispatcherServlet`、`HandlerMapping`、`HandlerAdapter`，先把请求主流程讲顺。
- Spring MVC 不是脱离 Servlet 体系单独存在的。
- 过滤器、拦截器、AOP 都能做横切，但拦截位置不同，不能混成一个概念。
- JSON 返回、参数绑定、异常处理这些能力，最好放在同一条请求链路里解释。

## 5. 自测清单

- 你能不能先不背组件名，直接讲 HTTP 请求进入 Spring MVC 之后的主流程。
- 你能不能说明为什么 Spring MVC 建立在 Servlet 容器之上。
- 你能不能解释参数绑定、返回值处理、异常处理为什么能统一治理。
- 你能不能说清 Filter、Interceptor、AOP 各自拦在哪一层。
- 你能不能用 1-2 分钟把“Spring MVC 请求是怎么处理的”讲顺。
