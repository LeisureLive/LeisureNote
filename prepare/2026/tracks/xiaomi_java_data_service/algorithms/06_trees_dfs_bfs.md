# 算法专题 06：二叉树 / DFS / BFS

适用目标：

- 建立树题的统一认知：节点问题往往不是按下标处理，而是按“子树”和“层次”处理
- 分清什么时候优先用 DFS，什么时候优先用 BFS
- 建立遍历、递归、信息归并、路径处理这几类高频树题模板
- 能把常见树题从“会背代码”提升到“能解释为什么这样定义状态和返回值”

## 1. 树题的核心特点

- 树的本质是递归结构：
  - 一个节点的问题，往往可以拆成左子树和右子树的问题
- 树题最重要的不是数组式随机访问，而是：
  - 当前节点是谁
  - 左右孩子是谁
  - 当前子树能返回什么信息
- 很多树题表面不同，本质都在做下面几件事：
  - 遍历所有节点
  - 逐层处理节点
  - 从子节点收集信息再汇总给父节点
  - 沿路径维护某种状态
- 写树题时，真正决定难度的通常不是代码量，而是：
  - 递归函数的定义是否清楚
  - 返回值到底表示什么
  - 当前层和子问题之间的职责是否分清

## 2. 什么题应该优先想到树 / DFS / BFS

如果题目有下面这些信号，通常优先往树的方向想：

- 输入结构是 `TreeNode`
- 题目出现：
  - 前序 / 中序 / 后序遍历
  - 层序遍历
  - 最大深度 / 最小深度
  - 路径和
  - 最近公共祖先
  - 翻转 / 合并 / 序列化树
- 问题描述里强调：
  - 某个节点和它的左右子树关系
  - 某一层的节点
  - 从根到叶子的路径
- 需要判断“这个节点能向父节点返回什么信息”

如果题目虽然也有 DFS / BFS，但输入不是树，而是网格、图、状态空间，那属于图或搜索专题，不归到这个专题里。

## 3. 树题常考模式

看这一节时，建议你每个模式都固定追问三件事：

- 当前题是更像“遍历所有节点”，还是“逐层处理节点”
- 递归函数到底在解决哪个子问题
- 当前节点处理逻辑发生在前序、中序，还是后序位置

### 3.1 BFS：层序遍历和按层处理

适用场景：

- 按层输出节点
- 求最短层数 / 最小步数
- 需要区分“当前层”和“下一层”
- 题目天然按层推进，而不是按子树汇总

代表题：

- `102. 二叉树的层序遍历`
- `107. 二叉树的层序遍历 II`
- `199. 二叉树的右视图`
- `111. 二叉树的最小深度`

核心理解：

- BFS 不是“树题的默认答案”，而是“按层处理”时最自然的答案
- 队列里保存的是“下一批待处理节点”
- 真正重要的是“这一轮循环处理的节点，恰好属于同一层”

通用套路：

1. 先判断题目是不是在问：
   - 每一层有哪些节点
   - 第一个满足条件的层在哪里
   - 某一层的聚合结果是什么
2. 如果是，就优先考虑队列 + 分层遍历。
3. 常见主流程是：
   - 根节点入队
   - 进入 `while (!queue.isEmpty())`
   - 先记录当前层节点数 `size`
   - 用 `for (size)` 只处理这一层节点
   - 处理当前节点后，把左右孩子放进队列
4. 如果题目要求“最短层数 / 最早到达”，BFS 往往比 DFS 更直观，因为它天然一层层扩展。

你真正要会的是：

- 能解释为什么 `size` 能把当前层和下一层隔开
- 能区分“层序遍历输出”和“最短层数搜索”其实共用一套 BFS 框架

### 3.2 DFS：递归遍历和子树问题拆解

适用场景：

- 前序 / 中序 / 后序遍历
- 求深度、高度、节点数
- 翻转树、合并树、判断结构相同
- 题目天然可以拆成“左子树 + 右子树”

代表题：

- `104. 二叉树的最大深度`
- `94. 二叉树的中序遍历`
- `226. 翻转二叉树`
- `100. 相同的树`

核心理解：

- DFS 的重点不只是“递归访问”，而是“把整棵树问题拆成子树问题”
- 递归函数必须先定义清楚：
  - 参数表示处理哪棵子树
  - 返回值表示这棵子树处理后的结果
- 树题里前序、中序、后序的区别，本质是“当前节点处理逻辑放在递归的哪个位置”

通用套路：

1. 先定义函数：`dfs(node)` 到底返回什么。
2. 再写递归终止条件：
   - `node == null` 时返回什么。
3. 再思考当前节点逻辑应该放在哪：
   - 先处理当前节点：前序
   - 左子树和右子树之间处理当前节点：中序
   - 先拿到左右子树结果，再处理当前节点：后序
4. 如果题目要求的是“遍历顺序”，通常以前 / 中 / 后序视角思考。
5. 如果题目要求的是“子树汇总结果”，通常优先考虑后序 DFS。

你真正要会的是：

- 先定义 `dfs(node)` 的含义，再写代码
- 能解释为什么这题要在前序做，还是必须等左右子树返回后再做

### 3.2.1 树遍历到底怎么记：先看访问时机，再写递归或迭代

如果你写树遍历时总觉得前序、中序、后序容易混，最稳的处理方式不是先背代码，而是先回答：

1. 当前节点什么时候加入结果集。
2. 左右子树什么时候处理完，当前节点才允许被处理。

树遍历的本质可以统一压成一句话：

- 递归看“访问时机”。
- 迭代看“入栈、出栈、何时访问”。

三种遍历只差在“当前节点的访问语句放在哪”：

- 前序：`根 -> 左 -> 右`
  - 第一次到当前节点就访问。
- 中序：`左 -> 根 -> 右`
  - 左子树处理完后访问当前节点。
- 后序：`左 -> 右 -> 根`
  - 左右子树都处理完后再访问当前节点。

换句话说，不要把前序、中序、后序记成三套分裂模板，而要先记：

- 当前节点是“进门就处理”
- 还是“左边回来再处理”
- 还是“左右都做完再处理”

#### 递归版怎么想

递归版最稳的思路是：

1. 先定义函数：
   - 它负责遍历“以当前节点为根”的整棵子树。
2. 再决定访问语句放在哪：
   - 前序：先访问当前节点，再递归左、右子树。
   - 中序：先递归左子树，再访问当前节点，再递归右子树。
   - 后序：先递归左、右子树，最后访问当前节点。

可以把递归遍历记成：

- 前序：进门就记。
- 中序：左边回来再记。
- 后序：左右都做完再记。

所以递归版真正要先确定的，不是“代码怎么写”，而是：

- `result.add(node.val)` 应该放在递归的什么位置。

#### 迭代版怎么想

迭代版本质上是在用栈模拟递归调用过程。

最需要想清楚的是两件事：

1. 栈里保存的是什么：
   - 栈里保存的是“还没完全处理完的祖先节点”。
2. 当前节点什么时候可以访问：
   - 是第一次碰到就访问，还是必须等左边回来，还是必须等左右都处理完。

三种遍历的迭代版可以这样记：

- 前序：
  - 第一次碰到当前节点就访问。
  - 然后继续一路向左，回退后再切右子树。
  - 可以压成一句：能看见就记，记完往左，回退转右。
- 中序：
  - 当前节点不能一碰到就访问。
  - 必须先把左链走完，回退到当前节点时再访问，然后切右子树。
  - 可以压成一句：先压左，回退记，记完转右。
- 后序：
  - 当前节点必须等左右子树都处理完后才能访问。
  - 这是最难稳定写对的一种。
  - 面试里最稳的记法通常有两种：
    - 单栈 + `prev` 指针，判断右子树是否已经处理过。
    - 先按“根 -> 右 -> 左”收集结果，最后整体倒序。

如果当前目标是先把模板打稳，后序遍历优先建议记：

- 根右左 + 倒序

因为它更容易快速写对，不容易在“右子树是否已经访问过”这个状态判断上出错。

#### 树遍历时最值得固定下来的统一判断

以后每次写树遍历，先别急着写代码，先口头回答下面两个问题：

1. 当前节点什么时候加入结果集。
2. 回退到父节点后，下一步是访问父节点，还是转去右子树。

如果这两个问题能先回答清楚，递归和迭代代码都会稳定很多。

### 3.3 后序 DFS：信息归并题

适用场景：

- 当前节点答案依赖左右子树返回信息
- 需要从子节点收集结果，再决定当前节点状态
- 需要在遍历过程中顺手更新全局最优值

代表题：

- `543. 二叉树的直径`
- `110. 平衡二叉树`
- `124. 二叉树中的最大路径和`
- `236. 二叉树的最近公共祖先`

核心理解：

- 这类题不是“遍历题”，而是“子树信息汇总题”
- 关键不在于会不会递归，而在于：
  - 每棵子树要返回什么信息给父节点
  - 哪些信息只在当前层使用，不需要继续向上返回
- 很多资深面试中，树题的分水岭就在这里：
  - 初级候选人只会写遍历
  - 稍强一点的候选人能先定义返回值，再做信息归并

通用套路：

1. 先问：父节点为了完成自己的计算，最需要从左右孩子拿到什么。
2. 把这个信息定义成 `dfs(node)` 的返回值。
3. 递归拿到：
   - `left = dfs(node.left)`
   - `right = dfs(node.right)`
4. 用左右信息计算：
   - 当前节点要返回给父节点的值
   - 当前节点能否更新全局答案
5. 区分两类结果：
   - 向上返回的结果
   - 只在当前节点结算一次的结果

你真正要会的是：

- 一眼识别“这题答案依赖左右子树返回信息”，从而直接想到后序 DFS
- 能把“返回值”和“全局答案”分清，而不是全都塞在一个变量里硬写

### 3.3.1 树上的信息归并题怎么识别

如果普通遍历题的重点是“节点按什么顺序访问”，那么信息归并题的重点就是：

- 左子树先返回什么。
- 右子树先返回什么。
- 当前节点如何把左右子树结果合并起来。

这类题真正的关键，不是“会不会递归访问所有节点”，而是：

- 你能不能先定义 `dfs(node)` 到底返回什么。
- 你能不能在当前节点把左右子树信息归并成新的结果。

#### 最常见的识别信号

如果题目出现下面这些信号，通常就不是普通遍历题，而是树上的信息归并题：

1. 当前节点的答案依赖左右子树先返回信息。
   - 也就是必须先拿到：
     - `left = dfs(node.left)`
     - `right = dfs(node.right)`
   - 当前节点才能决定自己怎么做。
2. 题目要求“先算子节点，再算当前节点”。
   - 这类题天然更像后序 DFS。
3. 递归函数返回值不再只是“有没有访问过”，而是某种子树摘要信息。
   - 例如：
     - 高度
     - 最大单边长度
     - 是否平衡
     - 某个候选节点
     - 当前子树是否包含目标节点
4. 题目里同时存在两类结果：
   - 返回给父节点继续使用的结果
   - 当前节点用来更新全局答案或形成局部答案的结果

一句话识别：

- 只要当前节点依赖左右子树结果，就优先考虑后序 DFS。

#### 做题时最先要问的 3 个问题

碰到这类题时，建议固定先问自己下面三句：

1. `dfs(node)` 返回什么。
2. 当前节点怎么用左右子树结果做归并。
3. 哪些信息要继续向上返回，哪些信息只在当前节点结算一次。

如果这三句先想清楚，代码通常就不会乱。

#### 和普通遍历题最容易混淆的地方

普通遍历题更关注：

- 当前节点什么时候访问。
- 结果什么时候加入结果集。
- 是否要沿路径维护状态。

信息归并题更关注：

- 左右子树分别给当前节点提供什么信息。
- 当前节点如何组合这些信息。
- 当前节点是否需要顺手更新全局答案。

所以后面看到树题时，不要只问：

- 这是前序、中序，还是后序。

更要先问：

- 当前节点是不是必须等左右子树先算完，自己才能决定答案。

如果是，这题就优先按“信息归并题”来想。

#### 用 Day17 的两道题来理解

`543. 二叉树的直径`

- `dfs(node)` 返回的是：
  - 当前节点往下的最长单边长度。
- 当前节点的局部答案是：
  - `left + right`
- 但返回给父节点的不是 `left + right`，而是：
  - `max(left, right) + 1`

这题的关键就是：

- 返回给父节点的结果
- 当前节点用来更新全局直径的结果

这两者不是同一个东西。

`236. 二叉树的最近公共祖先`

- `dfs(node)` 返回的是：
  - 当前子树中找到的有效节点引用：
    - `p`
    - `q`
    - 或已经确定的最近公共祖先
- 如果左右都返回非空，说明当前节点第一次同时接住了左右两边的有效结果，当前节点就是答案。

这题的关键是：

- 递归返回值本身就可以直接承载“候选答案节点”
- 不一定非要额外维护全局变量

#### 最后压成一句话

树上的信息归并题，可以统一记成：

- 先定义 `dfs(node)` 返回什么
- 再用后序 DFS 拿左右结果
- 最后在当前节点做归并

如果还要再压缩一层，就是：

- 先定义返回值，再写后序归并

### 3.4 路径问题：沿递归路径维护状态

适用场景：

- 根到叶路径和
- 列出所有路径
- 路径是否满足某种约束
- 路径字符串 / 路径数字拼接

代表题：

- `112. 路径总和`
- `113. 路径总和 II`
- `257. 二叉树的所有路径`
- `129. 求根节点到叶节点数字之和`

核心理解：

- 这类题不只是“遍历到所有节点”，而是“沿当前路径累积状态”
- 维护的状态可能是：
  - 当前路径和
  - 当前路径节点列表
  - 当前构造出的字符串 / 数字
- 如果题目要求“收集所有路径”，通常要注意回溯；如果只求是否存在，状态维护会简单很多

这里再压一条最容易混的判断：

- 如果维护的是“共享可变状态”，通常需要回溯。
  - 典型如 `List path`、`StringBuilder`、计数数组、访问标记。
  - 原因是左右子树会共用同一个对象；当前分支改过之后，返回父节点时要恢复现场，避免污染兄弟分支。
- 如果维护的是“值状态”，通常不需要显式回溯。
  - 典型如剩余目标和、当前路径和、当前路径构成的数字、布尔标记。
  - 原因是递归调用拿到的是当前值推导出的新状态，各分支天然隔离。

可以直接记成一句话：

- 共享可变对象，通常要回溯。
- 纯参数传值状态，通常不用回溯。

对应到代表题上：

- `113` 收集路径时，需要维护同一个 `path` 容器，所以要 `add` 后再 `remove`。
- `129` 求路径数字和时，只要沿递归传 `current = current * 10 + node.val`，不需要真的保存整条路径，也不需要回溯。

通用套路：

1. 先定义当前递归路径上要维护什么状态。
2. 每进入一个节点，就把当前节点纳入状态。
3. 到叶子节点时判断：
   - 是否满足条件
   - 是否需要收集结果
4. 返回上一层前，如果用了可变容器保存路径，通常要做回溯。
5. 如果状态可以直接通过参数传值，不一定非要用全局变量或可变列表。

你真正要会的是：

- 能区分“路径存在性判断”和“收集所有路径”两类题的写法差异
- 知道什么时候必须回溯，什么时候传值就够

### 3.5 BST：利用有序性剪枝和定位

适用场景：

- 二叉搜索树中的查找、插入、删除
- 第 `k` 小 / 第 `k` 大
- 验证是否为 BST
- 利用大小关系剪枝

代表题：

- `98. 验证二叉搜索树`
- `230. 二叉搜索树中第 K 小的元素`
- `700. 二叉搜索树中的搜索`
- `235. 二叉搜索树的最近公共祖先`

核心理解：

- BST 不是普通二叉树，它额外提供了有序性：
  - 左子树都更小
  - 右子树都更大
- 这意味着很多题不需要遍历整棵树，可以基于大小关系直接剪枝
- 但也最容易犯一个错误：
  - 只比较当前节点和左右孩子
  - 却没有维护整棵子树的上下界

通用套路：

1. 先确认题目是不是明确给了 BST 条件。
2. 如果是，就优先考虑有序性的价值：
   - 能不能直接往左或往右走
   - 能不能减少遍历范围
3. 验证 BST 时，重点不是“左孩子小、右孩子大”，而是：
   - 整个左子树都要小于当前节点
   - 整个右子树都要大于当前节点
4. 第 `k` 小元素常用中序遍历，因为 BST 的中序结果天然有序。

你真正要会的是：

- 一看到 BST 就想到“有序性”和“剪枝”，而不是还按普通树全遍历
- 能说清为什么验证 BST 不能只看相邻父子节点

## 4. DFS 和 BFS 怎么选

可以先用下面这套判断：

- 如果题目强调“按层处理 / 最短层数 / 某层结果”，优先 BFS
- 如果题目强调“子树递归定义 / 左右子树汇总 / 深度高度”，优先 DFS
- 如果题目要求“收集所有路径”，通常优先 DFS
- 如果题目要求“最早到达某层叶子”或“逐层扩展”，通常优先 BFS

一个常见误区：

- 看到树题就默认 DFS
- 其实像 `102`、`199`、`111` 这种题，BFS 更自然

另一个常见误区：

- 看到 DFS 就只会写前序遍历
- 但很多高频题真正需要的是后序信息归并

## 5. 树题的高频坑点

- 递归函数定义不清，写到一半不知道该返回什么
- 前序、中序、后序位置混乱，导致逻辑放错位置
- 只会遍历，不会定义“子树返回信息”
- 把 `null` 节点的返回值随便写，导致边界全乱
- 层序遍历时没有按层隔离，结果把不同层节点混在一起
- 收集路径时忘记回溯，导致结果串路径
- 验证 BST 时只比较父子节点，遗漏整棵子树约束

## 6. 树题的面试表达模板

```text
这题我会先判断它更像哪一类树题：
如果题目强调按层处理，我会优先考虑 BFS；
如果题目更像“当前节点依赖左右子树结果”，我会优先考虑 DFS，
并先定义递归函数 dfs(node) 到底返回什么。

如果当前节点的答案需要左右子树先算完，我会用后序 DFS，
先拿到 left 和 right，再在当前节点做信息归并。
如果题目只是要求遍历或维护路径，我会再判断是前序处理还是沿路径传状态。
```
