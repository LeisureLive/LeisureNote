# 项目追问清单

这份文档只做一件事：把项目面高频追问先收口。

用法：

- 先配合 `04_project_story_stable_draft.md` 讲主线
- 再用这份文档做追问演练
- 每次模拟后，把最容易讲散的问题补到对应项目下面

## 1. 通用追问框架

项目面高频追问，基本都落在这 6 类：

1. 为什么要做
2. 为什么这样设计
3. 为什么不用另一种方案
4. 这套方案的边界和代价是什么
5. 出问题时怎么排查和止损
6. 如果量级再涨 10 倍怎么办

## 2. 实时数据订阅 + 实时查询

### 2.1 必问级问题

- 这个系统服务谁，为什么不能让业务方自己消费数据？
- 实时订阅和实时查询为什么适合放在同一条数据服务主线里？
- 发送链路 6w/s 的瓶颈主要在哪，是 CPU、网络、序列化还是下游能力？
- 上游数据为什么会拆成数仓链路和导入流链路两类？
- 查询侧为什么选 Pegasus，而不是直接放 MySQL 或其他引擎？
- 元数据依赖为什么会成为真实难点？
- 元数据更新没有通知机制时，你们怎么控制错误影响范围？
- 慢下游或异常下游会不会拖垮主链？怎么隔离？
- 为什么支持横向扩展，扩的是哪一层？

### 2.2 取舍级问题

- 为什么这里用缓存刷新，而不是强一致通知？
- 为什么这里要做统一平台能力，而不是每个业务自己适配？
- 如果强调一致性，吞吐会受什么影响？
- 如果强调吞吐，准确性或治理会付出什么代价？

### 2.3 故障与演进问题

- 如果下游 Kafka/HTTP 目标持续变慢，你们怎么止损？
- 如果订阅规则变复杂，规则匹配层会不会成为瓶颈？
- 如果新增更多目标端类型，这套设计还能不能继续扩？
- 如果查询流量也明显上涨，缓存、主存储和分析引擎怎么分工？
- Flink 为什么默认这样配并行度和 slotNum，扩容时优先调什么？

## 3. id-mapping

### 3.1 必问级问题

- 这个系统的业务目标是什么，为什么难点不只是“查映射关系”？
- 原链路为什么只能做到 3k-4k/s？
- 高性能模式具体做了哪些关键取舍？
- 为什么可以去掉或弱化对 `skv-proxy` 的依赖？
- 单机 10k+/s 的提升主要来自哪里？
- 为什么支持横向扩展，扩容后瓶颈会转移到哪里？

### 3.2 兼容性与边界问题

- 为什么不能直接彻底重写老系统？
- 放弃部分低频功能会不会影响线上业务？
- 怎么保证高性能模式可灰度、可回退？
- 新模式和旧逻辑并存时，怎么避免行为不一致？

### 3.3 深挖问题

- 你负责的核心模块到底是哪一块？
- 最怕被追问的点是什么？
- 如果量级再涨 10 倍，这个系统下一步先改什么？
- 这个项目里最能体现你技术判断力的取舍是什么？

## 4. 当前最该优先练的 Top 8

先别平均练所有追问，优先练这 8 个：

1. 为什么实时数据订阅不是简单消息转发
2. 元数据依赖为什么会成为瓶颈和一致性风险
3. 上游数据为什么会分成数仓链路和导入流链路
4. 慢下游拖垮主链时怎么隔离
5. 查询为什么选 Pegasus，以及它的数据模型怎么支撑查询
6. id-mapping 原链路为什么慢
7. 高性能模式到底舍弃了什么、换来了什么
8. 为什么不能直接推倒重写

## 5. 使用要求

- 每次项目模拟至少从这份清单里抽 3-5 个问题
- 答不稳的问题，不要只记结论，要回补到：
  - `04_project_story_stable_draft.md`
- 目标不是“背答案”，而是做到：
  - 主线稳定
  - 追问能接
  - 能讲取舍和边界
