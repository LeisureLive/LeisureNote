## 1. 这一块在解决什么问题

Spring AOP 这块不是在考你记住了几个注解或通知类型，而是在考你能不能把下面四件事讲清楚：

1. 为什么业务系统需要一种“不改业务代码也能加公共逻辑”的机制。
2. Spring 为什么没有直接改原对象，而是大多通过代理做增强。
3. JDK 动态代理和 CGLIB 各自依赖什么前提，分别适合什么场景。
4. AOP 能解决什么，不能解决什么，为什么它经常和事务、日志、鉴权一起出现。

把这四个问题串起来，主线其实很稳定：

- 业务代码里总会出现日志、事务、权限、监控这类横切逻辑。
- 如果把这些逻辑硬写进每个方法，代码会重复、耦合、难维护。
- 所以 Spring 需要一种在目标方法调用前后统一追加公共逻辑的机制。
- 这种机制在 Spring 里主要靠代理对象实现，而不是直接改写业务代码主干。
- 代理方案不同，就会继续引出 JDK Proxy 和 CGLIB 的差异。
- 代理边界不同，又会继续引出自调用失效、方法可见性、final 限制这些常见追问。

所以 AOP 这块真正要建立的是三种能力：

- 能解释 AOP 在解决什么工程问题。
- 能解释 Spring AOP 为什么通常是“代理增强”而不是“直接织入全部字节码”。
- 能解释 AOP 的边界，以及它为什么经常和事务失效问题连在一起问。

## 3. 核心重点展开

### 3.1 为什么系统需要 AOP，而不是把公共逻辑直接写进业务代码

这一节在回答什么问题：

- AOP 到底在解决什么问题。
- 为什么它不是一个“语法花活”，而是工程化分层手段。

这个问题是怎么解决的：

系统需要 AOP，本质上是因为很多逻辑不是业务主流程的一部分，但又会反复出现在大量方法里，比如日志、事务、权限、监控、审计。如果把这些逻辑都硬塞进业务代码，结果通常会变成代码重复、主流程被横切逻辑淹没、公共能力改动需要改很多地方。AOP 的价值，不是让代码“更高级”，而是把横切关注点从业务主逻辑里抽出来，让业务代码更聚焦，也让公共能力更容易统一治理。

把这件事放回一个最小场景里更容易理解。比如订单服务里的多个方法都要做入口日志、鉴权、事务和耗时统计。如果这些逻辑都写在每个方法里，业务代码很快就会变成“重复模板 + 少量真实逻辑”。AOP 做的事情，就是把这类重复模板从业务方法里抽走，让方法调用进入前后自动执行公共逻辑。

从机制上看，可以把 AOP 理解成“把横切逻辑挂到目标方法调用链上”的机制。它不是替代 OOP，而是在 OOP 之上补一层跨多个对象统一生效的增强能力。所以 AOP 解决的是：公共逻辑重复、业务代码和治理逻辑耦合、横切能力难以统一维护。收益是分层更清晰、公共能力更集中；边界是它更适合横切问题，不适合替代明确的业务主流程。

### 3.2 Spring 为什么大多通过代理实现 AOP，而不是直接改原对象

这一节在回答什么问题：

- Spring 为什么能在不侵入业务代码的情况下做增强。
- 为什么“代理”会成为 Spring AOP 的核心实现思路。

这个问题是怎么解决的：

Spring 大多通过代理实现 AOP，是因为代理方案足够通用、和容器结合自然，而且能在运行时对 Bean 做增强。它的核心不是改掉原对象，而是让外部调用先经过代理对象，再由代理决定是否追加前置、后置或环绕逻辑。这意味着 Spring AOP 的关键不是“目标对象被改写了”，而是“调用路径被接管了”。

放回场景里看，一个 Service Bean 被 Spring 容器托管后，如果它匹配某个切面规则，Spring 可以在创建 Bean 时额外生成一个代理对象。调用方注入的是代理对象，代理对象内部再持有目标对象，每次方法调用先经过代理，代理再判断是否要执行增强逻辑。这就是为什么你的业务代码里没有显式写日志或事务模板，但运行时这些逻辑仍然能生效。

这种设计的价值在于三点：

1. 不侵入业务代码主干。
2. 容器可以在创建 Bean 时统一决定是否增强。
3. 事务、日志、权限这类公共能力都能走同一条接入方式。

代价也很明确：AOP 的能力边界天然受代理边界约束。也就是说，后面你看到的很多 AOP 失效问题，本质上都不是“增强逻辑没写对”，而是调用没有真正经过代理对象。

### 3.3 JDK 动态代理和 CGLIB 的区别，为什么这题经常和 AOP 一起问

这一节在回答什么问题：

- 两种常见代理方案分别依赖什么前提。
- 为什么代理边界会直接影响 AOP 是否生效。

这个问题是怎么解决的：

JDK 动态代理是基于接口生成代理，前提是目标对象实现了接口；CGLIB 是基于继承目标类生成子类代理，不依赖接口，但会受到 final 类和 final 方法限制。它们之所以总和 AOP 一起问，是因为 Spring AOP 落地时必须选一种代理方式，而代理方式决定了增强边界。

把这件事放回实际场景会更直观。如果你的 Service 通过接口暴露能力，JDK 动态代理通常就够用；如果目标类没有接口，Spring 就更可能退回到 CGLIB。面试官追问这题，不是为了听你背定义，而是想看你是否知道为什么有时增强能生效，为什么有时因为方法边界或类定义方式，增强会失效。

从机制上可以这样拆：

1. JDK 动态代理生成的是“实现同一组接口的代理对象”，所以调用必须通过接口方法进入。
2. CGLIB 生成的是目标类的子类，通过重写可代理的方法来做增强。
3. 正因为 CGLIB 依赖继承和重写，所以 final 类不能被继承，final 方法和 private 方法也不在可增强边界上。

这也是两种方案的主要取舍：JDK Proxy 更依赖接口抽象，CGLIB 更灵活但受继承边界影响。真正重要的，不是背结论，而是把“代理生成方式”和“增强生效边界”连起来理解。

### 3.4 为什么 AOP 这题总会继续追到事务、自调用失效这些问题

这一节在回答什么问题：

- 为什么 AOP 题不会只停留在“能不能做增强”。
- 为什么事务题和自调用失效题，本质上经常和 AOP 走的是同一条主线。

这个问题是怎么解决的：

AOP 这题总会继续追到事务和自调用失效，是因为 Spring 里很多横切能力本来就是靠代理挂到调用链上的，事务只是其中最常见的一种。如果你只会说“Spring AOP 用代理实现”，但说不清调用到底什么时候会绕过代理、哪些方法边界会让增强失效，那你的 AOP 理解就还停留在概念层。

所以这类追问真正想确认的，不只是你知不知道“Spring 用代理”，而是你能不能顺着下面这条主线往下讲：

1. 这个 Bean 有没有生成代理对象。
2. 这次调用有没有先经过代理对象。
3. 当前方法是不是处在代理可拦截边界里。
4. 如果是事务题，再额外判断：是不是代理已经进来了，但回滚规则、线程边界、异常处理方式不符合预期。

也就是说，AOP、自调用失效、事务失效这些问题，不应该拆成三套孤立知识点去背，而应该顺着“代理调用链”统一理解。只要调用没走代理，AOP 和事务这类增强就都可能一起失效；而即使已经走到代理，事务也还可能因为回滚规则或线程边界表现得“不像生效”。

#### 3.4.1 先把最核心的一句话记住

- Spring AOP 接管的是“外部通过代理对象进入目标方法”的调用路径。
- 一旦调用没有经过代理，或者当前方法不在代理可拦截边界里，增强就可能失效。
- 所以 AOP、事务、自调用这些题，本质都在问：这次调用到底有没有经过代理链。

#### 3.4.2 会导致代理路径失效的高频场景汇总

1. 对象根本不是 Spring 容器里的代理 Bean

   典型场景：

   - 自己 `new` 对象。
   - 类没有被 Spring 扫描或注册。
   - 对应 Bean 没有走到容器创建和后置处理流程。

   为什么会失效：

   - 因为代理对象本来就是 Spring 在创建 Bean 时额外包出来的。对象不在容器里，就没有代理，更谈不上 AOP 调用链。

   例子：

   - `new OrderService().createOrder()` 不会触发 `@Transactional`，因为这就是普通对象调用。

2. 同类内部自调用：显式 `this.method()` 和隐式 `method()`

   典型场景：

   - 一个方法里直接写 `this.otherMethod()`。
   - 一个方法里直接写 `otherMethod()`。

   为什么会失效：

   - 这两种写法本质上都是当前对象内部对自己的调用。`method()` 只是隐式的 `this.method()`。
   - 代理对象只能拦截“外部进入代理”的调用；一旦已经进入目标对象内部，后续内部方法互调就不会重新绕回代理。

   例子：

   - `createBatch()` 内部调用 `createOne()`，如果 `createOne()` 上标了 `@Transactional`，这次内部调用通常不会触发事务增强。

3. 初始化阶段内部调用：构造器、`@PostConstruct`、`afterPropertiesSet()`

   典型场景：

   - 在构造器里调用本类另一个带事务或切面的方法。
   - 在 `@PostConstruct` 里调用本类另一个带事务的方法。
   - 在 `afterPropertiesSet()` 里直接触发本类其他代理能力。

   为什么会失效：

   - 这类调用发生在 Bean 初始化过程中，Bean 还处在创建链路里，代理通常还没有以“外部入口”的方式稳定参与调用。
   - 即使方法上有 `@Transactional`，这时也很容易因为没有走完整代理链而达不到预期。

   例子：

   - `@PostConstruct` 里调用 `loadData()`，而 `loadData()` 上标了 `@Transactional`，这时不要默认认为事务一定稳定生效。

4. 方法边界不在代理可拦截范围内

   典型场景：

   - `private` 方法上标注事务或切面。
   - `final` 方法希望被 CGLIB 增强。
   - `final` 类希望走基于继承的代理增强。
   - JDK 动态代理场景下，调用没有通过接口方法边界进入。

   为什么会失效：

   - Spring 代理不是任意字节码织入，它有明确的可拦截边界。
   - JDK 动态代理依赖接口方法。
   - CGLIB 依赖子类继承和方法重写，所以 `final` / `private` 会限制增强。

   例子：

   - `private void createOne()` 上标 `@Transactional`，通常不能按预期生效，因为这个方法本来就不在常规代理增强边界内。

5. 拿到的不是代理引用，而是目标对象本身

   典型场景：

   - 通过非常规方式绕过容器持有了原始目标对象。
   - 在一些底层扩展或调试场景里，直接操作 target 而不是操作 proxy。

   为什么会失效：

   - 代理增强真正挂在 proxy 上，不挂在原始 target 上。
   - 只要最终调用是直接打到 target，增强链就会被整体绕过。

   例子：

   - 业务代码里正常很少主动这么做，但从原理上必须知道：增强生效依赖的是“你拿到的是代理对象”，不是“目标类上写了注解”。

#### 3.4.3 容易混淆，但不完全等于“代理路径失效”的事务问题

- 异常被自己 `catch` 住了：
  - 这更多是事务回滚条件没满足，不一定是代理没进来。
- 抛的是 checked exception，但没有配 `rollbackFor`：
  - 这通常是回滚规则问题，不是代理路径问题。
- 开了新线程，或者和 `@Async` 混用：
  - 这更多是事务上下文和线程边界问题，不一定是代理没有拦截到外层方法。
- 传播行为理解错了：
  - 这属于事务语义问题，不是 AOP 代理本身失效。

所以面试里最好先把问题分层：

- “增强完全没触发”：
  - 先查代理有没有生成、调用有没有走代理、方法边界是否可拦截。
- “事务看起来不符合预期”：
  - 再查回滚规则、异常处理、线程边界、传播行为。

#### 3.4.4 这一节真正要记住什么

- AOP、`@Transactional`、`@Async`、`@Cacheable` 这类能力，很多时候共用同一条代理调用链。
- 高概率导致代理路径失效的场景，可以先按四类记：
  - 对象不在容器里，没有代理。
  - 同类内部自调用，没有重新经过代理。
  - 初始化阶段调用，代理还没稳定参与外部调用链。
  - 方法不在代理可拦截边界里。
- 面试里如果你能先说清“这次调用有没有经过代理”，后面再展开事务、自调用、final / private、线程边界，主线通常就不会乱。

## 4. 边界、误区、常见追问

- AOP 不适合替代明确业务逻辑，它更适合横切关注点。
- Spring AOP 重点是运行时代理，不要和织入能力更强的 AspectJ 混成一个概念。
- 不要把 `method()` 和 `this.method()` 分开理解，在同类内部它们都属于自调用，都会绕过代理。
- 自调用、初始化阶段调用、手动 `new` 对象，都是高频的“没走代理链”场景。
- final 类、final 方法、private 方法会影响代理增强边界。
- 事务“不回滚”不一定等于代理失效，也可能是异常处理、回滚规则或线程边界问题。
- 面试官经常从 AOP 直接追到事务，因为两者很多时候共用代理主线。

- 你能不能按“有没有代理 -> 有没有经过代理 -> 方法边界能不能拦截”这条顺序分析失效场景。
