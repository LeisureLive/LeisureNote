# 算法题出题规则与答题框架

适用目标：

- 后续算法练习时，题目描述更清楚，减少来回补充约束
- 保证每道题都能自然进入“先识别题型，再写代码”的训练节奏
- 让题目、样例、限制条件、答题要求保持统一风格

## 1. 为什么需要这份规则

算法练习里最容易出现两个问题：

- 题目描述不完整：
  - 没说清输入输出
  - 没说清是否允许额外空间
  - 没说清时间复杂度要求
  - 没给样例，导致理解偏差
- 直接进入写代码：
  - 还没确认题型
  - 还没明确窗口、指针、状态定义
  - 还没说复杂度和边界

所以后续出题时，要先把题面描述清楚，再固定进入答题框架。

## 2. 强制出题结构

每道算法题至少要包含下面六部分。

### 2.1 题目标题

建议格式：

- `题号. 题目名`
- 如有必要，可补充题型标签

示例：

- `3. 无重复字符的最长子串`
- `209. 长度最小的子数组`

说明：

- 默认不要只说“做一道滑动窗口题”
- 题号和题名尽量一起给，方便后续复盘和检索

### 2.2 题目描述

要求：

- 说明输入是什么
- 说明输出是什么
- 说明目标是什么
- 描述尽量避免歧义

合格写法：

- 给定一个字符串 `s`，请你找出其中不含有重复字符的最长子串的长度。

不合格写法：

- 给你一个字符串，求最长不重复子串。

问题在于：

- 没明确输出的是“子串本身”还是“长度”

### 2.3 输入输出与方法签名

如果是 Java 练习，建议同步给出方法签名。

示例：

```java
public int lengthOfLongestSubstring(String s)
```

要求：

- 输入参数类型要明确
- 返回值类型要明确
- 如果题目要求原地修改数组，要明确返回什么

例如：

- 返回新长度
- 返回结果数组
- 直接修改原数组，不额外返回

补充要求：

- 出题模式下，默认只给方法骨架，不给具体实现
- 题目文件里可以保留：
  - 类名
  - 方法签名
  - 题目描述
  - 样例
  - 复杂度要求
  - 答题框架
- 题目文件里默认不直接写完整解法
- 如果后续需要标准答案，单独在问答中提供，或放到单独的题解文件中

### 2.4 输入输出样例

要求：

- 视情况尽量提供 `1-2` 个具体样例
- 样例尽量覆盖典型情况，而不是只给最简单情况
- 如有必要，补一句样例解释

推荐格式：

```text
示例 1：
输入：s = "abcabcbb"
输出：3
解释：最长不含重复字符的子串是 "abc"，长度为 3。

示例 2：
输入：s = "bbbbb"
输出：1
解释：最长不含重复字符的子串是 "b"，长度为 1。
```

### 2.5 额外要求

这是最容易缺失、但最应该提前说明的部分。

至少要明确下面几项中的相关内容：

- 时间复杂度要求：
  - 如目标为 `O(n)`、`O(n log n)`，要明确写出
- 空间复杂度要求：
  - 如要求 `O(1)` 额外空间，要明确写出
- 是否允许使用额外空间：
  - 如允许哈希表、辅助数组、队列等
- 是否要求原地修改：
  - 如数组必须原地处理
- 是否允许暴力解：
  - 如不能接受 `O(n^2)` 枚举
- 是否保证有解：
  - 如默认至少存在一个解
- 下标约定：
  - 返回 `0-based` 还是 `1-based`
- 结果是否唯一：
  - 如果不唯一，返回任意一个还是全部返回

推荐写法：

```text
额外要求：
- 目标时间复杂度为 O(n)
- 允许使用额外空间
- 不接受暴力枚举所有子串的 O(n^2) 解法
```

### 2.6 答题顺序要求

出题时默认要说明答题顺序，避免一上来直接写代码。

默认流程：

1. 先复述题目要求和限制条件
2. 先判断题型
3. 先口头说思路
4. 再写代码
5. 最后补复杂度、边界和易错点

## 3. 默认要补清楚的约束清单

下面这些点，如果题目里有歧义，默认必须补清：

- 输入规模：
  - 是否需要按大输入规模选择线性或对数复杂度解法
- 输入元素范围：
  - 是否可能有负数、重复值、空字符串、空数组
- 字符范围：
  - 只包含小写字母，还是任意 ASCII / Unicode
- 返回结果形式：
  - 返回长度、下标、区间，还是实际内容
- 多解场景：
  - 返回任意一个，还是全部返回
- 有序性：
  - 输入是否已经有序
- 稳定性要求：
  - 结果是否要保持原相对顺序

## 4. 不建议的出题写法

后续出题时，尽量避免下面这些问题。

### 4.1 只给题型，不给题面

不建议：

- 做一道滑动窗口题

更合理：

- `3. 无重复字符的最长子串`
- 给定字符串 `s`，求不含重复字符的最长子串长度

### 4.2 不给样例

不建议：

- 请实现最优解

问题：

- 容易误解输出形式
- 不方便快速校验边界

### 4.3 不写复杂度要求

不建议：

- 求解这道题

问题：

- 容易写成暴力解
- 没法训练“看到题就先判断复杂度边界”的习惯

### 4.4 不写能否使用额外空间

问题：

- 同一题可能有：
  - 哈希表解法
  - 原地解法
  - 排序解法
- 如果不提前说明，很容易在答题中途反复调整方案

## 5. 默认答题框架

每道题默认按下面结构回答。

### 5.1 先确认题目要求

至少先确认：

- 输入是什么
- 输出是什么
- 复杂度要求是什么
- 是否允许额外空间

### 5.2 先识别题型

默认先说一句：

- 这题为什么更像滑动窗口 / 双指针 / 哈希 / 前缀和 / 二分

### 5.3 再说核心状态定义

根据题型说明：

- 指针怎么定义
- 窗口怎么定义
- 哈希表存什么
- 前缀和表示什么
- 队列 / 栈维护什么

### 5.4 再说转移或移动规则

例如：

- 什么时候扩张窗口
- 什么时候收缩窗口
- 什么时候更新答案
- 为什么这样移动不会漏解

### 5.5 再写代码

要求：

- 先保证思路和代码一致
- 再关注边界条件

### 5.6 最后补三件事

- 时间复杂度
- 空间复杂度
- 边界条件和易错点

## 6. 推荐出题模板

后续算法题默认优先按下面模板出。

```text
题目：
`3. 无重复字符的最长子串`

题目描述：
给定一个字符串 s，请你找出其中不含有重复字符的最长子串的长度。

方法签名：
public int lengthOfLongestSubstring(String s)

示例 1：
输入：s = "abcabcbb"
输出：3
解释：最长不含重复字符的子串是 "abc"，长度为 3。

示例 2：
输入：s = "bbbbb"
输出：1
解释：最长不含重复字符的子串是 "b"，长度为 1。

额外要求：
- 目标时间复杂度为 O(n)
- 允许使用额外空间
- 不接受 O(n^2) 的暴力枚举所有子串解法

答题顺序：
1. 先说这题为什么是滑动窗口
2. 再说窗口里维护什么状态
3. 再说什么时候扩张、什么时候收缩
4. 再写代码
5. 最后补复杂度、边界和易错点
```

## 7. 对当前训练节奏的建议

结合当前仓库和练习节奏，后续默认这样执行：

- 工作日每题都给清楚题面、样例和复杂度要求
- 默认先口头答，再写代码
- 如果当天目标是恢复手感，优先 1 道题讲清，不追求题量
- 如果题目本身就是模板题，优先训练：
  - 题型识别
  - 状态定义
  - 移动规则
  - 复杂度表达
- 如果你答题明显卡在题面不清楚，就先补出题规则，不继续硬做

## 8. 文档放置建议

这份文档更适合放在：

- `prepare/2026/tracks/xiaomi_java_data_service/algorithms/rules/`

原因：

- `algorithms/` 根目录更适合放算法专题正文
  - 如 `00_roadmap`、`01_arrays`、`02_hashing`
- 出题规则属于“训练规则”，不是具体算法知识点
- 单独放到 `rules/` 子目录后，后续继续补：
  - `03_linked_list.md`
  - `04_stack_queue.md`
  - `05_binary_search.md`
  这类专题时不会发生编号冲突

因此默认建议：

- 算法知识正文放 `algorithms/`
- 算法训练规则放 `algorithms/rules/`
