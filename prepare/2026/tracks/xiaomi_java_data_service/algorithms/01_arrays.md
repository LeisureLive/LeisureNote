# 算法专题 01：数组

适用目标：

- 系统理解数组题的核心特点
- 能识别数组题常见模式
- 为后续刷题和面试表达提供模板

## 1. 数组的核心特点

- 内存连续
- 支持按下标 `O(1)` 访问
- 尾部追加通常较高效
- 中间插入 / 删除通常需要搬移元素，代价较高
- 很多题的本质是“如何在遍历过程中维护一段有意义的区间”

## 2. 什么题应该优先想到数组

如果题目有下面这些特征，通常优先考虑数组思维：

- 输入本身就是数组，要求原地修改
- 需要找连续区间、子数组、窗口
- 需要按位置关系处理：左边、右边、前缀、后缀
- 需要通过下标做状态表达
- 需要频繁访问相邻元素

## 3. 数组题常考模式

### 3.1 遍历与模拟

适用场景：

- 规则明确，按顺序处理每个元素
- 更考查边界控制和实现细节

典型题：

- `88. 合并两个有序数组`
- `54. 螺旋矩阵`
- `59. 螺旋矩阵 II`

### 3.2 双指针

适用场景：

- 左右收缩
- 快慢指针
- 原地去重 / 原地删除
- 有序数组上的搜索和配对

常见信号：

- “有序数组”
- “原地删除 / 原地去重”
- “找两数 / 多数之和”

典型题：

- `26. 删除有序数组中的重复项`
- `27. 移除元素`
- `167. 两数之和 II - 输入有序数组`
- `11. 盛最多水的容器`

### 3.3 滑动窗口

适用场景：

- 求满足某条件的最短 / 最长连续子数组
- 窗口内需要动态维护某种状态

常见信号：

- “连续子数组 / 子串”
- “最短 / 最长”
- “满足某个约束”

典型题：

- `209. 长度最小的子数组`
- `3. 无重复字符的最长子串`
- `438. 找到字符串中所有字母异位词`

### 3.4 前缀和

适用场景：

- 频繁查询某个区间和
- 子数组求和类问题
- 需要把“区间信息”转成“两个前缀的差”

典型题：

- `303. 区域和检索 - 数组不可变`
- `724. 寻找数组的中心下标`
- `560. 和为 K 的子数组`（与哈希表结合）

### 3.5 前后缀分解

适用场景：

- 每个位置的答案依赖左边信息和右边信息
- 不适合直接暴力枚举左右两边

典型题：

- `238. 除自身以外数组的乘积`
- `42. 接雨水`

### 3.6 原地交换 / 原地标记

适用场景：

- 要求 `O(1)` 额外空间
- 元素值和下标之间存在天然映射关系

典型题：

- `41. 缺失的第一个正数`
- `448. 找到所有数组中消失的数字`

### 3.7 二分查找

适用场景：

- 数组有序
- 答案空间具有单调性

典型题：

- `704. 二分查找`
- `34. 在排序数组中查找元素的第一个和最后一个位置`
- `875. 爱吃香蕉的珂珂`

## 4. 数组题的高频坑点

- 下标越界
- 左闭右闭、左闭右开不统一
- 原地修改时覆盖掉后面还要用的数据
- 忘记处理空数组、长度为 `1` 的数组
- 写出了正确思路，但“定义”和“代码实现”不一致
- 复杂度判断不严谨，以为用了数组就默认 `O(n)`

## 5. 数组题的面试表达模板

面试里可以优先用这个结构表达：

1. 先说输入特征
2. 再说为什么想到某种数组技巧
3. 再说核心状态定义
4. 再说遍历顺序
5. 最后给复杂度和边界

例子：

```text
这题输入是数组，而且每个位置的答案依赖左边和右边的信息，所以我会考虑前后缀分解。
我定义 prefix[i] 表示 i 左边元素的乘积，suffix[i] 表示 i 右边元素的乘积，
那么答案就是 prefix[i] * suffix[i]。
这样只需要线性遍历两到三次，时间复杂度 O(n)。
```
