## 3. 核心重点展开

### 3.1 JMM、`volatile`、`happens-before`

#### 1）先把 JMM 和 JVM 区分清楚

- JVM 内存结构：
  - 讨论运行时区域怎么划分，比如堆、栈、方法区
- JMM：
  - 讨论多线程下共享变量怎么保证可见性、有序性和原子性语义边界

这两个概念经常被混淆，但完全不是一回事。

#### 2）JMM 的核心问题

JMM 主要回答三件事，但这三件事不是“列出来就完了”，而是有一套统一的回答方式。

先抓一个总认识：

- JMM 不是在描述“物理内存真的长什么样”
- 它是在定义：
  - 线程如何和共享变量交互
  - 什么情况下写入必须对别的线程可见
  - 什么重排序允许发生，什么不允许
  - 多线程程序要想正确，最少要建立哪些同步关系

所以 JMM 真正回答的是：

- 可见性问题：
  - 线程 A 改了共享变量，线程 B 什么时候必须看见
- 有序性问题：
  - 编译器和 CPU 能否重排指令，重排到什么边界为止
- 原子性边界问题：
  - 哪些操作天然是不可再分的，哪些复合操作必须借助同步手段

#### 3）JMM 用什么抽象来回答这三件事

第一层要先建立一个抽象模型：

- 主内存：
  - 共享变量的“规范意义上的共享位置”
- 工作内存：
  - 每个线程对共享变量的本地视图

你可以把它先粗略理解成：

- 共享变量的最终正确值要以主内存为准
- 线程执行时，未必每次都直接从主内存读
- 线程可能先读到自己的工作内存副本，再在合适时机刷新或回写

这个抽象的意义不是模拟真实 CPU 缓存细节，而是解释一个核心事实：

- 多线程下，线程之间默认不是“天然立刻互相可见”的

也正因为如此，JMM 才需要规定：

- 什么同步动作会把修改刷新出去
- 什么同步动作会让线程重新读取共享变量
- 什么重排序虽然在单线程里没问题，但在多线程里必须禁止

#### 4）JMM 是怎么回答“可见性”的

可见性的核心不是“值变了”，而是：

- 别的线程什么时候必须看到这个变化

JMM 对可见性的回答方式是：

1. 默认情况下，线程可以只基于自己的工作内存继续执行
2. 如果线程之间没有建立同步关系，那么一个线程的写入，另一个线程什么时候看到，是不可靠的
3. 只有当程序通过某些同步动作建立了可见性关系后，前一个线程的写入结果才必须对后一个线程可见

常见能建立可见性的同步动作包括：

- 释放锁 -> 随后获取同一把锁
- `volatile` 写 -> 随后对同一个变量的 `volatile` 读
- 线程 `start()`
- 线程结束后，其他线程 `join()`

所以“可见性”不是一个线程自己努力一下就能保证的，而是：

- 需要程序建立一条明确的同步链路

#### 5）JMM 是怎么回答“有序性”的

先分两层看：

- 单线程语义：
  - 只要最终结果和按源码顺序执行一致，编译器和 CPU 可以做一定重排序
- 多线程语义：
  - 如果重排序会破坏线程间同步语义，就必须被禁止

这就是为什么经常会同时听到两句话：

- Java 允许重排序
- Java 又能通过 `volatile`、锁等机制限制重排序

这两句话并不矛盾。

更准确的理解是：

- JMM 允许“对单线程结果无害”的优化重排
- 但当程序通过同步原语表达了线程间顺序要求时，JMM 必须维护这条顺序

例如：

- `volatile` 写之前的普通写，不能被重排到 `volatile` 写之后
- `volatile` 读之后的普通读，不能被重排到 `volatile` 读之前
- 解锁之前对共享变量的修改，不能被“挪到”释放锁之后才对后续加锁线程可见

所以“有序性”在 JMM 里不是保证所有代码逐条按源码执行，而是保证：

- 不会因为重排序破坏多线程程序已经声明出来的同步语义

#### 6）JMM 是怎么回答“原子性边界”的

JMM 不会替你把所有并发操作都变成原子操作，它只是定义边界：

- 某些基本读写操作可以视为原子
- 但复合操作默认不是原子

比如：

- 读取一个变量
- 给一个变量赋值

这类单次操作可以理解成“一个动作”。

但下面这些都不是原子操作：

- `i++`
- `count = count + 1`
- 先判断再更新
- 先读多个变量，再统一做业务决策

也就是说，JMM 对原子性的回答不是“自动帮你保证”，而是：

- 明确告诉你默认能保证到哪一层
- 超出这层，就要靠 `synchronized`、`Lock`、CAS、原子类等机制补足

#### 7）`happens-before` 到底是什么

`happens-before` 不是一个底层实现组件，而是 JMM 用来判断“线程间结果是否必须可见、顺序是否必须被维护”的规则。

如果操作 A `happens-before` 操作 B，那么可以认为：

- A 的执行结果对 B 可见
- A 在内存语义上先于 B

这里最容易说错的点有两个：

- 它不是“时间上一定更早发生”
- 它不是“代码写在前面就天然成立”

它更接近一种规范判断：

- 如果 A 和 B 之间建立了 `happens-before` 关系，那么 JMM 必须保证 A 的结果能被 B 正确观察到

如果没有这条关系，会怎样？

- 线程之间的读写可能彼此不可见
- 重排序也可能让结果看起来“违反直觉”
- 程序就会落入数据竞争状态，行为不再可靠

#### 8）为什么 `happens-before` 是 JMM 的核心抓手

因为 JMM 前面那三个问题，本质上都可以落到 `happens-before` 上判断：

- 可见性：
  - 前一个写对后一个读是否可见
- 有序性：
  - 前一个操作是否必须排在后一个操作之前
- 原子性边界：
  - 哪些地方仅靠 `happens-before` 不够，还必须加互斥或原子更新手段

你可以把它理解成：

- JMM 先允许底层做优化
- 然后用 `happens-before` 给出“哪些边界绝不能破坏”

#### 9）高频 `happens-before` 规则要怎么理解

不要只背名字，关键是知道每条规则都在建立一条“从写到读”的同步链。

- 程序次序规则：
  - 同一线程内，前面的操作 `happens-before` 后面的操作
  - 它主要保证单线程语义
- 锁规则：
  - 对同一把锁的解锁，`happens-before` 后续对这把锁的加锁
  - 这就是为什么锁不仅能互斥，还能保证可见性
- `volatile` 变量规则：
  - 对某个变量的 `volatile` 写，`happens-before` 后续对这个变量的 `volatile` 读
  - 所以 `volatile` 常用来做状态发布
- 线程启动规则：
  - 线程 A 调用线程 B 的 `start()` 之前的操作，对线程 B 启动后可见
- 线程终止规则：
  - 线程中的所有操作，`happens-before` 其他线程成功从 `join()` 返回
- 传递性：
  - 如果 A `happens-before` B，B `happens-before` C，那么 A `happens-before` C

传递性非常重要，因为很多线程间可见性不是一跳完成的，而是靠多条规则串起来的。

#### 10）用一个最小例子理解 `happens-before`

例子一：锁

- 线程 A 在同步块里写共享变量，然后退出同步块
- 线程 B 随后进入同一把锁保护的同步块，再读取该变量

为什么 B 能看到 A 的更新？

- 不是因为“锁很神秘”
- 而是因为：
  - A 的解锁 `happens-before` B 的加锁
  - 所以 A 在临界区内的写入，对 B 是可见的

例子二：`volatile`

- 线程 A 先写普通变量 `value = 42`
- 然后写 `volatile` 变量 `ready = true`
- 线程 B 读到 `ready == true` 后，再去读 `value`

如果这段代码按 JMM 正确使用，那么：

- A 对 `ready` 的 `volatile` 写
- `happens-before` B 对 `ready` 的 `volatile` 读

再结合程序次序规则和传递性，可以得到：

- A 在写 `ready` 之前对 `value` 的写入
- 对 B 在读到 `ready == true` 之后读取 `value`
- 也是可见的

这就是 `volatile` 作为“发布信号”最经典的用法。

#### 11）`volatile` 到底能保证什么

- 保证可见性
- 保证和它相关的关键重排序不会破坏并发语义

它不能保证的东西也要说得非常明确：

- 不能保证复合操作原子性
- 不能保证多个变量的一致性维护

从机制上理解，`volatile` 的价值主要在两点：

- 写 `volatile` 变量时，前面的普通写不能随意跑到后面去
- 读 `volatile` 变量时，后面的普通读不能随意跑到前面来

所以它更像：

- 一个轻量级的“可见性 + 顺序性发布工具”

而不是互斥锁。

#### 12）为什么 `i++` 不能靠 `volatile`

`i++` 至少包含三步：

1. 读
2. 改
3. 写

`volatile` 只能保证：

- 读到的是较新值
- 写回后别人能较快看到

但不能把“读改写”三步变成一个不可分割的整体。

所以 `volatile` 在这里缺的不是“看不见”，而是：

- 缺少把这三步绑成一个原子单元的能力

#### 13）双重检查单例为什么必须加 `volatile`

对象创建可以粗略看成：

1. 分配内存
2. 初始化对象
3. 把引用赋给变量

如果发生重排序变成 `1 -> 3 -> 2`：

- 另一个线程可能先看到一个非空引用
- 但这个对象还没初始化完成

所以 DCL 里的 `volatile` 核心作用是：

- 禁止实例发布过程中的关键重排序
- 保证看到引用非空时，对象已完成初始化

如果想把这题讲得更完整，可以把逻辑说成：

- DCL 外层第一次判空，是为了减少不必要加锁
- 真正的风险不在“多个线程都进来了”
- 而在“某个线程先看到了非空引用，但对象还没初始化完成”
- `volatile` 在这里修的是发布顺序，而不是替代锁做互斥

#### 14）`volatile` 的适用边界

适合：

- 状态标记位
- 单写多读
- 不依赖旧值的开关变量
- DCL 单例引用

不适合：

- `i++`
- 多步业务逻辑
- 多变量整体一致性
### 3.2 CAS、原子类、AQS、`ConcurrentHashMap`

#### 1）CAS 的本质

CAS 是 Compare-And-Set。

核心思想不是“先读再写”，而是：

- 只有当前值仍等于预期值，才允许更新

它属于乐观并发控制：

- 默认冲突不高，先尝试更新
- 失败了再重试

#### 2）CAS 的边界和代价

优点：

- 冲突不高时性能通常不错
- 不需要像悲观锁那样锁住整段代码

缺点：

- 高竞争时会频繁自旋，浪费 CPU
- 更适合单点原子更新
- 不适合复杂临界区一致性控制

#### 3）ABA 问题要会说“为什么会出错”

CAS 只比较“当前值是不是预期值”，并不知道中间发生过什么。

所以从 `A -> B -> A` 的变化，对 CAS 来说仍然像“没变过”。

常见解决方式：

- 给值加版本号
- Java 里常见是 `AtomicStampedReference`

#### 4）`AtomicLong` 和 `LongAdder` 的区别

- `AtomicLong`：
  - 更像单热点共享值
  - 适合需要 CAS 语义的场景
- `LongAdder`：
  - 把热点拆散到多个槽位上做局部累加
  - 更适合高并发统计

选型不要背成一句口号，而要记住本质：

- 要严格 CAS 语义，优先 `AtomicLong`
- 要高并发计数吞吐，优先 `LongAdder`

#### 5）AQS 的定位

AQS 不等于某一把锁，而是一套同步器基础框架。

你只要抓住三个核心点：

- 一个共享状态 `state`
- 一个等待队列
- 一套阻塞 / 唤醒机制

它支撑了很多 JUC 同步器：

- `ReentrantLock`
- `Semaphore`
- `CountDownLatch`
- `ReentrantReadWriteLock`

继续深挖时，优先讲：

- 独占和共享两种模式
- 获取失败为什么要入队
- 为什么需要阻塞和唤醒，而不是一直自旋

#### 6）`ConcurrentHashMap` 为什么线程安全

这题要区分 JDK 版本：

- JDK 7：
  - `Segment` 分段锁
- JDK 8：
  - 主要是 `CAS + synchronized`

JDK 8 核心思路：

- 空桶插入时尽量先 CAS
- 发生冲突时只锁桶头或链表 / 红黑树局部结构
- 读路径尽量无锁或低锁

这题要顺手补一句边界：

- 容器线程安全，不代表“先判断再更新”这种复合业务逻辑就线程安全
## 4. 推荐表达顺序

如果面试官问 JMM 或无锁并发，推荐按下面顺序讲：

1. 先说问题背景：
   - 多线程共享变量为什么会出可见性、顺序性和原子性问题
2. 再说语义规则：
   - JMM、`happens-before`、`volatile` 分别在保证什么
3. 再说工具选择：
   - 什么时候用锁，什么时候用 CAS / 原子类 / AQS
4. 最后补边界：
   - `volatile` 为什么不能替代互斥，容器线程安全为什么不等于业务逻辑线程安全
