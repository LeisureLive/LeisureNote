## 3. 模拟问答复盘

### 3.1 讲得相对顺的点

- 能抓住“慢 SQL 先区分稳定慢还是抖动慢，再区分执行慢还是等待慢”这条主线
- 能说出 `type`、`key`、`key_len`、`rows`、`filtered` 这些 `EXPLAIN` 核心字段
- 能意识到 `EXPLAIN` 更偏执行计划视角，不能单独判断等待慢
- 能说出 MVCC 的主线是：
  - `undo log`
  - `Read View`
  - 快照读
  - 当前读
- 能区分快照读和当前读，也知道 MVCC 不等于“完全无锁”
- 能说出“两个 SQL 都走索引，扫描行数仍可能差很多”主要和联合索引利用、范围条件、排序和深分页有关

### 3.2 讲散了的点

- `EXPLAIN` 字段：
  - 第一轮只说到了 `type`、`key`、`key_len`、`rows`、`filtered`
  - `possible_keys`、`ref`、`Extra` 这几个关键字段还不熟
- 等待慢：
  - 一开始把等待慢想得太窄，主要只想到锁
  - 没有主动补出：
    - 元数据锁
    - 长事务
    - 连接池排队
    - CPU / 磁盘 IO 资源争用
- `Read View`：
  - 把可见性判断过度简化成“比较事务版本号和记录版本号”
  - `RC` 也说成了“读当前最新版本”，不够准确
  - 更准确应该是：
    - `RC` 一般每次快照读生成新的 `Read View`
    - `RR` 一般复用第一次快照读生成的 `Read View`
    - 当前版本不可见时，要沿 `undo log` 找更老的可见版本
- 索引扫描代价：
  - 主线是对的
  - 但“范围条件后，后续列通常很难继续参与高效定位，但仍可能参与 ICP 过滤”这层边界今天没有稳定讲出来

## 4. 算法复盘

### 4.2 当前算法薄弱点

- 题型识别：
  - 能识别“有序数组适合二分”
  - 但“这是边界二分，不是普通命中后二侧扩展”还不是第一反应
- 思路建立：
  - 没有第一时间把复杂度约束压到方案里
  - 导致先想到了会退化成 `O(n)` 的做法
- 边界处理：
  - 左边界命中后继续往哪边收缩，第一反应还不够稳
  - `while (left <= right)` 这种基础区间模板也还没完全自动化
- 代码实现：
  - 修正后能写对
  - 说明真正的问题不在编码，而在边界模板没有内化
- 口头解释：

    - 左边界命中后为什么继续向左收缩
