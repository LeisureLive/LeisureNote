## 1. 这一块在解决什么问题

循环依赖这块不是在考你会不会背 `singletonFactories`、`earlySingletonObjects`、`singletonObjects`，而是在考你能不能把下面四件事讲清楚：

1. 什么叫循环依赖，为什么 Spring 容器创建 Bean 时会遇到这个问题。
2. 为什么 Spring 不是“所有循环依赖都能解决”，而是只能解决一部分。
3. 三级缓存到底在解决什么问题，为什么不是二级缓存就够了。
4. 构造器注入、原型 Bean、AOP 代理介入时，为什么循环依赖会有不同结果。

把这些点串起来，可以这样理解：

- Spring 容器要创建 Bean，同时还要处理依赖注入。
- 如果 A 依赖 B，B 又依赖 A，容器在创建过程中就会出现“对象还没完全创建好，但另一个 Bean 已经要用它”的问题。
- 直接等对象完全创建完再暴露已经来不及，所以 Spring 需要“提前暴露一个可引用对象”。
- 但提前暴露的不一定是原始对象，还可能是后续需要增强的代理对象，这就引出了三级缓存。
- 即便如此，Spring 也不是万能的，因为有些依赖阶段太早，有些作用域不适合缓存复用。

所以这块真正要建立的是三种能力：

- 能解释循环依赖为什么会发生在 Bean 创建期。
- 能解释三级缓存是在解决“提前暴露”和“可能需要代理”这两个问题。
- 能说明 Spring 能解决什么场景，不能解决什么场景。

## 3. 核心重点展开

### 3.1 循环依赖为什么会出现在 Spring Bean 创建过程中

这一节在整条主线里解决的是第一个问题：

- 循环依赖为什么是容器创建期问题。
- 为什么它不是一个“普通引用关系”这么简单。

#### 先直接回答这个问题

循环依赖会出现在 Spring Bean 创建过程中，是因为 Bean 的“实例化”和“依赖注入完成”不是同一时刻。如果 A 在创建时需要 B，而 B 在创建时又反过来需要 A，就会出现一个典型问题：A 还没完全初始化完成，但 B 已经需要拿到 A 的引用。

所以循环依赖的本质不是“两个类互相认识”，而是“两个 Bean 都在创建过程中，同时依赖对方已经可用”。

#### 再用最小场景把问题钉住

比如：

- `ServiceA` 注入 `ServiceB`
- `ServiceB` 注入 `ServiceA`

如果是 Setter 注入，Spring 可能先把 `ServiceA` 实例化出来，再在属性填充阶段注入 `ServiceB`。当它去创建 `ServiceB` 时，又需要 `ServiceA`，这时如果没有提前暴露机制，就会发现 `ServiceA` 还没走完整个生命周期。

#### 是什么

循环依赖可以先理解成：

- 容器在创建一个对象时，需要另一个对象。
- 另一个对象在创建时，又回头需要第一个对象。

#### 为什么会有它

因为 Spring 支持依赖注入，而依赖注入和对象创建是分阶段进行的，这就给了“提前拿引用”的需求空间。

#### 怎么理解它的机制

第一层，Spring 创建 Bean 不是一步到位，而是：

- 先实例化。
- 再填充属性。
- 再初始化。

第二层，只要在属性填充阶段出现互相依赖，就可能触发循环依赖问题。

第三层，问题的关键不在于“有没有对象”，而在于“对方要的那个对象能不能在当前阶段被安全引用”。

#### 面试时怎么讲才不空

“循环依赖本质上发生在 Bean 创建期，因为 Spring 的实例化、属性注入、初始化是分阶段的。A 在创建过程中需要 B，B 在创建过程中又反过来需要 A，这时问题不是两个类互相引用，而是对象还没完整创建好，另一个 Bean 就已经要使用它。”

### 3.2 为什么 Spring 只能解决一部分循环依赖，而不是所有情况都能解决

这一节在整条主线里解决的是第二个问题：

- Spring 的能力边界到底在哪。
- 为什么面试里不能把这题答成“Spring 可以解决循环依赖”。

#### 先直接回答这个问题

Spring 只能解决一部分循环依赖，因为它依赖的前提是：对象能够先被实例化，再在后续阶段把“早期引用”暴露出去。这通常只适用于单例 Bean、且依赖发生在属性填充阶段的场景。对于构造器注入、原型 Bean 等情况，这个前提不成立，所以 Spring 解决不了。

#### 再用场景把问题钉住

如果 A 和 B 都通过构造器注入依赖彼此，那么 A 连第一步实例化都过不去，因为创建 A 前就必须先拿到完整 B，而创建 B 前又必须先拿到完整 A。这个时候连“提前暴露一个半成品对象”的机会都没有。

#### 是什么

Spring 的循环依赖处理能力，本质上依赖于“能否先得到一个早期引用，再完成后续初始化”。

#### 为什么会有这个边界

因为容器不是在任意阶段都能安全地把对象交出去。太早拿不到对象，太晚又已经错过注入时机。

#### 怎么理解这个边界

可以先记三个关键词：

- 单例。
- 属性注入。
- 允许提前暴露。

缺一个，循环依赖处理能力都可能不成立。

#### 面试时怎么讲才不空

“这题不能答成 Spring 能解决循环依赖，更准确的说法是：Spring 能解决一部分单例 Bean、属性注入场景下的循环依赖。因为它依赖的是提前暴露早期引用这个前提。构造器注入和原型 Bean 这类场景，前提不成立，所以 Spring 解决不了。”

### 3.3 三级缓存到底在解决什么问题，为什么不是二级缓存就够了

这一节在整条主线里解决的是第三个问题：

- 三级缓存的本质作用是什么。
- 为什么三级缓存不只是“多放一层 Map”。

#### 先直接回答这个问题

三级缓存的核心不是为了把缓存层数搞复杂，而是为了同时处理两件事：

1. Bean 还没完成初始化时，要能提前暴露一个可引用对象。
2. 这个提前暴露出去的对象，后面可能不是原始对象，而是需要经过代理包装的最终对象。

所以三级缓存的关键价值在于：它既支持提前暴露，又给“是否需要返回代理对象”留出了延迟决策空间。

#### 再用场景把问题钉住

如果 A 需要 B，B 又需要 A，而 A 最终还要被 AOP 代理，那么 B 在注入 A 时，最好拿到的不是“将来会失效的原始对象”，而是和最终容器里一致的那个引用。三级缓存里对象工厂的作用，就是在真正需要早期引用时，按需决定返回原始对象还是代理对象。

#### 是什么

可以先这样理解三级缓存：

- 一级缓存：已经创建完成的单例成品对象。
- 二级缓存：已经提前暴露出来的早期对象。
- 三级缓存：还没真正暴露对象前，用来按需生产早期引用的对象工厂。

#### 为什么会有它

如果没有第三级，只靠二级缓存，很多情况下你只能提前暴露原始对象，而无法优雅处理“这个 Bean 其实应该提前暴露代理对象”的问题。

#### 怎么理解它的机制

第一层，Bean 实例化后，如果可能被其他 Bean 提前引用，Spring 会先把一个对象工厂放进三级缓存。

第二层，当别的 Bean 真来取这个对象时，Spring 再通过对象工厂生成早期引用，并把它放入二级缓存。

第三层，等 Bean 完整初始化并成为成品对象后，再进入一级缓存。

第四层，这种设计让 Spring 不必过早决定“暴露原始对象还是代理对象”，而是把决策推迟到真正需要早期引用的时候。

#### 面试时怎么讲才不空

“三级缓存不是为了记 3 个 Map，而是在处理两个问题：一是循环依赖下要提前暴露对象，二是这个对象后面可能还要被代理。三级缓存里的对象工厂相当于把‘早期引用怎么生成’这件事延迟到真正需要时再决定，所以它不是简单的缓存分层，而是在兼顾提前暴露和代理一致性。”

## 4. 边界、误区、常见追问

- 不要把循环依赖讲成“两个类互相引用”，重点是 Bean 创建期和依赖注入期的冲突。
- 不要把三级缓存答成单纯记忆题，关键是“提前暴露 + 代理一致性”。
- 构造器注入场景通常解决不了，因为连实例化都过不去。
- Prototype Bean 不适合依赖这种单例缓存思路。
- 面试官经常继续追问：如果有 AOP 代理，提前暴露出来的到底应该是谁。

## 5. 推荐表达顺序

建议这样讲：

1. 先讲循环依赖为什么出现在 Bean 创建期。
2. 再讲 Spring 只能解决一部分循环依赖的前提。
3. 再讲为什么必须提前暴露对象。
4. 再讲三级缓存分别存什么，以及对象工厂为什么重要。
5. 最后补不能解决的边界场景。
